## 端到端工作流概览

Kiro 的核心设计理念是"规格驱动开发"（Spec-Driven Development），与传统的 Vibe Coding（随意对话式编程）有本质区别。在 Kiro 的工作流中，每一个功能的开发都经历清晰的三个阶段：需求文档化、设计文档化、任务执行。这三个阶段分别对应 `.kiro/specs/{feature}/` 目录下的 `requirements.md`、`design.md` 和 `tasks.md` 三个文件。

整个端到端工作流可以概括为以下协同链路：

**Spec（规格）** 负责定义"做什么"——通过结构化的需求文档和设计文档，让 Agent 在执行前就对目标有清晰认知，避免反复返工。

**Steering（引导）** 负责注入"背景知识"——通过持久化的上下文文件，让 Agent 在每次对话中都能感知项目的技术栈、代码规范和业务约束，不需要每次重复说明。

**Hooks（钩子）** 负责"自动化触发"——在特定事件发生时（如文件保存、任务完成、Agent 停止）自动执行预设动作，减少人工干预，形成流水线式的开发体验。

**MCP（模型上下文协议）** 负责"工具扩展"——通过接入外部服务（数据库、API、文件系统等），让 Agent 具备超出默认能力范围的操作能力。

四者协同工作，构成了 Kiro 完整的智能开发闭环。理解这个闭环是高效使用 Kiro 的关键前提。

## 实战：从零开始一个功能

下面以"为博客系统添加文章搜索功能"为例，完整演示 Kiro Spec 驱动开发的端到端流程。

### 第一步：创建 Feature Spec

在 Kiro 的 Chat 面板中，使用自然语言描述需求，触发 Spec 创建流程：

```
# 完整工作流 Prompt 示例序列

## Step 1 - 触发 Spec 创建（需求阶段）
用户：
我想为博客系统添加一个全文搜索功能，用户可以在首页搜索框输入关键词，
实时过滤文章列表，支持按标题、标签、摘要搜索，结果高亮匹配词。
请帮我创建一个 Feature Spec。

Kiro：
好的，我将为"文章搜索功能"创建 Feature Spec。
正在生成 .kiro/specs/article-search/requirements.md ...
[生成完毕，请确认需求是否准确]

## Step 2 - 确认需求，进入设计阶段
用户：
需求准确。请继续生成设计文档，技术栈是 Vue 3 + Vite，
搜索逻辑在前端完成，数据来源是 notes-index.json。

Kiro：
正在生成 .kiro/specs/article-search/design.md ...
[设计文档包含：组件结构、数据流、搜索算法、UI 交互规范]

## Step 3 - 生成任务列表
用户：
设计文档没问题，请生成任务列表。

Kiro：
正在生成 .kiro/specs/article-search/tasks.md ...
[共 8 个任务，含子任务，已标注优先级]

## Step 4 - 执行任务
用户：
请执行任务 1：创建 SearchBar 组件。

Kiro：
[任务 1 状态 → in_progress]
正在创建 src/components/SearchBar.vue ...
[任务 1 状态 → completed]

## Step 5 - 验证与迭代
用户：
搜索结果没有高亮，请修复任务 3 中的高亮逻辑。

Kiro：
[任务 3 状态 → in_progress]
正在修改 src/utils/searchHighlight.js ...
[任务 3 状态 → completed]
```

### 第二步：理解 Spec 三层文档

在上述流程中，Kiro 自动生成了三个核心文档：

`requirements.md` 使用 EARS（Easy Approach to Requirements Syntax）格式描述验收标准，每条需求都有明确的"当……时，系统应……"结构，便于后续验证。

`design.md` 包含架构决策、组件接口、数据模型和测试策略，是 Agent 执行任务时的技术参考手册。

`tasks.md` 将设计拆解为可执行的原子任务，每个任务有明确的状态标记（`[ ]` 未开始、`[-]` 进行中、`[x]` 已完成），支持断点续传。

### 第三步：任务执行与状态追踪

Kiro 在执行任务时会实时更新 `tasks.md` 中的状态标记。这意味着即使中途中断，下次打开项目时 Agent 也能从上次停止的地方继续，不会重复已完成的工作。这是 Kiro 相比普通 AI 对话工具最显著的工程优势之一。

## Steering 最佳实践

Steering 是 Kiro 的持久上下文注入机制，合理配置 Steering 文件能显著提升 Agent 的输出质量和一致性。

### 建立三个基础 Steering 文件

推荐为每个项目建立以下三个基础文件，放置于 `.kiro/steering/` 目录：

**product.md**：描述产品定位、目标用户、核心功能边界。Agent 在生成需求文档和设计文档时会参考这里的产品背景，避免提出与产品方向不符的技术方案。

**tech.md**：描述技术栈、依赖版本、代码规范、禁止使用的库或模式。例如"本项目使用 Vue 3 Composition API，禁止使用 Options API；CSS 使用 Tailwind，禁止引入额外 UI 框架"。

**structure.md**：描述目录结构、模块职责、命名规范。Agent 在创建新文件时会遵循这里定义的规范，保持代码库的一致性。

### 善用 fileMatch 模式

对于特定类型文件的专属规范，使用 `fileMatch` 模式按需注入，避免污染所有对话的上下文：

```yaml
# .kiro/steering/vue-component-rules.md
---
inclusion: fileMatch
fileMatchPattern: "src/components/**/*.vue"
---
# Vue 组件规范

- 所有组件使用 <script setup> 语法
- Props 必须使用 defineProps 并标注类型
- Emits 必须使用 defineEmits 显式声明
- 组件文件名使用 PascalCase
- 每个组件只做一件事，超过 200 行考虑拆分
- 样式使用 scoped，避免全局污染
```

```yaml
# .kiro/steering/test-rules.md
---
inclusion: fileMatch
fileMatchPattern: "src/**/__tests__/**/*.test.js"
---
# 测试规范

- 测试框架：Vitest，环境：jsdom
- 测试文件与源文件同目录，放在 __tests__ 子目录
- 测试名称使用中文描述，清晰表达测试意图
- 禁止使用 mock 绕过真实逻辑
- 每个测试函数只验证一个行为
```

### Steering 最佳实践要点

**保持文件精简**：每个 Steering 文件聚焦一个主题，避免把所有规范堆在一个文件里。Agent 处理短小精悍的上下文比处理冗长文档效果更好。

**定期审查更新**：随着项目演进，技术决策会发生变化。建议每个迭代周期检查一次 Steering 文件，确保内容与当前代码库保持一致。

**避免矛盾规则**：多个 Steering 文件之间不应存在相互矛盾的规则。工作区级别的 Steering 优先级高于全局级别，但同级别内的冲突会让 Agent 产生困惑。

**利用文件引用**：使用 `#[[file:path/to/file]]` 语法在 Steering 文件中引用实际代码文件，让 Agent 直接参考真实的代码示例，比文字描述更准确。

## Hooks 最佳实践

Hooks 是 Kiro 的事件驱动自动化机制，合理使用 Hooks 能将重复性工作从开发流程中剥离出去，让开发者专注于核心逻辑。

### 哪些场景适合用 Hooks

**代码质量保障**：在文件保存后自动运行 lint 和格式化，确保每次提交的代码都符合规范，无需手动执行。

**测试自动触发**：在相关源文件修改后自动运行对应的测试套件，及时发现回归问题。

**文档同步**：在 API 接口文件修改后自动提醒 Agent 检查并更新对应的文档文件，保持文档与代码同步。

**任务完成通知**：在 Spec 任务全部完成后自动触发代码审查或生成变更摘要，形成完整的开发闭环。

**构建验证**：在关键配置文件修改后自动运行构建，提前发现配置错误。

### Steering + Hooks 协同配置示例

下面展示一个将 Steering 规范与 Hooks 自动化结合的完整配置，实现"文件保存后自动检查是否符合 Steering 规范"的效果：

```json
// .kiro/hooks/auto-lint-on-save.json
// Hook 1：文件保存后自动运行 lint
{
  "id": "auto-lint-on-save",
  "name": "保存后自动 Lint",
  "description": "Vue/JS 文件保存后自动运行 ESLint，确保代码风格一致",
  "event": "fileEdited",
  "filePatterns": ["src/**/*.vue", "src/**/*.js"],
  "action": {
    "type": "runCommand",
    "command": "npx eslint --fix ${file}"
  },
  "enabled": true
}
```

```json
// .kiro/hooks/spec-task-complete-review.json
// Hook 2：Spec 任务全部完成后触发 Agent 代码审查
{
  "id": "spec-task-complete-review",
  "name": "任务完成后代码审查",
  "description": "所有 Spec 任务完成后，自动触发 Agent 对变更文件进行代码审查",
  "event": "postTaskExecution",
  "action": {
    "type": "askAgent",
    "prompt": "所有任务已完成。请对本次 Spec 涉及的所有变更文件进行代码审查，重点检查：1) 是否符合 tech.md 中的技术规范；2) 是否有明显的性能问题；3) 错误处理是否完善。输出审查报告。"
  },
  "enabled": true
}
```

```yaml
# .kiro/steering/tech.md（与上述 Hooks 协同工作的 Steering 文件）
---
inclusion: always
---
# 技术规范

## 代码风格
- 缩进：2 空格，单引号，无行尾分号
- 模块导入使用 @/ 别名（src 目录）
- 异步操作统一使用 async/await，禁止裸 Promise.then 链

## 禁止事项
- 禁止在组件中直接操作 DOM（使用 Vue 响应式 API）
- 禁止硬编码 API 地址（统一在 src/config.js 中管理）
- 禁止提交 console.log 调试语句

## 测试要求
- 新增工具函数必须附带单元测试
- 测试覆盖率不低于 80%
```

### Hooks 使用注意事项

**避免过度自动化**：不是所有操作都适合自动化。对于破坏性操作（如删除文件、推送代码），建议保留人工确认步骤，不要完全交给 Hook 自动执行。

**控制 Hook 数量**：过多的 Hook 会导致每次操作都触发大量自动化流程，反而降低开发效率。建议只为高频且重复性强的操作配置 Hook。

**测试 Hook 逻辑**：在正式使用前，先在测试环境验证 Hook 的触发条件和执行结果，避免误触发导致意外修改。

## MCP 最佳实践

MCP（Model Context Protocol）让 Kiro 的 Agent 能够调用外部工具和服务，极大扩展了其能力边界。

### 工具选择原则

**按需接入，不贪多**：每个 MCP 服务器都会占用上下文空间并增加工具调用的复杂度。建议只接入当前项目实际需要的工具，避免"工具过载"导致 Agent 选择混乱。

**优先使用官方维护的服务器**：社区维护的 MCP 服务器质量参差不齐，优先选择有官方背书或活跃维护的实现，降低安全风险和兼容性问题。

**敏感操作配置 autoApprove 白名单**：对于数据库写入、文件删除等高风险操作，不要将其加入 `autoApprove` 列表，保留人工确认环节。

### 工作区级 vs 用户级配置

**工作区级**（`.kiro/mcp.json`）：适合项目专属的 MCP 工具，如项目数据库连接、项目特定的 API 服务。这类配置应提交到版本控制，让团队成员共享相同的工具集。

**用户级**（`~/.kiro/mcp.json`）：适合个人通用工具，如个人 GitHub Token、个人笔记服务。这类配置不应提交到版本控制，避免泄露个人凭证。

### 环境变量管理

MCP 配置中的敏感信息（API Key、Token、密码）必须通过环境变量引用，不得硬编码在配置文件中：

```json
// .kiro/mcp.json - 推荐配置示例
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}"
      },
      "autoApprove": [
        "get_file_contents",
        "search_repositories",
        "list_issues"
      ],
      "disabledTools": ["create_or_update_file", "delete_file"]
    },
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/username/projects/current-project"
      ],
      "autoApprove": ["read_file", "list_directory"],
      "disabledTools": ["delete_file"]
    },
    "fetch": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-fetch"],
      "autoApprove": ["fetch"]
    }
  }
}
```

### MCP 常见问题排查

当 MCP 服务器无法连接时，优先检查以下几点：环境变量是否正确设置、`command` 指定的可执行文件是否在 PATH 中、网络代理是否影响了远程服务器的连接。Kiro 的 MCP 面板会显示每个服务器的连接状态，红色图标表示连接失败，点击可查看详细错误信息。

## Kiro vs Claude Code vs Codex 横向对比

三款工具都是面向开发者的 AI 编程助手，但定位和适用场景有明显差异。以下对比基于 2026 年 4 月的公开信息：

| ------------ | --------------------------- | ---------------------- | -------------------- |
| 发布方       | AWS                         | Anthropic              | OpenAI               |
| 底层模型     | Claude Sonnet 4             | Claude Sonnet 4 / Opus | GPT-4o / o3          |
| 运行形态     | 独立 IDE（VS Code 内核）    | CLI 工具               | CLI 工具 + Web       |
| 核心范式     | Spec 驱动开发               | 对话式 + CLAUDE.md     | 对话式 + AGENTS.md   |

| 任务追踪     | tasks.md 状态机             | 无内置任务追踪         | 无内置任务追踪       |

| 适合快速原型 | 一般（Spec 流程有开销）     | 强                     | 强                   |

### 三工具的核心差异解读

**Kiro 的独特价值**在于其结构化的 Spec 系统。当项目复杂度上升、团队规模扩大时，Spec 文档能有效降低沟通成本，让 AI 的每次操作都有据可查、可回溯。Hooks 机制则让 Kiro 具备了其他两款工具不具备的"主动自动化"能力。

**Claude Code 的优势**在于轻量和灵活。作为纯 CLI 工具，它可以无缝集成到任何现有工作流中，不需要切换 IDE。对于习惯终端操作的开发者，Claude Code 的上手成本极低，适合快速迭代和探索性开发。

**Codex 的差异化特性**是沙箱隔离执行环境。Codex 在 Docker 容器中运行代码，天然隔离了风险，适合需要执行不可信代码或在受控环境中进行实验的场景。同时 Codex 支持 Web 界面，对不熟悉命令行的用户更友好。

## 适合与不适合 Kiro 的场景

### 适合 Kiro 的场景

**中大型功能开发**：当一个功能涉及多个文件、多个模块的协同修改时，Spec 系统的价值最为突出。需求文档和设计文档能确保 Agent 在执行过程中不偏离方向，任务列表能保证每个细节都被覆盖。

**团队协作项目**：Spec 文档存储在版本控制中，团队成员可以共同审查需求和设计，对 AI 的工作有清晰的预期和记录。Steering 文件也能确保不同成员使用 Kiro 时得到一致的输出风格。

**需要长期维护的项目**：Steering 文件积累的项目知识会随着时间增长，让 Agent 对项目的理解越来越深入。新加入的团队成员也可以通过阅读 Steering 文件快速了解项目规范。

**有明确质量标准的项目**：通过在 Steering 中定义代码规范，在 Hooks 中配置自动化检查，Kiro 能持续保障代码质量，适合对工程质量有较高要求的项目。

**需要工具集成的复杂场景**：通过 MCP 接入数据库、API、文件系统等外部工具，Kiro 能处理需要跨系统操作的复杂任务，如数据迁移、API 集成测试等。

### 不适合 Kiro 的场景

**快速原型验证**：如果目标是在几小时内验证一个想法，Spec 流程的文档化开销可能得不偿失。这种场景下 Claude Code 或 Codex 的直接对话方式更高效。

**单文件小修改**：对于修改一个函数、修复一个 bug 这类小任务，启动完整的 Spec 流程是过度设计。直接在 Chat 面板对话即可，不需要创建 Spec。

**纯探索性研究**：当目标是探索某个技术方向、学习新框架时，结构化的 Spec 流程反而会限制探索的自由度。

**对 IDE 有强依赖的场景**：Kiro 基于 VS Code 内核，但并非完整的 VS Code。部分 VS Code 插件可能不兼容，对于深度依赖特定 VS Code 插件的开发者，切换成本较高。

## 常见坑与解决方案

### 坑一：Spec 文档与实际代码脱节

**现象**：随着开发推进，`requirements.md` 和 `design.md` 中的内容与实际实现越来越不一致，导致后续任务执行时 Agent 参考了过时的设计。

**解决方案**：将 Spec 文档视为"活文档"，每次重大设计变更后及时更新。可以配置一个 `postTaskExecution` Hook，在任务完成后提醒 Agent 检查 Spec 文档是否需要更新。

### 坑二：Steering 文件过于冗长

**现象**：把所有规范都堆在一个 `always` 模式的 Steering 文件中，导致每次对话都携带大量无关上下文，影响 Agent 的响应质量和速度。

**解决方案**：按主题拆分 Steering 文件，合理使用 `fileMatch` 和 `manual` 模式，只在需要时注入相关上下文。`always` 模式的文件应保持精简，只包含最核心的全局规范。

### 坑三：Hooks 触发频率过高

**现象**：为 `fileEdited` 事件配置了耗时较长的 Hook（如运行完整测试套件），导致每次保存文件都要等待很长时间，严重影响开发体验。

**解决方案**：对于耗时操作，改用 `agentStop` 或 `postTaskExecution` 事件触发，而不是 `fileEdited`。或者在 Hook 中添加文件路径过滤，只对关键文件的修改触发完整检查。

### 坑四：MCP 服务器权限过宽

**现象**：将所有 MCP 工具都加入 `autoApprove`，导致 Agent 在未经确认的情况下执行了破坏性操作（如删除文件、修改数据库记录）。

**解决方案**：严格区分只读操作和写入操作。只读操作（查询、列举、读取）可以加入 `autoApprove`，写入和删除操作必须保留人工确认。使用 `disabledTools` 禁用当前项目不需要的高风险工具。

### 坑五：任务粒度过粗

**现象**：`tasks.md` 中的任务描述过于笼统（如"实现搜索功能"），导致 Agent 在执行时做了过多假设，产生大量不符合预期的代码。

**解决方案**：在生成任务列表时，要求 Kiro 将每个任务拆解到足够细的粒度（单个函数、单个组件），并在任务描述中明确输入输出和验收标准。可以在 `design.md` 中提供足够详细的接口定义，让任务执行有明确的参考。

### 坑六：忽略 Spec 的 Requirements-First 流程

**现象**：直接跳过需求文档，让 Kiro 直接生成设计文档和任务，导致后期发现需求理解有偏差，需要大量返工。

**解决方案**：严格遵循 Requirements-First 流程。在确认需求文档准确无误之前，不要进入设计阶段。需求文档的每条验收标准都应该是可验证的，模糊的需求会直接导致模糊的实现。

## 注意事项与常见问题

### 注意事项

**版本兼容性**：Kiro 处于快速迭代阶段，功能和配置格式可能随版本更新而变化。本篇内容基于 2026 年 4 月的版本，使用时请以官方最新文档为准。

**数据安全**：Kiro 会将代码发送到 AWS 的 Claude 服务进行处理。对于包含敏感信息的代码库，在使用前应了解 AWS 的数据处理政策，并确保符合所在组织的安全合规要求。

**网络依赖**：Kiro 需要持续的网络连接才能正常工作，不支持离线使用。在网络不稳定的环境下，Agent 的响应可能出现延迟或中断。

**资源消耗**：作为基于 VS Code 内核的 IDE，Kiro 的内存占用高于纯 CLI 工具。在配置较低的机器上，同时运行多个大型项目可能影响性能。

### 常见问题

**Q：Spec 文档应该有多详细？**

A：需求文档应详细到每条验收标准都可以独立验证；设计文档应详细到 Agent 执行任务时不需要做重大架构决策。过于简略的文档会导致 Agent 做过多假设，过于详细则会增加维护成本。找到适合项目规模的平衡点需要一定的实践积累。

**Q：可以在 Kiro 中使用 VS Code 插件吗？**

A：Kiro 基于 VS Code 内核，支持大多数 VS Code 插件，但并非全部兼容。建议在使用前测试关键插件的兼容性。

**Q：Steering 文件的修改何时生效？**

A：Steering 文件的修改在下一次对话开始时生效。如果在对话进行中修改了 Steering 文件，需要开启新的对话才能让 Agent 感知到变化。

**Q：如何处理 Agent 执行任务时产生的错误？**

A：当 Agent 执行任务出错时，不要直接修改代码，而是在 Chat 面板描述错误现象，让 Agent 分析原因并提出修复方案。这样能保持任务状态的一致性，也能让 Agent 从错误中学习。
