# 第四篇：Cursor 接手陌生项目的实战流程（2026-04）

> 本篇是基于 Cursor 官方能力整理出的中文实战流程，不是官方逐字教程。目标是回答一个真实问题：拿到陌生仓库时，怎样让 Cursor 帮你稳定推进，而不是一上来就乱改。

[[toc]]

---

## 1. 实战目标

假设你接手的是一个已经在运行的 Web 项目，你知道它大概能跑，但你并不熟悉它的代码组织。此时最容易踩的坑不是“不会改”，而是：

- 还没看懂就开始改
- 一次让 AI 干太多事
- 没有限定范围，结果越改越散
- 改完没验证，最后不知道到底修没修好

所以整个流程要围绕四个字：**先收再放**。

先收：先收窄目标、边界、上下文  
再放：再让 Agent 真正执行

---

## 2. 第一步：先让 Ask 画地图

刚接手项目时，先别让 Agent 修改代码，而是用 Ask 做“项目定向”。

推荐第一问：

```text
请先不要修改代码。
帮我梳理这个仓库的：
1. 技术栈
2. 启动与构建命令
3. 入口文件
4. 路由位置
5. 与“文章详情页”最相关的目录和文件
如果有不确定项，请直接标注不确定。
```

这一步的目标不是让它“解决问题”，而是让它先证明自己真的看懂了项目的大体结构。

### 为什么这一步不能省

因为只要地图没画清楚，后面所有“帮我改”都容易演变成盲改。

---

## 3. 第二步：把项目边界写死

在正式修改前，尽量把项目长期约束写进规则。

推荐最低配做法：

1. 在根目录准备 `AGENTS.md`
2. 把技术栈、目录规则、禁改目录、必跑命令写进去
3. 如果项目复杂，再拆分为 `.cursor/rules`

这一步的目标不是做复杂治理，而是防止 Agent 犯低级错，比如：

- 去改生成文件
- 忘记跑必须执行的索引脚本
- 用错导入路径
- 不按既有代码风格输出

对于长期维护仓库，这一步往往比 prompt 本身更值钱。

---

## 4. 第三步：只给当前任务需要的上下文

官方文档提供的 `@Files`、`@Folders`、`@Link` 非常适合做“任务级补上下文”。

推荐顺序：

1. 先 `@Files` 关键入口和疑点文件
2. 如果只知道模块位置，再补 `@Folders`
3. 如果有外部需求文档或报错说明，再补 `@Link`

一个稳妥的 bug 修复 prompt 长这样：

```text
@src/views/NoteDetailPage.vue
@src/components/ReadingToolbar.vue
@src/components/TableOfContents.vue
任务：修复移动端阅读页滚动容器切换后目录高亮不准的问题
范围：只改阅读相关模块
约束：不要改首页，不要新增依赖
输出：先给原因和最小改动方案，确认后再实施
```

这个阶段的关键不是“上下文越多越好”，而是“上下文越相关越好”。

---

## 5. 第四步：先让 Agent 出方案，再执行

复杂任务不要直接让它开改，最好先多加一句：

```text
先分析原因并给出实施计划，不要立即修改文件。
```

这样你能先检查它是否：

- 找对了问题根因
- 选对了文件
- 计划是否过大
- 有没有误伤风险

如果计划靠谱，再进入执行阶段：

```text
按你刚才的最小方案开始修改。
修改后请说明影响文件、核心变化和验证方式。
```

---

## 6. 第五步：把“大改”拆成“小闭环”

这是最容易决定成败的一步。

错误做法：

- “把整个详情页体验全部优化一下”

推荐做法：

1. 先修滚动容器判断
2. 再修阅读进度绑定
3. 再修目录高亮
4. 最后再看样式和交互细节

为什么要这样拆：

- 便于验证
- 便于回退
- 便于你判断它到底改对没有
- 不会把一个问题变成一串新问题

---

## 7. 第六步：局部收尾交给 Inline Edit

Agent 适合主线任务，收尾细节往往更适合 Inline Edit。

例如这些动作，就不一定还要再开一轮 Agent：

- 调整一段提示文案
- 把一段条件判断写得更清楚
- 规范某个函数命名
- 优化单个组件内注释

你可以直接选中代码按 `Ctrl/Cmd + K`，让修改范围保持在眼前。

---

## 8. 第七步：验证不能省

一个完整闭环至少要有一项验证：

- 跑测试
- 跑构建
- 人工验证页面
- 对比修改前后行为

可以直接在 prompt 里把这件事写死：

```text
改完后请执行最小必要验证。
如果无法运行测试或构建，请明确说明原因，不要假设通过。
```

这句话非常重要，因为很多“看起来改好了”的问题，最后都死在没有验证。

---

## 9. 第八步：把重复经验沉淀下来

如果你发现下面这些内容会反复解释，就应该沉淀：

- 哪些文件是生成的
- 哪些目录不能动
- 哪些命令改完必须执行
- 某个业务模块的关键约束

可沉淀到：

- `AGENTS.md`
- `.cursor/rules`
- Memories

这样下次继续做同一仓库时，启动成本会越来越低。

---

## 10. 一套可以直接复用的实战脚本

### 阶段一：读项目

```text
请先不要改代码。
告诉我这个项目的技术栈、入口、路由和与阅读页最相关的文件。
按“确定信息 / 需要补充的信息”输出。
```

### 阶段二：定边界

```text
基于这个项目，请告诉我为了安全修改阅读页，我还缺哪些上下文。
如果需要，请列出你建议我补充引用的文件。
```

### 阶段三：提方案

```text
@相关文件
先分析问题原因，并给出最小修改方案。
暂时不要改代码。
```

### 阶段四：执行

```text
按刚才确认的方案修改代码。
范围只限上述文件。
改完后总结“改动文件、关键变更、验证方式”。
```

### 阶段五：复盘

```text
这次改动里有哪些信息适合沉淀为项目规则或记忆？
请给出建议清单。
```

---

## 11. 这套流程最适合哪些场景

适合：

- 接手个人旧项目
- 接手同事留下的中小仓库
- 维护一段时间没碰的业务模块
- 要求风险可控的功能修复

不太适合：

- 完全没有边界的大而空需求
- 明显需要先做产品决策的任务
- 你自己都还没搞清楚目标的场景

---

## 12. 一句话总结

拿 Cursor 接手陌生项目，不要追求“第一次就让它把事做完”，而要追求“每一轮都在缩小不确定性”。

先看懂，再限定，再执行，再验证，再沉淀。  
你会发现 Cursor 最强的地方，不是一次性产出很多代码，而是帮你把复杂项目拆成可以连续推进的小闭环。
