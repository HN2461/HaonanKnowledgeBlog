# 第二篇：Cursor 常用操作与高频工作流（2026-04）

> 本篇以 Cursor 官方文档为主，重点整理最常用的几个入口和它们的分工。

[[toc]]

---

## 1. 先记住 Cursor 里最常用的四种能力

如果只从工作方式来理解，Cursor 日常最常用的是这四类能力：

1. Agent：适合多文件任务、搜索代码、执行命令、连续完成一串动作
2. Ask：适合理解项目、做方案、做风险评估，偏只读
3. Inline Edit：适合局部改写、解释、补全、重构当前选择
4. Tab：适合高频补全和顺手续写

很多人用不好 Cursor，不是不会提问，而是把所有任务都丢给一种模式。实际上，模式分工越清楚，输出越稳定。

---

## 2. Agent 负责“动手做”

官方文档把 Agent 定位为可以自主完成复杂编码任务的助手。它能搜索、读文件、改文件，还能跑终端命令，所以适合这类事情：

- 横跨多个文件的功能开发
- 带排查步骤的 bug 修复
- 需要先搜代码再改代码的任务
- 需要顺手执行测试、构建、格式化的任务

一个稳妥的 Agent prompt，最好同时包含：

- 目标
- 范围
- 不允许动的部分
- 验收标准
- 是否允许运行命令

示例：

```text
请修复文章详情页在移动端内容被截断的问题。
范围只限阅读相关组件和详情页，不要改首页。
改完后请运行相关测试或说明为什么没法运行。
最后用“原因、改动点、验证结果”三段总结。
```

---

## 3. Ask 负责“先想清楚”

Ask 更适合拿来做这些事情：

- 读懂陌生仓库
- 比较两种实现方案
- 解释一段代码为什么这么写
- 先列风险，再决定要不要动手

最适合 Ask 的心法是：**先让它缩小问题，再进入 Agent。**

例如你接手一个项目，不要一开始就说“帮我优化这个仓库”，而是先问：

```text
请先不要修改代码。
告诉我这个项目中和文章详情页滚动相关的文件有哪些，并说明每个文件负责什么。
如果存在不确定项，请明确标出来。
```

等它把地图画出来，再进入真正的修改阶段，准确率会明显更高。

---

## 4. Inline Edit 负责“就地处理”

Cursor 官方文档里，`Ctrl/Cmd + K` 是最核心的局部编辑入口。它特别适合这些场景：

- 改写选中代码
- 把注释翻成更清晰的说明
- 把一段逻辑抽成函数
- 把一段代码从“能跑”整理成“更规范”

几个很实用的操作：

- `Ctrl/Cmd + K`：对当前选择做就地编辑
- `Ctrl/Cmd + Shift + Enter`：对整个文件做大范围编辑
- `Alt + Enter`：对选中代码快速提问
- `Ctrl/Cmd + L`：把选中代码发送到聊天上下文

我的建议是：

- 局部、明确、可视化的改动，用 Inline Edit
- 一旦任务跨文件、跨目录，回到 Agent

---

## 5. Tab 负责“把重复动作变轻”

Tab 不只是单行补全。官方文档强调，它可以给出多行建议，甚至在合适场景下感知相邻文件和你最近的编辑行为。

适合 Tab 的场景：

- 续写重复结构
- 补齐 if/else、错误处理、类型或注释
- 在已有模式下跟随式扩写

不适合 Tab 的场景：

- 你自己还没想清楚目标
- 需要跨文件理解
- 需要设计权衡
- 需要严格遵守复杂项目约束

一句话总结：Tab 用来加速手上已经很明确的动作，不用来替代思考。

---

## 6. `@` 上下文引用是命中率的关键

Cursor 官方文档明确提供了 `@Files`、`@Folders`、`@Link` 等上下文引用方式。你可以把它理解为“主动告诉 AI 这次该看什么”。

### 6.1 `@Files`

适合：

- 明确知道相关文件名
- 想把单文件精准塞进上下文
- 避免它在大型仓库里自己乱找

优点是最稳，尤其适合 bug 修复和小功能。

### 6.2 `@Folders`

适合：

- 你只知道功能大概在哪个目录
- 任务涉及一组相关文件
- 你要让它先理解某个模块的组织方式

但目录引用要克制。官方文档也提醒，开启 Full Folder Content 后，目录内容可能大幅消耗上下文和 token。

### 6.3 `@Link`

适合：

- 贴公开文档链接
- 贴设计文档或 PDF
- 让它遵循外部接口说明

这在“按文档接入 SDK”或“按 PRD 改页面”时非常好用。

---

## 7. 高频工作流 1：接手陌生项目

推荐顺序：

1. Ask：先看结构、依赖和模块关系
2. `@Files` / `@Folders`：补足关键上下文
3. Agent：执行真正修改
4. Inline Edit：局部收尾
5. 测试或构建验证

示例 prompt：

```text
@src/views @src/components
请先梳理和阅读页相关的文件结构，不要修改代码。
然后告诉我如果要修复移动端阅读体验，最可能需要动哪些文件。
```

---

## 8. 高频工作流 2：修一个具体 bug

推荐顺序：

1. Ask：先定位可能原因
2. `@Files`：喂给它报错文件和调用链
3. Agent：让它在限定范围内修
4. Inline Edit：对单个文件细修

示例 prompt：

```text
@src/views/NoteDetailPage.vue
@src/components/ReadingToolbar.vue
用户反馈移动端滚动到底部后目录定位不准。
请先说明最可能的原因，再给出最小改动方案。
确认后再开始改代码。
```

---

## 9. 高频工作流 3：写一个小功能

推荐顺序：

1. Ask：先拆范围
2. Agent：实现主逻辑
3. Inline Edit：抠文案、样式、命名
4. Tab：补重复结构

如果需求不是特别大，尽量把任务写成“一个功能 + 一组验收标准”，不要一次塞五六个相互独立的小需求。

---

## 10. 一个很实用的模式切换原则

可以直接记成一句话：

- 想清楚，用 Ask
- 真动手，用 Agent
- 改局部，用 `Ctrl/Cmd + K`
- 赶节奏，用 Tab

如果你每次都先判断“这次任务是理解、执行、局部处理，还是续写补全”，Cursor 的稳定度会比“全都塞给 Agent”高很多。

---

## 11. 常见误区

### 11.1 一句话扔一个大目标

比如“帮我把整个后台重构一下”。这种话没有范围、没有约束、没有验收标准，很容易漂。

### 11.2 不给文件上下文

明明知道相关文件，还让它全仓库自己猜，成功率会明显下降。

### 11.3 用 Agent 做纯局部改字

局部改文案、调一段 CSS、改一个函数名，Inline Edit 更顺手。

### 11.4 开了大目录全量上下文还不自知

目录引用越大，token 消耗越高，模型越容易被噪声干扰。
