# 第六篇：Cursor MCP 工具扩展与安全配置（2026-04）

> 本篇基于 Cursor 官方 MCP 文档整理，资料快照时间：2026-04-14。

[[toc]]

---

## 1. MCP 是什么

MCP 全称 Model Context Protocol，可以让 Cursor 连接外部工具和数据源。

你可以把它理解为：给 Agent 增加一组可调用工具。比如：

- 查数据库
- 读设计系统资料
- 调内部 API
- 连接浏览器自动化工具
- 连接 issue、任务、文档系统

Cursor 官方文档里说得很清楚：MCP 的价值在于让 Cursor 直接连接外部系统，而不是每次靠你手工复制粘贴上下文。

---

## 2. 什么时候需要 MCP

适合用 MCP：

- Agent 需要访问项目之外的数据
- 任务依赖内部系统或 SaaS 工具
- 你经常重复复制同一类外部信息
- 需要让 Agent 调用专门工具完成动作

不一定需要 MCP：

- 只是改一个本地文件
- 只是让 Cursor 解释代码
- 只是贴一份公开文档链接
- 项目还没有形成稳定的外部工具调用需求

一句话：先把 Cursor 自带的 `@Files`、`@Folders`、`@Link` 用好，再考虑 MCP。

---

## 3. Cursor 支持的三种传输方式

官方文档列出了三种 MCP 传输方式：

| 方式 | 适合场景 | 简单理解 |
| --- | --- | --- |
| `stdio` | 本地单用户工具 | Cursor 通过命令启动本地 MCP 进程 |
| `SSE` | 本地或远程服务 | 通过 SSE endpoint 连接服务 |
| `Streamable HTTP` | 本地或远程服务 | 通过 HTTP endpoint 连接服务 |

个人本地开发最常见的是 `stdio`；团队共享服务更可能用远程 HTTP 或 SSE。

---

## 4. MCP 配置文件位置

官方文档提供两个配置位置：

| 位置 | 适合内容 |
| --- | --- |
| `.cursor/mcp.json` | 项目专属工具，适合跟仓库相关的 MCP |
| `~/.cursor/mcp.json` | 全局工具，适合个人所有项目都要用的 MCP |

选择原则：

- 只服务当前仓库：放项目级
- 所有项目都常用：放全局
- 含个人密钥：优先全局或环境变量，不要提交到仓库
- 团队共享配置：项目级可以提交，但不要提交真实 secret

---

## 5. `stdio` 本地服务器示例

下面是官方风格的简化示例：

```json
{
  "mcpServers": {
    "server-name": {
      "command": "npx",
      "args": ["-y", "mcp-server"],
      "env": {
        "API_KEY": "value"
      }
    }
  }
}
```

逐项理解：

1. `mcpServers`：所有 MCP 服务器的配置入口
2. `server-name`：你给这个工具起的名字
3. `command`：启动服务器的命令
4. `args`：命令参数
5. `env`：传给服务器的环境变量

真实项目里，`API_KEY` 不建议直接写死在可提交文件中，优先使用环境变量或本地私有配置。

---

## 6. 远程服务器示例思路

远程 MCP 一般会配置一个 URL，并配合认证信息使用。适合：

- 团队统一维护的内部工具
- 需要 OAuth 的 SaaS 工具
- 不希望每个人本地都安装一套服务的场景

但远程服务也意味着更严格的安全审查：

- 它能访问什么数据
- 谁维护这个服务
- 认证方式是否安全
- 返回内容是否可能包含敏感信息
- 是否会执行写操作

---

## 7. 在聊天里怎么使用 MCP

官方文档说明，Agent 会在相关时自动使用 `Available Tools` 中的 MCP 工具。你也可以在 prompt 里明确点名某个工具，或者描述你需要它做什么。

例如：

```text
请使用可用的 issue 工具查看当前任务的验收标准，然后只基于这些标准给出实现计划。
暂时不要修改代码。
```

工具越危险，prompt 越要写清楚：

- 只能读，不能写
- 先列计划，不要执行
- 需要修改外部系统时先询问

---

## 8. 工具审批与自动运行

默认情况下，Agent 使用 MCP 工具前会请求审批，你可以查看参数和调用内容。

官方文档还提到可以启用 auto-run，让 Agent 不再每次询问就直接使用工具。这个能力很方便，但风险也明显增加。

建议：

- 新工具先不要开 auto-run
- 涉及写操作的工具不要轻易开 auto-run
- 第三方来源不明的工具不要装
- 关键工具先在测试仓库里试

---

## 9. 安全清单

官方文档在安全注意事项里强调：安装 MCP 服务器前要理解它会访问什么，并只信任可靠来源。

可以按这个清单检查：

1. 工具来源是否可信
2. 是否读写本地文件
3. 是否能访问网络
4. 是否能访问外部账号或 API
5. API Key 是否最小权限
6. 是否会把数据发送到第三方
7. 是否需要代码审计
8. 是否应该放到项目级配置中共享

如果答案不清楚，就先不要接入生产项目。

---

## 10. 项目里怎么落地

个人项目可以这样做：

1. 先用 `@Files` / `@Link` 验证需求是否稳定
2. 稳定后找可信 MCP 服务器
3. 先配到 `~/.cursor/mcp.json`
4. 确认有价值后，再考虑项目级共享
5. 把密钥管理方式写到项目说明里

团队项目可以这样做：

1. 由负责人确认 MCP 服务来源
2. 明确读写权限和审批策略
3. 项目级配置只提交无敏感信息的部分
4. 在 README 或 `AGENTS.md` 写清楚配置步骤
5. 定期审计 MCP 工具清单

## 11. 常见误区

### 11.1 把 MCP 当成 Prompt 替代品

MCP 是工具扩展，不是需求表达替代品。任务目标、范围、验收标准仍然要写清楚。

### 11.2 随便安装第三方服务器

MCP 服务器可能访问外部服务或代表你执行代码，不能把它当成普通文档插件。

### 11.3 把密钥写进仓库

项目级 `.cursor/mcp.json` 如果要提交，必须避免提交真实 API Key。

### 11.4 一上来就自动运行

auto-run 适合你已经非常信任的工具链，不适合刚接入时直接打开。
