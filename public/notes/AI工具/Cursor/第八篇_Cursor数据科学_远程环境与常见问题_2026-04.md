# 第八篇：Cursor 数据科学、远程环境与常见问题（2026-04）

> 本篇依据 Cursor 官方 `Data Science`、`Common Issues`、`Troubleshooting Guide` 文档整理，资料检索时间：2026-04-14。

[[toc]]

## 1. 为什么把数据科学和排障收在最后

这两个主题以前分散成多篇，会让目录显得很“杂”。但它们其实都属于：

- 不是每个 Cursor 用户第一天就会用到
- 一旦用到，往往会牵涉环境、扩展、网络和系统边界

所以保留一篇进阶收口稿，比拆成更多小主题更实用。

---

## 2. Data Science 在 Cursor 里现在怎么做更稳

官方 `Data Science` 指南里已经把主线说得比较清楚：

- 先安装 `Jupyter` 扩展（`ms-toolsai.jupyter`）
- Cursor 同时支持 `.ipynb` 和 `.py` 的单元执行
- `Tab`、`Inline Edit`、`Agent` 都能在 notebook 中工作

这意味着它不是“只能看 notebook，不能像代码一样协作”的状态了。对 Python、R、SQL 混合流来说，Cursor 已经更像一个能吃到 AI 上下文的通用工作台。

---

## 3. 数据库和可视化，不一定非要靠聊天框

官方把数据库接入分成两条路：

- 通过 `MCP servers`
- 通过 IDE 扩展

如果目标是让 Agent 主动查询数据库、分析结果、继续执行下一步，用 MCP 更自然；如果只是想在 IDE 里稳定执行查询并配合 AI 做解释和优化，扩展就够用了。

官方还特别强调了数据可视化能力，尤其适合：

- Matplotlib
- Plotly
- Seaborn

这类场景里，Cursor 的价值不是替你做结论，而是更快生成可复现的分析和可视化代码。

---

## 4. Remote SSH、容器和远程环境，先认清支持边界

官方排障文档里有几个边界非常值得提前知道：

- GitHub Codespaces 目前仍不支持
- Remote SSH 到 `Mac` 或 `Windows` 机器不支持
- Windows 上如果看到 “SSH is only supported in Microsoft versions of VS Code”，官方建议卸载旧的 Remote-SSH，并改装 `Anysphere Remote SSH`

所以，远程环境不是“理论上都能连”，而是要先看官方支持边界，再决定是否投入时间调环境。

---

## 5. 常见问题里，最常用的是这一组排查动作

### 5.1 网络与代理

官方明确写到，Cursor 的 AI 功能高度依赖 `HTTP/2`。如果你在企业网络、VPN 或某些代理环境下遇到：

- 索引失败
- Tab / Inline Edit 不稳定
- AI 相关请求异常

优先去：

- `Cursor Settings -> Network -> Run Diagnostics`

如果确认是网络层对 `HTTP/2` 支持差，官方建议在应用设置里开启：

- `Disable HTTP/2`

让它回落到 `HTTP/1.1`。

### 5.2 登录问题

如果点 `Sign In` 会跳到 `cursor.com`，但客户端里还是没登录成功，官方建议优先检查：

- 防火墙
- 杀毒软件

### 5.3 更新问题

如果 changelog 里已经看到新版本，但本机没有更新，官方说明通常是 staged rollout，也就是分批推送，不一定是本地坏了。

---

## 6. 当你怀疑“不是网络，是环境本身有问题”时

官方排障材料里还建议做这几件事：

- 用 `cursor --disable-extensions` 测试是否是扩展冲突
- 打开 `Developer: Open Process Explorer`
- 打开 `Developer: Open Extension Monitor`
- 做最小安装测试

这套顺序的意义在于，先判断问题来自：

- Cursor 本体
- 扩展
- 终端/pty host
- 操作系统资源

而不是一上来就重装。

---

## 7. 哪些人最值得保留这一篇

如果主人属于下面几类使用者，这一篇建议直接收藏：

- 经常在 notebook 里做分析或实验
- 会让 Agent 触碰数据库或可视化代码
- 需要 Remote SSH 或远程开发环境
- 常在公司网络、代理、VPN 环境里使用 Cursor

它不是基础入门篇，但能帮你少踩很多“和功能本身无关，却最影响体验”的坑。
