# 第一篇：Cursor 快速上手与初始化设置（2026-04）

> 本篇以 Cursor 官方文档为主线整理，资料快照时间：2026-04-14。

[[toc]]

---

## 1. Cursor 是什么

Cursor 可以理解为一个把 AI 编程能力深度做进编辑器的 IDE。它不只是“聊天框 + 代码补全”，而是把聊天、编辑、终端、代码索引、规则系统和上下文引用都接到了同一套工作流里。

如果你以前主要用 VS Code，那么上手 Cursor 的成本并不高。真正决定体验差异的，不是界面，而是下面这几件事有没有配好：

1. 项目有没有完成代码库索引
2. 哪些文件应该给 AI 看，哪些文件应该屏蔽
3. 你的项目规则是否有持久化载体
4. 你是否清楚 Agent、Ask、Inline Edit 分别适合什么任务

---

## 2. 安装与首次启动

### 2.1 下载安装

官方下载安装页：

- `https://www.cursor.com/downloads`

常见安装方式：

- Windows：下载 `.exe` 安装包后按向导完成安装
- macOS：下载对应芯片版本并拖入 Applications
- Linux：根据发行版使用对应安装包或解压版

### 2.2 首次打开时建议完成的事

第一次启动 Cursor，建议按这个顺序完成初始化：

1. 登录账号，确保 AI 功能可用
2. 导入你习惯的 VS Code 快捷键和主题
3. 打开一个真实项目，而不是先在空目录里试
4. 等待代码库索引开始工作
5. 先把忽略文件和项目规则配好，再开始让 Agent 改代码

很多人刚装完就直接丢一句“帮我修 bug”，结果体验不好，原因通常不是模型不行，而是上下文还没准备好。

---

## 3. 代码库索引是第一优先级

Cursor 官方文档说明，打开项目后会自动开始做 Codebase Indexing，也就是为文件计算语义索引，后续 Ask、Agent、搜索和相关文件召回都会受它影响。

你可以在设置中的 `Indexing & Docs` 查看状态。实操上建议关注三件事：

1. 索引是否真的开始了
2. 哪些文件被纳入了索引
3. 大体量无关内容是否已经排除

### 3.1 为什么索引很重要

如果索引不完整，常见表现是：

- 它只会围绕当前打开文件回答
- 能看到目录名，但抓不到真正相关实现
- 相似问题在别处明明有现成代码，它却找不到

### 3.2 哪些文件不该让它乱读

官方提供了两个很有用的忽略机制：

- `.cursorignore`：直接限制 AI 访问、引用和索引这些文件
- `.cursorindexingignore`：只从索引里排除，但文件本身仍可被其他 AI 功能访问

适合优先排除的内容：

- `dist/`
- `coverage/`
- `node_modules/`
- 构建缓存和日志
- 含密钥、证书、环境变量的文件
- 超大体积、与当前任务无关的生成文件

一个很实用的思路是：先把“噪声”和“敏感内容”清掉，再让 Cursor 学你的仓库。

---

## 4. 初始化阶段就该建立规则

Cursor 官方文档把规则分成几层：

- Project Rules：放在 `.cursor/rules`，适合项目级、团队级约束
- User Rules：写在个人设置里，适合你的回复风格和个人习惯
- Memories：从对话中沉淀的项目记忆
- `AGENTS.md`：适合简单项目说明的 Markdown 方案

如果你是第一次用 Cursor，我更建议这样起步：

1. 先写一个简洁的 `AGENTS.md`，把项目风格、目录约定、构建命令写清楚
2. 等需求变复杂，再把内容拆到 `.cursor/rules` 里做作用域管理

原因很简单：刚开始最需要的是“让它别跑偏”，而不是一上来就追求复杂规则体系。

### 4.1 一个适合新项目的最小说明清单

你可以先准备下面这些信息：

- 技术栈和版本，比如 Vue 3、Vite、Pinia
- 代码风格，比如 2 空格、单引号、无分号
- 路由方式和路径别名
- 哪些目录不能动，比如 `dist/`、脚本生成文件
- 修改后必须执行的命令，比如 `npm run generate:index`

这类信息越稳定，越值得写进规则，而不是每次临时 prompt 重复一遍。

## 5. 第一次打开一个仓库，建议这样试

### 5.1 第一步：先问结构，不急着改

先用 Ask 或 Agent 做代码地图：

```text
请先不要改代码。
用 5 到 8 条说明这个仓库的技术栈、入口文件、路由位置、数据来源和构建命令。
如果你不确定，请明确说不确定，不要猜。
```

### 5.2 第二步：让它确认上下文是否够

```text
基于当前索引和已读取文件，告诉我你还缺哪些上下文才能安全修改这个项目。
请按“必须补充”和“可选补充”分组。
```

这一步很有价值，因为它会逼着你先补上下文，再改代码。

### 5.3 第三步：从小任务验证它

别一开始就让它做大重构。先给一个很小、很容易验收的任务，例如：

```text
只修改文章详情页顶部的日期显示格式，其他文件不要改。
改完后说明你改了什么、为什么这样改。
```

如果这一步都经常跑偏，那说明是上下文、规则或者任务边界没有给清楚。

---

## 6. 初始化后的高频配置建议

### 6.1 把“可复用信息”从 Prompt 挪到规则里

适合长期存在的信息：

- 项目约束
- 目录规范
- 命名规范
- 提交前必跑命令
- 输出语言和说明风格

### 6.2 把“本次任务特有信息”留在 Prompt 里

适合每次单独说清楚的信息：

- 本次只改哪些文件
- 禁止动哪些模块
- 验收标准
- 是否要补测试
- 是否允许运行命令

这两类信息分开，Cursor 才不容易在长期对话里越来越混乱。

## 7. 常见问题

### 7.1 为什么它总是回答得很泛

大概率是这几个原因：

1. 仓库还没索引完
2. 你没有引用相关文件或目录
3. prompt 里只有“帮我看看”，没有目标、范围和验收标准
4. 仓库里噪声文件太多，真正重要的内容被淹没了

### 7.2 为什么它会碰到我不想让它看的文件

优先检查：

- 有没有配置 `.cursorignore`
- 是否误把过大的目录通过 `@Folders` 全量塞进上下文
- 是否用过过宽泛的“全项目重构”式指令

### 7.3 Memories 要不要开

如果你做的是持续迭代项目，Memories 很有价值，因为它能跨会话保留项目偏好。官方文档同时说明，Memories 与隐私模式相关，具体是否可用要看你的设置状态。

---

## 8. 新手可直接照抄的初始化清单

1. 安装 Cursor 并登录
2. 打开真实项目，等待索引开始
3. 用 `Indexing & Docs` 确认索引状态
4. 新建 `.cursorignore`，先排除噪声与敏感文件
5. 在项目根目录写 `AGENTS.md` 或建立 `.cursor/rules`
6. 先用 Ask 让它画项目结构图
7. 先用一个小改动验证它是否稳定
8. 稳定后再进入多文件修改和重构
