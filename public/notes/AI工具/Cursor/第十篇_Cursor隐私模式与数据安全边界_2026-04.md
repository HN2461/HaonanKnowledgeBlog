# 第十篇：Cursor 隐私模式与数据安全边界（2026-04）

> 本篇基于 Cursor 官方 Privacy & Security 文档整理，资料快照时间：2026-04-14。隐私政策、企业合规和套餐能力可能变化，实际以官方页面和组织安全要求为准。

[[toc]]

---

## 1. 先明确一件事：AI 编程工具不是本地纯离线工具

使用 Cursor 的 AI 能力时，代码片段、上下文、prompt、索引请求等都可能经过 Cursor 的服务端流程。官方文档也明确说明，即使使用你自己的 API key，请求仍会经过 Cursor backend，因为 Cursor 会在后端完成最终 prompt 构建。

所以在企业项目或敏感项目里，不能只问“模型强不强”，还要问：

- 哪些数据会发出去
- 是否开启 Privacy Mode
- 代码索引如何处理源码
- 哪些文件必须屏蔽
- 团队是否允许接入外部 AI 工具

---

## 2. Privacy Mode 是什么

官方文档说明，Cursor 提供隐私选项，让用户控制代码如何被处理。设置入口通常在：

```text
Cursor Settings -> General -> Privacy Mode
```

官方页面提到的隐私选项包括：

- Share Data
- Privacy Mode with Storage
- Privacy Mode

具体可用项和命名可能随版本变化，使用时以当前设置页为准。

### 2.1 实操理解

如果你处理的是个人学习项目，风险较低，可以按个人偏好选择。

如果你处理的是公司代码、客户数据、内网接口、密钥、日志或生产问题，应该优先：

1. 了解组织安全政策
2. 开启更严格的隐私选项
3. 使用 `.cursorignore` 屏蔽敏感文件
4. 避免把生产数据和密钥贴进聊天

---

## 3. 代码索引会不会长期保存源码

官方 Privacy & Security FAQ 对 Codebase Indexing 做了说明：索引时 Cursor 会上传代码库的小块内容来计算 embeddings，但明文代码在请求生命周期结束后不再保留；数据库中保存的是 embeddings 和元数据，例如 hashes、混淆后的文件名等，而不是源码本身。

这对理解索引边界很重要：

- 索引确实需要把代码片段发到服务端计算
- 官方说明不会长期保存明文源码
- 但 embeddings 和元数据会被保存
- 敏感文件仍然应该主动排除

所以安全策略不应该是“反正官方说不存源码，就什么都给它读”，而应该是“能排除的敏感内容先排除”。

---

## 4. `.cursorignore` 和 `.cursorindexingignore`

这两个文件适合配合隐私设置使用。

| 文件 | 作用 |
| --- | --- |
| `.cursorignore` | 限制 Cursor AI 访问、引用和索引对应文件 |
| `.cursorindexingignore` | 只从索引中排除，但不一定阻止其他 AI 功能访问 |

建议优先加入：

```gitignore
.env
.env.*
*.pem
*.key
*.p12
*.pfx
secrets/
credentials/
dist/
coverage/
node_modules/
```

真实项目要结合业务再补：

- 数据库导出
- 生产日志
- 客户资料
- 私有证书
- 内部配置
- 大型二进制和构建产物

---

## 5. 敏感信息不要靠模型“自觉”

不要把下面内容直接贴进 Chat：

- API Key
- 数据库密码
- SSH 私钥
- Cookie / Token
- 客户隐私数据
- 生产事故日志中的身份信息
- 内部未公开接口地址

如果必须让 Cursor 分析日志，先脱敏：

```text
用户 ID 已替换为 USER_ID_x
Token 已替换为 TOKEN_REDACTED
内部域名已替换为 internal.example
请只分析错误类型和调用链，不要尝试还原敏感信息。
```

## 6. MCP 与 Background Agents 的额外风险

只使用 Chat 和本地索引是一类风险；接入 MCP、Background Agents、Bugbot 后，边界会继续扩大。

### 6.1 MCP

MCP 服务器可能连接外部系统，例如数据库、文档系统、SaaS API。接入前要确认：

- 工具来源是否可信
- API Key 权限是否最小化
- 是否会执行写操作
- 是否会把数据传给第三方

### 6.2 Background Agents

Background Agents 会在远程环境运行代码，需要考虑：

- GitHub 仓库授权
- 远程 VM 环境
- 自动运行命令
- 网络访问能力
- 隐私模式对运行中 Agent 的生效边界

这些功能很强，但也更需要明确团队规则。

---

## 7. 团队使用安全清单

团队引入 Cursor 前，建议最少确认：

1. 是否允许使用 AI 编程工具处理当前仓库
2. 是否强制开启 Privacy Mode
3. `.cursorignore` 是否覆盖敏感文件
4. 是否允许代码库索引
5. 是否允许 MCP 连接内部系统
6. 是否允许 Background Agents 访问仓库
7. Bugbot 是否只能审查特定仓库
8. 日志、密钥、客户数据是否有脱敏流程
9. 代码合并前是否仍保留人工 review

---

## 8. 个人项目也要注意的事

个人项目也可能有敏感信息：

- 本地 `.env`
- 第三方平台 key
- 个人服务器地址
- 登录 token
- 自动部署凭据

建议一开始就把 `.cursorignore` 建好，而不是等泄漏后再补。

---

## 9. 常见误区

### 9.1 开了 Privacy Mode 就不用管文件屏蔽

Privacy Mode 是隐私策略的一部分，不替代 `.cursorignore`。

### 9.2 用自己的 API Key 就不经过 Cursor

官方 FAQ 说明，即使使用自己的 API Key，请求仍会经过 Cursor backend。

### 9.3 只屏蔽 `.env` 就够了

证书、日志、数据库导出、客户数据、构建产物都可能需要屏蔽。

### 9.4 把敏感数据贴给 AI 后再要求“不要保存”

最稳妥的做法是不要发送敏感数据，必要时先脱敏。

---

## 10. 一句话总结

Cursor 的安全用法不是“完全不用高级能力”，而是把数据边界写清楚：隐私模式、忽略文件、敏感数据脱敏、MCP 权限和远程 Agent 授权都要纳入同一套规则。
