# 第十七篇：Cursor Tabs 并行会话与多任务管理（2026-04）

> 本篇基于 Cursor 官方 Tabs 文档整理，资料快照时间：2026-04-14。

[[toc]]

---

## 1. 为什么要学会用 Tabs

Cursor 的每个 Agent Chat Tab 都是一个独立工作区，不只是一个“标签页”。

官方文档说明，每个 Tab 都有：

- 自己的对话历史
- 自己的上下文
- 自己的模型配置

这意味着如果你把不同问题混在一个对话里，AI 很容易上下文串台；而把不同任务拆到不同 Tabs，稳定度会高很多。

---

## 2. 新建、切换和命名

官方文档给出的常见操作：

| 操作 | 快捷键 / 方式 |
| --- | --- |
| 新建 Tab | `Ctrl + T` |
| 切换 Tab | 点击标题或 `Ctrl + Tab` |
| 重命名 | 右键 Tab 标题 |

实操建议：

1. 一个 Tab 只做一个任务
2. 第一条消息就把任务边界写清楚
3. 做完及时关闭或重命名

这样后续回来看聊天记录时，成本会低很多。

---

## 3. 每个 Tab 的上下文是隔离的

官方文档明确说，每个 Tab 保留自己的上下文。

这对多任务工作流很关键：

- 修 bug 的对话不会污染文档编写对话
- 前端样式调试不会带着后端排障历史
- 不同模型配置也可以分开试

但要注意，文件系统还是同一个工作区，所以“上下文隔离”不等于“改动隔离”。

---

## 4. 冲突提示

官方文档说明，Cursor 会阻止多个 Tabs 同时编辑相同文件，并提示你解决冲突。

这说明 Tabs 可以并行，但不是完全无约束并行。

推荐做法：

- 不同 Tab 处理不同功能块
- 大型任务按模块拆 Tab
- 避免两个 Tab 同时盯着同一组核心文件

如果真要并行同一块内容，更适合切 Git 分支，而不是强靠多个 Tab。

---

## 5. `@Past Chats` 怎么配合用

官方文档建议，可以用 `@Past Chats` 把其他 Tab 或之前会话的上下文引入当前任务。

适合：

- 一个方案已在旧 Tab 里讨论完
- 新 Tab 要继续实现旧 Tab 的结论
- 想保留独立对话，又不想完全丢掉旧推理

不建议：

- 把所有旧对话都塞进来
- 用历史摘要代替项目文档

重要结论还是应该沉淀成规则、Notepad 或正式文档。

---

## 6. 推荐的并行工作流

### 6.1 分任务建 Tab

例如：

- Tab A：排查移动端滚动 bug
- Tab B：梳理目录与 frontmatter 规范
- Tab C：写新文章或文档

### 6.2 先分析后执行

可以让一个 Tab 只做 Ask/Plan，另一个 Tab 再专门执行 Agent 改动。

### 6.3 做完立即收口

任务完成后：

1. 关闭 Tab
2. 或重命名为“已完成”
3. 关键结论转存到规则、文档或提交说明

---

## 7. 常见误区

### 7.1 一个 Tab 从头干到尾

越大的任务越应该拆，不然会话历史会越来越乱。

### 7.2 把 Tabs 当分支

Tabs 是对话隔离，不是代码隔离。真正要隔离代码改动，还是用 Git 分支。

### 7.3 旧结论不沉淀

只靠 `@Past Chats` 续上下文，后期会越来越难维护。

---

## 8. 一句话总结

Tabs 最适合做的是“任务隔离”，不是“代码隔离”。一个 Tab 一个目标，必要时用 `@Past Chats` 续上下文，做完及时收口，稳定度会高很多。
