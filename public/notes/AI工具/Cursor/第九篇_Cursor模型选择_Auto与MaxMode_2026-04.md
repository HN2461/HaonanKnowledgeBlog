# 第九篇：Cursor 模型选择、Auto 与 Max Mode（2026-04）

> 本篇基于 Cursor 官方 Models 与 Team Pricing 文档整理。模型列表、价格和额度规则变化较快，资料快照时间：2026-04-14，实际以官方页面为准。

[[toc]]

---

## 1. 为什么要理解模型选择

Cursor 不是只有一个固定模型。官方文档说明，Cursor 支持来自主要模型提供商的前沿编程模型，并提供 Auto 与 Max Mode 等能力。

如果不理解这些选项，常见问题是：

- 小任务用了过重模型，速度慢且成本高
- 大任务用了过小上下文，模型读不全项目
- 长会话一直堆上下文，后面回答越来越不稳定
- 看到 Max Mode 就默认打开，结果任务并不需要

模型选择的核心不是“永远选最强”，而是“任务难度、上下文规模、速度、成本”之间做平衡。

---

## 2. 先理解上下文窗口

官方文档把 context window 定义为模型一次能考虑的 token 范围，包括：

- 你的 prompt
- 附加文件
- 代码片段
- 对话历史
- 模型输出

Cursor 中每个 chat 都维护自己的上下文窗口。也就是说，同一个项目里，不同聊天会话的上下文状态可能不同。

### 2.1 上下文窗口不等于“越大越好”

更大的上下文窗口能读更多内容，但也会带来：

- 响应更慢
- 成本更高
- 噪声更多
- 关键文件被无关信息稀释

实战里更推荐“精准上下文”优先于“超大上下文”。

---

## 3. Auto：让 Cursor 自动选模型

官方文档说明，启用 Auto 后，Cursor 会根据当前需求和实时可用容量自动选择更合适的 premium model，并在需要时切换模型以保持稳定性。

适合用 Auto 的场景：

- 日常编码
- 小中型 bug 修复
- 你不想花时间手动选模型
- 对速度和稳定性有平衡要求

不适合完全依赖 Auto 的场景：

- 你明确知道任务需要某个特定模型
- 你要做稳定可复现的模型对比
- 团队有严格成本或模型策略

一句话：Auto 适合大多数日常工作，但不是所有任务都必须交给它自动判断。

---

## 4. Max Mode：扩展上下文窗口

官方文档说明，Cursor 默认通常以约 200k tokens 作为工作上下文窗口；开启 Max Mode 后，会尽量把窗口拉到当前所选模型支持的最大范围。

官方也提醒：Max Mode 会更慢、更贵。

### 4.1 什么时候适合开 Max Mode

适合：

- 需要读大量跨目录代码
- 需要长文档 + 大量代码一起分析
- 大型重构前做整体设计
- 长日志、长 PR、长规格文档的综合分析

### 4.2 什么时候不建议开

不建议：

- 改一个小函数
- 改一段 CSS
- 只问某个文件的局部逻辑
- 你还没有收敛任务范围

如果问题可以通过 `@Files` 精准补上下文解决，就不要先开 Max Mode。

---

## 5. 模型选择的实用分层

可以按任务类型粗分：

| 任务类型 | 建议 |
| --- | --- |
| 简单解释、小改动 | Auto 或常规模型 |
| 多文件 bug 修复 | Auto，必要时指定更强模型 |
| 大型重构设计 | 更强模型，必要时 Max Mode |
| 长文档和长代码联合分析 | 支持长上下文的模型 + Max Mode |
| 成本敏感的日常维护 | Auto 或普通模式优先 |

关键是先判断“难点在哪”：

- 难在推理：选更强模型
- 难在读得多：考虑 Max Mode
- 难在上下文噪声：先缩小引用范围
- 难在执行闭环：用 Agent + 测试验证

---

## 6. 成本意识

截至 2026 年 4 月 14 日的官方 Team Pricing 页面，Auto 在达到月度包含请求上限后仍可继续使用，但不会继续走额外的 usage-based pricing；如果你手动指定模型或继续使用 Max Mode，则会进入按量计费。官方页面同时写明，Max Mode 按 token 计费，通常是在模型提供商 API 价格基础上加价 20%。价格和额度后续仍可能变化，所以这里不写死具体金额。

实操建议：

1. 小任务不要默认 Max Mode
2. 长会话定期新开对话，减少历史噪声
3. 明确引用关键文件，不要整目录乱塞
4. 大任务先用 Ask 做方案，再让 Agent 执行
5. 成本敏感团队要在规则或协作说明里写清模型策略，并明确何时允许切到手动选模或 Max Mode

---

## 7. Prompt 里怎么表达模型需求

通常不用在 prompt 里说“必须用最强模型”，而是把任务难点写清楚：

```text
这是一次跨多个模块的重构设计任务。
请先读取我引用的目录和接口文档，给出风险清单和分阶段方案。
暂时不要修改代码。
```

如果你已经知道上下文很长，可以配合 UI 里选择 Max Mode，再写：

```text
请基于这组长文档和相关代码做一致性检查。
重点输出冲突点、缺失实现和建议修改顺序。
```

---

## 8. 常见误区

### 8.1 模型越强越不用写清楚需求

更强模型也需要清晰目标、边界和验收标准。

### 8.2 Max Mode 能解决所有上下文问题

Max Mode 只是扩大窗口，不会自动帮你去掉噪声。无关上下文太多仍然会影响结果。

### 8.3 一条会话从早聊到晚

长会话会不断积累历史。任务切换明显时，建议新开会话。

### 8.4 把 Auto 当成完全不可控

Auto 是日常工作里很好的默认项。真正需要指定模型时，再手动选择即可。

---

## 9. 一句话总结

Cursor 模型选择的顺序建议是：

先缩小上下文，再选择模型；先判断任务难点，再决定是否开启 Max Mode。
