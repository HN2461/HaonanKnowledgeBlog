## 1. 为什么同一个 Cursor，有人觉得很强，有人觉得很飘

核心差别通常不在模型，而在“上下文组织能力”。

高质量输出，往往不是因为你写了很华丽的 prompt，而是因为你把下面几件事说清楚了：

1. 目标是什么
2. 改动范围是什么
3. 哪些限制必须遵守
4. 它应该参考哪些文件或规则
5. 最后要交付成什么样

Cursor 官方文档里，Rules、Memories、`@Files`、`@Folders`、`@Link` 本质上都在解决同一个问题：**把该长期保存的信息和该临时补充的信息分开管理。**

---

## 2. 一条高质量 Prompt 的 5 个组成部分

最稳的写法一般包含这 5 段：

1. 任务目标
2. 作用范围
3. 约束条件
4. 参考上下文
5. 输出格式

可以套这个骨架：

```text
任务：
范围：
约束：
参考：
输出：
```

例如：

```text
任务：修复文章详情页移动端滚动容器判断错误
范围：只改阅读相关组件和详情页
约束：不要改首页，不要删现有注释
参考：@src/views/NoteDetailPage.vue @src/components/ReadingToolbar.vue
输出：改完后说明原因、改动点、验证方式
```

这种写法的好处是：你自己也会被迫先想清楚。

## 3. 哪些信息该写进 Rules，哪些该留给 Prompt

### 3.1 适合写进 Rules 的信息

- 长期稳定的代码风格
- 项目目录约定
- 常见禁区
- 构建、测试、发布前置命令
- 固定的架构原则

### 3.2 适合留在单次 Prompt 的信息

- 本次任务目标
- 本次只动哪些文件
- 本次验收标准
- 本次是否允许执行命令
- 本次是否需要补测试和文档

如果你把所有东西都堆在每次 prompt 里，会越来越长，也越来越容易漏。

## 4. Cursor 的规则分层怎么理解

官方文档里，Cursor 主要有这些长期上下文载体：

- Project Rules：项目级规则，放在 `.cursor/rules`
- User Rules：个人全局规则，适合回复风格和通用习惯
- Memories：从对话中沉淀出来的项目记忆
- `AGENTS.md`：适合简单场景的纯 Markdown 指令文件

### 4.1 `.cursor/rules` 适合什么场景

适合：

- 团队项目
- 规则较多，且需要按目录或场景分作用域
- 需要把不同规范拆开管理

官方文档提到的常见类型包括：

- `Always`
- `Auto Attached`
- `Agent Requested`

你可以把它理解成“总是带上”“命中文件再带上”“让 AI 自己按描述决定要不要带上”。

### 4.2 `AGENTS.md` 适合什么场景

适合：

- 个人项目
- 想先快速立规则，不想一开始就搭复杂结构
- 项目规范本身就不多

如果你的仓库已经有一份写得很清楚的 `AGENTS.md`，那它就是 Cursor 很好的第一层项目说明。

### 4.3 Memories 什么时候值得开

适合：

- 长期迭代的同一个仓库
- 经常重复解释相同偏好
- 项目知识会不断沉淀

官方文档说明，Memories 可以在对话中自动生成，也可以在你明确要求“记住这件事”时创建，并且可以在 `Settings -> Rules` 里管理。

## 5. 一个很稳的项目级规则最小模板

如果你暂时不想写复杂规则文件，可以先写一份 `AGENTS.md`：

```markdown
# Project Instructions

## Code Style
- 2 空格缩进
- 单引号
- 不加行尾分号

## Structure
- src 内优先使用 @/ 别名导入
- 路由使用 hash history

## Safety
- 不要手动修改 dist
- public/notes 变更后运行 npm run generate:index

## Workflow
- 修改前先说明影响范围
- 修改后说明验证结果
```

这个版本的重点不是“大而全”，而是足够清晰、足够稳定。

## 6. 6 个高频 Prompt 模板

下面这些模板是整理版，适合直接改字后用。

### 6.1 模板一：接手陌生项目

```text
请先不要改代码。
基于当前仓库，帮我梳理：
1. 技术栈与入口文件
2. 核心页面和路由
3. 数据流或状态管理位置
4. 可能的构建与发布命令
如果你不确定，请明确说不确定，不要猜。
```

### 6.2 模板二：定位 bug

```text
任务：定位并修复 [具体问题]
范围：只检查 [文件/目录]
约束：先分析原因，不要立刻大改
参考：@[相关文件]
输出：按“现象、原因、最小修复方案”回答
```

### 6.3 模板三：实现小功能

```text
任务：实现 [功能名]
范围：只允许修改 [文件/目录]
约束：
1. 保持现有风格
2. 不改无关文件
3. 如需新增文件请先说明
验收标准：
1. [标准1]
2. [标准2]
3. [标准3]
改完后总结改动点并给出验证方式。
```

### 6.4 模板四：安全重构

```text
请帮我重构这段代码，但不要改变行为。
目标是提升可读性并减少重复。
先给出你的重构计划和风险点，确认后再改。
```

### 6.5 模板五：生成测试

```text
请为这次改动补充最小必要测试。
优先覆盖新增逻辑和已修复问题。
如果现有测试体系不方便落地，请说明阻碍点，不要编造通过结果。
```

### 6.6 模板六：代码评审

```text
请只做代码评审，不要修改文件。
重点检查：
1. 行为回归风险
2. 边界条件
3. 是否缺少测试
4. 是否有不必要的复杂度
按严重程度排序输出。
```

## 7. 让 Prompt 更稳的三个小技巧

### 7.1 明确“不要做什么”

很多跑偏问题，不是因为目标没说，而是因为禁区没说。

例如：

- 不要改 UI，只修逻辑
- 不要改公开接口
- 不要新增依赖
- 不要碰生成文件

### 7.2 明确“先分析再执行”

尤其是复杂任务，最好先让它给计划或风险清单，再决定是否执行。

### 7.3 明确“用什么格式回报”

比如要求它按“原因、改动点、验证方式”或“风险、建议、结论”输出，沟通会清楚很多。

---

## 8. Rules 和 Prompt 的推荐组合

一个简单但很有效的组合是：

- User Rules：只放你的回复风格、语言偏好
- `AGENTS.md` 或 `.cursor/rules`：放项目长期规范
- 本次 Prompt：放当前任务目标、范围、验收标准
- `@Files` / `@Folders`：补足本次任务的局部上下文

这样做的结果是：长期信息不丢，短期信息不乱。

---

## 9. 常见误区

### 9.1 把 Rules 写成空话

比如“注意代码质量”“保持优雅”。这种话没有可执行性。

### 9.2 一个规则塞所有内容

规则越大越难维护，也越不容易命中真正有价值的部分。

### 9.3 Prompt 太长但没有结构

堆很多自然语言不等于清晰。结构化往往比长度更重要。

### 9.4 指望 Memories 替代项目规则

Memories 适合沉淀偏好，不适合承载完整项目规范。
