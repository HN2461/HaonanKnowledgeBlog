# 第七篇：Cursor 模型、隐私与团队企业治理（2026-04）

> 本篇依据 Cursor 官方 `Models`、`Team Pricing`、`Privacy & Security`、`Teams Setup`、`Dashboard`、`Analytics`、`Admin API` 文档整理，资料检索时间：2026-04-14。

[[toc]]

---

## 1. 模型选择现在最重要的不是“背型号”，而是认清计费方式

官方当前资料里，模型和计费逻辑已经比早期清晰很多：

- `Normal mode`：按请求数计费
- `Max Mode`：按 token 计费
- `Auto-select`：让 Cursor 自动挑当前最适合、且可用性更高的模型

官方明确写到：

- 到达月度额度后，继续开 `Auto` 不会额外产生 usage-based pricing
- 如果额度用完但你还想指定某个模型，可以开启 usage-based pricing
- usage-based pricing 目前是 API 成本基础上再加 `20%`

所以，主人真正需要记住的不是某个型号今天是不是最强，而是：

- 日常开发优先 `Auto` 或普通模式
- 复杂推理、超长上下文、异步 Agent 再考虑 `Max`
- 长上下文不是越大越好，输入 token 越多，成本也越高

## 2. 隐私这一块，最重要的是理解“代码如何被处理”

官方 `Privacy & Security` 文档给了一个关键表述：

- 如果你选择为代码库建立索引，Cursor 会把代码按小块上传到服务端计算 embeddings
- 纯文本代码只在请求生命周期内存在

这说明两件事：

1. 索引不是“完全不上传”
2. 但它和“长期明文存整份源码”不是一回事

另外，Background Agents 官方也明确说明：

- 在 Privacy Mode 下可用
- 但因为任务需要执行，会有几天量级的数据保留

所以，隐私治理不能只停留在“开没开 Privacy Mode”，还要结合：

- 是否索引
- 是否启用 Background Agents
- 是否把敏感目录通过 ignore 排除
- 团队有没有统一策略

## 3. Dashboard 是团队治理入口，不只是看账单

官方 `Dashboard` 页面里，团队后台至少承担这些职责：

- billing
- usage-based pricing
- team management
- privacy settings
- active sessions
- invite code management
- Cursor Admin API Keys

Enterprise 相关控制项更值得集中看：

- `Model Access Control`
- `Auto Run Configuration`
- `Repository Blocklist`
- `MCP Configuration`
- `Cursor Ignore Configuration`
- `.cursor Directory Protection`

这些能力合在一起，才是真正的“团队级 Cursor 治理面板”。

---

## 4. Analytics 和 Admin API，分别解决什么问题

### 4.1 Analytics

官方 `Analytics` 文档强调，团队管理员可以在后台看：

- 总体使用量
- 每活跃用户平均指标
- 周活和月活

导出后的 analytics 报告还会包含更细的行为与使用字段，用来判断：

- 谁在高频使用
- 哪类功能使用最多
- 请求和代码接受情况如何

### 4.2 Admin API

官方 `Admin API` 当前主打的是：

- 团队成员信息
- 使用量数据
- 花费数据

而且它有几个边界值得记住：

- 只有管理员能创建和管理 key
- key 绑定组织，不绑定创建人个人生命周期
- key 对所有管理员可见

如果主人后面要做内部可视化报表、用量预警或和已有平台打通，这一组接口会比手工导出更合适。

---

## 5. 一套更实用的团队落地顺序

如果站在团队治理视角，我更建议这样分阶段上 Cursor：

1. 先统一 `Rules / AGENTS.md / ignore` 基础约束
2. 再统一模型和 usage-based pricing 策略
3. 接着在 Dashboard 里打开必要的团队控制项
4. 需要观测数据时，再启用 Analytics 和 Admin API
5. 对企业仓库再补 Model Access Control、Repository Blocklist、MCP Configuration 这类硬护栏

这样做比一开始就把所有企业配置打开，更容易让团队真正理解每个控制项在防什么风险。
