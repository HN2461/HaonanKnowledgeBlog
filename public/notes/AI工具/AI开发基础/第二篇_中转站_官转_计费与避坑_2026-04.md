# 第二篇：中转站、官转与计费规则到底怎么回事（2026-04核对）

> 资料核对时间：2026-04-27  
> 适用对象：准备把 Codex、Claude、Gemini 接到开发工具里，但看中转站手册和价目表看不明白的人。  
> 本文优先引用官方定价页、官方文档和 CC Switch 官方仓库；涉及“市场常见卖法”的部分，会明确标注为归纳判断。

## 先说结论

如果你只想先记住最核心的一句话：

**中转站本质上不是“另一个模型”，而是站在你和 OpenAI / Anthropic / Google 之间的接口网关或代计费层。**

所以你实际要同时看三层：

1. **上游官方模型怎么收费**：输入、输出、缓存、长上下文、搜索、图片、Batch 等
2. **中转站怎么转卖**：是否加价、是否包月、是否限速、是否偷偷切模型
3. **你用的客户端怎么耗费 token**：Codex、Claude Code、Gemini CLI、Cursor、CC Switch 本地路由都会影响真实消耗

你之所以觉得手册很绕，通常不是你笨，而是卖家把这三层混在一起写了。

---

## 1. 中转站到底是什么

### 1.1 最常见的三种形态

为了方便理解，我把开发者常见的“中转站”分成三类。  
这一部分是基于官方 API 形态和常见售卖方式做的归纳，不是某一家厂商的标准术语。

| 类型 | 本质 | 你拿到什么 | 典型特点 |
|------|------|-----------|---------|
| 官方直连 | 你直接向 OpenAI / Anthropic / Google 付款 | 官方 API Key / 官方账号 | 最稳定、最透明、价格最清楚 |
| 聚合网关 / 路由商 | 它替你对接多个上游，再统一卖给你 | 一个 Base URL + 一个 Key | 方便切模型、统一余额、统一协议 |
| 代充 / 共享池型中转 | 它可能自己维护上游账号、额度池或代理层 | 一个看起来像官方 API 的入口 | 价格可能便宜，但风控、稳定性、模型真实性风险最高 |

### 1.2 中转站卖给你的通常不是“模型”，而是这几样东西

- 一个 `Base URL`
- 一个 `API Key`
- 一套“兼容协议”
- 一份价格表
- 一条限速规则
- 一套余额 / 会员 / 套餐系统

你最终接入开发工具时，真正要填进去的往往只有两样：

```text
Base URL
API Key
```

但真正容易出问题的，恰恰是你看不见的后面三样：**协议兼容、模型映射、计费结算**。

---

## 2. 为什么同样叫 Claude、Gemini、GPT，价格能差这么多

因为中转站给你看的价格，未必等于上游官方原价。

你可以把一次请求的真实成本拆成：

```text
你的最终支出
= 上游官方模型成本
+ 中转站加价 / 套餐摊销 / 会员费
+ 额外能力费用（搜索、图片、文件、路由、代理）
+ 因客户端行为多出来的 token 消耗
```

### 2.1 上游官方模型成本

这是最底层，也是你判断“中转站报价是否离谱”的基准线。

### 2.2 中转站自己的售卖方式

开发者市场里最常见的几种卖法：

| 卖法 | 你表面上看到的东西 | 实际要注意 |
|------|------------------|-----------|
| 余额充值 | 充 10 美元、50 美元、100 美元 | 一般最接近按量付费，先看是否有充值手续费、是否会过期 |
| 按 token 加价 | “官方价 1.2 倍”“输入 1.5 倍” | 要问清是所有模型统一倍率，还是每个模型单独定价 |
| 包月 / 会员 | “Pro”“无限用”“开发者版” | 几乎一定有软限制、公平使用、并发限制或高峰限速 |
| 套餐包 | “每月 5000 次”“每月 50M token” | 先看超额后怎么收费，没用完是否结转 |
| 混合模式 | 会员 + 超额按量 | 最复杂，最容易看漏“超额段价格” |

### 2.3 客户端自己的消耗

这是很多人忽略的点。

你不是只为“这一句提问”付钱，而是在为整段上下文付钱，包括：

- System Prompt
- 历史对话
- 贴进去的代码和文档
- 工具调用产生的附加内容
- 模型输出
- 某些平台的缓存写入 / 读取

所以哪怕你问的是同一个问题：

- 在网页聊天里可能很便宜
- 在 Codex、Claude Code、Gemini CLI 里可能更贵
- 在长会话、Agent、多工具调用下可能会明显更贵

## 3. 先学会读官方计费，再回头看中转站就容易了

下面只整理和开发最相关、最容易影响中转站报价理解的规则。

## 3.1 OpenAI：输入、缓存输入、输出通常分开计费

根据 OpenAI 官方 API 定价页（2026-04-27 核对），当前旗舰编码模型大致是：

| 模型 | 输入 | 缓存输入 | 输出 |
|------|------|---------|------|
| GPT-5.5 | $5.00 / 1M tokens | $0.50 / 1M | $30.00 / 1M |
| GPT-5.4 | $2.50 / 1M | $0.25 / 1M | $15.00 / 1M |
| GPT-5.4 mini | $0.75 / 1M | $0.075 / 1M | $4.50 / 1M |

你要记住的不是每个数字，而是这几个规则：

1. **输出通常远贵于输入**
2. **缓存输入明显更便宜**
3. **Batch API 官方写明可省 50%**
4. **某些工具能力单独收费**

OpenAI 官方定价页当前还单独列出了：

- `Web search`：`$10 / 1k calls`
- `Batch API`：输入和输出都可打 5 折

所以如果中转站跟你说“支持 GPT，价格超便宜”，你至少要问一句：

**它是正常按输入/缓存输入/输出分别结算，还是偷懒按一口价糊成一个倍率？**

如果它把缓存优惠吃掉不让利，你看到的“便宜”未必真的便宜。

## 3.2 Anthropic：最关键的是缓存写入、缓存命中、长上下文加价

Anthropic 官方文档里，开发者最容易忽略三点：

1. Prompt caching 默认 **5 分钟**
2. 也支持 **1 小时** TTL
3. 如果启用 Claude Sonnet 4 的 **1M 上下文**，且输入超过 **200K tokens**，会进入长上下文加价档

按 Anthropic 官方定价文档（2026-04-27 核对）：

| 项目 | Claude Sonnet 4 |
|------|------------------|
| 基础输入 | $3 / MTok |
| 5 分钟缓存写入 | $3.75 / MTok |
| 1 小时缓存写入 | $6 / MTok |
| 缓存命中 / 刷新 | $0.30 / MTok |
| 输出 | $15 / MTok |

Anthropic 官方还明确写了缓存倍率：

- 5 分钟缓存写入 = 基础输入价格的 **1.25 倍**
- 1 小时缓存写入 = 基础输入价格的 **2 倍**
- 缓存读取 = 基础输入价格的 **0.1 倍**

如果启用 Sonnet 4 的 1M 上下文窗口并超过 200K 输入 token，Anthropic 文档写明：

- 输入从 `$3 / MTok` 提升到 `$6 / MTok`
- 输出从 `$15 / MTok` 提升到 `$22.50 / MTok`

这就是为什么很多人会觉得：

> “明明还是 Claude，怎么某次突然贵了两倍？”

原因往往不是中转站乱扣钱，而是**你真的打进了长上下文档位**。

## 3.3 Gemini：不只是输入输出，缓存存储本身也收费

Google Gemini Developer API 定价页（2026-04-27 核对）里，对开发者最重要的点是：

1. Gemini 有 **免费层** 和 **付费层**
2. **Context caching** 不只是“读缓存收费”，还有**存储费用**
3. Gemini 2.5 Pro 在 **200K tokens** 以内和以外是两档价
4. `thinking tokens` 已经包含在输出价格里

几个最常用模型的标准档价格：

| 模型 | 输入 | 输出 | 缓存价格 | 备注 |
|------|------|------|---------|------|
| Gemini 2.5 Pro | $1.25 / 1M（<=200K） | $10 / 1M（<=200K） | $0.125 / 1M + $4.50 / 1M tokens/小时存储 | 超过 200K 后单价上升 |
| Gemini 2.5 Flash | $0.30 / 1M | $2.50 / 1M | $0.03 / 1M + $1 / 1M tokens/小时存储 | 速度和价格平衡 |
| Gemini 2.5 Flash-Lite | $0.10 / 1M | $0.40 / 1M | $0.01 / 1M + $1 / 1M tokens/小时存储 | 最便宜 |

Gemini 2.5 Pro 超过 200K prompt 后，官方定价页当前写的是：

- 输入：`$2.50 / 1M`
- 输出：`$15 / 1M`
- 缓存：`$0.25 / 1M`

所以 Gemini 这边你一定要看清：

**卖家给你的“超长上下文支持”到底只是能发得进去，还是价格也跟官方长上下文档一致。**

## 4. 把中转站价格表翻译成人话

你以后看到价目表，按下面顺序读，基本不会再乱。

### 4.1 先找“真实模型 ID”

不要只看卖家写的：

- `Claude 高级版`
- `GPT 官方`
- `Gemini 稳定版`
- `编程模型`

这些都不够。

你真正要找的是类似下面这种能落到接口里的模型标识：

```text
gpt-5.4
gpt-5.5
claude-sonnet-4
gemini-2.5-pro
gemini-2.5-flash
```

如果它不给真实模型 ID，只给营销名字，这种价格表基本没法严肃比较。

### 4.2 再找计费单位

常见单位有：

- 每 `1M tokens`
- 每次请求
- 每月套餐
- 每日额度
- 每小时并发包
- 每张图片 / 每次搜索

如果一个中转站同时写了“包月”和“按 token”，你要继续追问：

**包月覆盖哪些模型，哪些功能超出后要另算？**

### 4.3 再找是否区分输入 / 输出 / 缓存

如果价格表没有区分这三项，就很容易出现两种情况：

1. 卖家把最坏情况按一个均价卖给你
2. 卖家根本没打算把官方缓存优惠给你

对于开发场景来说，这个差异很大，因为代码工具往往有大量可缓存前缀。

### 4.4 再看有没有“隐藏档位”

最常见的隐藏档位：

- 超过 200K token 后涨价
- 高峰时段限速
- 高级模型要单独开通
- 免费模型和稳定模型不是一回事
- `thinking`、`search`、`tool use`、`image` 另计费

## 5. 主人买中转站前后，最该问卖家的 10 个问题

只要这 10 个问题问清楚，基本能过滤掉大部分糊涂账。

1. **你给我的是真实模型 ID 还是自定义别名？**  
   如果是别名，请给一张“别名 -> 实际上游模型”的对照表。

2. **你兼容的是哪种协议？**  
   是 OpenAI 兼容、Anthropic 兼容，还是 Gemini 原生接口？别只说“兼容官方”。

3. **输入、输出、缓存是不是分开计费？**  
   如果不是，要求对方解释它的统一单价怎么算出来。

4. **是否支持官方缓存机制，优惠会不会传递给我？**  
   OpenAI、Anthropic、Gemini 的缓存规则并不一样。

5. **长上下文有没有额外档位？**  
   例如 Claude / Gemini 超过 200K token 后是否涨价。

6. **限速规则是什么？**  
   包括 RPM、TPM、并发数、峰值限制、排队策略。

7. **会不会自动降级或切换模型？**  
   例如高峰时把 Sonnet 切成更便宜的小模型，或者把官方源切到别家源。

8. **余额怎么查，是否有接口或面板？**  
   最好能查剩余额度、调用明细、失败记录。

9. **充值是否有手续费、最低充值额、有效期、退款规则？**

10. **日志和隐私怎么处理？**  
    是否保留 prompts、代码片段、文件内容，保留多久，能否关闭。

## 6. 对开发者最实用的购买建议

### 6.1 想稳定开发，优先选“透明计费”的中转

比起“便宜”两个字，更要优先看：

- 模型 ID 是否透明
- 是否写清输入 / 输出 / 缓存
- 是否给出余额查询和调用日志
- 是否说明协议兼容范围
- 是否支持故障切换和备用线路

### 6.2 “包月无限”最适合轻量聊天，不一定适合重度开发

重度开发场景常见这些情况：

- 长上下文
- 持续多轮会话
- Agent 自动循环
- 反复改文件
- 工具调用、搜索、日志追踪

这些都比普通聊天更烧 token。  
因此“无限”通常会带来：

- 更严格的限速
- 更高的排队
- 更强的高峰管控
- 更容易触发公平使用策略

如果你要长期跑 Codex、Claude Code、Gemini CLI，**按量透明 + 可查日志** 往往比“无限套餐”更适合开发。

### 6.3 预算有限时，先把任务分层

一个很实用的思路：

| 任务类型 | 优先模型 |
|----------|---------|
| 批量总结、改小 bug、提取信息 | Gemini 2.5 Flash-Lite / Flash、小型 GPT |
| 常规代码开发主力 | Claude Sonnet 4、GPT-5.4、Gemini 2.5 Pro |
| 超长仓库阅读、大上下文分析 | Claude Sonnet 4 长上下文、Gemini 2.5 Pro |
| 真正高价值难题 | 更强旗舰模型，再开高配 |

不要把所有任务都扔给最贵模型，否则无论官方还是中转，账单都会失控。

## 7. 如果你要配合 CC Switch 使用，中转站该怎么理解

CC Switch 的作用不是“替你省钱”，而是**把多个 Provider 的配置、切换、路由、用量观察统一起来**。

根据 CC Switch GitHub Releases 当前可见的最新发布信息，`2026-04-10` 的 `v3.13.0` 已强调了这些能力：

- Provider 管理
- 本地接管 / 路由
- 用量与成本追踪
- 配额与余额可视化
- OpenAI 兼容 `/v1/models` 自动拉模型列表
- 故障转移

所以对开发者来说，CC Switch 更像“中转管理中控台”，而不是“模型本身”。

你可以把它理解成：

```text
Codex / Claude / Gemini CLI
        ↓
     CC Switch
        ↓
    中转站 / 官方 API
        ↓
  OpenAI / Anthropic / Google
```

这也意味着：

- **CC Switch 能帮你切换 Provider**
- **CC Switch 能帮你观察余额 / 用量（若 Provider 支持）**
- **但 CC Switch 不能把一个不兼容的烂中转 magically 变成完全兼容的好接口**

如果上游协议不完整、模型映射错误、卖家偷切模型，CC Switch 也只能帮你更快发现问题，不能替它修正商业规则。

## 8. 给主人的最终判断标准

以后再看任何一个中转站，先别急着买，直接套这张判断表。

| 维度 | 合格表现 | 风险信号 |
|------|---------|---------|
| 模型透明度 | 给出真实模型 ID | 只写“官方 / 高级 / 稳定” |
| 协议兼容 | 明确 OpenAI / Anthropic / Gemini 哪一种 | 只写“全兼容” |
| 计费规则 | 输入、输出、缓存、长上下文写清楚 | 只有一个总价 |
| 风控说明 | 明确限速、套餐、公平使用 | 不写限制，只写“无限” |
| 可观测性 | 有余额、日志、失败记录 | 只能问客服 |
| 开发友好度 | 支持模型列表、稳定返回错误 | 报错含糊、经常超时 |

只要它在“真实模型 ID、计费拆分、限速、余额查询”这四项里有两项说不清，主人就尽量别拿它当开发主力。
