# 统一登录中心与子系统切换实战说明

> 适用场景：
>
> - 公司有一个统一登录页，页面里先输入账号密码
> - 登录页上可以选择不同的子系统
> - 登录成功后，当前标签页直接切换到所选子系统的站点，并且目标站点已经是登录态
> - 可能还要求“从 A 子系统跳到 B 子系统时，不再重新输账号密码”

这类需求本质上不是“两个网站互相登录”，而是 **统一登录中心给多个子系统签发同一份身份凭证**。  
更准确地说，它属于 **单点登录（SSO）** 或 **联合登录** 的范畴。

## 零、先记最短版

如果你现在只想先记一句最容易复述的话，可以先记这个：

**用户先在统一登录页登录，系统再发一个一次性的 `code` 给目标子系统，目标子系统拿这个 `code` 去换自己的登录态，所以看起来像“一次登录，多个系统都能进”。**

如果再压缩一点，就是：

**不是直接共享登录态，而是先发临时 `code`，再让目标系统自己换登录态。**

## 零点五、3 分钟回想卡片

如果你只剩 3 分钟复习时间，就回想下面 6 句话：

1. 统一登录页先负责把“人”登录成功。
2. 登录成功后，不是直接把登录态塞给别的系统。
3. 系统会先发一个一次性的 `code`。
4. 浏览器带着这个 `code` 跳到目标系统。
5. 目标系统再拿 `code` 去换它自己系统的 `token/session`。
6. 所以看起来像“一次登录，到处都能进”，但实际上每个系统最终还是用自己的登录态。

如果这 6 句话你能自己顺出来，说明主线已经记住了。

## 一、先把这件事说成人话

你现在看到的流程，通常是这样的：

1. 用户打开公司统一登录页。
2. 用户输入账号密码，或者走企业微信、钉钉、短信等其他登录方式。
3. 登录页让用户先选择要进入哪个子系统，比如“教务系统”“财务系统”“后台管理系统”。
4. 登录成功后，系统不是把账号密码直接发给另一个网站，而是生成一个 **一次性票据**。
5. 浏览器被重定向到目标子系统，目标子系统拿票据去统一登录中心验票。
6. 验票成功后，目标子系统自己建立登录态，比如写自己的 session 或 token。
7. 浏览器停留在目标子系统页面，用户看起来像是“直接进了另一个网站，而且已经登录好了”。

关键点只有一个：

**不是把一个网站的登录态直接搬到另一个网站，而是两个网站都信任同一个登录中心。**

---

## 一点五、再说得更白一点：它既能进自己家子系统，也能进公司外部系统

很多人会把这类功能只理解成“公司内部几个页面互相跳”。

其实你们现在这类登录页，通常同时支持两种目标：

### 1. 登录你们自己项目里的子系统

比如：

- 教务模块
- 学工模块
- 财务模块
- 招生模块

这些系统有时还是同一个前端项目里的不同菜单、不同路由，或者是同公司维护的多个站点。

这种情况下，登录成功后可能只是：

- 跳本项目内部路由
- 或跳公司自己另一个业务系统

### 2. 登录公司对接的第三方系统

比如：

- 外部办事大厅
- 合作厂商系统
- 第三方平台
- 另一个公司提供的业务站点

这种情况下，登录页本身并不会把密码给第三方系统，而是：

- 先在你们自己的统一登录中心登录
- 再由你们系统生成一个一次性 `code`
- 最后把浏览器带到第三方系统指定的回调地址

第三方系统收到这个 `code` 后，再去调用它自己的后端接口完成登录。

所以从用户角度看，表现都是一样的：

- 先选应用
- 再点登录
- 当前标签页直接进入目标系统

但从技术上看，背后可能有两种完全不同的跳法：

- 跳你们自己项目内部页面
- 跳公司外部第三方系统，并附带一次性授权码

这也是为什么这个功能不能只叫“菜单跳转”，它本质上是“**登录后按目标系统类型做不同分发**”。

## 一点七、它和企业微信、微信、QQ 官方登录其实很像

很多人第一次接触公司内部单点登录时，会觉得：

- 这和企业微信官方登录怎么这么像？
- 这和微信、QQ 第三方登录是不是差不多？

答案是：**对，骨架真的差不多。**

它们共同的核心流程都是：

1. 先去一个“认证中心”完成身份确认
2. 认证通过后拿到一个一次性的 `code`
3. 浏览器带着 `code` 回到目标系统
4. 目标系统后端再拿 `code` 去换用户身份
5. 最后目标系统生成自己系统的 `token/session`

真正的区别只在于：“认证中心是谁”。

### 1. 企业微信 / 微信 / QQ 登录

认证中心是官方平台：

- 企业微信官方
- 微信开放平台
- QQ 开放平台

也就是说：

- 你们要先去官方平台注册应用
- 拿 `corpId`、`agentId`、`clientId`、`secret` 这类信息
- 再按官方规则拼授权地址
- 最后由官方把 `code` 回调给你们

### 2. 公司内部统一登录

认证中心不是微信官方，而是你们自己公司的登录中心。

也就是说：

- 用户不是去企业微信官方登录
- 而是去你们自己的统一登录页登录
- 登录成功后，由你们后端签发一次性的 `code`
- 再把这个 `code` 带给目标子系统

所以你可以这样理解：

**企业微信官方登录，是“官方给我们发 `code`”；公司统一登录，是“我们自己的登录中心给子系统发 `code`”。**

换句话说，前者是“接第三方官方认证中心”，后者是“自己当认证中心”。

## 二、为什么不能直接“登录完就把 cookie 给另一个网站”

这是很多人第一次接触时最容易误解的地方。

### 1. 浏览器天然不允许跨域直接共享登录态

`A.com` 的 cookie，不能直接让 `B.com` 读取和使用。  
`localStorage`、`sessionStorage` 也一样，都是按域名隔离的。

所以你不能这样做：

- 在公司统一登录页登录成功
- 直接把 cookie 塞给另一个站点
- 让另一个站点“凭空变成登录状态”

浏览器的安全模型不会允许这种事。

### 2. 账号密码也不能直接转发给别的网站

更不能做的是：

- 统一登录页拿到密码
- 再把密码发给另一个网站

这样会有几个问题：

- 密码暴露面太大
- 目标站点会拿到不该知道的敏感信息
- 用户无法信任“到底谁保存了密码”
- 后续无法统一注销、统一风控、统一审计

所以正规做法一定是：

- 用户只和统一登录中心交互
- 统一登录中心发一次性凭证
- 子系统拿凭证换自己的登录态

## 三、标准架构怎么设计

这类系统通常分成三层：

### 1. 统一身份提供方

也叫 `IdP`，可以理解成“总登录中心”。

它负责：

- 校验账号密码
- 校验短信、企业微信、钉钉或其他二次认证
- 生成一次性 `code` 或 `ticket`
- 维护统一的用户身份

### 2. 各个子系统

也叫 `SP` 或业务系统。

它们负责：

- 提供自己的业务页面
- 接收来自统一登录中心的回跳
- 用一次性票据换取当前子系统自己的登录态
- 记录用户在本系统内的权限、角色、菜单

### 3. 浏览器

浏览器只是跳转通道。

它会在这些页面之间来回切换：

- 统一登录页
- 登录中心回调页
- 子系统回调页
- 子系统首页

## 三点五、把几个名词翻译成人话

这几个词第一次看很容易绕，直接翻译成人话最省事。

### 1. `IdP`

全称可以理解成“身份提供方”。

人话就是：

**谁负责证明‘你是谁’。**

在不同场景里，它可能是：

- 企业微信官方
- 微信官方
- QQ 官方
- 你们公司自己的统一登录中心

### 2. `SP`

可以理解成“业务系统”或“服务提供方”。

人话就是：

**真正要让你进去用的那个系统。**

比如：

- 教务系统
- 财务系统
- 公司外部对接平台
- 某个合作厂商系统

### 3. `code`

人话就是：

**一次性的临时通行证。**

它不是最终登录态，只是一个“你可以去换登录态”的凭证。

### 4. `token` / `session`

人话就是：

**目标系统真正发给你的正式门禁卡。**

你真正进入目标系统后，靠的是它，不是前面的 `code`。

## 四、真实流程长什么样

下面是最常见的一种流程。

### 方案 A：统一登录中心 + 授权码

```text
用户 → 统一登录页
      选择子系统 B
      输入账号密码
      登录成功
      ↓
统一登录中心生成一次性 code
      ↓
浏览器跳转到子系统 B 的回调地址
      ↓
子系统 B 后端拿 code 去统一登录中心换票
      ↓
子系统 B 创建自己的 session / token
      ↓
浏览器进入子系统 B 首页
```

这个方案最常见，也最稳。

### 方案 B：登录中心先登录，子系统再静默换票

有些系统会先让用户在登录中心登录一次，然后后续进入任何子系统都不用再次输入密码，只需要自动跳转并换票。

流程大概是：

```text
用户先登录统一登录中心
      ↓
统一登录中心在自己的域名下保存登录态
      ↓
用户点击子系统 A / B / C
      ↓
浏览器跳到子系统
      ↓
子系统再跳回统一登录中心换一次票据
      ↓
子系统建立自己的登录态
```

这个体验上很像“所有子系统都已经登录了”，但底层仍然是 **各自建立各自的会话**。

---

## 五、为什么看起来像“直接进了对方网站”

因为子系统做了两件事：

1. 它先拿到了统一登录中心签发的票据。
2. 它在自己的域名下立刻完成了登录态初始化。

所以用户体感上就是：

- 还在当前标签页
- 地址栏跳到了目标子系统
- 页面内容马上变成目标系统首页
- 没有再次输密码

这不是“跨站共享 cookie”，而是“**目标站点自己完成了登录**”。

---

## 五点五、如果你想用“钥匙”来理解，这个比喻可以这样讲

你前面提到“是不是像先拿一把钥匙”，这个比喻其实挺好，只要稍微修一下就更准确了。

可以这样理解：

- 用户在统一登录页登录成功
- 统一登录中心先发一个**临时钥匙**，这就是 `code`
- 浏览器拿着这把临时钥匙去目标系统门口
- 目标系统不会直接让你永久进入，而是先检查这把钥匙真不真
- 检查没问题后，目标系统再给你发它自己家的**正式门禁卡**
- 这张正式门禁卡就是它自己的 `token/session`

所以：

- `code` 不是最终登录态
- `code` 更像“临时钥匙”或“通行条”
- 真正长期用来访问系统的，是目标系统自己签发的 `token`

这个比喻对应到企业微信官方登录也一样：

- 企业微信官方发临时钥匙 `code`
- 你们后端拿 `code` 去官方换身份
- 你们自己系统再发正式登录态

对应到公司内部统一登录也一样：

- 统一登录中心发临时钥匙 `code`
- 子系统后端拿 `code` 去验票
- 子系统自己再发正式登录态

## 六、这类系统最推荐的协议

### 1. 首选：OAuth2 + OpenID Connect

如果是现代 Web 系统，最推荐的是：

- OAuth2 负责授权流程
- OpenID Connect 负责“我是谁”的身份层

它的优点是：

- 生态成熟
- 前后端分工清晰
- 支持多子系统接入
- 很适合做统一登录中心

### 2. 传统企业系统：SAML 或 CAS

如果公司已经有老系统，常见还会看到：

- `SAML`
- `CAS`

它们也能做单点登录，只是协议风格更偏企业内网和传统平台。

### 3. 自研票据系统

有些公司会自己做一套：

- 登录中心
- 一次性 ticket
- 签名参数
- 回调换 token

这也能跑，但本质上仍然是在模仿同样的登录交换模型。

## 七、前后端应该怎么配合

### 1. 前端负责什么

前端一般只做这些事：

- 把用户带到统一登录页
- 帮用户选择子系统
- 登录成功后把用户带到回跳地址
- 处理回调参数，比如 `code`、`state`、`ticket`

### 2. 后端负责什么

后端一般负责：

- 校验账号密码
- 生成一次性票据
- 票据换登录态
- 根据子系统返回不同权限、菜单、首页
- 处理统一登出

### 3. 子系统负责什么

子系统负责：

- 接收回调参数
- 调统一登录中心验票
- 写自己的 session / token
- 按自己的权限模型渲染页面

## 八、接口设计最常见的样子

你可以把它想成下面这组接口。

### 1. 统一登录接口

```http
POST /auth/login
```

入参通常包括：

- `username`
- `password`
- `systemId` 或 `targetSystem`
- `rememberMe`

返回通常不是直接给业务系统 token，而是给：

- `ticket`
- `redirectUrl`

### 2. 子系统回调地址

```http
GET /callback?ticket=xxxx&state=yyyy
```

子系统拿到后会：

- 校验 `state`
- 向统一登录中心验 `ticket`
- 获取用户身份和权限
- 在自己域名下落登录态

### 3. 票据校验接口

```http
POST /auth/exchange
```

子系统后端拿 `ticket` 去换：

- `userId`
- `username`
- `roles`
- `permissions`
- `sessionToken`

## 九、安全上一定要注意什么

### 1. `ticket` 必须一次性、短时效

这是最重要的一条。

如果票据可以重复使用，那就很容易被截获重放。

所以一般要求：

- 只能用一次
- 几分钟内过期
- 用过就立即失效

### 2. 必须校验 `state`

`state` 的作用是防 CSRF 和防回跳被篡改。

如果没有 `state`，攻击者可能构造一个恶意回跳链接，把用户带到错误站点或者恶意站点。

### 3. 回跳地址必须白名单

统一登录中心不能让用户随便跳任何地址。

要提前配置：

- 允许哪些子系统域名
- 允许哪些回调路径
- 允许哪些环境

否则会变成开放重定向漏洞。

### 4. 子系统自己的 session 不能直接依赖别的域名 cookie

每个系统都应该有自己的登录态。

统一登录中心只负责“证明你是谁”，不负责替你长期持有每个业务系统的会话。

## 十、如果你要做的是“登录页选子系统”

这类页面最标准的 UX 是：

1. 用户输入账号密码。
2. 页面选择目标子系统。
3. 登录按钮提交到统一登录中心。
4. 统一登录中心确认登录成功后，返回一个 `redirectUrl`。
5. 浏览器跳到该 `redirectUrl`。
6. 子系统拿 `ticket` 换自己的登录态。
7. 最终停留在目标子系统首页。

如果用户想在同一个登录页里切换不同子系统，常见做法是：

- 子系统列表从后端配置读取
- 不同子系统对应不同 `systemCode`
- 统一登录中心根据 `systemCode` 决定回跳地址和权限范围

## 十点二、第一次做这类功能时，最容易绕的其实就三句话

如果你已经大致明白，但脑子里还是有点拧巴，通常卡在下面三句话没彻底分清：

### 1. 登录页不是直接把用户“塞进”子系统

而是：

**登录页先把用户认证通过，再决定把用户送去哪个系统。**

### 2. `code` 不是最终 token

而是：

**`code` 只是一个拿来换 token 的一次性凭证。**

### 3. 最终登录态一定属于目标系统自己

也就是说：

**真正让用户保持登录状态的，一定是目标系统自己签发的 token/session，而不是来源系统的 cookie。**

这三句话一通，这个问题基本就通了。

---

## 十点五、结合 `wisdomCampusClient` 现有代码，这套逻辑在项目里是怎么落地的

下面这段不是概念，而是你们现有项目里已经落地的真实思路。

### 1. 登录页先让用户选“应用”

在 `wisdomCampusClient` 里，登录表单组件 `passwordForm2.vue` 顶部就有一个“请选择应用”的下拉框。

这说明登录页不是只负责“账号密码”，它还承担了一个动作：

**先决定这次登录完成后，要把用户送去哪个目标系统。**

目标系统既可能是：

- 本项目内部子系统
- 公司自己另一个站点
- 公司对接的第三方系统

### 2. 登录成功后，会根据所选应用的 `path` 判断怎么跳

现有代码会先找到当前选中的应用配置，然后判断它属于哪一类：

- 如果是内部路由，就直接 `router.replace(...)`
- 如果是普通外链，就新窗口打开
- 如果是“单点登录中转链接”，就走授权码流程

这里最关键的是第三种。

代码里已经专门写了识别逻辑：

- 链接里包含 `/login`
- 同时包含 `secretID`
- 同时包含 `redirectUrl`

一旦命中这个特征，就不把它当普通链接，而是当成“**要去别的系统登录**”。

### 3. 当前系统不会把密码给对方系统，而是先去换一个授权码

现有前端不是这样做：

- 用户登录成功
- 直接把用户名密码拼到第三方地址上

而是这样做：

1. 当前系统先调用 `thirdpartyapp.authorize`
2. 传 `secretID + redirect_uri`
3. 后端返回一个一次性 `Authorize_code`
4. 浏览器再跳到目标系统的 `redirectUrl?code=xxx`

这一步就已经非常接近标准的“授权码模式”了。

### 4. 目标系统收到 `code` 后，再建立它自己的登录态

目标系统并不是拿到 `code` 就直接显示页面，而是：

1. 读取 URL 里的 `code`
2. 读取可能一起带过来的 `secretID`
3. 把这些参数发给它自己的后端回调接口
4. 后端校验成功后返回本系统的 `TOKEN`、用户信息、菜单权限
5. 前端把这些数据写到当前系统自己的 Cookie / Storage
6. 再跳首页或业务页

这一步特别重要，因为它说明：

**目标系统最终用的还是“它自己域名下的登录态”，不是来源系统的 cookie。**

### 5. 用人话总结这套真实落地

你们当前项目里，“登录页选应用”的本质不是简单的页面导航，而是：

- 如果目标是自己家系统里的功能页，就走内部路由
- 如果目标是普通外部地址，就当外链打开
- 如果目标是需要单点登录的外部系统，就先换授权码，再带着 `code` 跳过去

所以更准确的人话应该是：

**这个登录页既能把人送进我们自己项目里的某些子系统，也能把人送进公司对接的第三方系统；如果对方系统支持单点登录，就不是普通跳转，而是先拿一次性授权码再跳过去。**

## 十一、什么时候别自己造轮子

如果你的公司有这些需求，优先考虑标准协议而不是自研：

- 多个业务系统共享一套账号体系
- 以后还要接钉钉、企业微信、LDAP、AD、短信登录
- 需要统一注销
- 需要审计和风控
- 需要给多个前端项目共用登录态

这时候最稳的是：

- 统一登录中心
- OAuth2 / OpenID Connect
- 子系统各自消费授权码

## 十二、一句话总结

你描述的“登录页选子系统，登录成功后当前标签页直接变成对方系统”的本质是：

**用户只登录一次统一登录中心，系统通过一次性票据把身份安全地交给目标子系统，目标子系统再在自己的域名下建立登录态。**

不是把一个网站的 cookie 直接搬到另一个网站，而是 **通过受控的授权交换完成单点登录**。

如果结合你们当前项目的真实做法，再翻译成人话就是：

**一个登录页，既能负责把用户送进自己家的子系统，也能负责把用户送进公司对接的第三方系统；区别只在于，前者可能是普通路由跳转，后者通常要先换一次性 `code`，再跳到对方系统完成登录。**

如果你以后要口头解释给同事听，还可以直接用下面这段：

**可以把它理解成统一登录页先把人认证通过，然后按目标系统发一张一次性的临时通行证 `code`。如果目标是自己家子系统，可能直接跳内部页面；如果目标是第三方系统，就带着 `code` 跳过去。目标系统再拿这个 `code` 去换它自己系统的登录态，所以用户看起来像“一次登录，到处都能进”。**

如果后面要落地实施，优先按这三个关键词去设计：

- 统一登录中心
- 一次性票据
- 子系统回跳换票

## 复习时最该记住的 4 个关键词

如果过几天你只记得 4 个词，也够把整篇文章重新串起来：

- 统一登录中心
- 一次性 `code`
- 回调地址
- 目标系统自己换登录态

把这 4 个词连成一句话就是：

**统一登录中心登录成功后，签发一次性 `code`，浏览器带着它回到目标系统的回调地址，目标系统再自己换登录态。**
