# Vue后台标签栏TagsView中$route、fullPath、href与监听路由的理解

## 先说结论

这次真正让我打通的，不是某一家公司的标签栏写法，而是下面这组以前零散、这次第一次在真实项目里串起来的知识点：

- 普通标签通常不是后端菜单直接渲染出来的，而是监听 `$route` 后，根据当前访问的路由动态加入的
- 首页标签常常是特殊处理，因为它希望默认就存在，所以很多项目会从 `menu` 里手工补进去
- `fullPath` 往往才是标签栏真正的唯一标识，不是单纯的 `path`
- `href` 更偏向“浏览器可直接打开的链接”，常用于“在新窗口中打开”
- 标签栏本质上不是菜单栏复制品，而是“访问过的页面状态”可视化

所以这篇笔记重点不是讲某个项目的业务细节，而是整理出阅读 `TagsView` 源码时最容易第一次碰到、也最容易混乱的几个 Vue Router 运行时知识点。

## 我这次真正困惑的点是什么

不是源码看不懂。

相反，恰恰是因为源码看懂了，我才第一次意识到，原来后台标签栏还会这样做：

- 可以直接监听 `$route`
- 路由对象里不只有 `path`，还有 `fullPath`、`name`、`meta`、`query`、有时还有 `href`
- 标签不是菜单原始数据本身，而是“路由运行时对象”和“少量菜单补丁数据”的混合结果

以前自己写小页面时，接触最多的是：

```js
this.$router.push('/user/list')
this.$route.path
this.$route.query
```

但看到后台标签栏源码时，才第一次真正碰到这一层设计：

- 路由变化会驱动标签栏变化
- 标签关闭后要跳回哪个页面，要看当前保存下来的页面状态
- 同一个页面带不同参数时，标签要不要算同一个，不能只看 `path`
- 首页为什么经常单独补一套逻辑

也就是说，这次不是“不会实现”，而是第一次系统理解这些原本分散的知识点。

## 标签栏不是菜单栏复制品

这是最该先记住的一句。

很多人第一次看后台标签栏，容易默认以为：

“左侧菜单里有什么，顶部标签栏就显示什么。”

但真实项目里，标签栏往往不是这样工作的。

标签栏更像：

- 记录你访问过哪些页面
- 展示当前打开了哪些页面状态
- 允许你在这些状态之间来回切换

所以它更接近“访问历史状态管理”，而不是“菜单结构渲染”。

于是就会出现一个关键区别：

- `menu` 负责提供入口和层级
- `$route` 负责表达当前页面状态
- `tags` 负责把访问过的状态保存并展示出来

## 为什么要监听 `$route`

因为普通标签通常就是随着页面跳转动态生成的。

典型逻辑大致就是这样：

```js
watch: {
  $route(to) {
    this.addViewTags(to)
  }
}
```

这段代码的人话翻译就是：

“只要当前路由变了，就把新的页面状态加入标签栏。”

这里要分清两个对象：

- `$router`：负责跳转，是“操作工具”
- `$route`：负责被读取和监听，是“当前页面状态结果”

所以：

```js
this.$router.push('/user/list')
```

是在操作路由。

而：

```js
this.$route.fullPath
this.$route.meta
this.$route.query
```

是在读取当前页面已经解析出来的结果。

这也是我这次补到的第一个关键知识点：

**原来路由不只是拿来跳页面，还可以作为界面状态源被监听。**

## `$route` 驱动普通标签，`menu` 补齐首页标签

这句可以直接当成阅读后台 `TagsView` 源码的主线。

### 1. 普通标签怎么来

普通标签一般来自当前访问的运行时路由对象，也就是 `$route`。

流程通常是：

1. 用户进入某个页面
2. 当前路由变化
3. 监听 `$route`
4. 执行 `addViewTags($route)`
5. 当前页面进入标签栏

所以普通标签的根数据源，不是菜单接口，而是当前实际访问过的路由对象。

### 2. 首页标签为什么常常特殊处理

因为很多后台项目都希望首页标签默认存在，而且常驻不关闭。

但首页未必总是靠一次标准跳转自然生成，所以经常会在初始化时：

- 先从菜单树里找到首页节点
- 手工给它补一些标签栏需要的字段
- 再把它推入标签栏列表

于是标签列表里就可能同时存在两类对象：

- 一类更像运行时 `$route`
- 一类更像菜单节点改造后的对象

这就带来一个非常重要的事实：

**标签对象的数据来源并不完全统一。**

后面很多兼容逻辑，都是从这里开始的。

## 为什么标签栏不能只看 `path`

因为后台页面经常会有参数状态。

比如同一个列表页，可能有这些不同访问状态：

```text
/user/list?page=1
/user/list?page=2
/user/list?page=1&keyword=admin
```

如果只看 `path`，它们可能都是：

```text
/user/list
```

那标签栏就分不清：

- 这是同一个标签
- 还是同一个页面下的不同查询状态

所以标签栏通常不会只看 `path`，而是更偏向看 `fullPath`。

## `fullPath` 为什么常作为标签唯一标识

因为 `fullPath` 比 `path` 更完整。

可以这样理解：

- `path`：基础路径
- `fullPath`：路径 + 查询参数 + 其他完整路由信息

例如：

```js
$route.path
// /user/list

$route.fullPath
// /user/list?page=1&keyword=admin
```

于是很多标签栏逻辑都会围绕 `fullPath`：

- 判断当前激活标签是不是自己
- 判断一个标签是否已存在
- 关闭标签时定位当前项
- 关闭后决定跳回哪个标签
- 刷新某个标签对应的具体页面状态

所以可以直接记一句：

**标签页认状态，优先看 `fullPath`。**

这也是我这次第一次真正理解的地方。以前知道有这个字段，但不知道为什么它在后台项目里这么重要。

## `name`、`meta`、`query` 在标签栏里通常怎么配合

虽然这次最核心的是 `fullPath`，但只记它一个也不够，因为真实项目里标签栏一般会同时依赖多个字段。

### `name`

`name` 很常用来做：

- 命名路由跳转
- `keep-alive` 缓存标识

所以经常能看到项目里把 `route.name` 存进缓存列表。

### `meta`

`meta` 经常承载业务配置，例如：

- `meta.title`：标签显示标题
- `meta.affix`：标签是否固定、是否允许关闭
- `meta.fullpage`：某些页面是否不加入标签栏

所以 `meta` 更像“给路由挂上的页面规则”。

### `query`

`query` 代表问号参数，例如：

```js
$route.query
// { page: '1', keyword: 'admin' }
```

刷新标签、切回标签时，如果想保留分页、搜索条件、筛选条件，就往往不能丢掉它。

## `href` 到底是什么，为什么它不是每个标签都有

这是我这次第二个很有感觉的点。

很多后台项目会出现这种代码：

```js
var url = nowTag.href || '/'
window.open(url)
```

这段代码本身就在告诉我们两件事：

### 第一件事：`href` 不是所有标签对象都有

如果每个标签对象都稳定存在 `href`，就没必要写兜底。

之所以要：

```js
nowTag.href || '/'
```

往往正是因为：

- 普通标签更像运行时路由对象，字段可能更完整
- 首页标签可能是菜单对象手工补出来的，只补了 `fullPath`，没有补 `href`

### 第二件事：`href` 更偏浏览器可直接打开地址

`fullPath` 更偏“路由内部完整状态标识”。

而 `href` 更偏：

- 当前路由模式下可以直接打开的地址
- 更适合拿给 `window.open()`

尤其在 `hash` 模式下，这种区别会更明显。

所以可以这样记：

- `fullPath`：内部状态识别用
- `href`：浏览器直开链接用

## 为什么这里会出现很多“兜底”和“兼容”

因为标签对象不是单一来源。

只要一个系统里同时存在：

- 运行时 `$route` 生成的普通标签
- 菜单节点改造出来的首页标签

那字段不完全一致几乎就是必然的。

于是就很容易看到下面这些处理：

- 没有 `href` 就回首页
- 首页单独补 `fullPath`
- 固定标签不能关闭
- 关闭当前标签时跳到左侧标签或默认页

这些写法看起来像“作者习惯不同”，但往深一点看，本质是：

**标签栏正在兼容多种来源的标签数据结构。**

## 这次最值得记住的几组概念

### 1. 菜单和标签不是一回事

- `menu` 是导航配置
- `tags` 是访问状态记录

### 2. `$router` 和 `$route` 不是一回事

- `$router` 用来跳
- `$route` 用来读、用来监听

### 3. `path` 和 `fullPath` 不是一回事

- `path` 只够表示页面路径
- `fullPath` 才足够表示具体页面状态

### 4. `fullPath` 和 `href` 也不是一回事

- `fullPath` 更偏路由内部标识
- `href` 更偏浏览器直接打开链接

## 如果把后台标签栏逻辑压成一句话

可以记成下面这句：

**普通标签由 `$route` 驱动生成，首页标签常由 `menu` 手工补齐，`fullPath` 用来识别和切换具体页面状态，`href` 只在新窗口打开时优先尝试使用，并做好兼容兜底。**

这句话一通，很多源码细节都会跟着通：

- 为什么监听 `$route`
- 为什么首页处理经常不同
- 为什么 `fullPath` 很关键
- 为什么 `href` 不一定存在
- 为什么标签栏逻辑总会有一层兼容代码

## 我这次真正补到的不是某种写法，而是一层认识

不同公司实现标签栏的细节当然会不一样。

但只要是 Vue 后台项目，很多实现方式的底层思路其实都差不多：

- 用菜单决定入口
- 用路由对象承载当前页面状态
- 用标签栏记录访问过的页面
- 用缓存机制保留部分页面实例

所以我这次真正补到的，不是“某家公司标签栏怎么写”，而是：

**原来 Vue Router 的运行时路由对象，真的可以驱动一整套后台界面状态。**

这一步理解打通以后，后面再看这些模块会轻松很多：

- 标签栏 `TagsView`
- 面包屑 `Breadcrumb`
- 动态菜单高亮
- `keep-alive` 页面缓存
- 权限路由
- 标签关闭后的跳转逻辑

## 最后给自己留一版极简复习卡片

以后如果忘了，先回想这几句：

- 标签栏不是菜单复制品，而是访问状态管理
- 普通标签通常监听 `$route` 后动态加入
- 首页标签经常从 `menu` 手工补进去
- 标签唯一标识优先看 `fullPath`
- `name` 常拿去做缓存标识
- `meta` 常拿去做标题和固定标签等规则
- `href` 更适合新窗口打开，没有时要兜底

只要这几句还记得，再回头看 `TagsView` 源码，主线基本就不会乱。
