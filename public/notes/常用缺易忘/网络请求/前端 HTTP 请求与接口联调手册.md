# 前端 HTTP 请求与接口联调手册

> 这篇手册不再按单个术语拆，而按前端真正的工作流来收口：
>
> - 写请求时看什么
> - 打开 Network 面板时先看什么
> - 状态码和缓存该怎么解读
> - 下载、流式输出、实时通信和重试怎么选

## 一、先建立前端看 HTTP 的心智图

| 你现在在问什么 | 优先看哪一组概念 |
| --- | --- |
| 这次请求怎么发 | 方法、URL、`Content-Type`、`Accept`、`Authorization` |
| 为什么返回成功但页面没数据 | `202`、`204`、`206`、响应体解析方式 |
| 为什么跳转后方法变了 | `302`、`303`、`307`、`308` |
| 为什么请求被拦或被重定向登录 | `401`、`403`、`404`、`409`、`422` |
| 为什么缓存和浏览器表现对不上 | `Cache-Control`、`ETag`、`Last-Modified`、Memory/Disk/Service Worker Cache |
| 为什么下载和流式返回行为不一样 | `Range`、`Content-Range`、`Transfer-Encoding: chunked` |
| 为什么“实时”方案选不明白 | 轮询、长轮询、`SSE`、`WebSocket` |
| 为什么重试越写越乱 | 超时、重试、幂等、`AbortController` |

前端最容易踩的坑，不是某个术语不认识，而是把本来应该一起看的概念拆散了。

## 二、写请求前先把这些高频头看明白

### 1. 最常一起混的请求头与响应头

| 名词 | 它真正决定什么 | 前端最常误解 |
| --- | --- | --- |
| `Content-Type` | 请求体或响应体是什么格式 | 以为它表示“我想要什么”，其实它表示“我现在发的是什么”或“服务器回的是什么” |
| `Accept` | 客户端希望服务端优先返回什么格式 | 以为它会强制服务端按这个格式返回，实际上只是协商意愿 |
| `Content-Disposition` | 浏览器把响应当内联内容还是附件处理 | 只在下载时想到它，忽略了文件名通常也靠它控制 |
| `Accept-Encoding` | 客户端能接受哪些压缩方式 | 不是内容格式协商，它关心的是传输压缩 |
| `Content-Encoding` | 服务端最终用了哪种压缩方式 | 不是文件原始类型，比如 `gzip` 不等于 `application/json` |
| `Connection` / `Keep-Alive` | 连接是否复用 | 现代浏览器大多自动处理，前端通常不需要手写 |
| `Transfer-Encoding: chunked` | 服务端按分块持续返回，长度可事先未知 | 不等于断点下载，它更接近“边生成边发” |
| `User-Agent` | 客户端身份描述 | 不是安全凭证，只能做兼容判断和日志分析，别拿它做强校验 |

### 2. 一句记住最常见的 3 组区别

- `Content-Type` 看“我发的是什么”
- `Accept` 看“我想收什么”
- `Content-Disposition` 看“浏览器怎么处理这份响应”

- `Accept-Encoding` 看“我能解哪些压缩”
- `Content-Encoding` 看“服务器到底压成了什么”

- `Content-Length` 看“总长度已知”
- `Transfer-Encoding: chunked` 看“边生成边传，长度不一定先知道”

### 3. 前端联调里最实用的提醒

- 用 `FormData` 上传时，不要手动写 `Content-Type: multipart/form-data`，浏览器会把 `boundary` 一起补上
- 返回 `204` 时不要再无脑 `response.json()`
- 下载文件名要优先和后端确认 `Content-Disposition`
- 流式输出、日志流、AI 输出经常会和 `chunked` 或 `SSE` 一起出现

## 三、状态码不要一篇一篇背，按场景记

### 1. 成功了，但成功方式不同

| 状态码 | 更像在说什么 | 前端常见场景 | 前端动作 |
| --- | --- | --- | --- |
| `202` | 我先收到了，但还没处理完 | 导出、转码、异步任务 | 提示“任务已受理”，然后轮询或订阅任务状态 |
| `204` | 成功了，但没有正文 | 删除、开关切换、无需回详情的更新 | 只看状态，不要再解析 body |
| `206` | 成功了，但我只回了一部分 | 断点下载、范围请求、媒体拖动预览 | 配合 `Range`、`Content-Range` 一起看 |

最容易错的点只有两个：

- `202` 不是最终完成
- `204` 不是“空对象”，而是“没有响应体”

### 2. 重定向先看方法会不会被改掉

| 状态码 | 语义 | 方法和请求体是否保留 | 前端怎么记 |
| --- | --- | --- | --- |
| `302` | 临时重定向 | 历史实现里经常被改成 `GET` | 常见，但别拿它表达“必须保留原方法” |
| `303` | 去另一个地址看结果 | 会改成 `GET` | 提交完跳结果页很常见 |
| `307` | 临时重定向 | 应保留原方法和请求体 | 临时搬家，但原请求别变 |
| `308` | 永久重定向 | 应保留原方法和请求体 | 永久搬家，但原请求别变 |

前端要真正记住的是：

- 提交后跳结果页，优先理解成 `303`
- 接口迁移又要求保留 `POST`、`PUT` 语义，优先看 `307`、`308`

### 3. 权限、资源、状态冲突不要混成“反正失败了”

| 状态码 | 核心含义 | 前端第一反应 |
| --- | --- | --- |
| `401` | 还没认证或凭证失效 | 清理登录态，触发刷新或跳登录 |
| `403` | 已认证，但没权限 | 做权限提示，不要强行重试 |
| `404` | 路径或资源不存在 | 先排查 URL、参数、环境地址 |
| `409` | 当前资源状态和操作冲突 | 看并发修改、重复提交、版本冲突 |
| `422` | 语法能读懂，但语义过不去 | 提示表单校验或业务规则失败 |

这组状态码最适合前端做成统一错误分层：

- `401` 归到登录态问题
- `403` 归到权限问题
- `404` 归到路由或资源问题
- `409` 归到业务状态竞争
- `422` 归到表单或业务校验

### 4. `429` 和 `503` 都可能要等，但原因不一样

| 状态码 | 更像是谁扛不住了 | 常见原因 | 前端处理 |
| --- | --- | --- | --- |
| `429` | 你这边打得太快了 | 频控、接口限流、短信发送过密 | 降频、退避、参考 `Retry-After` |
| `503` | 服务端当前不可用 | 服务维护、熔断、临时过载 | 展示服务不可用，必要时稍后重试 |

`Retry-After` 经常和它们一起出现：

- 秒数形式：告诉你多少秒后再试
- 日期形式：告诉你某个具体时间点后再试

前端别犯两个错：

- 看到 `429`、`503` 就立即无脑重试
- 完全忽略 `Retry-After`

## 四、缓存、下载和浏览器表现要放在一起看

### 1. `Cache-Control` 先记这些最常见取值

| 指令 | 最省事理解 | 前端场景 |
| --- | --- | --- |
| `no-store` | 完全别存 | 支付结果、隐私页面、敏感接口 |
| `no-cache` | 可以存，但用前要回源确认 | 需要协商缓存的 HTML 或接口 |
| `max-age=60` | 60 秒内直接用缓存 | 静态资源、变化不频繁的数据 |
| `private` | 只允许私有缓存保存 | 用户专属内容 |
| `public` | 共享缓存也能存 | 公共静态资源 |
| `immutable` | 一旦缓存命中，不用反复再问 | 带 hash 的静态资源 |

一句话理解：

- `no-store` 是不留
- `no-cache` 是留着，但每次问一句能不能继续用

### 2. `ETag` 和 `Last-Modified` 怎么配

| 方案 | 校验依据 | 优点 | 局限 |
| --- | --- | --- | --- |
| `ETag` / `If-None-Match` | 资源指纹 | 更精确 | 后端要生成和比对 |
| `Last-Modified` / `If-Modified-Since` | 最后修改时间 | 简单直观 | 秒级精度、部分场景不够精确 |

实战里常见的搭配是：

- 静态资源强缓存看 `Cache-Control`
- 需要协商时同时给 `ETag` 或 `Last-Modified`

### 3. 浏览器缓存分层不要只盯着 HTTP 头

| 层级 | 谁在管 | 你会看到什么现象 |
| --- | --- | --- |
| Memory Cache | 当前进程内存 | 同页或短时间内再次请求极快 |
| Disk Cache | 浏览器磁盘缓存 | 刷新后仍可能直接命中 |
| HTTP Cache | 协议层缓存策略 | 受 `Cache-Control`、协商缓存头影响 |
| Service Worker Cache | 业务脚本接管的缓存 | 结果可能和 Network 面板直觉不一致 |

前端排缓存问题时，顺序建议是：

1. 先看响应头
2. 再看请求是不是被 Service Worker 接管
3. 最后再看浏览器的 Memory / Disk 命中

### 4. `Range`、`206` 和断点下载是一组

前端只要看到下面这组组合，就要想到“不是整包下载”：

- 请求头：`Range: bytes=0-1023`
- 响应码：`206 Partial Content`
- 响应头：`Content-Range`

它常见于：

- 大文件断点下载
- 音视频拖动播放
- 预览文件局部内容

## 五、请求治理要一起看：取消、超时、重试、幂等

### 1. `AbortController` 是前端取消请求的标准做法

它最适合这几类场景：

- 路由切换后取消旧请求
- 搜索框连续输入，只保留最后一次
- 给 `fetch` 做超时控制

```js
async function fetchWithTimeout(url, options = {}, timeout = 8000) {
  const controller = new AbortController()
  const timer = setTimeout(() => controller.abort(), timeout)

  try {
    const response = await fetch(url, {
      ...options,
      signal: controller.signal
    })

    return response
  } finally {
    clearTimeout(timer)
  }
}
```

### 2. 超时、重试、幂等别写成一锅粥

| 请求类型 | 要不要超时 | 要不要自动重试 | 幂等要求 |
| --- | --- | --- | --- |
| `GET` / 查询类 | 要 | 可以，1 到 3 次 | 天然更接近幂等 |
| `PUT` / `DELETE` | 要 | 可以谨慎重试 | 后端通常应保证幂等 |
| 普通 `POST` 创建 | 要 | 非常谨慎 | 最好有幂等 key |
| 支付、发短信、扣库存 | 要 | 默认不要无脑重试 | 必须先对齐幂等方案 |

最容易把事情做坏的写法是：

- 超时了就立刻重试
- 所有 `POST` 都跟 `GET` 一样处理
- 不分错误类型地统一重试

### 3. 真要重试，至少做到这 3 件事

- 只对瞬时失败重试，比如超时、偶发网关抖动
- 用指数退避加一点抖动
- 对有副作用的请求先确认幂等标识，比如 `Idempotency-Key`

## 六、实时通信别默认一上来就 `WebSocket`

### 1. 先看 4 种方案的落点

| 方案 | 通信方向 | 实时性 | 成本 | 适合什么 |
| --- | --- | --- | --- | --- |
| 轮询 | 客户端反复发请求 | 低到中 | 简单但空请求多 | 任务状态、低频查询 |
| 长轮询 | 客户端发起，服务端等到有消息再回 | 中 | 比轮询省一些空跑 | 简单消息等待 |
| `SSE` | 服务端单向持续推送 | 中到高 | 比 WebSocket 更轻 | 通知流、日志流、AI 输出流 |
| `WebSocket` | 双向长连接 | 高 | 接入与维护成本最高 | 聊天、协作、双向实时互动 |

### 2. `SSE` 和 `WebSocket` 怎么选

优先选 `SSE` 的场景：

- 服务端单向推消息就够了
- 你想用更接近 HTTP 的方式接入流式输出
- 你不想一开始就处理双向协议、心跳、复杂重连

优先选 `WebSocket` 的场景：

- 客户端也要高频向服务端发消息
- 要做聊天室、协同编辑、在线对战这类双向互动

### 3. 一个很常见的误区

AI 输出流、日志流、任务进度流，不一定需要 `WebSocket`。

很多时候下面这个组合就够了：

- `SSE`
- 或普通 HTTP 流式响应
- 再配合 `Transfer-Encoding: chunked`

## 七、前端联调时照着过一遍就够了

1. 请求方法和接口语义对了吗，`GET`、`POST`、`PUT`、`DELETE` 没混吗
2. `Content-Type`、`Accept`、认证头和请求体格式对齐了吗
3. 状态码分层处理了吗，`401`、`403`、`409`、`422` 没被混成一种报错吗
4. 返回 `204` 时有没有还在强行解析 JSON
5. 遇到缓存问题时，有没有把 `Cache-Control`、协商缓存和 Service Worker 一起排查
6. 下载或媒体预览场景里，有没有确认 `Range`、`206`、`Content-Range`
7. 自动重试是否区分了可重试请求和有副作用请求
8. 需求只是单向推送时，有没有先评估 `SSE`，而不是直接上 `WebSocket`

如果只记一句，就记：

`前端看 HTTP，不要按术语碎背，要按“发请求、看状态、查缓存、做重试、选实时方案”这条链路一起看。`
