# IndexedDB 什么时候该上

> 很多人第一次接触 IndexedDB，都会觉得它像“浏览器里的小数据库”，然后马上问：
>
> `那我以后是不是都该用它，不用 localStorage 了？`

答案通常是：

`不是。`

IndexedDB 很强，但它不是默认选项。真正该上的时候，它特别值；不该上的时候，它会让简单问题变复杂。

## 一、先给结论

### 继续用 localStorage / sessionStorage 就够的情况

- 存主题色、语言、布局偏好
- 存很小的草稿或筛选条件
- 临时页面状态
- 少量字符串配置

### 该认真考虑 IndexedDB 的情况

- 数据量明显变大
- 要离线可用
- 要存结构化对象列表
- 要做索引查询
- 要存文件、二进制、图片或较大的缓存

## 二、它和 localStorage 最大的区别是什么

最重要的不是“它更高级”，而是下面这几件事：

| 维度 | localStorage | IndexedDB |
| --- | --- | --- |
| 数据规模 | 小 | 大得多 |
| 是否同步 | 同步 | 异步 |
| 数据类型 | 主要是字符串 | 结构化对象、Blob 等 |
| 查询能力 | 几乎没有 | 支持对象仓库和索引 |
| 事务能力 | 没有 | 有 |
| 适合场景 | 小配置、小状态 | 离线数据、本地缓存、复杂列表 |

所以可以很直接地记：

- `localStorage` 更像“小抽屉”
- `IndexedDB` 更像“浏览器里的本地数据库”

## 三、哪些场景非常适合 IndexedDB

## 1. 离线优先

比如：

- 离线笔记
- 待提交工单
- 表单草稿队列
- 本地缓存后再次打开还能继续看

这类场景需要的数据往往不只是一个字符串，而是成批对象和同步状态。

## 2. 列表缓存比较大

比如：

- 搜索结果缓存
- 商品列表缓存
- 大量字典数据
- 历史消息索引

当你开始出现这些想法时：

- “我想按 id 查”
- “我想按时间排序”
- “我想按某个字段过滤”

这就已经有数据库味道了，IndexedDB 会比 localStorage 更合适。

## 3. 需要存二进制资源

比如：

- 图片草稿
- 音频片段
- 离线附件
- 大体积导出缓存

这类内容放 localStorage 通常很别扭，甚至根本不应该转成 Base64 去硬塞。

## 4. 需要更稳的本地任务队列

比如：

- 待同步记录
- 离线提交后重试
- 本地编辑记录

这时 IndexedDB 的事务和对象存储会更像样。

## 四、哪些场景其实没必要上

## 1. 只存几个用户设置

比如：

- 深色模式
- 字体大小
- 当前语言

这类就别把问题做复杂了，localStorage 足够。

## 2. 临时页面态

比如：

- 当前筛选条件
- 这次打开页面的临时输入
- 标签页关闭就不想留的数据

这类更适合 `sessionStorage`。

## 3. 只是保存一个很小的 token 或标识

这不是 IndexedDB 的优势场。

尤其是认证信息，真正要优先考虑的是安全策略，不是“存哪种浏览器数据库显得更高级”。

## 五、什么时候会从“够用”变成“该上”

你可以观察几个信号。

### 1. 你开始频繁做 JSON.parse / JSON.stringify

而且存的不是单个对象，而是一大坨列表。

### 2. 你开始自己写“按字段查找、去重、分页、排序”

这说明你已经在手搓一个很弱的数据库层了。

### 3. 你担心 localStorage 同步阻塞

localStorage 是同步 API。数据一大、操作一多，就会更容易拖累主线程。

### 4. 你需要离线能力

一旦离线可用变成产品要求，IndexedDB 的价值就会明显上来。

## 六、IndexedDB 的代价也要看清

它不是零成本的。

### 1. API 更复杂

原生 IndexedDB 的 API 比较啰嗦，有：

- 打开数据库
- 版本升级
- 对象仓库
- 事务
- 索引

如果只是记一个主题色，真没必要把这些全搬出来。

### 2. 版本升级要设计

结构变更时，你要处理数据库升级逻辑。

### 3. 异步心智更重

它不是“我 set 一下、get 一下”那么轻。

### 4. 浏览器存储也不是无限且永久

IndexedDB 容量通常比 localStorage 大得多，但仍然受浏览器存储配额和清理策略影响。需要长期离线依赖时，最好关注：

- 配额
- 清理策略
- 是否需要持久化存储能力

## 七、工程里怎么用更舒服

如果项目真的要上 IndexedDB，通常不建议一上来手写大量原生 API。

更常见的做法是：

- 用 `idb`
- 用 `Dexie`

原因很简单：

- API 更清晰
- 事务和查询更顺手
- 可维护性更好

## 八、一个最朴素的判断公式

你可以直接这样判断：

### 继续用 localStorage

当你的需求是：

- 小
- 简单
- 配置型
- 不需要查询

### 上 IndexedDB

当你的需求是：

- 大
- 结构化
- 要索引
- 要离线
- 要缓存文件或对象列表

## 九、记忆版示例

### 不该上 IndexedDB 的例子

- “记住用户选的主题”
- “保存阅读字体大小”
- “缓存一个当前页的筛选条件”

### 该上 IndexedDB 的例子

- “离线保存 500 篇笔记”
- “缓存大量搜索结果并按时间查询”
- “本地保存图片草稿和待上传队列”
- “断网后继续操作，联网后再同步”

## 十、一句话收尾

IndexedDB 不是“更高级的 localStorage”，而是“当浏览器端真的开始像本地数据层时，你该上的数据库工具”。

如果只记一句，就记：

`小配置继续 localStorage，大缓存、离线和查询能力再上 IndexedDB。`
