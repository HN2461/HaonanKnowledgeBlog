# 前端开发常见面试题详解 (60问)
## 一、JavaScript 相关
### 1. JavaScript 作用域（Scope）
**概念/定义：** 作用域决定了变量和函数的可见性与生命周期。JavaScript 中主要有全局作用域、函数作用域、块级作用域等。其中全局作用域就是脚本的最外层作用域，函数作用域指函数内部，而 ES6 引入了块级作用域（如 `let`/`const` 在 `{}` 块中生效）。作用域遵循词法（静态）作用域规则：子作用域可以访问父作用域中的变量，而父作用域无法访问子作用域变量。  
**使用场景/原理：** 作用域用于变量存储和管理。例如，定义在函数内部的变量只能在该函数内部访问，避免与全局变量冲突；定义在块中的变量用 `let`/`const` 实现更精细的控制。JavaScript 在执行时，会根据作用域链查找变量，直到根作用域（全局）。  
**注意事项/误区：** 使用 `var` 声明的变量具有函数作用域而非块作用域，容易在循环或条件语句中产生意外行为。避免不必要的全局变量，尽量使用块级作用域和函数作用域控制变量范围。  
**示例代码：**

```javascript
function foo() {
  let x = 1; // 函数作用域
  if (true) {
    let y = 2; // 块级作用域，外部不能访问y
  }
  console.log(x); // 1
  // console.log(y); // ReferenceError
}
```

**开发建议：** 在实际开发中，应养成使用 `const`/`let` 代替 `var` 的习惯，以利用块级作用域带来的优势；尽量避免将变量挂载到全局对象 `window`，保持模块和函数内部的封闭。

### 2. `var`、`let`、`const` 区别
**概念/定义：** `var`、`let`、`const` 都用于声明变量，但行为不同。`var` 声明的变量具有函数作用域（或全局作用域）并且会被提升；而 `let` 和 `const` 是 ES6 引入的块级作用域声明，不会被提升到函数顶部。`const` 声明的是常量，声明后必须立即初始化且不可重新赋值。  
**使用场景/原理：** 在现代 JavaScript 开发中，推荐使用 `const` 声明不可变变量、`let` 声明可变变量，以利用块级作用域和暂时性死区（TDZ）机制提高代码可维护性。`var` 应尽量避免使用，以防止变量泄露到外层作用域或重复声明。  
**注意事项/误区：** `var` 声明的变量会被提升（hoisting），即变量声明会被移动到当前作用域顶部，导致在声明前使用变量不会报错（但值为 `undefined`）。同时，`var` 在相同作用域内允许重复声明，可能覆盖之前的变量。相反，`let`/`const` 在同一作用域中不可重复声明。`const` 并不意味着值不可变，仅指变量绑定不可更改；对于对象/数组类型变量，其内容仍可修改。  
**示例代码：**

```javascript
console.log(a); // undefined（变量提升）
var a = 5;

{
  let b = 3;
  const c = 4;
}
// console.log(b, c); // ReferenceError（b、c 均为块级作用域）

// const 示例
const PI = 3.14;
// PI = 3; // TypeError: Assignment to constant variable.
```

**开发建议：** 总结来说，日常开发应首选 `const`（默认值不可变），需要可变时用 `let`，尽量不用 `var`。这样可以避免作用域混淆、重复声明等问题，并通过 `const` 提高代码的可读性和安全性。

### 3. JavaScript 提升（Hoisting）
**概念/定义：** 提升指的是 JavaScript 在代码执行前，会将变量和函数的声明“上提”到当前作用域顶部的机制。实际上，在编译阶段，函数声明和 `var` 声明都会被先处理，从而可以在声明前使用。需要强调的是，“提升”只是编程术语描述的现象，并不是语言规范里的行为，规范中仅定义了哪些声明是可提升的。  
**使用场景/原理：** 由于提升，函数可以在其声明之前调用，而 `var` 声明的变量在访问时会返回 `undefined`（因为声明已被提升但赋值尚未执行）。例如：

```javascript
console.log(foo()); // "Hello"
function foo() { return "Hello"; }

console.log(x); // undefined（声明被提升，赋值还未执行）
var x = 10;
```

**注意事项/误区：** 只有函数声明和 `var` 声明会被“提升”。`let`、`const` 以及类声明不会被提升到块顶部；在声明之前访问会产生暂时性死区错误（ReferenceError）。滥用提升可能导致代码可读性差，容易引发错误，建议始终在使用前声明变量/函数。  
**示例代码：**

```javascript
function testHoisting() {
  console.log(a); // undefined (var a 提升了声明)
  var a = 5;
  console.log(a); // 5

  // let/const 没有提升：
  // console.log(b); // ReferenceError
  let b = 3;
}
```

**开发建议：** 尽量避免依赖提升带来的副作用，建议在使用变量或函数之前先声明，从而使代码更直观。ES6 模块本身也强制使用声明和导入顺序，减少了提升带来的混淆。

### 4. JavaScript 闭包（Closure）
**概念/定义：** 闭包是指函数和其形成时所引用的词法环境的组合。换句话说，当一个内部函数访问到外部函数的变量时，就形成了闭包，这使得该函数即使在外部函数执行结束后，依然可以访问外部作用域的变量。在 JavaScript 中，每次函数创建都会生成一个闭包。  
**使用场景/原理：** 闭包常用于数据封装、实现私有变量或函数。例如可以通过闭包生成私有计数器：

```javascript
function createCounter() {
  let count = 0;
  return function() {
    count++;
    console.log(count);
  };
}
const counter = createCounter();
counter(); // 1
counter(); // 2
```

此时 `count` 变量对外部不可见，但每次调用 `counter` 时都能保留和修改它。  
**注意事项/误区：** 使用闭包时要注意内存泄漏：持续的闭包引用可能导致变量无法被垃圾回收。另一个常见问题是循环中使用闭包：如果用 `var` 声明循环变量，闭包会捕获同一个变量，导致外部使用时值已变。ES6 的 `let`/`const` 提供块级作用域，可以解决这个问题。  
**示例代码：**

```javascript
// 常见闭包示例：私有变量
function makeCounter() {
  let n = 0;
  return {
    increment() { n++; },
    get() { return n; }
  };
}
const counterObj = makeCounter();
counterObj.increment();
console.log(counterObj.get()); // 1

// 注意：循环中闭包捕获
for (var i = 1; i <= 3; i++) {
  setTimeout(() => console.log(i), 100); 
  // 全部输出 4，因为 i 已变为 4 才执行
}
// 使用 let 解决
for (let j = 1; j <= 3; j++) {
  setTimeout(() => console.log(j), 100); 
  // 正确输出 1,2,3
}
```

**开发建议：** 在实际开发中，闭包非常有用，但也要谨慎使用。尽量避免在需要频繁销毁变量的场景中使用闭包，使用时注意作用域链和内存。循环变量尽量使用 `let` 代替 `var`，防止闭包捕获同一个变量。对于简单的数据缓存或私有化使用闭包较为方便，但若逻辑复杂可以考虑使用类或模块模式。

### 5. 箭头函数（Arrow Function）及 `this` 绑定
**概念/定义：** 箭头函数是 ES6 提供的函数定义语法：`(参数) => {}`。与普通函数不同，箭头函数没有自己的 `this`、`arguments`、`super` 或 `new.target` 绑定。它会创建一个包含其定义时上下文的词法 `this` 绑定，也就是说箭头函数内部的 `this` 与外层作用域一致。此外，箭头函数不能用作构造函数，不能使用 `new` 调用。  
**使用场景/原理：** 箭头函数常用于回调函数场景，例如数组遍历、事件处理等，它使代码更加简洁。由于它的 `this` 是词法绑定，适合在回调函数中仍需访问外部 `this` 的场景，比如在类的方法中使用箭头函数避免手动 `.bind(this)`。  
**注意事项/误区：** 不要在需要动态 `this` 或构造函数的场景下使用箭头函数。比如 Vue 的生命周期钩子或组件方法中不要使用箭头函数，否则 `this` 将不指向 Vue 实例。此外，箭头函数的 `arguments` 不可用，如果需要参数对象，应显式使用剩余参数。  
**示例代码：**

```javascript
const add = (a, b) => a + b;
console.log(add(3, 4)); // 7

// 箭头函数的 this
const obj = {
  value: 10,
  regularFunc: function() {
    console.log(this.value); // 10
  },
  arrowFunc: () => {
    console.log(this.value); // undefined（this指向外层上下文，非obj）
  }
};
obj.regularFunc();
obj.arrowFunc();
```

**开发建议：** 在日常开发中，可广泛使用箭头函数简化回调。对于类方法或 Vue 的方法/钩子，避免使用箭头函数定义，否则 `this` 指向会有误。需要注意的是箭头函数缺少自身 `this` 绑定，使得它们“自动绑定”到定义时的上下文，这对许多场景很便利，但也要留意副作用和调试难度。

### 6. `call()`、`apply()` 与 `bind()` 区别
**概念/定义：** 三者都是函数方法，用来设置调用时的 `this` 指向：`func.call(obj, arg1, arg2…)` 按参数顺序传递；`func.apply(obj, [argsArray])` 参数以数组形式传递；而 `func.bind(obj, …)` 则返回一个新的函数，永久绑定了 `this` 对象与预置参数。  
**使用场景/原理：** 可以在动态改变函数所属对象时使用。例如，一个通用函数需要在不同上下文中执行，就可以用 `call/apply`。`bind` 则常用于函数柯里化，或在事件处理中预先绑定 `this` 和参数。  
**注意事项/误区：** `call/apply` 会立即执行函数，区别仅在于参数格式；`bind` 不会立即执行，而是生成一个新函数。`bind` 生成的函数可多次调用，并且其返回值不可再用 `call/apply` 更改 `this` 绑定（新绑定不会生效）。  
**示例代码：**

```javascript
function greet(greeting, punctuation) {
  console.log(`${greeting}, ${this.name}${punctuation}`);
}
const user = { name: 'Alice' };
greet.call(user, 'Hello', '!');             // "Hello, Alice!"
greet.apply(user, ['Hi', '...']);           // "Hi, Alice..."
const boundGreet = greet.bind(user, 'Hey');
boundGreet('?');                            // "Hey, Alice?"
```

**开发建议：** 实际开发中，如果只需要立即调用函数并改变 `this`，用 `call` 或 `apply`；如果需要生成新函数并稍后调用，使用 `bind`。注意性能：`bind` 会产生新函数，多次调用绑定可能产生内存开销。一般场景下，使用 ES6 箭头函数和模块化设计可以减少对 `call/apply/bind` 的依赖。

### 7. Promise 与异步（Async/Await）
**概念/定义：** Promise 是 ES6 引入的用于管理异步操作的对象，代表一个可能当前还未完成的操作，其状态由“未完成(pending)”、“已成功(resolved)”、“已失败(rejected)”等组成。可以通过 `.then()`、`.catch()` 链式处理异步结果。`async/await` 是 ES2017 语法糖，可以让异步代码写得像同步代码，底层还是基于 Promise。  
**使用场景/原理：** 当有异步操作（如网络请求、定时器、文件读取等）时，应使用 Promise 来避免回调地狱。使用 `async` 函数配合 `await` 关键字，可在异步操作完成前“暂停”函数执行，等待结果后再继续，语义更清晰。  
**注意事项/误区：** Promise 应总是处理异常，应在链尾使用 `.catch()` 来捕获错误，或者在 `async` 函数中使用 `try/catch` 包裹 `await`。避免在一个 Promise 内无限嵌套。另外，`async/await` 内的并行操作需要显式 `Promise.all` 等方式，否则多个 `await` 会串行执行，影响性能。  
**示例代码：**

```javascript
// 使用 Promise
function fetchData() {
  return new Promise((resolve, reject) => {
    setTimeout(() => resolve('data'), 1000);
  });
}
fetchData()
  .then(result => console.log(result))
  .catch(err => console.error(err));

// 使用 async/await
async function getData() {
  try {
    const result = await fetchData();
    console.log(result);
  } catch (error) {
    console.error(error);
  }
}
getData();
```

**开发建议：** 在实际开发中，优先使用 `async/await` 使代码结构更平坦易读，并且在遇到多个并行异步任务时用 `Promise.all` 或类似方法优化性能；同时确保对异常进行捕获处理，不要遗漏 `.catch()`。

### 8. JavaScript 事件循环（Event Loop）
**概念/定义：** 事件循环是 JavaScript 运行环境（浏览器或 Node）的核心机制之一，它允许单线程的 JavaScript 执行异步任务。详细描述：事件循环是一个不断运行的循环，它会从任务队列（macrotasks）中依次取出任务执行，然后执行所有可用的微任务（microtasks），接着进行渲染或下一轮任务。  
**使用场景/原理：** 当 JavaScript 执行异步操作时（如 `setTimeout`、Promise 回调、DOM 事件回调等），这些回调会被放入相应的队列中，等待当前执行栈（call stack）清空再执行。微任务队列（例如 Promise 的 `.then`）拥有更高优先级，在当前任务执行完成后立刻执行。  
**注意事项/误区：** 长时间的同步任务会阻塞事件循环，导致界面卡顿。应避免在主线程做过多计算，应将耗时操作拆分或使用 Web Worker。了解微任务与宏任务的区别：每轮事件循环完成后，会先执行所有微任务再渲染界面。常见误区是期望 `setTimeout(fn, 0)` 马上执行，实际上会延后到下一个宏任务队列。  
**示例代码：**

```javascript
console.log('Start');
setTimeout(() => console.log('setTimeout'), 0);
Promise.resolve().then(() => console.log('Promise'));
console.log('End');
// 执行结果：
// Start
// End
// Promise
// setTimeout
```

**开发建议：** 在开发中，应避免阻塞事件循环的操作。对于需要频繁更新 UI 的场景，可以使用 `requestAnimationFrame` 或分批执行任务，以保证浏览器有机会进行重绘。在使用 Promise 或异步函数时，注意 `.then` 或 `await` 会被放入微任务队列，应当理解事件循环的执行顺序。

### 9. 数组常用方法 (`map`、`filter`、`reduce`、`forEach` 等)
**概念/定义：** JavaScript 提供了多种数组方法用于遍历和处理数据。`map()` 会返回一个新数组，其中每一项是回调函数返回值；`filter()` 返回满足条件的项组成的新数组；`reduce()` 将数组归约成单个值（如求和）；`forEach()` 对每个元素执行回调，但不返回新数组。  
**使用场景/原理：** 使用这些高阶函数可以简洁地对数组进行转换和筛选。例如，对数组元素平方可用 `map`，筛选偶数用 `filter`。`reduce` 可用于累加、统计等操作。`forEach` 适用于遍历并执行副作用操作（如打印或修改外部变量），但其返回值为 `undefined`。  
**注意事项/误区：** `map` 和 `filter` 都不会修改原数组，而是返回新数组；`forEach` 不返回新数组，也不能通过 `return` 结束循环，仅用于副作用。`forEach` 不能使用 `break` 停止循环；若需要中途退出，可使用普通 `for` 循环或 `some/every`。`reduce` 回调函数有四个参数（累积值、当前值、索引、原数组），使用时要注意初始值的传递。  
**示例代码：**

```javascript
const arr = [1, 2, 3, 4];
// map: 每项乘2
const mapped = arr.map(x => x * 2); // [2,4,6,8]
// filter: 取偶数
const evens = arr.filter(x => x % 2 === 0); // [2,4]
// reduce: 求和
const sum = arr.reduce((total, x) => total + x, 0); // 10
// forEach: 打印每项
arr.forEach(x => console.log(x));
```

**开发建议：** 实际开发中，根据需求选择正确的方法：若需要返回变换后的数组用 `map`，筛选时用 `filter`，累加或聚合时用 `reduce`。避免误用 `forEach` 来替代 `map` 或 `filter`。如果性能成为瓶颈，可考虑原生循环或其它方法；但在可读性优先的场景，内置方法更简洁易懂。

### 10. 事件委托（Event Delegation）
**概念/定义：** 事件委托是一种常用的事件处理模式：利用事件冒泡机制，将多个子元素的事件统一委托到父元素上监听。这样无需为每个子元素都绑定事件，而是给共同祖先绑定一个监听函数，通过检查 `event.target` 来区分具体的子元素。  
**使用场景/原理：** 当有大量相似子元素或动态生成元素时，使用事件委托可以显著减少绑定数量和内存开销。例如，对一个表格或列表中的所有单元格/项进行点击操作，可以只在父容器（如 `<table>` 或 `<ul>`）上添加一个点击监听。事件发生时，事件对象会冒泡到父元素，在父元素的监听器中通过 `event.target` 确定哪个子元素触发了事件，并做相应处理。  
**注意事项/误区：** 事件委托依赖事件冒泡，因此仅适用于冒泡事件（大部分常见事件都是冒泡的）。需要在处理函数中判断 `event.target` 或使用 `matches` 等方法过滤出目标元素。如果事件被中途阻止冒泡（`stopPropagation`），委托将失效。务必注意事件兼容性：有些事件（如 `focus`/`blur`）不冒泡，不能通过委托处理。  
**示例代码：**

```html
<ul id="itemList">
  <li class="item">项1</li>
  <li class="item">项2</li>
  <!-- 可能动态添加更多<li> -->
</ul>
<script>
document.getElementById('itemList').addEventListener('click', function(e) {
  // 判断是否点击在 .item 元素上
  if (e.target && e.target.matches('li.item')) {
    console.log('点击了:', e.target.textContent);
  }
});
</script>

```

**开发建议：** 在实际开发中，遇到列表、表格或动态添加元素的场景时，优先考虑事件委托来简化代码并提高性能。同时注意在委托函数中精确识别目标元素，避免误触发。对于无法冒泡的事件，可选用等效冒泡事件（例如使用 `focusin` 代替 `focus`）或单独绑定。

## 二、CSS 相关
### 11. CSS 盒模型 (Box Model)
**概念/定义：** CSS 盒模型描述了元素占据空间的方式，每个元素由内容(content)、内边距(padding)、边框(border)和外边距(margin)构成。默认情况下，`width`/`height` 只指内容区域的大小；通过 `box-sizing: border-box` 可以让元素的总宽高包含内边距和边框。  
**使用场景/原理：** 了解盒模型有助于精确布局。例如，设置元素宽度时要考虑内边距和边框对盒子大小的影响。`border-box` 模式更常用，它使元素更容易控制实际占据的空间。  
**注意事项/误区：** 默认的 `content-box` 模式（未设置或使用 `box-sizing: content-box`）会让宽高不包括内边距和边框，常导致计算混乱。建议全局设置 `box-sizing: border-box`，使布局更直观。不同的浏览器或框架也可能有不同默认值，需统一处理。  
**示例代码：**

```css
.element {
  width: 200px;
  padding: 20px;
  border: 5px solid #000;
  box-sizing: content-box; /* 默认模式 */
}
/* 该元素的总宽度 = 200 + 2*20 + 2*5 = 250px */

.element-border {
  width: 200px;
  padding: 20px;
  border: 5px solid #000;
  box-sizing: border-box;
}
/* 总宽度 = 200px（已包含 padding/border） */
```

**开发建议：** 在项目中通常使用 `* { box-sizing: border-box; }` 重置样式，这样设置元素宽高时更符合直觉。布局时根据需要灵活使用外边距调整位置。始终考虑盒模型对元素真实占位的影响，以避免因边框或内边距超出预期范围造成布局错乱。

### 12. Flex 布局
**概念/定义：** Flexbox 是 CSS3 的一种一维布局模式，用来简便地控制容器内部元素的对齐、分布和大小。通过设置父容器的 `display: flex`（或 `inline-flex`），子元素自动成为弹性项，可以使用属性如 `justify-content`（主轴对齐）、`align-items`（交叉轴对齐）等来布局。  
**使用场景/原理：** Flex 布局非常适合做常见的水平/垂直居中、均分或自适应弹性布局。例如，一个导航条可以使用 `justify-content: space-between` 平均分布项，或者 `align-items: center` 垂直居中导航链接。Flex 的主轴（默认横向）和交叉轴（默认纵向）概念使布局更灵活。  
**注意事项/误区：** 旧浏览器对 Flex 支持有限，开发时可适当考虑兼容性。使用 Flex 做单个轴布局很方便，但多行时要使用 `flex-wrap` 开启换行，同时管理换行间距。Flex 子项的宽度默认会收缩以适应父容器，若不希望收缩可以设置 `flex-shrink: 0` 或固定宽度。  
**示例代码：**

```css
.navbar {
  display: flex;
  justify-content: space-between; /* 主轴两端对齐 */
  align-items: center;            /* 交叉轴居中 */
}
.navbar .item {
  flex: 1; /* 平均分布 */
}
```

**开发建议：** 在实际开发中，Flex 是大多数一维布局的首选方案。若需要在一个方向上排列或居中元素，优先考虑 Flexbox，配合媒体查询可实现响应式布局。在使用 Flex 时注意其默认伸缩行为，必要时锁定子元素大小来避免意外变形。对于复杂的二维布局，可结合 CSS Grid（网格布局）使用。

### 13. CSS 定位（Positioning）
**概念/定义：** CSS 定位包括 `static`、`relative`、`absolute`、`fixed`、`sticky` 等属性值，决定元素在文档流中的行为。`static` 是默认定位，元素正常流动。`relative` 相对定位，元素仍占据原位置但可偏移。`absolute` 绝对定位，使元素脱离文档流，相对于最近的定位祖先定位；`fixed` 固定定位，相对于视口定位，页面滚动时保持位置；`sticky` 粘性定位，在滚动达到指定位置时表现为固定。  
**使用场景/原理：** 定位用于精确放置元素：如弹框可用 `absolute` 放在父容器中；导航栏常用 `fixed` 固定在顶部；`relative` 经常和绝对定位配合使用，作为定位上下文（最近定位父元素）。粘性定位适用于滚动到一定位置再固定的场景，比如锚点导航条。  
**注意事项/误区：** 使用 `absolute`/`fixed` 时要注意元素脱离了文档流，会覆盖后续内容，因此布局时需留意可能的重叠。`absolute` 定位元素若没有定位父容器，会相对于文档根元素（`<html>`）定位。`sticky` 支持较好，但在某些旧浏览器中兼容性需注意。  
**示例代码：**

```css
.header {
  position: fixed;
  top: 0;
  left: 0;
  width: 100%;
}

.modal {
  position: absolute;
  top: 50px;
  right: 20px;
}
```

**开发建议：** 在开发中，定位常与 `z-index` 配合使用，实现元素层级管理。对于动态内容或弹窗，用 `absolute/fixed` 定位并注意滚动行为对布局的影响。避免过度使用绝对定位导致布局不灵活，应尽量利用 Flex 或 Grid 布局完成大部分需求，仅在必要时使用定位。

### 14. 响应式设计（响应式布局与适配）
**概念/定义：** 响应式设计通过 CSS 媒体查询（`@media`）、弹性布局、视口单位等技术，实现不同屏幕尺寸下布局和样式的自适应。通常结合 `<meta name="viewport" content="width=device-width, initial-scale=1.0">` 确保移动端按比例显示。常用单位包括百分比、`vw/vh`、`em/rem` 等相对单位。  
**使用场景/原理：** 针对手机、平板、桌面等不同设备，需要调整布局和元素尺寸。例如，使用媒体查询在宽度小于某个值时改变列数或隐藏某些元素；使用 `flex-wrap`、弹性盒子或栅格系统使元素自动换行。`rem` 单位可通过根元素字体大小动态缩放，实现整体缩放。  
**注意事项/误区：** 响应式需优先移动端（Mobile-first）的设计思路，小屏样式放在基础样式，大屏通过 `min-width` 媒体查询逐步增强。要避免使用固定像素值，应尽量使用相对单位。切勿忘记设置视口元标签，否则移动设备可能默认缩放。  
**示例代码：**

```css
/* 移动端基础样式 */
.container { width: 100%; padding: 10px; }
/* 1024px 以上平板/桌面 */
@media (min-width: 1024px) {
  .container { width: 960px; margin: 0 auto; }
}
```

**开发建议：** 实际开发时，结合响应式框架或栅格系统（如 Bootstrap），并自行设置媒体查询和弹性布局。使用 `rem` 结合 JS 动态改变根 `font-size`（响应式字体或布局）也常见。对关键元素进行横竖屏适配测试，并使用开发者工具模拟各种屏幕来验证布局的可用性。

### 15. 选择器优先级（Specificity）
**概念/定义：** CSS 选择器优先级决定了当多个规则作用于同一元素时，哪条规则生效。它根据选择器类型（元素、类、ID、内联样式等）计算优先级数值。例如，ID 选择器权重最高，其次是类/属性/伪类选择器，再次是标签/伪元素选择器，最低是通配符。内联样式优先级最高。  
**使用场景/原理：** 理解优先级可帮助调试样式冲突问题。可以通过制定权重更高的选择器或者添加 `!important` 来覆盖已有规则（不推荐大量使用 `!important`）。良好的做法是使用语义化和层级清晰的选择器，将样式作用域局限在需要的元素。  
**注意事项/误区：** 过度依赖高优先级选择器或 `!important` 会导致维护困难。避免写过深的嵌套选择器（如 `.a .b .c span`)；这样会降低可读性且难以覆盖。如果出现样式无法生效的情况，应检查是否被更高权重的规则覆盖。  
**示例代码：**

```css
/* ID 选择器优先级高于类选择器 */
#main .button { color: red; }
.button { color: blue; }
<!-- 生效的颜色为红色 -->
```

**开发建议：** 在实际开发中，可采用 BEM 命名法或 CSS Modules 等技术来避免选择器冲突和优先级问题。编写 CSS 时尽量保持简单明确的选择器，不要过分层叠。在遇到复杂的样式覆盖问题时，可使用浏览器开发者工具查看元素的具体样式来源，以判断优先级关系。

### 16. 块级格式化上下文（BFC）
**概念/定义：** 块级格式化上下文（Block Formatting Context，BFC）是一种渲染布局模式，它独立于常规的文档流。元素只与自身的内容发生关系，不受外界浮动影响，同时其内部元素也不会影响外部元素的布局。可通过某些 CSS 属性触发生成 BFC，如 `overflow: hidden/auto`、`float`、`display: flex` 等。  
**使用场景/原理：** BFC 在清除浮动和避免边距重叠（margin collapse）时非常有用。常用手法是为父容器设置 `overflow: hidden` 或添加空块 `::after` 来自动清除浮动。BFC 还可用于防止垂直外边距相邻元素合并。  
**注意事项/误区：** 生成 BFC 会脱离普通文档流，对布局有影响。例如，`overflow: hidden` 同时可能裁剪子元素溢出部分。应谨慎使用。浮动元素需要在父容器建立 BFC 或使用清除浮动的技术，否则父容器高度可能塌陷（无法包裹浮动子元素）。  
**示例代码：**

```css
.container {
  overflow: hidden; /* 触发 BFC, 清除内部浮动 */
}
.container::after {
  content: "";
  display: block;
  clear: both; /* 另一种清除浮动的方式 */
}
```

**开发建议：** 在实际开发中，布局常常需要“清除浮动”来使父容器包裹浮动元素。利用 BFC 生成方式（如给父容器 `overflow: auto`）来清除浮动是一种简便方案。了解 BFC 原理对控制父子元素的外边距和响应式布局效果也有帮助。

## 三、Vue 相关
### 17. Vue 框架简介
**概念/定义：** Vue.js 是一个渐进式的前端框架，用于构建用户界面。它基于 MVVM 模式，提供声明式渲染和组件化开发模型。Vue 的核心特色包括响应式数据绑定、虚拟 DOM 和灵活的组件系统。由于其轻量、高性能和易用性，Vue 既可作为简单页面交互层，也可用于大型单页应用开发。  
**使用场景/原理：** Vue 适合需要快速构建交互丰富的用户界面或单页应用。它通过双向数据绑定（用于表单输入）和单向数据流（父组件到子组件的 props）来管理视图与数据同步。Vue 的模板语法直观易懂，也支持基于函数式渲染。组件间可以复用和嵌套，提高了开发效率。  
**注意事项/误区：** 在 Vue 中，尽量避免直接操作 DOM，应通过 Vue 的响应式数据改变来更新界面。理解响应式原理（Vue 2 使用 `Object.defineProperty`，Vue 3 使用 `Proxy`）有助于优化性能。不要在生命周期钩子中使用箭头函数来定义回调，否则 `this` 会出现指向错误。  
**示例代码：**

```javascript
import { createApp } from 'vue';
const app = createApp({
  data() {
    return { message: 'Hello Vue' };
  }
});
app.mount('#app');
```

**开发建议：** 实际开发时，利用 Vue 的组件系统将功能拆分，保持每个组件的关注点单一。常用工具链（如 Vue CLI、Vite）可以提高开发效率。多借助 Vue DevTools 进行调试，并遵循官方风格指南（例如尽量使用 `camelCase` 命名组件等）。了解核心概念（响应式、虚拟 DOM、组件通信）将有助于在面试中表现出扎实的框架基础。

### 18. Vue 2 与 Vue 3 区别
**概念/定义：** Vue 3 相较于 Vue 2 做了底层和语法的改进。最大的变化之一是响应式系统从 `Object.defineProperty` 转为使用 ES6 `Proxy` 实现；这使得 Vue 3 在跟踪依赖、性能和对新特性的支持上更优。Vue 3 引入了 Composition API（组合式 API）等新特性，允许用函数式风格组织组件逻辑。此外，Vue 3 对 TypeScript 支持更友好，框架包体积更小、性能更好。  
**使用场景/原理：** Vue 2 继续在许多项目中使用，而 Vue 3 新项目可以利用其新特性：如 Composition API 使逻辑复用更灵活，响应式系统更高效。要注意迁移时组件生命周期钩子的顺序变化和废弃特性。  
**注意事项/误区：** 由于实现机制不同，某些 Vue 2 模式在 Vue 3 下需调整。例如，Vue 2 中利用数组索引设置新属性无法触发更新（需要 `Vue.set`），而 Vue 3 的 `Proxy` 可自动检测。Vue 3 不再支持一些 Vue 2 的全局 API（如 `Vue.config.ignoredElements` 变更位置）。在使用 Composition API 时，需要注意 `setup()` 中访问 `this` 的可用性已改变。  
**示例代码：**

```javascript
// Vue2 组件选项写法
export default {
  data() { return { count: 0 }; },
  methods: { inc() { this.count++; } }
};

// Vue3 组合式写法
<script setup>
import { ref } from 'vue';
const count = ref(0);
function inc() { count.value++; }
</script>

```

**开发建议：** 在实际开发中，新项目推荐使用 Vue 3 来获得更好的性能和新特性支持。对已有 Vue 2 项目，可采用官方迁移指南升级。开发者应熟悉 Vue 3 的变化（如全局 API 改动、生命周期钩子顺序改动等），并逐步使用新的 Composition API 和 `<script setup>` 等语法糖来重构组件。

### 19. 双向绑定（v-model 原理）
**概念/定义：** Vue 的 `v-model` 指令用于创建表单元素（或组件）的双向数据绑定。对于原生 `<input>` 等元素，`v-model` 相当于绑定值和监听输入事件；对于组件，Vue 会将 `v-model` 解构为绑定 `modelValue` 属性和监听 `update:modelValue` 事件的语法。在 Vue3 中，推荐使用 `defineModel()` 宏来定义组件的 `v-model` 支持。  
**使用场景/原理：** 在表单输入场景下，`v-model` 简化了数据读写。例如 `<input v-model="name">` 会根据 `name` 的值设置输入框显示，并在用户输入时更新 `name`。其原理是在底层给组件或元素绑定 `value` 和 `input/change` 事件（组件版则是 `modelValue` 和 `update:modelValue`）。  
**注意事项/误区：** 绑定组件时，需在子组件内正确发射事件来更新父组件数据，否则父子数据不同步。不同版本 Vue 对 `v-model` 的底层实现稍有差别，Vue 3 以上用 `modelValue/update:modelValue`，Vue 2 用 `value/input`。另外，表单元素还提供 `.lazy`、`.number`、`.trim` 修饰符，实现延迟更新、数字转换等功能。  
**示例代码：**

```vue
<!-- 父组件 -->
<template>
  <child-component v-model="username" />
</template>
<script>
import ChildComponent from './Child.vue';
export default {
  data() { return { username: '' }; },
  components: { ChildComponent }
}
</script>
<!-- 子组件 Child.vue -->
<template>
  <input :value="modelValue" @input="$emit('update:modelValue', $event.target.value)" />
</template>
<script setup>
const props = defineProps(['modelValue']);
const emit = defineEmits(['update:modelValue']);
</script>

```

**开发建议：** 在实际开发中，使用 `v-model` 可以快速实现表单双向绑定，提高效率。对于自定义组件，按照 Vue 文档要求实现 `modelValue` 属性和 `update:modelValue` 事件。避免在父组件直接修改子组件的 prop，而应通过事件通知父组件变更数据。对于复杂情况，可使用多个参数的 `v-model` 或分离事件实现多对多绑定。

### 20. 生命周期钩子（Lifecycle Hooks）
**概念/定义：** Vue 组件从创建到销毁经历多个阶段，每个阶段都有对应的生命周期钩子函数。Vue 3 提供了例如 `setup`、`onBeforeMount`、`onMounted`、`onBeforeUpdate`、`onUpdated`、`onBeforeUnmount`、`onUnmounted` 等钩子。每个钩子函数会在组件的特定时刻被调用。  
**使用场景/原理：** 生命周期钩子提供了插入代码的时机。例如，在组件创建后加载数据可放在 `onMounted`（DOM 已生成）或 `onCreated` 钩子；在组件销毁前做清理（如移除全局事件、取消定时器）可在 `onBeforeUnmount` 钩子完成。  
**注意事项/误区：** 使用钩子函数时避免使用箭头函数，否则 `this` 指向可能不正确。也不要在异步操作中误用钩子顺序，比如在 `mounted` 钩子中直接返回一个 `Promise` 并期望组件挂载后等待其完成，这是不准确的做法。  
**示例代码：**

```javascript
import { onMounted, onBeforeUnmount } from 'vue';
export default {
  setup() {
    onMounted(() => {
      console.log('组件已挂载');
      // 例如: 发起 AJAX 请求或订阅事件
    });
    onBeforeUnmount(() => {
      console.log('组件即将销毁');
      // 清理操作: 例如取消订阅、停止定时器等
    });
  }
}
```

**开发建议：** 实际开发时，在合适的钩子中处理对应逻辑：异步数据获取可放在 `onMounted`，DOM 操作也应在 `onMounted` 后执行；在 `onUnmounted` 或 `onBeforeUnmount` 中执行清理操作以防内存泄漏。注意不要在钩子内部使用 `await` 使逻辑混乱；如果使用 `setup()`，应通过 `import { onMounted } from 'vue'` 等组合式 API 来注册钩子。

### 21. 组件通信（Props 与事件、Provide/Inject）
**概念/定义：** Vue 中组件间通信方式包括：父组件通过 `props` 传递数据给子组件；子组件通过 `$emit` 触发自定义事件通知父组件；另外还有 Vue 2 的全局事件总线（现不推荐）和 Vue 3 的 `provide/inject` 机制。`provide/inject` 允许祖先组件提供数据，任何层级的后代组件都可以注入并使用。  
**使用场景/原理：** 常见的父子通信：父组件在使用子组件时绑定属性（`<child :msg="parentMsg" />`），子组件在 `props: ['msg']` 中声明并使用；子组件想改变父组件的数据时可在子上使用 `$emit('update', newValue)`，父组件监听并处理。对于多层级需要传递的依赖，可以在顶层组件中使用 `provide` 提供数据/方法，在任意后代组件中使用 `inject` 接收。  
**注意事项/误区：** `props` 传递的数据是单向的（父到子），子组件不应直接修改 prop 的值；若需要修改，应在子组件中发出事件通知父组件更新。过度使用 `$emit` 进行跨多层通信会导致复杂，可以考虑将共享状态放入 Vuex 或使用 `provide/inject` 以简化。  
**示例代码：**

```vue
<!-- 父组件 -->
<template>
  <child-component :msg="message" @update-msg="message = $event" />
</template>
<script>
import ChildComponent from './Child.vue';
export default {
  data() { return { message: 'Hello' }; },
  components: { ChildComponent }
}
</script>
<!-- 子组件 Child.vue -->
<template>
  <div>
    <p>{{ msg }}</p>
    <button @click="$emit('update-msg', 'New value')">Change</button>
  </div>
</template>
<script>
export default {
  props: ['msg']
};
</script>

```

```javascript
// Provide/Inject 示例
// 祖先组件
import { provide } from 'vue';
setup() {
  const shared = reactive({ count: 0 });
  provide('sharedData', shared);
}
// 后代任意组件
import { inject } from 'vue';
const shared = inject('sharedData');
```

**开发建议：** 在实际开发中，尽量使用清晰的单向数据流：父组件通过 props 下传静态数据或状态，子组件通过 `$emit` 提交事件修改父状态。对于非常深层级的数据传递，优先考虑 Vuex 或 `provide/inject` 而非事件链式传递。Vuex （或 Pinia）则适用于更复杂的全局状态管理需求。

### 22. 计算属性 vs 侦听器（Computed vs Watch）
**概念/定义：** 计算属性是基于其响应式依赖进行缓存的属性，只要相关依赖未变，重复访问会立即返回旧值。侦听器（`watch`）用于对数据进行监视，在数据变化时执行回调。计算属性通常用于模板中直接绑定，而侦听器用于在数据变更时执行异步或开销较大的操作。  
**使用场景/原理：** 如果需要对数据进行处理并在模板中展示，且处理过程无副作用，应使用计算属性；它会根据依赖自动更新，且高效缓存。如果需要在数据变化时执行异步请求、手动操作 DOM 或复杂逻辑，则适合使用 `watch`。例如，监听用户输入并调用 API，或监听路由参数变化进行相应处理。  
**注意事项/误区：** 不要把所有逻辑都写在 `watch` 回调里：简单的绑定、格式化等可使用计算属性以便自动更新和缓存。计算属性不能替代所有场景，比如非同步操作必须使用 `watch`。  
**示例代码：**

```javascript
data() {
  return { firstName: 'John', lastName: 'Doe', fullName: '' };
},
computed: {
  fullNameComputed() {
    return `${this.firstName} ${this.lastName}`; // 缓存依赖 firstName/lastName
  }
},
watch: {
  // 监控 firstName 或 lastName 变化时更新 fullName
  firstName(newVal) { this.fullName = newVal + ' ' + this.lastName; },
  lastName(newVal) { this.fullName = this.firstName + ' ' + newVal; }
}
```

**开发建议：** 在实际开发中，大多数对数据的直接转换应使用计算属性，因为计算属性免除了手动更新过程并拥有缓存机制。仅当确实需要在数据变化时执行异步任务或有显著副作用时才使用 `watch`。掌握两者的区别可以让数据驱动更加清晰和高效。

### 23. Vue 插槽（Slot）
**概念/定义：** 插槽（Slot）是 Vue 提供的组件内容分发机制，用来让父组件可以在子组件指定的插槽位置插入自定义内容。默认插槽使用 `<slot>`，具名插槽使用 `<slot name="...">`，作用类似 React 的 `props.children`。作用域插槽（Scoped Slot）允许父组件在插槽模板中访问子组件传出的数据。  
**使用场景/原理：** 插槽使组件更加灵活：父组件通过插槽向子组件传入任意模板内容。例如，常见的布局组件 `Modal`、`Table` 等会定义插槽来让调用者填充具体内容。作用域插槽可以让子组件在插槽处“传递”数据给父组件模板使用，实现双向数据传递。  
**注意事项/误区：** 使用作用域插槽时注意数据的命名和 `v-slot` 指令语法，避免混淆。默认插槽如果没有内容则显示 `<slot>` 标签中的默认内容。多余的插槽名称和 `v-slot` 指令需严格对应，且编译后会消耗一定性能，不要滥用。  
**示例代码：**

```vue
<!-- Parent.vue -->
<custom-card>
  <template #header>
    <h1>标题</h1>
  </template>
  <p>主要内容</p>
  <template #footer>
    <button>提交</button>
  </template>
</custom-card>
<!-- CustomCard.vue -->
<template>
  <div class="card">
    <header><slot name="header"></slot></header>
    <section><slot>默认内容（未提供时）</slot></section>
    <footer><slot name="footer"></slot></footer>
  </div>
</template>

```

**开发建议：** 在实际开发中，常使用插槽来构建可复用的高级组件，例如自定义弹框、表格、表单项等。设计组件时预留好插槽位，以提升灵活性。使用作用域插槽可将子组件的数据暴露给父组件，增强定制能力。要注意插槽会增加模板层级，适当简化插槽结构以保证可读性。

### 24. v-if 与 v-show 区别
**概念/定义：** `v-if` 与 `v-show` 都可根据表达式条件控制元素显示，但原理不同：`v-if` 是真正地在条件为假时销毁和移除元素（改变渲染状态），当条件再次为真时再重新创建；`v-show` 则是通过切换元素的 CSS `display` 属性来隐藏/显示元素。  
**使用场景/原理：** 当需要频繁切换元素显示状态时，`v-show` 性能更优（只切换样式）；当切换次数较少且初始渲染开销需要控制时，`v-if` 更合适。举例，当弹窗、标签页等组件初始不可见，切换时使用 `v-show` 更流畅；而动态切换路由视图时使用 `v-if` 节省内存。  
**注意事项/误区：** 尽量不要在两者间替换使用，否则可能导致无谓开销。要明确区分：`v-if` 对应的是真正的添加/移除 DOM；`v-show` 始终渲染元素，只是简单地切换样式。如果DOM结构很大，频繁使用 `v-if` 会导致频繁创建销毁影响性能。  
**示例代码：**

```vue
<!-- v-if -->
<div v-if="isVisible">仅在 isVisible 为 true 时渲染</div>
<!-- v-show -->
<div v-show="isVisible">始终渲染，通过 display 控制可见</div>

```

**开发建议：** 在实际开发中，根据需求选择使用：频繁切换的列表项或低性能损耗偏好的场景使用 `v-show`，初始化时隐藏或显示次数少时使用 `v-if`。避免混用这两者在同一元素上，清晰明确判断条件逻辑即可。

## 四、Vuex 相关
### 25. Vuex 状态管理 (State Management)
**概念/定义：** Vuex 是 Vue 官方提供的状态管理库，用于在 Vue 应用中集中管理组件间的共享状态。它采用单一状态树（single state tree），即用一个对象包含应用所有层级状态，使得状态可以集中管理和跟踪。Vuex 将状态变更逻辑通过“变更事件”显式记录，从而在开发者工具中便于调试和追踪。  
**使用场景/原理：** 在大型应用中，多个组件可能需要访问、修改同一份状态时，使用 Vuex 可以避免使用事件总线或层层回调传值，保证状态管理的可预测性和一致性。Vuex 包含 **state**（状态）、**getter**（派生状态）、**mutation**（同步变更）、**action**（可异步操作）四大核心概念。组件通过 `commit` 触发 mutation 改变 state，通过 `dispatch` 触发 action 执行逻辑，action 内最终还是通过 `commit` 调用 mutation。  
**注意事项/误区：** 不要直接在组件中修改 Vuex state，所有状态更改应在 mutation 中发生（可维护记录）。Mutation 必须是同步函数，如果有异步需求，应在 action 中完成后再 commit。牢记单一状态树原则，一个应用只有一个 store 实例（可以用模块分割）。避免在组件中复制 Vuex 状态来修改，应直接通过 store 提交变更。  
**示例代码：**

```javascript
// store.js
export const store = new Vuex.Store({
  state: { count: 0 },
  mutations: {
    increment(state) { state.count++; }
  },
  actions: {
    incrementAsync({ commit }) {
      setTimeout(() => commit('increment'), 1000);
    }
  }
});

// 在组件中使用
methods: {
  add() {
    this.$store.commit('increment');
  },
  addAsync() {
    this.$store.dispatch('incrementAsync');
  }
}
```

**开发建议：** 在实际开发中，如果只是简单组件通信，可不使用 Vuex；但随着业务复杂度增加，共享状态通过 Vuex 管理可使项目更可维护。使用 Vuex 时可结合 [mapState]、[mapActions] 辅助函数简化代码。对于大型应用，可以开启严格模式检测非法状态修改，并使用 Vuex 插件或持久化解决 page 刷新状态丢失问题。

### 26. Vuex Modules（模块化管理）
**概念/定义：** Vuex 模块化是指将 Vuex 的 store 拆分为多个模块，每个模块拥有自己的 state、mutation、action、getter，从而实现对大型项目状态的分割管理。这样依然只有一个根状态树，但每个模块可看作子模块独立管理自身数据。  
**使用场景/原理：** 当应用状态较多时，将状态根据业务领域拆分为模块（如 auth、products、cart 等）可以使代码结构更清晰。Vuex 的 module 配置支持命名空间（namespaced），可以让模块拥有独立的命名空间，从而在提交 mutation 或 dispatch action 时更明确地定位到模块。  
**注意事项/误区：** 模块化管理并不是多 store，而是单个 store 下的多模块。命名空间模块内的 mutation/action 不会自动注册到全局，需要在调用时加上模块前缀（或用 `namespaced: true` 明确）。避免在模块间不清晰的交互：一个模块想使用另一个模块的 state，应通过根级 `rootState` 或根级 getters。  
**示例代码：**

```javascript
const cartModule = {
  namespaced: true,
  state: { items: [] },
  mutations: { addItem(state, item) { state.items.push(item); } },
  actions: { addToCart({ commit }, item) { commit('addItem', item); } }
};
const store = new Vuex.Store({
  modules: {
    cart: cartModule
  }
});
// 组件中使用时：
this.$store.dispatch('cart/addToCart', newItem);
```

**开发建议：** 在实际开发中，尽早规划 Vuex 的模块结构。对于大团队项目，可以让不同团队维护不同模块。谨慎使用全局状态（root state），尽量把数据放在对应模块，保持关注点单一。必要时可以使用插件（如 `vuex-persistedstate`）对模块状态进行持久化管理，防止页面刷新导致状态丢失。

### 27. Vuex 持久化 (Token 管理等)
**概念/定义：** Vuex 状态默认存在内存中，页面刷新会丢失。持久化 Vuex 通常指将状态保存到本地存储（如 localStorage）或通过插件自动同步，实现跨刷新保持状态。Token 管理通常与此相关：将登录获取的 Token 存在 Vuex，并借助本地存储保持状态。  
**使用场景/原理：** 常见做法是结合浏览器存储：用户登录时，Vuex 存储 Token，同时同步到 localStorage。页面加载时读取 localStorage 初始化 Vuex 中的 Token。也可使用 Vuex 插件（如 vuex-persist）自动完成此过程。  
**注意事项/误区：** 不要将敏感的长期凭证放在易被 XSS 攻击读取的地方；如果需要安全性更高，可将 Token 存入 HttpOnly Cookie，并通过服务端验证。持久化过多数据会增加 localStorage 体积，性能和安全风险，建议只存储必要数据（如用户基本信息、Token），不建议直接持久化整个 state。  
**示例代码：**

```javascript
// 每次登录后 commit 和 localStorage 写入
this.$store.commit('setToken', token);
localStorage.setItem('token', token);
// 初始化应用时读取并设置
const token = localStorage.getItem('token');
if (token) {
  store.commit('setToken', token);
}
```

**开发建议：** 在实际应用中，Token 推荐优先通过 HttpOnly 安全 Cookie 存储。如果使用 localStorage，应确保项目有足够的 XSS 防范措施。在 Vuex 中，尽量不要在 mutation 中访问浏览器 API；可以在 action 中处理持久化。使用持久化插件时，也要留意刷新状态与登录状态同步的时序问题。

## 五、认证与通信
### 28. Token 与认证（JWT 机制）
**概念/定义：** Token（令牌）常指用于身份验证的随机字符串或 JWT（JSON Web Token）。JWT 是一种自包含的认证方案，包含了用户身份信息和签名，前端收到后可在后续请求中携带此令牌进行身份校验。  
**使用场景/原理：** 常见场景是在用户登录后，服务端返回一个 JWT。前端存储该 Token（如在 localStorage 或 Cookie 中）并在每次请求时通过 HTTP Header（`Authorization: Bearer <token>`）发送给后端。后端通过验证签名确认身份，有效期内无需再次登录。Token 可以包含过期时间（exp）和刷新机制。  
**注意事项/误区：** 存储位置关系到安全性：**避免将敏感令牌存储在易被脚本访问的地方（如 **`localStorage`**）而不做好 XSS 防护。** 可以考虑将 Token 放在加 `HttpOnly` 的 Cookie 中防止 XSS 攻击，但这会引入 CSRF 问题。不同方案有权衡：**localStorage** 容易使用且避开 CSRF，但受 XSS 威胁；**Cookie** 可设置 `HttpOnly` 防 XSS，但需额外防范 CSRF（如使用同源策略或设置 `SameSite`）。  
**示例代码：**

```javascript
// 登录后保存 Token
localStorage.setItem('token', jwtToken);
// 请求拦截器中加入
axios.interceptors.request.use(config => {
  const token = localStorage.getItem('token');
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});
```

**开发建议：** 实际开发中，对 JWT 的管理策略要全面考虑安全性和用户体验。尽量减少高权限操作直接依赖前端 Token，后台应校验并控制敏感操作。对长期 Token 可设置合理过期，并通过刷新 Token 机制延长会话。保证前端代码防 XSS（Content Security Policy、输入验证）和后端防 CSRF。

### 29. Token 存储方式比较（LocalStorage vs Cookie）
**概念/定义：** 浏览器提供了多种存储方案：`localStorage` 和 `sessionStorage`（Web Storage）、以及 Cookie。Token 存储时，需要权衡安全和便利。`localStorage` 容量大、使用方便，但数据会随脚本访问；Cookie（可设置为 `HttpOnly`）在请求时自动携带。  
**使用场景/原理：** - **Cookie**：可设置 `HttpOnly`，防止 JS 访问，从而减轻 XSS 攻击对 Token 的风险；可设置 `Secure`、`SameSite` 属性进一步提高安全性。缺点是 Cookie 会随每次 HTTP 请求自动发送，可能导致请求负担增加，并需防范 CSRF。

+ **LocalStorage**：容量较大，数据不会自动发送，调用简单，是单页应用常用做法。但易受 XSS 攻击，一旦页面脚本被注入，Token 可被偷取。LocalStorage 对 CSRF 天然免疫。  
**注意事项/误区：** 无论哪种方式，都要做好前后端安全措施。使用 Cookie 存储时需要 CSRF Token 或 `SameSite=strict/lax` 来抵御 CSRF；使用 `localStorage` 时必须消除 XSS 漏洞（输入检查、内容安全策略等）。不要在不可信的域下共享 Cookie，也要谨慎设置过期时间和平衡持久登录需求。  
**示例代码：**

```javascript
// 使用 HttpOnly Cookie 存储（前端无法直接设置，只能通过后端响应头）
axios.post('/login', { user, pass })
  .then(res => {
    // 服务器设置 Set-Cookie 响应头，无需前端处理
  });

// 使用 localStorage 存储
localStorage.setItem('token', jwtToken);
```

**开发建议：** 建议配合应用场景选择存储：如果应用不易发生 XSS 攻击，使用 `localStorage` 更灵活；如果对安全性要求高，可使用后端 Set-Cookie 发放 `HttpOnly` Token，并结合同源策略和 CSRF 保护。在前端开发时，配合响应式技术及时更新 Vuex 中的登录状态，并在页面加载时从存储中读取初始状态。

### 30. 刷新令牌（Refresh Token）
**概念/定义：** 刷新令牌是一种长期有效的令牌，用于在访问令牌（access token）过期后获取新的访问令牌。访问令牌通常有效期短（如 15 分钟），刷新令牌有效期更长（如几天）。当访问令牌过期，前端可自动用刷新令牌向后端申请新令牌，而无需用户重新登录。  
**使用场景/原理：** 登录成功后，服务器同时返回短期访问令牌和长期刷新令牌。前端将两者保存（通常将访问令牌保存在内存/短期存储，刷新令牌保存在更安全的位置）。每次 API 请求都带访问令牌。后端验证过期后若发现令牌过期且发现有效的刷新令牌，则颁发新的访问令牌。  
**注意事项/误区：** 刷新令牌也需保护，一般保存在 HttpOnly Cookie 或安全存储中。若刷新令牌也泄露，应在后端废止该刷新令牌，使得无法再换取访问令牌。刷新操作应防止频繁调用，可限制刷新频率。后端应考虑刷新令牌的最大有效期（如每刷新 N 次或 N 天后强制重新登录）。  
**示例代码：**

```javascript
// 伪代码示例：拦截响应中处理 401 错误
axios.interceptors.response.use(null, async error => {
  if (error.response.status === 401) {
    // 假设 refreshToken 存在
    const refreshToken = localStorage.getItem('refreshToken');
    const response = await axios.post('/refresh', { token: refreshToken });
    const newToken = response.data.accessToken;
    localStorage.setItem('token', newToken);
    // 重新发起原始请求
    error.config.headers['Authorization'] = `Bearer ${newToken}`;
    return axios(error.config);
  }
  return Promise.reject(error);
});
```

**开发建议：** 实际开发中，刷新令牌机制可以显著提升用户体验。但也带来风险：后端需确保刷新令牌的安全，前端要合理存储和使用刷新令牌。如使用 Cookie，刷新逻辑可在服务端完成；如使用 `localStorage`，则要防范 XSS 窃取。定期清理旧的令牌和及时更新状态能够保证系统安全。

### 31. WebSocket 简介
**概念/定义：** WebSocket 是 HTML5 提供的协议，使得浏览器（客户端）与服务器之间能够建立持久化的双向通信通道。通过 WebSocket，可以在客户端和服务端之间实时双向推送数据，而无需频繁轮询。  
**使用场景/原理：** WebSocket 在连接建立时通过 HTTP/HTTPS 进行握手（升级到 `ws://` 或 `wss://`），之后客户端和服务器可以随时互发消息。常见应用包括实时聊天、消息推送、多人协作、金融行情更新等场景。相比长轮询和 HTTP 请求，WebSocket 更高效地减少了网络开销和延迟。  
**注意事项/误区：** 必须在后端支持 WebSocket 协议并进行握手。使用时注意：`wss://` 是加密传输（需要 SSL），`ws://` 是明文。长连接需要考虑心跳、重连机制：当断线时要重建连接；对实时性要求较高的场景，可在客户端设置定时 ping/pong 保持活跃。不同浏览器可能对 WebSocket 有差异，应注意兼容性问题。  
**示例代码：**

```javascript
// 创建 WebSocket 连接
const socket = new WebSocket('wss://example.com/socket');
// 打开连接
socket.addEventListener('open', () => {
  console.log('连接已打开');
  socket.send('Hello server');
});
// 接收消息
socket.addEventListener('message', event => {
  console.log('收到消息:', event.data);
});
// 关闭连接
socket.addEventListener('close', () => {
  console.log('连接已关闭');
});
```

**开发建议：** 在实际开发中，使用 WebSocket 时要统一服务器和客户端协议，保持版本兼容。对网络不稳定的应用，应实现自动重连和断线提示。若 WebSocket 不可用时，可考虑退回到轮询方案。开发中可使用封装库（如 Socket.IO）简化使用，并配合后端使用（例如 Node.js 的 ws 库）进行可靠通信。

### 32. WebSocket 与 HTTP 的区别
**概念/定义：** WebSocket 与 HTTP 都建立在 TCP 之上，但设计目的不同：HTTP 是无状态的单向请求/响应协议，每次请求后连接可关闭；WebSocket 是双向通信协议，一旦建立连接（握手升级后）即保持连接，双方可以主动推送数据。  
**使用场景/原理：** HTTP 适合传统请求场景，如页面加载、表单提交等。WebSocket 适合实时双向交互场景，如在线聊天、游戏实时通知、股票价格更新等。WebSocket 握手基于 HTTP/HTTPS，然后协议升级。  
**注意事项/误区：** WebSocket 连接会一直保持，占用服务器资源；在短期或单次交互场景使用 WebSocket 反而增加复杂度。设计时要清楚是需要实时通信才用 WebSocket，否则用普通 AJAX/Fetch 更简单。由于 WebSocket 连接不同于 HTTP 请求，需要在应用中管理好连接的生命周期（如组件挂载时创建连接、卸载时关闭）。  
**示例代码：**

```javascript
// 服务器端（Node.js ws 库示例）
const WebSocket = require('ws');
const wss = new WebSocket.Server({ port: 8080 });
wss.on('connection', ws => {
  ws.on('message', message => { console.log(`Received: ${message}`); });
  ws.send('Welcome client');
});
```

**开发建议：** 实际开发时，评估应用需求：对于实时性要求不高的业务，使用 RESTful 或 Fetch API；对于聊天、直播等需实时推送的业务，使用 WebSocket。确保服务器端支持 WebSocket，可使用反向代理（如 nginx）正确转发 `Upgrade` 请求。为避免连接泄露，需要正确关闭不再使用的 WebSocket 连接。

### 33. 柔性布局 vs 绝对布局 vs CSS Grid
**概念/定义：** 前端常见布局方式包括 Flexbox（柔性盒）和 CSS Grid（网格布局），它们都可以取代传统的浮动/绝对布局。Flexbox 适用于一维（行或列）布局，Grid 适用于二维布局，功能更强大。  
**使用场景/原理：** 对于简单的一行或一列排列，使用 Flexbox 快速简单。需要更复杂的网格布局时使用 CSS Grid，可以指定行列位置和跨行跨列。Flexbox 在移动端和简单组件非常灵活；CSS Grid 在页面整体布局、可视化仪表盘等场景下更高效。  
**注意事项/误区：** 旧版浏览器对 CSS Grid 支持有限，需要留意兼容性。过度依赖绝对定位可能导致响应式问题。应根据需求选择合适方案：优先考虑 Flex 或 Grid，而非再使用浮动和绝对定位“手动”计算位置。  
**示例代码：**

```css
/* Flex 示例：水平居中 */
.container { display: flex; justify-content: center; align-items: center; }
/* Grid 示例：两列布局 */
.grid { display: grid; grid-template-columns: 1fr 1fr; grid-gap: 10px; }
```

**开发建议：** 在项目中，建议尽量使用现代布局模块（Flex 和 Grid）完成布局，大幅度提升开发效率和可维护性。只有在特殊需求时才使用绝对定位。例如，覆盖层或固定元素。始终测试不同屏幕与浏览器下的兼容效果，保证布局的灵活性与稳定性。

### 34. 清除浮动（clearfix）和百分比布局
**概念/定义：** 在使用浮动元素时，父容器的高度会塌陷（无法包裹浮动子元素）。清除浮动（clearfix）是一种通用的解决方案，使父容器包含浮动子元素。常用方法是在父元素末尾插入伪元素（`::after`）并设置 `clear: both`。  
**使用场景/原理：** 当布局需要使用浮动排列子元素（如并列的图片或卡片）时，父元素需要清除浮动保持高度。CSS Clearfix 一般写法：

```css
.clearfix::after {
  content: "";
  display: block;
  clear: both;
}
```

父元素使用 `.clearfix` 类即可自动清除内部浮动。此外，也可用 `overflow: hidden` 触发 BFC 来清除浮动。  
**注意事项/误区：** 现代布局往往用 Flexbox 或 Grid，减少对浮动的依赖。因此，清除浮动主要在维护旧代码或特定需求时使用。使用伪元素清除浮动时要确保不影响布局，也可采用 `display: flow-root`（支持现代浏览器）直接创建 BFC。  
**示例代码：**

```css
/* 清除浮动 */
.container::after {
  content: "";
  display: table;
  clear: both;
}
```

**开发建议：** 在实际项目中，优先使用 Flexbox 而不是浮动来做并列布局。如果必须使用浮动，确保父容器应用清除浮动策略，或者给父容器设置 `overflow: auto;` 触发 BFC。保持布局层次清晰，减少意外的浮动影响。

### 35. CSS 单位（px、em、rem、vw、vh）
**概念/定义：** CSS 提供多种长度单位：`px` 是绝对像素；`em`、`rem` 是相对单位，`em` 相对于当前元素的字体大小，`rem` 相对于根元素（`<html>`）的字体大小；`vw`、`vh` 是视口单位，分别为视口宽度和高度的百分比。  
**使用场景/原理：** 在响应式设计中，常用 `rem` 和 `vw/vh` 使页面随屏幕缩放。设置根字体大小后，使用 `rem` 控制整体缩放。例如，移动端可将根字体大小设置为视口宽度除以固定值，动态适配不同屏幕宽度。`vh` 常用于全屏组件（如满屏背景高度）。  
**注意事项/误区：** 使用 `px` 可以精确控制元素大小，但不易响应式调整。`em` 因为依赖父元素大小，嵌套过深可能难以计算。`rem` 推荐用于全局字体大小和布局，简化适配。`vw`、`vh` 在一些小屏设备上可能不完全可靠（如移动端地址栏高度变化）。  
**示例代码：**

```css
html { font-size: 16px; }
.title { font-size: 1.5rem; /* 24px */ }
.section { width: 80vw; /* 相当于视口宽度的80% */ }
```

**开发建议：** 在实际开发中，为了统一管理尺寸，通常结合根元素动态设置字体大小（如用 JavaScript 将 `html` 的 `font-size` 设置为 `clientWidth / 10`），然后用 `rem` 进行布局。这样可以在保持设计比例的同时实现自适应。对于微调或固定布局用 `px`，对于自适应比例用 `rem/vw/vh`。在项目初期最好制定一致的单位使用规范。
