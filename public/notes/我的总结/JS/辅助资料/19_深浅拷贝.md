# 前端深浅拷贝（由浅到深全面讲解）
全程遵循**从基础到进阶、从概念到实战、从简单实现到复杂场景**的逻辑，先讲透底层基础，再依次讲解浅拷贝、深拷贝，最后落地实战选型和面试考点，所有代码示例极简可运行，新手也能一步步理解。

## 前置基础：搞懂这两点，才懂拷贝（最浅核心）
深浅拷贝的本质是**JavaScript对不同数据类型的内存处理规则**，这是所有拷贝知识的根基，先把这部分吃透，后续所有内容都会迎刃而解，无任何前置知识门槛。

### 1. JS的两大数据类型（按存储方式分）
| 类型 | 包含内容 | 存储位置 | 赋值规则 |
| --- | --- | --- | --- |
| 基本类型 | Number/String/Boolean/Null/Undefined/Symbol/BigInt | 栈内存 | 直接拷贝**值本身**，互不影响 |
| 引用类型 | Object/Array/Date/RegExp/Set/Map/Function 等 | 堆内存 | 栈存**地址指针**，堆存实际数据；赋值仅拷贝**地址**，共享堆内存 |


### 2. 最易混淆的「赋值」≠ 拷贝（核心痛点）
先看一个简单例子，理解**为什么需要拷贝**：

```javascript
// 1. 基本类型赋值：互不影响（无需拷贝）
let a = 18;
let b = a;
b = 20;
console.log(a); // 18 → b修改不影响a

// 2. 引用类型赋值：仅拷贝地址，共享堆内存（核心问题）
const obj1 = { name: '张三', info: { age: 18 } };
const obj2 = obj1; // 仅把obj1的堆内存地址赋值给obj2
obj2.name = '李四';
obj2.info.age = 20;
console.log(obj1); // { name: '李四', info: { age: 20 } } → 全变了！
console.log(obj1 === obj2); // true → 指向同一个堆内存
```

**核心痛点**：开发中我们需要**修改引用类型数据，但不影响原数据**，「赋值」做不到，这就需要**拷贝**来解决。

## 第一部分：浅拷贝（入门级，解决第一层问题）
先从最简单的**浅拷贝**入手，它是解决引用类型赋值问题的**基础方案**，先讲清定义、实现、效果，再讲它的局限性，为后续深拷贝做铺垫。

### 1. 浅拷贝的通俗定义（一句话懂）
创建一个**新的引用类型对象**，**仅拷贝原对象的第一层属性**：

+ 第一层是**基本类型**：拷贝「值本身」（和基本类型赋值一样，互不影响）；
+ 第一层是**引用类型**：仍拷贝「地址指针」（和原对象共享堆内存，修改会相互影响）。  
**核心作用**：解决**第一层基本类型**的引用问题，是最简单、性能最高的拷贝方式。

### 2. 浅拷贝的3种常用实现（从最简单到常用，逐个验证）
所有实现从**最简洁的扩展运算符**开始，代码极简，运行后可直观看到效果，建议跟着敲一遍。

#### （1）对象浅拷贝：扩展运算符 `{...obj}`（最常用，推荐）
```javascript
// 原数据：第一层有基本类型（name）和引用类型（info）
const obj1 = { name: '张三', info: { age: 18 } };
// 浅拷贝：用扩展运算符创建新对象
const obj2 = { ...obj1 };

// 验证1：修改第一层基本类型 → 原对象不受影响（浅拷贝的作用）
obj2.name = '李四';
console.log(obj1.name); // 张三 → 没问题！

// 验证2：修改第一层引用类型 → 原对象受影响（浅拷贝的局限性）
obj2.info.age = 20;
console.log(obj1.info.age); // 20 → 变了！
console.log(obj1 === obj2); // false → 是新对象（和单纯赋值的核心区别）
```

#### （2）对象浅拷贝：Object.assign({}, obj)（经典方式，和扩展运算符等价）
`Object.assign` 作用是**将源对象属性复制到目标对象**，当目标对象是空对象 `{}` 时，就是纯浅拷贝，效果和 `{...obj}` 完全一致：

```javascript
const obj1 = { name: '张三', info: { age: 18 } };
const obj2 = Object.assign({}, obj1); // 浅拷贝

obj2.name = '李四'; // 原对象不受影响
obj2.info.age = 20; // 原对象受影响
console.log(obj1); // { name: '张三', info: { age: 20 } } → 和扩展运算符结果一致
```

#### （3）数组浅拷贝：扩展运算符/ slice(0)/ concat()（3种方式，效果相同）
数组是特殊的对象，浅拷贝实现更简单，3种方式任选其一，核心效果和对象浅拷贝一致：**第一层基本类型互不影响，第一层引用类型相互影响**。

```javascript
const arr1 = [1, 2, { num: 3 }]; // 数组：第一层基本类型(1/2) + 引用类型({num:3})

// 方式1：扩展运算符（最推荐，简洁）
const arr2 = [...arr1];
// 方式2：slice(0)（无参数/0表示拷贝整个数组）
// const arr2 = arr1.slice(0);
// 方式3：concat()（无参数表示拼接空数组，返回新数组）
// const arr2 = arr1.concat();

// 验证
arr2[0] = 100; // 修改第一层基本类型 → 原数组不受影响
arr2[2].num = 300; // 修改第一层引用类型 → 原数组受影响
console.log(arr1); // [1, 2, { num: 300 }] → 符合浅拷贝特性
```

### 3. 浅拷贝的核心坑点（只记1条，永不踩坑）
**唯一坑点**：**浅拷贝只能处理第一层**，如果原数据有**嵌套的引用类型**（比如对象里的对象、数组里的对象），修改嵌套部分会直接影响原数据。  
**举例**：`obj1 = { a: { b: { c: 1 } } }`，浅拷贝后修改 `obj2.a.b.c`，原对象 `obj1.a.b.c` 一定会变，因为浅拷贝只拷贝了 `a` 的地址，`b`/`c` 仍共享内存。

### 4. 浅拷贝的适用场景（前端开发80%的场景都能用）
浅拷贝性能远高于深拷贝（无需递归，仅遍历第一层），满足以下场景直接用，不要无脑用深拷贝：

1. 引用类型数据**只有一层**（比如 `{ name: '张三', age: 18 }`/`[1,2,3]`）；
2. 只需修改**第一层的基本类型属性**，无需修改嵌套的引用类型；
3. 性能优先的场景（比如大数据列表、高频渲染的组件数据）。

## 第二部分：深拷贝（进阶级，解决所有层级问题）
在浅拷贝的基础上，自然引出**深拷贝**：既然浅拷贝解决不了**嵌套引用类型**的问题，那如何让**所有层级的引用类型**都互不影响？这就是深拷贝的核心作用。

### 1. 深拷贝的通俗定义（基于浅拷贝递进）
创建一个**新的引用类型对象**，**递归拷贝原对象的所有层级属性**：  
无论原数据嵌套多少层（对象里的对象、数组里的数组、对象里的数组），**每一层的引用类型**都会创建**新的堆内存空间**，拷贝后的数据和原数据**完全独立**，修改任何层级的属性，都不会影响原数据。  
**核心作用**：解决**所有层级引用类型**的修改问题，是彻底的拷贝方案（代价是性能比浅拷贝低）。

### 2. 深拷贝的4种实现（由浅到深，从简单到实用）
深拷贝的实现遵循**「先能用」→「再好用」→「最后定制化」**的逻辑，先讲最简单的快捷方案，再讲生产环境的稳定方案，最后讲手写进阶方案（面试必备），逐步递进。

#### （1）入门级：JSON.parse(JSON.stringify())（最简单，80%简单场景首选）
这是前端开发**最常用的快捷深拷贝方式**，无需手写复杂代码，一行搞定，先看用法和效果，再讲局限性（关键）。

##### 基本用法（解决嵌套问题，核心效果）
```javascript
// 原数据：两层嵌套的引用类型
const obj1 = { name: '张三', info: { age: 18, score: { math: 90 } } };
// 深拷贝：一行代码
const obj2 = JSON.parse(JSON.stringify(obj1));

// 验证：修改任意嵌套层级的属性，原对象都不受影响
obj2.name = '李四';
obj2.info.age = 20;
obj2.info.score.math = 100;
console.log(obj1); // { name: '张三', info: { age: 18, score: { math: 90 } } } → 完全独立！
console.log(obj1 === obj2); // false → 新对象
```

##### 核心局限性（必须记牢，否则必踩坑）
该方式通过**将对象转为JSON字符串，再转回对象**实现深拷贝，因此无法处理**非JSON标准的类型**，以下场景会失效/报错，这是面试高频考点：

1. 会**丢失函数、undefined、Symbol** 类型的属性；
2. 会把**Date对象转为字符串**（丢失Date的方法，比如getTime()）；
3. 会把**RegExp正则转为空对象 {}**；
4. 无法处理**循环引用**（比如 `obj1.self = obj1`，直接报错）；
5. 特殊数值**NaN/Infinity/-Infinity** 会被转为 `null`；
6. 不支持**BigInt** 类型（直接报错）。

**验证局限性（代码示例）**：

```javascript
const obj1 = {
  fn: () => console.log('函数'), // 函数
  und: undefined, // undefined
  sym: Symbol('test'), // Symbol
  date: new Date(), // Date
  reg: /abc/g, // 正则
  nan: NaN, // 特殊数值
  loop: null // 循环引用占位
};
obj1.loop = obj1; // 循环引用

// 尝试拷贝（直接报错：Converting circular structure to JSON）
// const obj2 = JSON.parse(JSON.stringify(obj1));

// 去掉循环引用后再试
delete obj1.loop;
const obj2 = JSON.parse(JSON.stringify(obj1));
console.log(obj2);
// 结果：{ date: "2026-02-05Txx:xx:xx.xxxZ", reg: {}, nan: null }
// 函数、undefined、Symbol 全部丢失！Date成字符串，正则成空对象，NaN成null
```

**适用场景**：嵌套引用类型的**简单数据**（无函数、无Date/RegExp、无循环引用），比如表单数据、接口返回的普通JSON数据。

#### （2）进阶级：structuredClone（浏览器原生API，比JSON方式强，无需第三方库）
ES2022新增的**浏览器原生深拷贝API**，专门解决JSON方式的局限性，无需引入任何库，直接调用，是**无库项目的首选深拷贝方案**。

##### 基本用法（和JSON方式一样简单，一行搞定）
```javascript
const obj1 = { name: '张三', info: { age: 18, score: { math: 90 } } };
const obj2 = structuredClone(obj1); // 原生深拷贝

obj2.info.score.math = 100;
console.log(obj1.info.score.math); // 90 → 完全独立
```

##### 核心优势（对比JSON方式，解决大部分局限性）
1. 支持**Date、RegExp、Set、Map、ArrayBuffer**（二进制数据）等特殊引用类型；
2. 原生支持**循环引用**（不会报错）；
3. 正确处理**NaN/Infinity/-Infinity**（不会转为null）。

**验证优势（代码示例）**：

```javascript
const obj1 = {
  date: new Date(),
  reg: /abc/g,
  set: new Set([1,2,3]),
  map: new Map([['key', 'value']]),
  nan: NaN,
  loop: null
};
obj1.loop = obj1; // 循环引用

const obj2 = structuredClone(obj1);
console.log(obj2.date); // Date对象（不是字符串）
console.log(obj2.reg); // /abc/g（不是空对象）
console.log(obj2.set); // Set(3) {1,2,3}（正常）
console.log(obj2.nan); // NaN（正常）
console.log(obj2.loop === obj2); // true（循环引用处理正常）
```

##### 剩余局限性（需注意）
1. 仍**不支持函数、Symbol** 类型的属性（会丢失/报错）；
2. 不支持**原型链属性**（无法拷贝对象的原型方法）；
3. 不支持**WeakSet/WeakMap** 类型；
4. 兼容性：Chrome 98+、Firefox 94+、Safari 15.4+、Node.js 18+（老浏览器/低版本Node不支持）。

**适用场景**：无第三方库的现代项目、浏览器端开发、需要处理Date/RegExp/循环引用的场景。

#### （3）生产级：lodash.cloneDeep（工业级方案，覆盖所有场景，推荐）
以上两种方案都有局限性，而**lodash的cloneDeep方法**是前端**生产环境的首选深拷贝方案**，成熟、稳定、覆盖**所有边界场景**（包括函数、特殊类型、循环引用），是各大框架/项目的通用方案。

##### 基本用法（需引入lodash，一行搞定）
```bash
# 先安装lodash（npm/yarn/pnpm任选）
npm install lodash
```

```javascript
// 引入
import _ from 'lodash';
// 或CDN引入（无需安装）
// <script src="https://cdn.jsdelivr.net/npm/lodash@4.17.21/lodash.min.js"></script>
// 深拷贝
const obj1 = { name: '张三', info: { age: 18 }, fn: () => console.log('函数'), loop: null };
obj1.loop = obj1;
const obj2 = _.cloneDeep(obj1);

// 验证：所有类型都正常，循环引用不报错
obj2.info.age = 20;
console.log(obj1.info.age); // 18 → 独立
console.log(obj2.fn); // [Function: fn] → 函数保留
console.log(obj2.loop === obj2); // true → 循环引用正常
```

##### 核心优势（工业级，无明显局限性）
1. 覆盖**所有引用类型**：函数、Date、RegExp、Set/Map、循环引用、BigInt等；
2. 处理**所有特殊数值**：NaN、Infinity、-Infinity等；
3. 体积小（单独引入cloneDeep仅2KB左右），性能稳定。  
**适用场景**：前端生产环境、企业级项目、需要处理复杂数据类型的场景（**推荐首选**）。

#### （4）面试级：手写深拷贝（从基础版到进阶版，由浅到深）
实际开发中直接用lodash/structuredClone即可，但**手写深拷贝是前端面试高频题**，要求从**基础版**（处理对象/数组）到**进阶版**（处理循环引用、特殊类型）逐步实现，这里分两步写，层层递进。

##### 手写基础版深拷贝（处理对象/数组，面试入门）
**核心思路**：

1. 判断数据类型，基本类型直接返回，引用类型创建新对象/数组；
2. 遍历原数据的所有属性，**递归调用深拷贝**，赋值给新对象/数组。

```javascript
/**
 * 基础版深拷贝：处理对象、数组、基本类型
 * @param {*} target 要拷贝的目标
 * @returns 拷贝后的新对象/数组
 */
function deepClone(target) {
  // 1. 基本类型/Null：直接返回值
  if (target === null || typeof target !== 'object') {
    return target;
  }

  // 2. 引用类型：创建新对象/数组
  const cloneTarget = Array.isArray(target) ? [] : {};

  // 3. 遍历属性，递归拷贝（核心：递归处理每一层）
  for (const key in target) {
    // 只拷贝自身属性，不拷贝原型链属性
    if (target.hasOwnProperty(key)) {
      cloneTarget[key] = deepClone(target[key]);
    }
  }

  return cloneTarget;
}

// 验证：处理嵌套对象/数组正常
const obj1 = { a: 1, b: { c: 2 }, d: [3, 4] };
const obj2 = deepClone(obj1);
obj2.b.c = 20;
obj2.d[0] = 30;
console.log(obj1); // { a: 1, b: { c: 2 }, d: [3, 4] } → 正常
```

**局限性**：无法处理循环引用、Date/RegExp/Set/Map等特殊类型，无法处理Symbol属性。

##### 手写进阶版深拷贝（处理循环引用+特殊类型，面试通关）
在基础版上，新增**3个核心优化**，覆盖面试要求的所有场景：

1. 用**WeakMap**做缓存，处理**循环引用**（避免递归栈溢出）；
2. 单独处理**Date、RegExp**等常见特殊类型；
3. 用**Reflect.ownKeys**遍历属性，支持**Symbol属性**和**不可枚举属性**。

```javascript
/**
 * 进阶版深拷贝：处理循环引用、Date/RegExp、Symbol、对象/数组、基本类型
 * @param {*} target 要拷贝的目标
 * @param {WeakMap} cache 缓存已拷贝对象，处理循环引用
 * @returns 拷贝后的新对象
 */
function deepClone(target, cache = new WeakMap()) {
  // 1. 基本类型/Null：直接返回
  if (target === null || typeof target !== 'object') {
    return target;
  }

  // 2. 处理循环引用：如果缓存中有，直接返回缓存的对象
  if (cache.has(target)) {
    return cache.get(target);
  }

  // 3. 处理特殊引用类型：Date/RegExp
  let cloneTarget;
  if (target instanceof Date) {
    cloneTarget = new Date(target);
  } else if (target instanceof RegExp) {
    cloneTarget = new RegExp(target.source, target.flags);
  } else {
    // 普通对象/数组：创建新实例
    cloneTarget = Array.isArray(target) ? [] : {};
  }

  // 4. 将新对象加入缓存，解决循环引用
  cache.set(target, cloneTarget);

  // 5. 遍历所有属性（包括Symbol、不可枚举属性），递归拷贝
  const keys = Reflect.ownKeys(target);
  keys.forEach(key => {
    cloneTarget[key] = deepClone(target[key], cache);
  });

  return cloneTarget;
}

// 验证：循环引用+特殊类型+Symbol都正常
const sym = Symbol('test');
const obj1 = {
  a: 1,
  b: { c: 2 },
  date: new Date(),
  reg: /abc/g,
  [sym]: 'symbol',
  loop: null
};
obj1.loop = obj1; // 循环引用

const obj2 = deepClone(obj1);
obj2.b.c = 20;
console.log(obj1.b.c); // 2 → 正常
console.log(obj2.date); // Date对象 → 正常
console.log(obj2[sym]); // symbol → 正常
console.log(obj2.loop === obj2); // true → 循环引用正常
```

**面试加分点**：用**WeakMap**而非**Map**做缓存，因为WeakMap的键是**弱引用**，不会阻止垃圾回收，避免内存泄漏。

### 3. 深拷贝的核心注意点（基于浅拷贝递进）
1. **性能代价**：深拷贝需要**递归遍历所有层级**，数据嵌套越深、体积越大，性能越低，因此**非必要不使用深拷贝**（能用水拷贝解决的就不用深拷贝）；
2. **响应式数据**：Vue的`reactive`/`ref`、React的`state`等响应式数据，深拷贝后会**丢失响应式**，需要重新用响应式API包装（比如Vue中`reactive(deepClone(obj))`）；
3. **特殊类型**：函数、Symbol、原型链属性等，大部分深拷贝方案不支持，需根据场景单独处理。

## 第三部分：深浅拷贝的核心对比（由浅到深总结）
用一张表把浅拷贝和深拷贝的核心差异总结，基于前面的知识，一眼就能看懂，重点记**拷贝层级**和**适用场景**：

| 对比维度 | 浅拷贝 | 深拷贝 |
| --- | --- | --- |
| 拷贝层级 | 仅拷贝**第一层**属性 | 递归拷贝**所有层级**属性 |
| 引用类型处理 | 第一层引用类型拷贝**地址**（共享内存） | 所有层级引用类型创建**新堆内存**（完全独立） |
| 性能 | 高（仅遍历第一层，无递归） | 低（递归遍历所有层级） |
| 修改影响 | 嵌套引用类型修改会影响原数据 | 任何层级修改都不影响原数据 |
| 实现复杂度 | 简单（一行代码） | 复杂（需递归/处理特殊场景） |
| 适用场景 | 一层数据、性能优先场景 | 多层嵌套引用类型、需要彻底独立的场景 |

## 第四部分：实战选型策略（由浅到深落地，前端必备）
开发中如何选择拷贝方式？**遵循「能用浅拷贝不用深拷贝，能用简单深拷贝不用复杂深拷贝」****的原则，根据****数据结构**和**场景**选择，以下是前端开发99%的场景选型，直接套用即可：

### 1. 选浅拷贝的场景（80%）
+ 数据只有**一层**（比如 `{ name: '张三', age: 18 }`/`[1,2,3]`）；
+ 只需修改**第一层基本类型**，无需修改嵌套引用类型；
+ 大数据量、高频渲染的性能优先场景（比如列表数据、表格数据）。  
**推荐实现**：对象用`{...obj}`，数组用`[...arr]`。

### 2. 选深拷贝的场景（20%）
#### 场景1：简单嵌套数据（无函数、无Date/RegExp、无循环引用）
**推荐实现**：`JSON.parse(JSON.stringify())`（一行搞定，无需引库）。

#### 场景2：现代项目/浏览器端（无第三方库，需处理Date/RegExp/循环引用）
**推荐实现**：`structuredClone`（原生API，比JSON方式强）。

#### 场景3：生产环境/企业级项目（复杂数据、特殊类型、无兼容性顾虑）
**推荐实现**：`lodash.cloneDeep`（工业级，覆盖所有场景，**首选**）。

#### 场景4：面试/定制化需求（需要手动处理特殊类型）
**推荐实现**：手写进阶版深拷贝（处理循环引用+常见特殊类型）。

## 第五部分：高频面试题（由浅到深，从基础到深挖）
基于前面的知识，面试题也遵循**由浅到深**的逻辑，先考基础概念，再考进阶实现，所有答案都能在前面找到，记牢即可：

### 基础题（必答，送分题）
1. **赋值、浅拷贝、深拷贝的区别？**
    - 赋值：仅拷贝地址，新旧对象指向同一堆内存，修改相互影响，无新对象创建；
    - 浅拷贝：创建新对象，仅拷贝第一层，第一层基本类型独立，嵌套引用类型共享内存；
    - 深拷贝：创建新对象，递归拷贝所有层级，所有属性完全独立，修改互不影响。
2. **浅拷贝的常用实现方式有哪些？**
    - 对象：扩展运算符`{...obj}`、`Object.assign({}, obj)`；
    - 数组：扩展运算符`[...arr]`、`slice(0)`、`concat()`。
3. **JSON.parse(JSON.stringify())的局限性有哪些？**
    - 丢失函数、undefined、Symbol；
    - Date转字符串，RegExp转空对象；
    - 无法处理循环引用，报错；
    - NaN/Infinity转为null，不支持BigInt。

### 进阶题（高频，区分度题）
1. **为什么手写深拷贝要用WeakMap做缓存？**
    - 核心：处理**循环引用**，避免递归栈溢出；
    - 加分：WeakMap的键是**弱引用**，不会阻止垃圾回收，避免内存泄漏（Map是强引用，会导致原对象无法被回收）。
2. **structuredClone相比JSON方式有哪些优势？**
    - 支持Date、RegExp、Set、Map、ArrayBuffer等特殊类型；
    - 原生支持循环引用，不报错；
    - 正确处理NaN/Infinity/-Infinity，不转为null。
3. **Vue的reactive对象深拷贝后为什么会丢失响应式？如何解决？**
    - 原因：深拷贝会创建新的普通对象，脱离了Vue的响应式代理；
    - 解决：深拷贝后重新用`reactive`/`ref`包装，比如`const newObj = reactive(_.cloneDeep(obj))`。

## 最终总结（由浅到深梳理核心）
1. **底层基础**：JS数据类型分栈存储的基本类型和堆存储的引用类型，**赋值仅拷贝引用类型的地址**，这是拷贝的核心痛点；
2. **浅拷贝**：入门级方案，解决第一层基本类型的引用问题，一行代码实现，性能最高，适用于80%的场景；
3. **深拷贝**：进阶级方案，解决所有层级引用类型的问题，从简单的JSON方式到原生的structuredClone，再到生产级的lodash.cloneDeep，最后到面试级的手写实现，由浅到深选择；
4. **实战原则**：**能浅不深**，性能优先，根据数据结构和场景选择最合适的方案，不无脑用深拷贝。
