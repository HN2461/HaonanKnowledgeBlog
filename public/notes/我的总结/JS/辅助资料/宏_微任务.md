# 前端异步操作「宏/微任务」精细化拆解（含同步/异步边界+执行时序，面试吃透版）
核心前提：**所有DOM操作、网络请求、Promise、定时器等，「本身执行代码」和「异步回调」要严格区分**——多数操作的**主逻辑是同步**，只有**后续的回调/触发的异步行为**才归为宏/微任务，这是最易踩坑的点！  
另外前端无原生IO操作，Node端IO（文件/流）属于宏任务，面试只需提一句区分即可，重点聚焦前端常用操作。

## 一、宏任务（Macrotask）：浏览器/node 主线程「分批执行」的异步任务，优先级低于微任务
每轮事件循环**仅执行1个宏任务**，执行完立刻清空所有微任务，再进行页面渲染（浏览器），最后取下一个宏任务，**前端高频宏任务**按面试常考度排序，精细化拆解如下：

### 1. 定时器/延时器：setTimeout、setInterval；
+ 同步部分：`setTimeout(()=>{}, 0)` 这行代码**执行本身是同步**，作用只是「把回调函数加入宏任务队列」，并设置延迟执行规则；
+ 异步归属于：**回调函数**是宏任务，即使延迟0ms，也不会立刻执行，必须等当前同步代码+所有微任务执行完，才会进入执行队列。
+ 关键考点：定时器的延迟时间是「**最早执行时间**」，而非「精准执行时间」，受前面任务执行时长影响。

:::warning
**启动定时器是同步，里面的回调是异步**

:::

### 2. 网络请求：XHR/fetch/axios/ajax
+ 同步部分：`fetch(url)`/`xhr.open()`/`xhr.send()` 这些**<font style="color:#DF2A3F;">发送请求的代码是同步</font>**，浏览器瞬间完成请求发送的指令操作；
+ 异步归属于：**<font style="color:#DF2A3F;">请求成功（onload/then）、失败（onerror/catch）、进度变化的回调</font>**是宏任务，回调的执行时机由「网络响应完成时间」和「事件循环队列」共同决定；
+ 补充：axios是XHR的封装，fetch是原生新API，二者的回调均为宏任务，无区别。

:::warning
**<font style="color:#DF2A3F;">发送请求的代码是同步,请求成功（onload/then）、失败（onerror/catch）、进度变化的回调是宏任务</font>**

:::

### 3. 浏览器UI页面渲染：布局（Layout）、绘制（Paint）
+ 无「同步代码」一说，属于浏览器**自带的宏任务行为**，执行时机固定：**每轮宏任务执行完毕 + 所有微任务清空后**，浏览器会判断是否需要渲染（如DOM样式/结构有变化），若需要则执行渲染，再开始下一轮事件循环；
+ 关键考点：渲染是宏任务，但**不会进入宏任务队列**，是浏览器的「内置步骤」，优先级高于下一个宏任务，这也是「微任务执行完会先渲染，再执行下一个宏任务」的原因。

### 4. DOM相关异步事件的回调
+ 同步部分：**<font style="color:#DF2A3F;">直接的DOM操作本身</font>**（如`document.getElementById('app')`、`el.innerHTML = '123'`、`el.onclick = function(){}）`是同步代码，执行时立刻修改/获取DOM，无异步；
+ 异步归属于：**<font style="color:#DF2A3F;">DOM操作触发的异步事件回调</font>**是宏任务，面试常考：`DOMContentLoaded`（DOM树构建完成回调）、`load`（页面所有资源加载完成回调）、点击/输入/鼠标移动等DOM事件回调（如`onclick`、`oninput`、`onchange`）、`onresize`（窗口大小变化）。
+ 举例：给按钮绑定`onclick = () => { console.log(1) }`是同步，**点击按钮后触发的回调函数执行**是宏任务。

:::warning
**<font style="color:#DF2A3F;">同步部分：直接的DOM操作本身;异步归属于：DOM操作触发的异步事件回调是宏任务</font>**

:::

### 5. 补充：Node端IO操作（前端面试仅作区分）
文件读写、流操作、数据库操作等IO操作，**本身执行是同步**，**成功/失败的回调**是Node端核心宏任务，前端无原生IO，无需深入记忆。

## 二、微任务（Microtask）：「本轮事件循环必清完」的异步任务，优先级最高
微任务队列的特点是**一次性清空**——只要微任务队列中有任务，就会依次执行，直到队列空为止，期间不会执行宏任务、不会进行页面渲染，**前端所有微任务均和Promise强相关，或为浏览器专属的DOM监听**，无其他类型，精细化拆解：

### 1. Promise 所有相关回调
+ 同步部分：`new Promise((resolve, reject) => { // 这里的代码是同步 })`，Promise构造函数内的执行器函数，会在创建Promise时**立刻同步执行**，哪怕里面有耗时操作（如循环），也会阻塞主线程；
+ 异步归属于：`Promise.then()`/`catch()`/`finally()` 回调、`Promise.all()`/`race()`/`allSettled()`/`any()` 当状态确定后的回调，**全部是微任务**；
+ 关键考点：只有Promise的**状态发生改变（pending→fulfilled/rejected）** 后，对应的回调才会被加入微任务队列，若状态已提前确定（如`Promise.resolve()`），则创建时直接加入微任务队列。

:::warning
**<font style="color:#DF2A3F;">同步部分：Promise构造函数内代码; </font>**

**<font style="color:#DF2A3F;">异步归属于：</font>**`**<font style="color:#DF2A3F;">Promise.then()</font>**`**<font style="color:#DF2A3F;">/</font>**`**<font style="color:#DF2A3F;">catch()</font>**`**<font style="color:#DF2A3F;">/</font>**`**<font style="color:#DF2A3F;">finally()</font>**`**<font style="color:#DF2A3F;"> 回调、</font>**`**<font style="color:#DF2A3F;">Promise.all()</font>**`**<font style="color:#DF2A3F;">/</font>**`**<font style="color:#DF2A3F;">race()</font>**`**<font style="color:#DF2A3F;">/</font>**`**<font style="color:#DF2A3F;">allSettled()</font>**`**<font style="color:#DF2A3F;">/</font>**`**<font style="color:#DF2A3F;">any()</font>**`**<font style="color:#DF2A3F;"> 当状态确定后的回调，</font>****全部是微任务****<font style="color:#DF2A3F;">；</font>**

:::

### 2. async/await 语法糖（本质是Promise.then）
+ 同步部分：`async function fn() {}` 函数声明是同步，`await` 左侧的代码是同步（如`await fn1()`，先同步执行`fn1()`）；
+ 异步归属于：**await 后面的所有代码**（即await的「后续执行体」）是微任务，等价于`Promise.then()`的回调；
+ 举例：`async () => { let a = await 1; console.log(a) }`，`await 1` 等价于`Promise.resolve(1)`，`console.log(a)` 是微任务。

:::warning
同步部分：`<font style="color:rgb(0, 0, 0);">async</font>`<font style="color:rgba(0, 0, 0, 0.85);"> 函数</font>**<font style="color:rgb(0, 0, 0) !important;">调用后</font>**<font style="color:rgba(0, 0, 0, 0.85);">，其内部</font>`<font style="color:rgb(0, 0, 0);">await</font>`<font style="color:rgba(0, 0, 0, 0.85);"> 之前的代码；</font>`<font style="color:rgb(0, 0, 0);">await</font>`<font style="color:rgba(0, 0, 0, 0.85);"> 右边是「普通值（123、'abc'）」还是「Promise 对象」，这个「表达式本身」都会</font>**<font style="color:rgb(0, 0, 0) !important;">同步执行</font>**<font style="color:rgba(0, 0, 0, 0.85);">；</font>

异步：`<font style="color:rgb(0, 0, 0);">await</font>`<font style="color:rgba(0, 0, 0, 0.85);"> 后面的代码会被「包装成微任务」，加入微任务队列，等待主线程和当前微任务队列清空后执行。</font>

:::

### 3. queueMicrotask() 手动创建微任务
浏览器原生API，**专门用于手动将一个函数加入微任务队列**，作用和Promise.then一致，但更直观，无同步部分，调用即把回调加入微任务队列。

+ 示例：`queueMicrotask(() => { console.log('微任务') })`，回调直接作为微任务执行。

### 4. MutationObserver DOM变化监听
+ 同步部分：`new MutationObserver(callback)` 实例化是同步，`observe(el, config)` 开启监听是同步；
+ 异步归属于：**当监听的DOM结构/属性发生变化时，触发的回调函数**是微任务；
+ 特点：DOM多次变化会被浏览器**批量监听**，最终只触发一次回调，回调执行在微任务阶段，这也是它能用来「监听DOM渲染完成」的原因（比宏任务更及时）。

## 三、核心执行时序（结合所有操作，面试必背完整流程）
以「同步代码+DOM操作+Promise+定时器+网络请求+渲染」为例，**完整的一轮事件循环**执行步骤如下，这是面试回答的核心逻辑，记死即可：

1. 执行**所有同步代码**（**<font style="color:#DF2A3F;">直接DOM操作、Promise构造函数内代码、定时器/网络请求的发送代码、变量声明等</font>**），执行过程中，将遇到的**宏任务回调**加入宏任务队列，**微任务回调**加入微任务队列；
2. 同步代码执行完毕，**立刻执行微任务队列的所有任务**（**<font style="color:#DF2A3F;">一次性清空</font>**，包括Promise.then、await后续、queueMicrotask、MutationObserver回调），<font style="color:#DF2A3F;">若微任务执行中又产生新的微任务，继续加入队列并执行，直到微任务队列为空；</font>
3. 微任务队列清空后，**浏览器执行UI页面渲染**（布局+绘制，若DOM无变化则跳过）；
4. 渲染完成后，从**宏任务队列**中**取出第一个任务执行**（如定时器回调、网络请求成功回调、DOM事件回调）；
5. 执行完这个宏任务后，**再次回到步骤2**，清空所有微任务→渲染→取下一个宏任务，循环往复（即「事件循环」）。

## 四、经典例题（结合所有操作，验证理解）
```javascript
// 同步代码
console.log('1-同步开始');
// 直接DOM操作（同步）
const app = document.getElementById('app');
app.innerHTML = '测试DOM';
// 定时器（同步发送，回调宏任务）
setTimeout(() => {
  console.log('2-定时器宏任务');
}, 0);
// 网络请求（同步发送，回调宏任务）
fetch('https://xxx.com/api')
  .then(res => {
    console.log('3-网络请求宏任务');
  });
// Promise（构造函数同步，then微任务）
new Promise((resolve) => {
  console.log('4-Promise构造函数同步');
  resolve();
}).then(() => {
  console.log('5-Promise.then微任务');
  // 微任务中创建新微任务
  queueMicrotask(() => {
    console.log('6-手动微任务');
  });
});
// async/await（本质微任务）
async function fn() {
  console.log('7-await前同步');
  await Promise.resolve();
  console.log('8-await后微任务');
}
fn();
console.log('9-同步结束');
```

### 执行结果（严格按时序）
1-同步开始 → 4-Promise构造函数同步 → 7-await前同步 → 9-同步结束 → 5-Promise.then微任务 → 6-手动微任务 → 8-await后微任务 → 页面渲染 → 2-定时器宏任务 → 3-网络请求宏任务（若网络先响应则先执行，否则后执行）

### 核心原因
同步代码先全部执行 → 清空所有微任务（**<font style="color:#DF2A3F;">包括微任务中新增的微任务</font>**） → 渲染 → 再执行宏任务（定时器和网络请求的回调执行顺序由「队列加入时间」决定，定时器是同步代码执行完立刻加入，网络请求需等响应完成后才加入）。

### <font style="color:rgb(0, 0, 0);">简化记忆：async/await 执行口诀</font>
1. <font style="color:rgb(0, 0, 0);">先跑「await 之前的同步代码」+「await 右边的表达式（同步）」；</font>
2. <font style="color:rgb(0, 0, 0);">遇到 await，暂停 async 函数，先跑「主线程剩余同步代码」；</font>
3. <font style="color:rgb(0, 0, 0);">等 await 的结果（Promise 状态改变），把「await 后面的代码」入微任务队列；</font>
4. <font style="color:rgb(0, 0, 0);">主线程空了 → 执行微任务 → 执行宏任务（定时器等）。</font>
