## uni-app 性能优化、包体治理与稳定性维护

> 真正做过一段时间 uni-app 项目的人，最后一定会发现：  
> 最难搞的从来不是“页面会不会写”，而是“为什么上线后越来越慢、越来越乱、越来越难改”。  
> 性能和稳定性这两块，往往是最能拉开项目维护水平差距的地方。  
> 因为它们不是某一个 API 能解决的，而是需要你对页面时序、资源、状态、宿主差异都有整体判断。

## 一、先把性能问题从“点状技巧”升级成“成本视角”
很多人一说性能优化，就会条件反射地想到：

+ 压缩图片
+ 减少请求
+ 加骨架屏

这些当然有用，但如果只停在这里，说明你看的还是“技巧”，不是“系统成本”。

我更建议把性能问题拆成 4 类成本：

| 成本类型 | 常见表现 | 典型排查方向 |
| --- | --- | --- |
| 包体成本 | 冷启动慢、首次进入慢、审核前突然超包 | 主包是否过大、低频资源是否混进主包 |
| 初始化成本 | 首屏白屏、页面进入慢、`onLoad` 特别重 | 页面刚进入时做了多少同步动作 |
| 渲染成本 | 长列表卡顿、频繁更新卡顿 | 视图层更新量、节点数量、更新频率 |
| 资源成本 | 图片大、内存高、真机卡 | 图片、音视频、静态资源组织方式 |

这个视角很重要，因为它能帮你少走很多弯路。  
有些“慢”根本不是接口慢，而是页面刚进来做了太多不该在首屏做的事。

## 二、包体治理为什么总是绕不过去
尤其是做微信小程序或类似宿主环境时，包体大小对冷启动体验影响很大。

### 1. 主包为什么要尽量轻
因为首次进入应用时，主包就是第一层成本。  
主包越重：

+ 冷启动越慢
+ 首次进入等待越长
+ 包体限制越容易踩线

### 2. 更稳的分包思路
我更推荐按“访问频率”来拆，而不是按“页面名好不好看”来拆。

优先留在主包的通常是：

+ 首页
+ 登录链路核心页
+ 高频列表页
+ 高频详情页

更适合放分包的通常是：

+ 低频活动页
+ 设置页
+ 后台管理页
+ 某个边缘业务模块

### 3. 分包不是补丁，而是页面组织的一部分
很多人等到包体爆了才想起分包，这时候往往已经很难拆。  
更稳的做法是一开始就规划：

+ 哪些页面属于高频路径
+ 哪些页面属于低频路径
+ 哪些资源只在分包业务里使用

## 三、首屏为什么会慢
很多页面慢，真不一定是接口慢。  
更常见的是：页面一进来做了太多同步动作。

典型问题包括：

+ `onLoad` 里并发发多个请求
+ 还没渲染就先做大量数据加工
+ 非关键模块和关键模块一起加载
+ 首屏一次性塞进大量图片或长列表

### 更稳的思路
1. 先渲染首屏关键内容
2. 非关键请求后置
3. 重计算延后
4. 长列表按需加载

这背后的核心思路其实只有一句：

`首屏阶段先保证“尽快可见”，而不是一口气把所有事全做完。`

## 四、长列表为什么是高频性能坑
长列表之所以容易卡，通常同时叠加了两类成本：

+ 节点太多
+ 数据更新太大、太频繁

### 常见优化手段
+ 分页加载
+ 触底再拉下一页
+ 控制单次更新量
+ 减少整数组整体替换
+ 避免滚动过程中频繁大范围更新

示例：

```javascript
export default {
  data() {
    return {
      list: [],
      page: 1,
      loading: false,
      finished: false
    }
  },
  onReachBottom() {
    if (this.loading || this.finished) return
    this.loadMore()
  },
  methods: {
    loadMore() {
      this.loading = true
      uni.request({
        url: `/api/list?page=${this.page + 1}`,
        success: (res) => {
          const nextList = res.data.list || []
          if (!nextList.length) {
            this.finished = true
            return
          }
          this.page += 1
          this.list = this.list.concat(nextList)
        },
        complete: () => {
          this.loading = false
        }
      })
    }
  }
}
```

## 五、为什么小程序体系下总会提到“通信成本”
因为在小程序体系里，逻辑层和渲染层之间的数据同步是有成本的。  
`uni-app` 做了封装，但底层的通信成本并没有神奇消失。

这就意味着：

+ 一次更新的数据太大，会增加同步压力
+ 更新频率太高，也会造成卡顿
+ 高频事件里频繁推大对象，很容易让页面发沉

所以你要避免：

+ 每次都整对象覆盖
+ 无脑整数组替换
+ 滚动或输入场景里频繁大范围更新数据

## 六、资源治理不要只盯图片压缩
图片当然重要，但资源治理并不只等于图片压缩。

还要同时看：

+ 图片是否原图直出
+ 是否按场景裁剪尺寸
+ 是否存在低频资源混进高频页面
+ 是否有大量资源被主包提前背上

如果是音视频或富媒体页面，还要额外注意：

+ 加载时机
+ 自动播放策略
+ 页面隐藏后的暂停与释放

## 七、为什么我会把“稳定性维护”放得和性能一样重
因为项目做久了你会发现：

`很多最烦人的线上问题，不是慢，而是乱。`

比如：

+ 登录态偶发失效
+ 页面回流后数据不对
+ 前后台切换后状态错乱
+ 权限明明授权过却还是取不到
+ 本地缓存结构升级后旧数据污染新逻辑

这些问题单看都像小 bug，但积累起来会非常难维护。

## 八、稳定性最容易混掉的是哪几套状态
我建议至少拆成下面 4 套来理解：

| 状态层 | 典型内容 | 常见风险 |
| --- | --- | --- |
| 微信登录态 | `code`、会话有效性 | 失效后要重新换取 |
| 业务登录态 | `token`、刷新机制 | 过期、踢下线、校验失败 |
| 本地缓存态 | 本地存储数据、草稿、历史信息 | 覆盖、污染、结构不兼容 |
| 权限状态 | 位置、手机号、隐私授权 | 用户拒绝、系统设置变更、恢复逻辑混乱 |

### 为什么一定要拆开
因为它们的：

+ 来源不同
+ 失效方式不同
+ 恢复方式不同

如果你脑子里只分“登录了没”“缓存有没有”，项目一复杂就会彻底写乱。

## 九、为什么登录态不能只靠本地缓存判断
因为缓存只能说明：

`本地曾经存过这个值`

它并不能说明：

+ 这个 token 现在还有效
+ 微信会话还没过期
+ 后端没有把你踢下线
+ 权限状态仍然和上次一致

更稳的做法一般是：

1. 启动应用先恢复本地缓存
2. 进入关键页面前校验业务登录态
3. 前后台切换时重新判断关键状态
4. 失效后走重新登录或刷新机制

## 十、前后台切换为什么特别容易出问题
因为你把应用切到后台的这段时间里，很多状态其实已经可能变化了。

例如：

+ token 过期
+ 权限被用户去系统设置里改掉
+ 某些页面参数已经失效
+ 业务数据已在别处更新

所以应用回前台时，绝不能默认“之前状态还完全可用”。

这也是为什么：

+ 应用级 `onShow`
+ 页面级 `onShow`

在稳定性治理里都很重要。

## 十一、性能优化和稳定性维护为什么应该一起看
因为真实项目里，它们往往不是两件完全分开的事。

比如：

+ 页面进来做太多校验，首屏会慢
+ 不做状态校验，回流又容易错
+ 缓存读写太粗暴，既可能卡，也可能污染状态

真正成熟的做法不是偏一头，而是在下面几条线之间找平衡：

+ 包体
+ 首屏
+ 渲染
+ 资源
+ 登录态
+ 本地缓存
+ 权限恢复

## 十二、这一篇最后要沉淀的核心共识
如果你读完只记住关键点，我建议记下面这些：

1. 性能问题不要只盯技巧，要先拆成包体、初始化、渲染、资源 4 类成本。
2. 分包越晚做越被动，最好一开始就按高频 / 低频路径规划。
3. 首屏优化的核心不是“把所有事都做完”，而是“先尽快可见”。
4. 长列表的核心问题通常是节点数量和更新频率叠加。
5. 小程序体系下要始终意识到逻辑层和渲染层的数据通信成本。
6. 稳定性最容易写乱的是微信登录态、业务登录态、本地缓存态、权限状态这 4 套状态。
7. 真正长期可维护的项目，性能优化和状态治理必须一起考虑。
