## uni-app 核心认知、跨端编译与平台边界

> 这一篇不只是在回答“uni-app 是什么”，更重要的是把 `uni-app 项目到底该怎么理解` 这件事一次说清楚。  
> 很多人学 `uni-app` 会卡住，不是因为 API 太多，而是因为一开始对它的定位就错了。  
> 如果你把它误解成“Vue 页面换个壳”，或者误解成“写一套就能自动完全抹平所有平台差异”，后面越学越别扭。

## 一、先把一个最重要的结论说清楚
`uni-app` 不是一个单独的平台，而是一套：

+ 跨端页面开发规范
+ 跨端 API 抽象
+ 多端编译输出体系
+ 平台差异收口方案

也就是说，你写的不是“微信小程序原生代码”，也不是“纯浏览器 Vue 页面”，而是一套遵循 `uni-app` 规范的跨端代码。  
这些代码在构建时会被编译成不同平台可运行的产物，最后再运行在对应宿主环境里。

这句话里一定要同时包含两个重点：

1. `uni-app` 解决的是“统一开发方式”
2. `uni-app` 并没有让所有平台真的变成同一个东西

## 二、很多人为什么会从一开始就理解错
最常见的两个误区是：

### 1. 误区一：我写的是 Vue，所以本质就是 Web 项目
这只说对了一半。

确实，`uni-app` 页面开发会带有明显的 Vue 风格，你会写：

+ `template`
+ `script`
+ `style`
+ 组件
+ 响应式数据
+ 事件绑定

但它最终并不一定运行在浏览器标准环境里。  
你写的是“Vue 风格的跨端页面”，不等于它最终就是普通浏览器页面。

### 2. 误区二：我最终发的是小程序 / App，所以 uni-app 只是语法糖
这也不完整。

因为 `uni-app` 不只是把代码翻译一下，它还统一了：

+ 页面组织方式
+ API 使用方式
+ 工程配置思路
+ 条件编译能力
+ 多端差异处理入口

如果没有这层规范，你做多端时会很快陷入“每个端一套写法、每个端一套工程结构、每个端一套工具链”的维护地狱。

## 三、把 uni-app 固定理解成 4 层，会清晰很多
这是我认为最重要的认知方式。

| 层级 | 你平时接触的内容 | 典型问题 | 优先查哪里 |
| --- | --- | --- | --- |
| 业务代码层 | 页面、组件、表单、列表、状态、请求 | 逻辑错、事件错、渲染错 | 业务代码本身 |
| uni-app 规范层 | `uni.*` API、页面生命周期、`pages.json`、条件编译 | 写法越界、跨端能力误用 | uni-app 官方文档 |
| 编译产物层 | 构建结果、平台工程、资源打包、配置映射 | 页面没进包、配置没落对 | 编译目标与构建配置 |
| 平台宿主层 | H5 浏览器、微信小程序宿主、App 容器 | 权限、登录、支付、性能、平台限制 | 对应平台官方文档 |

### 为什么这 4 层很重要
因为你以后一旦遇到问题，排错速度快不快，取决于你能不能先判断它属于哪一层。

比如：

+ 页面写了但跳不过去
+ 同一个 API 在 H5 可以，在微信小程序不行
+ 某个平台打包后资源丢失
+ 开发者工具里正常，真机上卡顿明显

这时不要立刻说“uni-app 有 bug”，而应该先判断：

1. 是业务代码自己写错了
2. 是 `uni-app` 规范边界没踩对
3. 是构建配置或编译产物有问题
4. 还是目标宿主本身就有限制

## 四、uni-app 真正统一了什么
`uni-app` 从来不是“所有东西都一模一样”，它真正统一的是“开发方式”，主要体现在下面 3 个层面。

### 1. 页面与组件的组织方式
你可以用一套相对统一的结构组织页面和组件，比如：

+ 页面目录
+ 组件拆分
+ `props`
+ 自定义事件
+ 常见表单、列表、卡片、弹窗页面

这些内容在多端里复用价值最高。

### 2. 一批高频通用 API
例如：

+ `uni.request`
+ `uni.navigateTo`
+ `uni.redirectTo`
+ `uni.setStorageSync`
+ `uni.getStorageSync`
+ `uni.uploadFile`
+ `uni.downloadFile`
+ `uni.showToast`
+ `uni.showModal`

这些 API 的意义不是“所有平台 100% 一模一样”，而是：

`先提供统一入口，让你能优先写通用代码。`

### 3. 一套统一的工程配置思路
例如：

+ `App.vue` 管应用入口和应用生命周期
+ `pages.json` 管页面注册和页面级配置
+ `manifest.json` 管应用级和平台级配置
+ `static` 管静态资源
+ 条件编译用来处理平台差异

这种统一的工程组织方式，在真实项目里非常重要，因为它决定了后期项目会不会越来越乱。

## 五、uni-app 为什么能跨端
从本质上说，因为它做了两件事：

### 1. 约束开发输入
也就是限制你“应该怎么写”。

你不是随便写任何浏览器原生能力、任何第三方 DOM 库、任何平台私有接口都指望它自动兼容，而是要尽量写在 `uni-app` 认可的规范范围里。

### 2. 针对目标平台做编译输出
也就是构建时根据目标端把代码转成对应产物。

可以把它粗略理解成：

`统一开发代码 -> 针对不同目标端编译 -> 运行在不同宿主里`

所以 `uni-app` 的跨端不是“同一份原始代码原封不动跑 everywhere”，而是“同一套开发方式 + 不同平台输出产物”。

## 六、宿主环境这个概念，一定要真正吃透
很多人学到后面会越来越混乱，就是因为只记住了 API，没记住“最终是谁在运行你的代码”。

### 1. H5
运行在浏览器里，受浏览器环境、跨域、安全策略、DOM、CSS 渲染规则影响。

### 2. 微信小程序
运行在微信宿主里，不是普通浏览器页面。  
它有自己的：

+ 登录能力
+ 支付能力
+ 页面栈规则
+ 分包限制
+ 审核规则
+ 组件协议和开放能力

### 3. App
运行在 App 容器里，通常和 WebView、原生容器、平台权限声明、打包配置相关。

### 为什么宿主环境重要
因为同样一段业务逻辑：

+ 在浏览器能写成一种方式
+ 到微信小程序会遇到宿主协议限制
+ 到 App 又会遇到权限、打包、容器能力的问题

这也是为什么跨端开发一定要承认差异存在。

## 七、uni-app 为什么不可能完全抹平平台差异
这不是 `uni-app` 做得不够好，而是问题本身就不可能被“完全消灭”。

### 1. 能力边界不同
比如：

+ 微信小程序有手机号授权、订阅消息、`web-view` 通信等宿主能力
+ 浏览器没有这些一模一样的能力
+ App 又会有另一套能力和权限模型

### 2. 渲染和性能模型不同
同一个页面：

+ 在浏览器可能主要是 DOM/CSS 性能问题
+ 在小程序更容易碰到逻辑层和渲染层通信成本问题
+ 在 App 又可能叠加容器渲染和平台资源开销

### 3. 配置和发布规则不同
例如：

+ 页面栈限制
+ 主包/分包规则
+ appid / 包名 / 签名
+ 网络白名单
+ 平台审核要求

这些都不是单靠 `uni.*` 就能“自动等价”的。

## 八、为什么真实项目里不能到处散写 `wx.*`
很多人一看到“目标端是微信小程序”，就会本能地在页面里直接写：

+ `wx.login`
+ `wx.requestPayment`
+ `wx.requestSubscribeMessage`
+ `wx.chooseMessageFile`

这不是绝对错，但如果你长期维护项目，这样写通常很危险。

### 1. 平台耦合会越来越重
一旦页面到处散着写 `wx.*`，以后你要：

+ 迁 H5
+ 迁 App
+ 做通用封装
+ 替换业务流程

都会很痛苦。

### 2. 平台差异会扩散到业务层
本来页面只该关心“我要登录”“我要上传”“我要发起支付”，结果最后页面自己要懂微信 API 细节、参数结构、错误处理和授权链路。

### 3. 后期治理难
项目刚开始只有两个页面时你可能不觉得，项目一旦写到几十个页面，平台专属代码散得满地都是，维护成本会指数级变差。

## 九、更稳的差异处理方式是什么
我更推荐按下面 3 层去拆：

### 1. 业务层
业务层只描述“我要什么能力”。

例如：

+ 登录
+ 上传
+ 支付
+ 分享
+ 取定位

### 2. 通用能力层
优先用 `uni.*` 封装通用 service。

比如：

```javascript
export function request(url, options = {}) {
  return new Promise((resolve, reject) => {
    uni.request({
      url,
      ...options,
      success: resolve,
      fail: reject
    })
  })
}
```

### 3. 平台差异层
确实属于微信专属能力的，再通过条件编译或适配层收口。

```javascript
// #ifdef MP-WEIXIN
export function requestWechatSubscribeMessage(tmplIds) {
  return wx.requestSubscribeMessage({ tmplIds })
}
// #endif
```

这个结构的重点不是代码多高级，而是：

+ 通用逻辑尽量共用
+ 差异能力集中管理
+ 页面不要直接承受所有平台细节

## 十、普通 Vue 项目和 uni-app 项目，真正的差别在哪里
很多人会说“都是 Vue，有什么区别”，其实差别非常大。

| 维度 | 普通 Vue Web 项目 | uni-app 项目 |
| --- | --- | --- |
| 目标环境 | 主要是浏览器 | H5、小程序、App 等多宿主 |
| API 入口 | 浏览器 API、第三方 Web 库 | 优先 `uni.*` 和平台适配 |
| 工程目标 | 构建 Web 产物 | 构建多个目标端产物 |
| 差异处理 | 浏览器兼容、接口差异 | 平台能力差异、宿主边界、发布规则 |
| 排错思路 | 多数问题停留在 Web 层 | 必须同时看规范层、编译层、宿主层 |

所以说，`uni-app` 更像“带平台目标和宿主边界的跨端工程项目”，而不是普通单端 Web 项目。

## 十一、这篇最后要沉淀成哪些核心共识
如果你读完只记住下面这些，我觉得就值了：

1. `uni-app` 不是单平台，而是跨端开发规范和编译体系。
2. 它统一的是开发方式，不是消灭所有平台差异。
3. 项目最好分成业务层、规范层、编译层、宿主层去理解。
4. 能用 `uni.*` 的地方优先用 `uni.*`，不要一上来散写 `wx.*`。
5. 平台专属能力并不可怕，可怕的是它们散落在业务页面里。
6. 跨端项目追求的不是“零差异”，而是“差异可控、结构清晰、后期好维护”。
