## uni-app 跨端开发方法与微信小程序常见能力接入

> 这一篇要解决的不是“某个 API 怎么调”，而是“公司里的 uni-app 项目做成微信小程序后，能力到底该怎么组织才稳”。  
> 很多人写到后面会越来越乱，不是因为不会调接口，而是因为跨端能力、微信协议能力、微信专属开放能力全都混在一起写。  
> 真正难的不是把一个功能跑通，而是让它在后续维护里不失控。

## 一、先记住一个总原则
做 `uni-app` 时，不要一看到目标端是微信小程序，就本能地到处写 `wx.*`。

更稳的思路是：

1. 通用能力优先看 `uni.*`
2. 平台协议能力按微信规则接
3. 平台专属能力用条件编译和适配层收口

这个顺序非常重要。  
因为它决定的是：你的项目以后是“多端可维护”，还是“业务页面里堆满微信细节”。

## 二、能力接入建议先分 3 类
这是我认为最值得固定下来的分类方法。

| 能力类型 | 典型能力 | 建议接法 |
| --- | --- | --- |
| 跨端通用能力 | 跳转、请求、缓存、上传、下载、提示 | 优先 `uni.*` |
| 微信页面 / 组件协议能力 | 头像昵称、手机号、页面分享 | 按微信页面协议或组件 `open-type` 接 |
| 微信专属开放能力 | 订阅消息、部分支付细节、部分专属文件和宿主能力 | `#ifdef MP-WEIXIN` 中集中使用 `wx.*` |

### 为什么这个分类很关键
因为很多项目一上来就不分层，结果最后变成：

+ 通用能力和微信能力混在一起
+ 页面写着写着就要懂微信开放平台细节
+ 切 H5 或 App 时几乎无从收口

## 三、为什么通用能力优先走 `uni.*`
这个问题不是“官方推荐所以这么写”，而是工程上这么做更稳。

### 1. 页面代码更统一
例如请求、跳转、缓存、上传下载这些能力，如果能统一走 `uni.*`，页面写法会更一致。

```javascript
uni.request({
  url: '/api/list',
  method: 'GET',
  success: (res) => {
    console.log(res.data)
  }
})
```

### 2. 迁其他端时改动更小
如果一开始就直接到处写 `wx.request`、`wx.navigateTo`、`wx.uploadFile`，那其实你写的已经不是“跨端业务页面”，而是“微信定制页面”。

### 3. 更容易做 service 封装
例如：

+ 请求统一封装
+ 登录统一封装
+ 上传统一封装
+ 异常处理统一封装

这样后面改规则时，修改范围不会扩散到所有页面。

## 四、条件编译到底是在解决什么问题
很多人把条件编译理解成“平台 if-else”，这只说对了一小部分。

更准确地说，条件编译的价值是：

`在编译阶段就把不属于当前平台的代码裁掉，让平台差异停留在少量边界里。`

示例：

```javascript
// #ifdef MP-WEIXIN
wx.requestSubscribeMessage({
  tmplIds: ['模板id1']
})
// #endif
```

这段代码的意义不是“我会写条件编译”，而是：

+ 微信专属能力只出现在微信构建产物里
+ 其他端不会带上这段无效代码
+ 平台差异可以被集中管理

所以条件编译不应该滥用。  
真正好的写法不是“满项目到处 ifdef”，而是“把 ifdef 尽量控制在少量适配层和平台边界里”。

## 五、登录：项目里最容易被混掉的不是流程，而是状态层次
做微信小程序登录时，很多人只记住了一句：

`先 uni.login，再发后端`

这句话没错，但太浅了。  
真实项目里真正容易写乱的，是下面 3 件事：

1. 分不清微信登录态和业务登录态
2. 把 `code` 错当成长期登录凭证
3. 把 `checkSession` 当成业务 token 的有效性判断

### 1. 一条更完整的登录链路
推荐你固定理解成下面这条链：

1. 前端调用 `uni.login`
2. 向微信宿主拿到临时 `code`
3. 前端把 `code` 发给后端
4. 后端调用微信侧接口换取会话信息
5. 后端基于业务体系签发自己的 `token`
6. 前端缓存的是业务 `token`，不是把 `code` 当长期身份

示例：

```javascript
export function loginByWechatMiniProgram() {
  return new Promise((resolve, reject) => {
    uni.login({
      provider: 'weixin',
      success: (loginRes) => {
        if (!loginRes.code) {
          reject(new Error('微信登录未拿到 code'))
          return
        }

        uni.request({
          url: 'https://api.example.com/auth/wechat-mini/login',
          method: 'POST',
          data: {
            code: loginRes.code
          },
          success: (requestRes) => {
            const data = requestRes.data || {}
            uni.setStorageSync('token', data.token)
            uni.setStorageSync('userInfo', data.userInfo)
            resolve(data)
          },
          fail: reject
        })
      },
      fail: reject
    })
  })
}
```

### 2. 这段代码真正要看懂的点
不是语法，而是分层：

+ `uni.login` 拿的是微信入口凭证
+ 最终真正能长期用的是后端签发的业务 `token`
+ 前端缓存的是业务态，不是微信底层态

### 3. 最容易踩的坑
+ `code` 有时效，不适合长期存着等以后再用
+ `uni.checkSession` 检查的是微信会话，不是业务 token
+ 业务 token 过期，不等于微信登录态也一起过期

## 六、用户信息和手机号：不要只记 API，要记“协议能力”
很多人一说用户信息，就本能想到 `getUserInfo` 一类接口。  
但从真实项目理解，更重要的是记住：

`头像昵称、手机号这类能力，本质上不是普通请求，而是微信页面 / 组件协议能力。`

也就是说，它们通常依赖：

+ 用户点击
+ 组件 `open-type`
+ 微信平台校验
+ 授权链路

所以不要把它们简单理解成“页面打开就自动拿”。

更稳的思路是：

+ 头像昵称按当前微信推荐能力设计交互
+ 手机号能力放在明确的用户触发链路里
+ 页面只负责业务流程，底层协议细节尽量封装

## 七、支付：为什么它一定属于“强平台能力”
支付不是普通按钮点击后发个请求那么简单，它背后带着整条平台安全链路。

推荐固定理解成下面 5 步：

1. 前端提交业务订单信息
2. 服务端生成支付所需参数
3. 前端调用支付 API 拉起支付
4. 前端根据结果做交互提示
5. 最终是否支付成功，以后端回调或后端最终确认结果为准

### 为什么不能只看前端支付回调
因为前端回调只代表客户端发起支付后当前拿到的结果，不等于服务端已最终确认支付成功。

这也是为什么支付一定不能只靠前端页面状态判断订单完成。

## 八、订阅消息、分享、`web-view` 为什么都属于高风险区域
这些能力之所以容易翻车，不是因为 API 难，而是因为它们都带有明显宿主属性。

### 1. 订阅消息
依赖用户交互、平台校验和模板能力，不适合散在业务页面里随便调。

### 2. 分享
不是单纯“给按钮绑个事件”就结束了，它往往还带着：

+ 分享路径
+ 参数回流
+ 登录承接
+ 业务落地页处理

### 3. `web-view`
它不是普通页面跳转，而是宿主里嵌套另一个页面。

真实项目里常见难点包括：

+ 登录态同步
+ H5 和小程序通信
+ 回跳逻辑
+ 分享链路
+ 参数同步

所以如果项目里用了 `web-view`，一定要把它当成“宿主型能力”看，而不是当成普通页面。

## 九、为什么真实项目里不能让页面自己散着拼能力细节
举个很常见的反例：

+ 页面 A 自己拼登录流程
+ 页面 B 自己拼支付流程
+ 页面 C 自己拼上传流程
+ 页面 D 自己拼订阅消息逻辑

这样刚开始看似快，后面会非常痛苦，因为：

+ 能力规则改一次，要改很多页面
+ 同一能力不同页面实现不一致
+ 平台差异散落在业务代码里
+ 后期几乎无法治理

## 十、我更推荐的项目组织方式
真实项目里，我更建议这样做：

1. 页面层只关心业务动作
2. 通用能力封装为 service
3. 微信专属能力单独收口
4. 条件编译尽量集中在平台边界

例如：

+ `loginByWechatMiniProgram`
+ `bindPhoneByWechat`
+ `uploadCommonFile`
+ `requestWechatSubscribeMessage`
+ `payByWechatMiniProgram`

这样后面如果微信规则、参数结构、错误处理方式变了，你改的是一层，而不是满项目翻。

## 十一、这一篇最后要沉淀的核心共识
如果你读完只记住最关键的，我建议记下面这些：

1. 跨端项目能力接入先分通用能力、协议能力、平台专属能力三类。
2. 请求、跳转、缓存、上传下载这类尽量优先走 `uni.*`。
3. 头像昵称、手机号、分享这类很多时候本质上是微信页面 / 组件协议能力。
4. 支付、订阅消息、`web-view` 这类能力宿主属性很强，一定要单点收口。
5. 登录最容易写乱的不是 API，而是微信登录态和业务登录态混在一起。
6. 条件编译的价值不是到处写平台判断，而是把差异控制在少量边界里。
