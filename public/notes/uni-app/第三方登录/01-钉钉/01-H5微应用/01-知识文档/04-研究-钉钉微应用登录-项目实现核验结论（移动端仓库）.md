# 研究-钉钉微应用登录-项目实现核验结论

## 1. 背景

本次核验聚焦项目里的钉钉微应用登录链路，目标不是重讲钉钉接入教程，而是回答三个问题：

1. 当前项目的钉钉 H5 微应用免登实现是不是和官方最新规则一致。
2. 当前项目的普通 H5 / App OAuth 登录实现是不是合理。
3. 现有知识库里哪些结论已经过时，需要同步修正。

本次核验涉及的项目文件：

- `pages/login/index.vue`
- `secondary/dingTalkH5Login/index.vue`
- `secondary/dingTalkLogin/index.vue`
- `api/dingtalk.js`
- `util/platform/dingtalk.js`
- `util/platform/auth.js`
- `util/platform/scan.js`
- `util/auth/dingtalkLogin.js`
- `configInfo/index.js`

---

## 2. 结论先行

### 2.1 当前项目主链路总体是对的

当前项目把钉钉登录拆成两条链路，这个方向和官方规则一致：

1. **钉钉容器内 H5 微应用**：走 `requestAuthCode` 静默免登。
2. **普通浏览器 H5 / App 原生端**：走 OAuth 授权页回调拿 `code` / `authCode`。

项目中对应实现如下：

- `pages/login/index.vue`：根据 UA 是否包含 `DingTalk`，在 H5 端分流到 `dingTalkH5Login` 或 `dingTalkLogin`。
- `secondary/dingTalkH5Login/index.vue`：负责钉钉容器内静默免登。
- `secondary/dingTalkLogin/index.vue`：负责普通 H5 和 App 端 OAuth 登录。

### 2.2 H5 微应用免登不需要先 `dd.config`

这是本次核验里最关键的结论。

根据钉钉官方当前文档：

- `requestAuthCode` / `getAuthCode` 这类免登接口**不需要**先调用 `dd.config`。
- 但 JSAPI 仍应放在 `dd.ready` 里调用，等钉钉容器初始化完成。

因此，项目当前 `secondary/dingTalkH5Login/index.vue` + `util/platform/auth.js` + `util/platform/dingtalk.js` 的做法是合理的：

- 动态加载 JSAPI SDK。
- 等待 `dd.ready`。
- 直接执行 `dd.runtime.permission.requestAuthCode({ corpId })`。
- 拿到 `authCode` 后交给后端换业务 token。

这条链路**不缺少 `dd.config`**，之前知识库里“免登前必须先做 JSAPI 鉴权”的表述需要拆开说，不能和通用业务 JSAPI 混为一谈。

### 2.3 OAuth 登录 URL 的参数方向基本正确

项目 `util/platform/auth.js` 当前 OAuth URL 组装方式与官方规则一致，关键点包括：

- 使用 `https://login.dingtalk.com/oauth2/auth`
- 带 `redirect_uri`
- 带 `client_id`
- 带 `response_type=code`
- 带 `state`
- 按需带 `corpId`
- 当限制企业范围时使用 `scope=openid corpid`

同时，项目回调解析兼容了 `code` 和 `authCode` 两种字段名，这一点也是对的。

### 2.4 OAuth `state` 之前没有真正校验，本次已补上

项目原先虽然生成了 `state`，但没有把它缓存并在回调时比对，等于没有真正完成这层校验。

本次已在以下文件补上本地缓存和回调比对：

- `util/platform/auth.js`
- `secondary/dingTalkLogin/index.vue`

现在流程变成：

1. 发起 OAuth 前生成并缓存 `state`
2. 回调后读取返回的 `state`
3. 与本地缓存比对
4. 比对通过才继续发起后端登录

这能避免“不是当前页面主动发起的回调 URL”被直接消费。

## 3. 官方依据与项目对照

## 3.1 H5 微应用免登调用规则

官方资料：

- [JSAPI 调用规则（官方文档镜像）](https://dingtalk.apifox.cn/doc-392795)
- [获取微应用免登授权码（官方文档镜像）](https://dingtalk.apifox.cn/doc-392803)
- [H5 微应用 FAQ（官方文档镜像）](https://dingtalk.apifox.cn/doc-392437)

核验结论：

- `requestAuthCode` / `getAuthCode` 不需要先 `dd.config`
- 仍要在 `dd.ready` 中调用
- `corpId` 是关键参数
- 如果配置不对，常见现象是 `errorCode=9` 或提示当前域名未关联微应用

项目对应：

- `util/platform/auth.js` 的 `requestDingTalkAuthCode` 会复用 `util/platform/dingtalk.js` 提供的 SDK 加载和 `dd.ready` 等待能力
- `secondary/dingTalkH5Login/index.vue` 已校验 `dingtalkCorpID`
- `pages/login/index.vue` 已先判断是否为钉钉容器，再决定是否进入 H5 免登页

结论：**实现正确。**

## 3.2 H5 微应用首页 URL 与 `corpId` 来源

官方 FAQ 提到，H5 微应用首页 URL 可以拼 `corpId=$CORPID$`，让钉钉在打开应用时自动把企业 CorpId 带进来。

项目当前不是走“URL 直接透传 CorpId”，而是走：

1. 登录页先确定 `schoolID`
2. `getSchoolInformation` 拉取学校配置
3. 再从学校配置里拿 `dingtalkCorpID`

这在“学校维度和企业维度一一对应”的当前项目里是可行的。

但如果后面要把同一套 H5 微应用做成更通用的多租户第三方企业应用，建议把首页 URL 也补成：

```text
https://你的H5域名/#/pages/login/index?corpId=$CORPID$
```

然后在前端把“路由参数 corpId”作为高优先级兜底来源，降低学校配置和实际打开企业不一致时的定位成本。

## 3.3 OAuth 浏览器登录规则

官方资料：

- [浏览器内获取用户 token（钉钉开发者百科）](https://opensource.dingtalk.com/developerpedia/docs/develop/permission/token/browser/get_user_app_token_browser/)
- [H5 网站应用授权登录（官方文档镜像）](https://dingtalk.apifox.cn/doc-392473)

核验结论：

- 浏览器内 OAuth 授权地址就是 `https://login.dingtalk.com/oauth2/auth`
- 回调参数里可能出现 `authCode`
- `state` 会原样回传，前端应自行校验

项目对应：

- `util/platform/auth.js`：已正确构建 OAuth 地址
- `util/platform/auth.js`：已兼容从 URL 中解析 `code` / `authCode`
- `secondary/dingTalkLogin/index.vue`：本次已补上 `state` 校验
- `pages/login/index.vue` / `secondary/dingTalkLogin/index.vue` / `secondary/dingTalkH5Login/index.vue`：已支持把路由中的 `corpId` 继续透传为组织兜底参数

结论：**修正后正确。**

## 4. 代码级核验清单

### 4.1 登录入口分流

文件：`pages/login/index.vue`

- H5 端点击“钉钉登录”后，会先判断 `isDingTalkContainer()`
- 如果当前就在钉钉容器内，跳到 `/secondary/dingTalkH5Login/index`
- 如果不在钉钉容器内，跳到 `/secondary/dingTalkLogin/index`

这个分流方式符合“钉钉内走静默免登，钉钉外走 OAuth”的常规设计。

### 4.2 H5 免登页

文件：`secondary/dingTalkH5Login/index.vue`

- 会先确保 `schoolID`
- 会请求学校配置
- 会检查 `dingtalkLoginApp`、`dingtalkCorpID`
- 会二次确认当前运行在钉钉容器里
- 会调用 `requestDingTalkAuthCode`
- 拿到 `authCode` 后调用 `/login/dingtalk/h5`

这里的 `dingtalkClientID` 校验属于**项目级配置完整性校验**，不是 `requestAuthCode` 自身的硬性要求。

### 4.3 OAuth 登录页

文件：`secondary/dingTalkLogin/index.vue`

- App 端用 WebView 打开 OAuth 页并拦截回调
- H5 端首次进入时跳转授权页，二次回调时再取 `code`
- 当前已补上 `state` 本地缓存与回调比对

这套流程适合当前项目的“普通浏览器 H5 + App 原生端”兜底路径。

### 4.4 登录成功后的会话落盘

文件：`util/auth/dingtalkLogin.js`

- OAuth 和 H5 免登两条链路复用同一套会话落盘逻辑
- 普通账号和家长账号走不同分支
- 家长账号会切换成关联学生身份再进入系统

这部分与钉钉无关，属于项目内部登录态处理，当前逻辑清晰且复用合理。

## 5. 注意事项

### 5.1 `requestAuthCode` 是当前官方推荐方式，但不排除未来变化

端内 H5 链路目前使用的是 `dd.runtime.permission.requestAuthCode`，这是钉钉官方当前推荐的端内授权方式。

如果后续钉钉官方推出新的授权组件或统一授权方案，需要单独评估是否升级，本文不会自动同步。

### 5.2 `state` 校验是 OAuth 规范要求，不能省略

`state` 是 OAuth 2.0 规范（RFC 6749）的标准安全机制，用于防止 CSRF 攻击。钉钉官方文档也明确说明"state 会原样回传，前端应自行校验"。

OAuth 链路里，`state` 不是可有可无的附加字段，而是前端区分"是不是本次登录自己发起的回调"的关键保护条件，不能删掉。

## 6. 后续建议

### 6.1 建议保留当前双链路设计

当前项目的双链路设计已经比较稳：

- 钉钉内：`requestAuthCode`
- 钉钉外：OAuth

不要把两条链路再揉回一套“万能登录页”里，否则后面排查容器差异会更难。

### 6.2 如果要做多租户长期治理，建议补 URL 级 `corpId` 兜底

如果未来一个钉钉应用要同时服务多个学校或多个企业，建议在 H5 首页地址里显式带 `corpId=$CORPID$`，让前端在学校配置之外再多一层兜底来源。

### 6.3 联调时重点盯三类问题

1. 域名是否和钉钉后台配置完全一致
2. 当前页面是否真的运行在钉钉容器里
3. 学校配置里的 `dingtalkCorpID`、`dingtalkClientID` 是否和后端使用的应用配置一致

---

## 7. 一句话总结

当前项目的钉钉微应用登录实现**总体正确**；本次主要修正的是知识库中过时的“免登必须先 `dd.config`”结论，并补上了 OAuth `state` 校验这一处安全细节。
