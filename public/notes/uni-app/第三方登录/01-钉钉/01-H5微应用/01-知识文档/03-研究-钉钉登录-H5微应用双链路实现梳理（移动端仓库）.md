# 研究-钉钉登录-当前项目双链路实现梳理

## 1. 背景

这份文档不是重讲钉钉开放平台教程，而是专门记录当前仓库里"钉钉登录"这条链路到底怎么跑。

适用场景：

- 新人接手登录模块时，先快速看清楚项目现在的实现。
- 排查"为什么钉钉里能登、浏览器里不能登"这类分流问题。
- 对接后端登录接口时，确认前端分别传了什么参数。
- 后续准备把当前实现替换成新方案时，先看清楚当前链路依赖点。

当前项目涉及的核心文件：

- `pages/login/index.vue`
- `secondary/dingTalkH5Login/index.vue`
- `secondary/dingTalkLogin/index.vue`
- `api/dingtalk.js`
- `util/platform/dingtalk.js`
- `util/platform/auth.js`
- `util/platform/scan.js`
- `util/platform/runtime.js`
- `util/auth/dingtalkLogin.js`
- `api/login.js`

## 2. 结论先行

当前项目采用的是一套明确的"双链路登录"方案：

1. 登录页点击"钉钉登录"时，先检查是否勾选隐私协议。
2. 再判断当前运行环境是否处于钉钉容器内部。
3. 如果处于钉钉内部 H5 微应用环境，则走"JSAPI 静默获取授权码"的免登链路。
4. 如果不是钉钉容器，而是普通浏览器 H5 或 App 原生端，则走 OAuth 授权页链路。
5. 两条链路在拿到后端返回的登录结果后，最终都会汇总到同一套会话处理工具里，统一写入项目当前使用的登录态字段。

也就是说，当前项目并不是"一套代码兼容所有钉钉场景"，而是按运行环境分流：

- 钉钉内：`authCode` 免登
- 钉钉外：OAuth 授权页回调拿 `code`

这个拆分方式，是当前仓库代码最重要的理解前提。

## 4. 入口分流总览

### 4.1 登录页入口职责

登录入口位于 `pages/login/index.vue`，职责很明确：

1. 用户点击钉钉登录按钮。
2. 先调用隐私协议校验逻辑，未勾选则直接中止。
3. 再检查当前学校 `schoolID` 是否存在。
4. 根据当前运行环境决定进入哪个钉钉登录页。

当前有两个入口方法：

- `loginWithDingTalkEntry`：H5 端点击钉钉登录时使用。
- `loginWithDingTalkApp`：App 端点击钉钉登录时使用。

其中 H5 端的关键分流规则是：

- 如果 `isDingTalkContainer()` 返回 `true`，跳转到 `secondary/dingTalkH5Login/index.vue`
- 如果 `isDingTalkContainer()` 返回 `false`，跳转到 `secondary/dingTalkLogin/index.vue`
- 如果登录页 URL 自带 `corpId`，还会继续透传给后续两个钉钉登录页，作为学校配置之外的组织兜底参数

### 4.2 运行环境判断规则

运行环境判断封装在 `util/platform/runtime.js`。

当前项目的判断依据是 User-Agent：

- `isDingTalkContainer()`：判断 UA 中是否包含 `DingTalk`
- `isAppPlusRuntime()`：判断是否为 uni-app App-Plus 原生端
- `isH5Runtime()`：判断是否为 H5 浏览器环境

这三个判断共同决定后面走哪条钉钉登录链路。

## 5. 钉钉内部 H5 微应用链路

这一条链路对应的是：**用户本来就在钉钉内打开系统 H5 微应用**。

核心页面：`secondary/dingTalkH5Login/index.vue`

核心目标：在钉钉容器内静默获取 `authCode`，再把 `authCode` 交给后端换取项目业务登录态。

### 5.1 页面启动时机

页面 `onLoad` 时会先从路由参数里读取 `schoolID`，然后直接调用登录主流程。

整体顺序如下：

1. 页面 `onLoad`
2. 读取 `schoolID`
3. 执行 `startLogin`

### 5.2 登录主流程

H5 微应用登录主流程可以概括为：

1. 检查学校 `schoolID` 是否存在。**（项目业务参数，用于区分多租户学校，非钉钉要求）**
2. 确保本地有学校上下文，便于后面获取学校配置。**（项目特有逻辑）**
3. 调用后端接口获取当前学校的钉钉配置。**（项目特有逻辑，其他项目可直接硬编码或从环境变量读取）**
4. 校验后端返回的钉钉配置是否完整。
5. 再次确认当前环境确实还是钉钉容器。
6. 调用钉钉 JSAPI 获取授权码。
7. 携带 `schoolID` 和 `authCode` 请求后端登录接口。**（`schoolID` 是项目参数，`authCode` 是钉钉返回的）**
8. 把后端返回的登录结果交给统一会话工具处理。
9. 登录成功后跳转系统首页。

### 5.3 学校上下文校验（项目特有逻辑）

> 这一节是本项目的多租户设计，不是钉钉要求的。其他项目可以跳过，直接从配置文件或环境变量读取 `corpId`。

在真正请求学校配置前，页面会先做 `schoolID` 兜底。

当前优先级是：

1. 路由参数里的 `schoolID`
2. 本地 `USER_INFO.schoolID`
3. `configInfo.DEFAULT_SCHOOL_ID`

如果三层兜底后仍然没有学校 ID，则直接中止流程。

### 5.4 拉取并校验学校钉钉配置（项目特有逻辑）

> 这一节是本项目的多租户设计：每个学校在后端有独立的钉钉应用配置，前端动态拉取。其他项目可以直接把 `corpId`、`clientId` 写在配置文件里，不需要这套动态拉取机制。

页面会通过 `getSchoolInformation` 拉取学校配置。

当前钉钉登录相关配置至少关注以下字段：

- `dingtalkLoginApp`：**项目自定义开关字段**，控制该学校是否启用钉钉登录
- `dingtalkClientID`：对应钉钉应用的 AppKey（`clientId`），**字段名是项目自定义的**
- `dingtalkCorpID`：对应钉钉企业 ID（`corpId`），**字段名是项目自定义的**

这里的校验目标有两层：

1. 当前学校是否开启了钉钉登录。
2. 当前学校的钉钉应用配置是否足够支撑后续登录流程。

对于当前 H5 免登页，前端要求：

- `dingtalkLoginApp === 1`
- `dingtalkCorpID` 必须存在
- `dingtalkClientID` 必须存在

其中：

- `dingtalkCorpID` 是前端调用端内授权接口时的关键参数（这个是钉钉要求的）。
- `dingtalkClientID` 虽然不是 `requestAuthCode` 这一步本身的入参，但项目后端登录依赖学校维度应用配置，因此前端也提前校验，避免晚失败。

### 5.5 再次确认钉钉容器环境

虽然登录页入口已经做过一次 `isDingTalkContainer()` 判断，但在 `secondary/dingTalkH5Login/index.vue` 中仍然会再次确认。

这样做的目的不是重复，而是防御性校验：

- 避免用户通过异常路由、收藏链接或手工输入地址误入该页面。
- 避免后续钉钉 JSAPI 在非钉钉容器里调用时报错。

### 5.6 获取授权码的细分流程

获取授权码封装在 `util/platform/auth.js` 的 `requestDingTalkAuthCode` 中，但底层的 SDK 加载和 `dd.ready` 等待，已经下沉到 `util/platform/dingtalk.js`。

这一步可以拆成四层。

#### 5.6.1 校验企业 CorpID

调用前首先校验 `corpId` 是否存在。

如果企业 `CorpID` 缺失，说明学校钉钉配置本身就不完整，前端不会继续往下走。

#### 5.6.2 懒加载钉钉 JS SDK

项目没有在全局 `main.js` 里提前注入钉钉 SDK，而是采用按需懒加载。

当前公共实现位置已经收口为：

1. `util/platform/dingtalk.js`：动态加载 SDK、等待 `dd.ready`、通用 JSAPI 鉴权
2. `util/platform/auth.js`：登录专用的免登授权码获取

具体逻辑如下：

1. 先判断当前是否处于浏览器环境。
2. 检查 `window` 上是否已经存在钉钉实例对象。
3. 如果已经存在，就直接复用钉钉对象，不重复加载。
4. 如果不存在，再检查全局是否已经存在 SDK 加载 Promise。
5. 如果 Promise 已存在，则直接返回同一个 Promise，避免重复创建 `script` 标签。
6. 如果 Promise 也不存在，就创建一个新的 Promise 作为全局唯一加载状态。
7. 创建前会先检查页面上是否残留了相同 ID 的 `script` 标签。
8. 如果已存在残留脚本，就监听其 `load` 和 `error` 事件。
9. 如果没有残留脚本，则动态创建一个新的 `script` 标签。
10. 给脚本绑定固定 ID、CDN `src`，并设置异步加载。
11. 最后把脚本插入 `head` 中开始加载。

这样设计的目的，是避免：

- 多次点击登录时重复插入 SDK 脚本
- 多个页面并发请求 SDK 时产生竞态问题
- 页面异常残留时重复监听和重复创建

#### 5.6.3 等待钉钉环境就绪

SDK 加载完成后，还不会立刻执行授权码获取，而是继续等待钉钉环境 ready。

当前项目通过 `waitForDingTalkReady` 做两件事：

1. 监听 `dd.ready`，确认钉钉容器初始化完成。
2. 监听 `dd.error`，捕获初始化失败情况。

同时额外设置了 5 秒超时保护，避免异常环境下无限等待。

#### 5.6.4 调用钉钉原生 API 获取授权码

前置条件都满足后，当前项目最终调用的是：

- `ddInstance.runtime.permission.requestAuthCode({ corpId })`

拿到结果后，会兼容读取：

- `result.code`
- `result.authCode`

最后统一返回一个授权码字符串。

### 5.7 拿授权码后请求后端登录

前端拿到 `authCode` 后，会调用：

- `dingTalkH5Login`
- 接口路径：`/login/dingtalk/h5`

当前前端传给后端的关键参数是：

- `authCode`：钉钉返回的授权码，**钉钉要求的**
- `schoolID`：**项目业务参数**，后端用来查找对应学校的钉钉应用配置

后端拿到这些参数后，会基于当前学校的钉钉应用配置继续换取用户信息，并返回项目自己的登录态数据。

### 5.8 统一会话落盘

前端拿到后端登录结果后，并不会在当前页面自己手工拼接一堆本地存储，而是统一交给：

- `util/auth/dingtalkLogin.js`

对外统一入口是：

- `completeDingTalkLogin(loginResponse)`

这一步的作用是把后端返回的数据，转换成项目当前约定的会话格式，例如：

- `USER_INFO`
- `USER_INFO_TOKEN`
- `currentUser`
- 家长模式相关字段

由于项目内不同账号类型的本地存储规则不完全相同，所以这里必须走统一工具，不能在钉钉登录页里临时散写。

## 6. 普通浏览器 H5 / App 原生端 OAuth 链路

这一条链路对应的是：**当前不在钉钉容器里，但仍然希望通过钉钉完成登录**。

核心页面：`secondary/dingTalkLogin/index.vue`

核心目标：跳转或内嵌钉钉授权页，拿到 `code` 或 `authCode`，再交给后端换取项目业务登录态。

### 6.1 页面启动时机

和 H5 微应用页一样，本页面也是 `onLoad` 时读取 `schoolID`，然后立即触发登录主流程。

### 6.2 登录主流程

普通浏览器 / App 的 OAuth 链路主流程如下：

1. 检查学校 `schoolID` 是否拿到。
2. 调用后端接口获取当前学校的钉钉配置。
3. 校验拼接授权 URL 所需配置是否完整。
4. 获取授权码。
5. 如果当前还没拿到授权码，就继续跳转授权页或等待回调。
6. 如果已经拿到授权码，就调用后端登录接口。
7. 登录成功后再走统一会话落盘逻辑。

### 6.3 OAuth 回调地址构建

获取授权码前，前端必须先构建 OAuth 回调地址 `redirect_uri`。

当前项目会根据运行环境分两种情况处理。

#### 6.3.1 App 端回调地址

App 端不会用当前页面 URL 直接做回调，而是使用后端域名下的固定地址。

设计目的有两个：

1. 便于在钉钉开放平台后台配置回调白名单。
2. 便于 App 里的 WebView 通过 `overrideUrlLoading` 拦截回调地址并提取授权码。

这个地址不一定需要真实可访问，但必须满足"WebView 能识别并拦截到"。

#### 6.3.2 H5 端回调地址

H5 环境则直接使用当前页面自身 URL 作为回调地址。

原因是：

- 钉钉授权成功后会把 `code` 或 `authCode` 回拼到 `redirect_uri`
- 页面会重新加载
- 重新加载后前端再从当前 URL 中把授权码解析出来

当前项目通过 `buildUniH5PageUrl` 生成完整的 uni-app H5 hash 路由地址。

### 6.4 H5 二次进入时的回调识别

对于浏览器 H5 场景，前端会先检查当前页面 URL 是否已经是钉钉回调返回后的地址。

这一步的目标是识别"当前是不是第二次进入页面"。

解析逻辑封装在 `resolveDingTalkAuthCodeFromUrl` 中，兼容以下场景：

- 普通 URL 查询参数里的 `code`
- 普通 URL 查询参数里的 `authCode`
- hash 路由里的 `code`
- hash 路由里的 `authCode`
- 回调参数中的 `state`

如果当前 URL 已经能解析出授权码，则说明这次进入是钉钉回调回来的第二次进入，此时会继续做两件事：

1. 解析并格式化 URL 回调参数
2. 校验回调中的 `state`

只有 `state` 校验通过，前端才会继续使用这个授权码。

另外，H5 hash 路由场景里还要额外注意一个隐藏坑：

- 如果钉钉把 `code` / `state` 拼在 URL search（`?`）里，而前端后续只是切换 `#` 后面的页面路由，旧参数可能继续残留在浏览器地址栏。
- 这会导致用户下次再次进入 `secondary/dingTalkLogin/index.vue` 时，被误判成"又一次 OAuth 回调"。
- 正确做法是在前端识别并消费完回调后，立即用 `history.replaceState` 清掉 `code`、`authCode`、`state`，只保留 `schoolID` 等业务参数。

### 6.5 首次进入时生成防伪状态码

如果当前页面不是回调后的第二次进入，或者当前 URL 中根本还没有授权码，则视为首次进入授权流程。

这时前端会：

1. 生成一个随机 `state`
2. 把 `state` 存入本地
3. 后续拼到钉钉 OAuth 授权地址里

这样做的原因是：

- `state` 由前端自己发起时生成
- 钉钉回调时会原样返回
- 前端回调后再拿返回值和本地缓存做比对

用途就是防止伪造或串流程。

### 6.6 动态拼接钉钉授权 URL

当前项目会用以下参数动态拼出钉钉 OAuth2.0 授权地址：

- `clientId`
- `redirectUri`
- `corpId`
- `state`

拼接结果是标准的：

- `https://login.dingtalk.com/oauth2/auth`

如果授权 URL 拼接失败，前端会立即：

1. 清理本地缓存的 `state`
2. 抛出错误并中止流程

此外，如果登录页入口已经透传了 `corpId`，它还会继续体现在 OAuth 授权 URL 和回调页 URL 上，避免多组织场景下前端只能完全依赖学校配置。

### 6.7 App 端 WebView 获取授权码

如果当前运行环境是 App 原生端，则前端不会直接整页跳转，而是：

1. 使用 `plus.webview.create` 打开钉钉 OAuth 授权页。
2. 通过 `overrideUrlLoading` 拦截页面跳转。
3. 当跳转地址命中 `redirectUri` 时，说明钉钉已经回调。
4. 从回调 URL 中提取 `code`、`authCode`、`state`。
5. 再校验回调 `state` 是否和本地一致。
6. 校验成功后返回授权码。
7. 最后关闭 WebView。

这里之所以单独放到 WebView，而不是整个 App 页面直接跳出，是为了把钉钉授权流程局部封装掉，避免影响原 App 页面栈。

### 6.8 H5 端跳转钉钉授权页

如果当前不是 App，而是普通浏览器 H5，则前端会直接调用 `redirectToDingTalkOAuth`。

当前实现使用的是：

- `window.location.replace(authUrl)`

不用 `href` 的目的，是避免用户点浏览器返回时回到授权中间态页面。

在这种首次跳转场景下，前端最后会返回 `null`，表示：

- 当前页面已经跳走
- 本次函数执行先结束
- 等钉钉授权完成后，浏览器再回调回来重新进入页面

### 6.9 回调后请求后端登录

拿到 `code` 或 `authCode` 后，前端会调用：

- `dingTalkOAuthLogin`
- 接口路径：`/login/dingtalk`

当前前端传给后端的关键参数包括：

- `code`：钉钉回调返回的授权码，**钉钉要求的**
- `schoolID`：**项目业务参数**，后端用来查找对应学校的钉钉应用配置
- `clientType`：**项目业务参数**，告知后端当前是 App 端还是 H5 端，用于区分登录场景

根据当前页面注释的设计意图，后端会继续：

1. 用学校配置里的 `clientID + clientSecret` 换取钉钉访问凭证
2. 再获取该用户信息
3. 最终返回项目自己的 `token`、`userInfo` 等业务登录结果

### 6.10 登录结果统一转项目格式

这一步和 H5 微应用链路一样，最终都走 `completeDingTalkLogin`。

原因是项目里不同用户类型对应的本地会话格式并不完全一样，例如：

- 普通账号
- 家长账号
- 企业账号

如果不统一收口，后续很容易在不同登录方式之间写出不同版本的登录态处理逻辑。

---

## 7. 共用会话处理逻辑

无论是钉钉内部 H5 免登，还是普通浏览器 / App OAuth 登录，后端返回结果后都会统一交给 `util/auth/dingtalkLogin.js`。

当前统一处理逻辑分两层：

1. `resolveLoginPayload`：解析后端返回结构，提取 `token` 和 `userInfo`
2. `completeDingTalkLogin`：按账号类型继续分发

账号类型又分成两类：

- 家长账号：走 `handleParentLogin`
- 其他账号：走 `handleStandardLogin`

这一步除了写入登录态，还会做：

- 获取通讯录完整信息
- 获取用户问候语
- 触发全局登录态就绪事件
- 根据账号类型决定后续跳转页

因此，钉钉登录页面的职责应该只停留在"拿授权码"和"调后端接口"，不要在页面里重复实现项目登录态落盘。

---

## 8. 前端依赖清单

要让当前钉钉登录链路跑通，至少要满足以下前提：

### 8.1 登录入口前提（项目特有）

- 用户已勾选隐私协议（**项目业务要求，非钉钉要求**）
- 当前页面已拿到 `schoolID`（**项目多租户参数，非钉钉要求**）

### 8.2 学校配置前提（项目特有）

以下字段均为**项目自定义字段名**，从项目后端接口动态拉取，不是钉钉官方字段：

- `dingtalkLoginApp` 已开启（项目开关）
- `dingtalkClientID` 已配置（对应钉钉 AppKey / clientId）
- `dingtalkCorpID` 已配置（对应钉钉企业 corpId）

### 8.3 钉钉内部 H5 微应用额外前提

- 当前页面确实运行在钉钉容器里
- 钉钉 JS SDK 能成功加载
- `dd.ready` 能正常完成初始化

### 8.4 普通浏览器 / App OAuth 额外前提

- 钉钉开放平台后台已配置正确回调白名单
- `redirect_uri` 与实际访问域名严格一致
- App 端 WebView 能成功拦截回调地址

## 9. 风险与边界

### 9.1 登录按钮显示和实际配置校验不是完全同一层

当前登录页按钮展示主要看：

- `dingtalkLoginApp === 1`
- `dingtalkClientID` 存在

但 H5 免登页面真正执行时，还会进一步要求：

- `dingtalkCorpID` 必须存在

这意味着存在一种情况：

- 登录页能看到钉钉登录按钮
- 但点击进入后才报"缺少钉钉CorpId"

后续如果要优化体验，可以考虑把入口显示条件和实际执行条件再统一一下。

### 9.2 `state` 校验不能省略

OAuth 链路里，`state` 并不是可有可无的附加字段，而是前端区分"是不是本次登录自己发起的回调"的关键保护条件。

如果后续有人改 OAuth 登录页，不能把这层逻辑删掉。

---

## 10. 联调排查顺序建议

排查钉钉登录问题时，建议按下面顺序看：

1. 登录页是否已勾选隐私协议。**（项目业务校验）**
2. 当前是否已经拿到正确的 `schoolID`。**（项目业务参数）**
3. 学校配置接口是否返回 `dingtalkLoginApp`、`dingtalkClientID`、`dingtalkCorpID`。**（项目特有字段）**
4. 当前环境是否真的处于钉钉容器中。
5. 如果是钉钉内，SDK 是否成功加载，`dd.ready` 是否成功。
6. 如果是钉钉外，OAuth 回调地址和钉钉后台白名单是否一致。
7. 回调 URL 中的 `code` / `authCode` / `state` 是否被正确解析。
8. 后端登录接口是否成功返回 `token` 和 `userInfo`。
9. `completeDingTalkLogin` 是否成功把登录结果转换为项目会话格式。

---

## 11. 最终结论

钉钉登录可以总结成四个核心点：

1. 先在登录页完成隐私协议校验和环境分流。
2. 钉钉内 H5 微应用走 `authCode` 静默登录。
3. 普通浏览器和 App 原生端走 OAuth 授权页回调登录。
4. 两条链路最终都统一落到同一套项目会话处理工具里。

后续无论是排查问题、补充注释，还是升级钉钉登录方案，都应该优先围绕这四个核心点来理解。
