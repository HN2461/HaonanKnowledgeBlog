# 钉钉 H5 微应用登录常见误区与官方规则核验

> 这篇文章不是钉钉登录入门教程，而是专门针对**实现过程中最容易搞错的几个点**，结合官方文档逐一核验，给出明确结论。
>
> 如果你已经在实现钉钉登录，但不确定某些做法是否正确，这篇文章适合你。

---

## 一、最关键的结论先看

| 问题 | 正确结论 |
|------|----------|
| `requestAuthCode` 需要先 `dd.config` 吗？ | **不需要**，但必须在 `dd.ready` 中调用 |
| OAuth `state` 可以不校验吗？ | **不能省**，是防 CSRF 的关键保护 |
| 回调参数字段名是 `code` 还是 `authCode`？ | **两种都可能出现**，需要兼容 |
| `corpId` 只能从学校配置里拿吗？ | **建议多层兜底**，URL 参数 > 学校配置 |
| 双链路设计可以合并成一个登录页吗？ | **不建议**，合并后排查容器差异更难 |

---

## 二、误区一：免登前必须先做 `dd.config`

### 错误认知

很多文章或老资料会说：

> "调用钉钉 JSAPI 前，必须先执行 `dd.config` 完成鉴权，否则接口会报错。"

然后有人就把这个规则套到免登上，在 `requestAuthCode` 前先做一遍 `dd.config`，结果发现：
- 要么 `dd.config` 本身报错（因为没有签名）
- 要么流程变得很复杂（需要后端提供签名接口）

### 正确规则

根据钉钉官方文档（JSAPI 调用规则 + 获取微应用免登授权码）：

**`requestAuthCode` / `getAuthCode` 这类免登接口不需要先调用 `dd.config`。**

`dd.config` 是用来声明**通用业务 JSAPI** 权限的（比如扫码、选人、分享等），和免登是两件独立的事。

**正确的免登流程：**

```js
// 1. 加载 SDK
const ddInstance = await loadDingTalkJsApi()

// 2. 等待 dd.ready（必须！）
await waitForDingTalkReady(ddInstance)

// 3. 直接调用 requestAuthCode（不需要先 dd.config）
ddInstance.runtime.permission.requestAuthCode({
  corpId,
  onSuccess: result => resolve(result?.code || result?.authCode || ''),
  onFail: error => reject(new Error(error?.errorMessage || '获取授权码失败'))
})
```

**两件事的区别：**

| 操作 | 用途 | 是否需要后端签名 |
|------|------|----------------|
| `dd.ready` | 等待钉钉容器初始化完成 | 不需要 |
| `dd.config` | 声明通用业务 JSAPI 权限（扫码、选人等） | 需要 |
| `requestAuthCode` | 获取免登授权码 | 不需要 |

**结论：免登只需要 `dd.ready`，不需要 `dd.config`。只有后续还要调用扫码、选人等通用业务 JSAPI 时，才需要额外做 `dd.config`。**

## 三、误区二：OAuth `state` 只是可选参数

### 错误认知

有些实现里，`state` 只是随便生成一个字符串拼到 URL 里，回调时根本不校验，或者根本不生成 `state`。

### 正确规则

`state` 是 OAuth 2.0 规范里的防 CSRF 参数，**不是可选的装饰品**。

**为什么必须校验：**

如果不校验 `state`，攻击者可以构造一个带有自己 `code` 的回调 URL，直接打到你的登录页，让你的系统用攻击者的 `code` 去换 token，造成账号劫持。

**正确的 state 校验流程：**

```js
// 发起 OAuth 前：生成并缓存 state
const state = `dingtalk_login_${Date.now()}_${Math.random().toString(36).slice(2, 10)}`
uni.setStorageSync('DINGTALK_OAUTH_STATE', state)

// 构建授权 URL 时带上 state
const authUrl = `https://login.dingtalk.com/oauth2/auth?...&state=${state}`

// 回调后：校验 state
const validateOAuthState = (callbackState) => {
  const expectedState = uni.getStorageSync('DINGTALK_OAUTH_STATE')
  uni.removeStorageSync('DINGTALK_OAUTH_STATE')

  // 本地没有 state：可能是历史残留参数，清理后重新发起
  if (!expectedState) {
    clearCallbackParamsFromUrl()
    return false  // 返回 false 表示"不是本次回调，重新发起"
  }

  // state 不匹配：可能是伪造请求
  if (!callbackState || callbackState !== expectedState) {
    clearCallbackParamsFromUrl()
    throw new Error('登录状态校验失败，请重试')
  }

  clearCallbackParamsFromUrl()
  return true  // 返回 true 表示"是本次回调，继续处理"
}
```

**一个容易忽略的细节：** 当本地没有 `state` 时（`expectedState` 为空），不应该直接报错，而应该清理 URL 参数后重新发起授权。这是因为 hash 路由下，历史回调参数可能残留在地址栏，导致下次进入登录页时被误判为"本次回调"。

## 四、误区三：回调参数字段名一定是 `code`

### 错误认知

按照 OAuth 2.0 标准，授权码回调参数应该叫 `code`。但有些实现只取 `code`，忽略了 `authCode`。

### 正确规则

钉钉在不同场景下，回调参数字段名可能是 `code`，也可能是 `authCode`。

**必须兼容两种字段名：**

```js
// 从 URL 中解析授权码
const resolveAuthCodeFromUrl = (url) => {
  const result = { code: '', authCode: '', state: '' }

  const parseQuery = (queryString) => {
    const params = new URLSearchParams(queryString.replace(/^\?/, ''))
    if (!result.code) result.code = params.get('code') || ''
    if (!result.authCode) result.authCode = params.get('authCode') || ''
    if (!result.state) result.state = params.get('state') || ''
  }

  try {
    const parsedUrl = new URL(url)
    parseQuery(parsedUrl.search)  // 解析 search 部分
    const hash = parsedUrl.hash || ''
    if (hash.includes('?')) {
      parseQuery(hash.substring(hash.indexOf('?') + 1))  // 解析 hash 部分
    }
  } catch (e) {}

  return result
}

// 使用时取任一非空值
const callbackResult = resolveAuthCodeFromUrl(window.location.href)
const code = callbackResult.code || callbackResult.authCode || ''
```

同样，JSAPI 免登的回调结果也需要兼容：

```js
onSuccess: result => {
  // 兼容 code 和 authCode 两种字段名
  resolve(result?.code || result?.authCode || '')
}
```

## 五、误区四：`corpId` 只从学校配置里拿

### 错误认知

有些实现里，`corpId` 完全依赖后端返回的学校配置，没有其他来源。

### 正确规则

`corpId` 应该有多层来源，按优先级兜底：

```
URL 路由参数 corpId  >  学校配置 dingtalkCorpID  >  空
```

**为什么需要 URL 参数兜底：**

钉钉官方 FAQ 提到，H5 微应用首页 URL 可以拼 `corpId=$CORPID$`，让钉钉在打开应用时自动把企业 CorpId 带进来：

```
https://你的域名/#/pages/login/index?corpId=$CORPID$
```

这样即使学校配置和实际打开的企业不一致，也能通过 URL 参数快速定位问题。

**实现方式：**

```js
const resolveCorpId = ({ routeCorpId, schoolConfig }) => {
  return String(
    routeCorpId ||                          // URL 路由参数（最高优先级）
    schoolConfig?.dingtalkCorpID ||         // 学校配置
    schoolConfig?.dingtalkCorpId ||         // 兼容字段名大小写
    ''
  ).trim()
}
```

**多租户场景下的建议：** 如果一个钉钉应用要同时服务多个企业，强烈建议在首页 URL 里带上 `corpId=$CORPID$`，让前端有一个独立于学校配置的 corpId 来源。

## 六、误区五：双链路可以合并成一个登录页

### 错误认知

有些人觉得两个登录页（H5 免登页 + OAuth 页）太麻烦，想合并成一个页面，在里面做 if/else 判断。

### 为什么不建议合并

1. **排查困难**：合并后，一个页面里同时有 JSAPI 免登逻辑和 OAuth 跳转逻辑，出问题时很难快速定位是哪条路出了问题
2. **状态管理复杂**：两条路的状态（loading 文案、错误信息、重试逻辑）不同，合并后代码会变得很乱
3. **防御性校验难做**：H5 免登页需要二次确认在钉钉容器内，OAuth 页需要处理两阶段（首次跳转 + 回调），合并后这些逻辑会互相干扰

**推荐的分流方式：**

```js
// 登录页入口：根据环境分流
const handleDingTalkLogin = () => {
  if (isDingTalkContainer()) {
    // 钉钉容器内：走 JSAPI 免登
    uni.navigateTo({ url: '/secondary/dingTalkH5Login/index' })
  } else {
    // 钉钉容器外：走 OAuth
    uni.navigateTo({ url: '/secondary/dingTalkLogin/index' })
  }
}
```

两个页面各自只负责一条路，逻辑清晰，排查方便。

## 七、常见报错与原因

### errorCode=9

**现象：** `requestAuthCode` 报错，errorCode 为 9。

**原因：** 通常是以下之一：
- 当前域名未在钉钉后台配置为可信域名
- 应用未发布
- `corpId` 不正确

**排查顺序：**
1. 检查钉钉后台可信域名配置
2. 确认应用已发布
3. 确认 `corpId` 和钉钉后台企业 ID 一致

### dd.ready 超时

**现象：** 等待 `dd.ready` 超时（5秒内没有触发）。

**原因：** 通常是以下之一：
- SDK 加载失败（CDN 不可访问）
- 不在钉钉容器内（UA 不含 `DingTalk`）
- 钉钉客户端版本过低

**排查顺序：**
1. 确认 `window.dd` 是否存在（SDK 是否加载成功）
2. 确认 UA 是否含 `DingTalk`
3. 让用户升级钉钉客户端

### OAuth 回调后 state 校验失败

**现象：** 回调后报"登录状态校验失败"。

**原因：** 通常是以下之一：
- 历史残留参数（上次登录的 code/state 没有清理）
- 跨 tab 授权（state 存在 A tab，但 B tab 收到了回调）
- 本地存储被清空

**排查顺序：**
1. 检查 URL 中是否有历史残留的 code/state
2. 确认是否在同一个 tab 里完成了授权
3. 检查本地存储是否正常

## 八、核验清单

在上线前，建议逐项核验：

**JSAPI 免登：**
- [ ] 没有在 `requestAuthCode` 前调用 `dd.config`
- [ ] `requestAuthCode` 在 `dd.ready` 回调中调用
- [ ] `corpId` 有多层来源（URL 参数 > 学校配置）
- [ ] 兼容了 `result.code` 和 `result.authCode` 两种字段名
- [ ] 有二次确认在钉钉容器内的防御性校验

**OAuth 授权：**
- [ ] 发起 OAuth 前生成并缓存了 `state`
- [ ] 回调后校验了 `state`，不匹配时抛出错误
- [ ] 本地没有 `state` 时（历史残留场景），清理参数后重新发起
- [ ] 兼容了 `code` 和 `authCode` 两种回调字段名
- [ ] 回调参数消费后立即清理（防止 hash 路由残留）
- [ ] 使用 `window.location.replace` 跳转（避免用户点返回回到中间态）

**通用：**
- [ ] 两条链路最终都走同一套会话落盘逻辑
- [ ] 登录页入口做了环境分流，不在同一个页面里混写两条路的逻辑
