# 第八篇：接口开发与会话控制

> 接口是前后端协作的契约，会话控制是用户认证的核心。这两块搞清楚，你就能独立完成一个完整的后端服务。

## 一、RESTful 接口设计规范

### 1.1 什么是 REST

REST（Representational State Transfer，表述性状态转移）是一种接口设计风格，不是协议，也不是标准。

核心思想：**用 URL 表示资源，用 HTTP 方法表示操作**。

### 1.2 RESTful 设计原则

**资源用名词，不用动词：**

```
❌ 非 RESTful 风格
GET  /getUsers
POST /createUser
POST /deleteUser?id=1
POST /updateUser

✅ RESTful 风格
GET    /users          获取用户列表
GET    /users/:id      获取单个用户
POST   /users          创建用户
PUT    /users/:id      更新用户（全量替换）
PATCH  /users/:id      更新用户（部分更新）
DELETE /users/:id      删除用户
```

**URL 层级表示资源关系：**

```
GET  /users/:userId/articles          获取某用户的所有文章
GET  /users/:userId/articles/:id      获取某用户的某篇文章
POST /users/:userId/articles          为某用户创建文章
```

**使用正确的 HTTP 状态码：**

| 状态码 | 含义 | 使用场景 |
|--------|------|----------|
| 200 | OK | GET 成功、PUT/PATCH 成功 |
| 201 | Created | POST 创建成功 |
| 204 | No Content | DELETE 成功（无响应体） |
| 400 | Bad Request | 参数错误、格式不对 |
| 401 | Unauthorized | 未登录、token 无效 |
| 403 | Forbidden | 已登录但无权限 |
| 404 | Not Found | 资源不存在 |
| 409 | Conflict | 数据冲突（如邮箱已存在） |
| 500 | Internal Server Error | 服务器代码报错 |

### 1.3 统一响应格式

团队开发时，建议统一响应结构：

```javascript
// 成功响应
{
  "code": 200,
  "message": "success",
  "data": { "id": 1, "name": "张三" }
}

// 列表响应（带分页）
{
  "code": 200,
  "message": "success",
  "data": {
    "list": [...],
    "pagination": {
      "page": 1,
      "limit": 10,
      "total": 100
    }
  }
}

// 错误响应
{
  "code": 400,
  "message": "邮箱格式不正确",
  "data": null
}
```

```javascript
// Express 中封装响应工具函数
// utils/response.js
const success = (res, data, message = 'success', code = 200) => {
  res.status(code).json({ code, message, data })
}

const fail = (res, message, code = 400, data = null) => {
  res.status(code).json({ code, message, data })
}

module.exports = { success, fail }

// 路由中使用
const { success, fail } = require('../utils/response')

router.get('/users', async (req, res) => {
  const users = await User.find()
  success(res, users)
})

router.post('/users', async (req, res) => {
  if (!req.body.name) {
    return fail(res, '姓名不能为空')
  }
  const user = await User.create(req.body)
  success(res, user, '创建成功', 201)
})
```

## 二、apipost 接口测试

### 2.1 为什么需要接口测试工具

后端接口写好后，不能每次都靠前端页面来测试。接口测试工具让你：
- 直接发送各种 HTTP 请求
- 设置请求头、请求体
- 查看响应内容、状态码
- 保存接口文档，团队共享

常用工具：**apipost**（国产，中文友好）、Postman（国际主流）、Insomnia。

### 2.2 apipost 基础使用

**安装**：去 [apipost.cn](https://www.apipost.cn) 下载安装。

**发送 GET 请求：**

```
方法：GET
URL：http://localhost:3000/api/users?page=1&limit=10
```

**发送 POST 请求（JSON 格式）：**

```
方法：POST
URL：http://localhost:3000/api/users
Headers：Content-Type: application/json
Body（raw JSON）：
{
  "name": "张三",
  "email": "zhangsan@example.com",
  "password": "123456"
}
```

**发送带认证的请求：**

```
Headers：Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

### 2.3 环境变量

apipost 支持设置环境变量，避免每次手动改 URL：

```
开发环境：baseUrl = http://localhost:3000
生产环境：baseUrl = https://api.example.com

接口 URL 写：{{baseUrl}}/api/users
```

## 三、Cookie

### 3.1 Cookie 是什么

HTTP 是**无状态协议**，每次请求都是独立的，服务器不知道两次请求是否来自同一个用户。

Cookie 是服务器通过响应头 `Set-Cookie` 发给浏览器的一小段数据，浏览器会自动保存，并在后续请求中自动携带。

```
服务器响应：
Set-Cookie: userId=123; HttpOnly; Max-Age=86400; Path=/

浏览器后续请求自动带上：
Cookie: userId=123
```

### 3.2 在 Express 中使用 Cookie

```bash
npm install cookie-parser
```

```javascript
const express = require('express')
const cookieParser = require('cookie-parser')

const app = express()
app.use(cookieParser())

// 设置 Cookie
// res.cookie(name, value[, options])
//   name: Cookie 名称
//   value: Cookie 值（字符串或对象，对象会被 JSON.stringify）
//   options:
//     maxAge: number，过期时间（毫秒），优先级高于 expires
//     expires: Date，过期时间（Date 对象）
//     httpOnly: boolean，禁止 JS 读取（防 XSS，默认 false）
//     secure: boolean，只在 HTTPS 下发送（默认 false）
//     path: string，Cookie 生效路径（默认 '/'）
//     domain: string，Cookie 生效域名（默认当前域名）
//     sameSite: 'strict' | 'lax' | 'none'，防 CSRF
//       strict: 完全禁止跨站发送
//       lax: 允许导航跳转时发送（推荐）
//       none: 允许跨站发送（必须同时设置 secure: true）
//     signed: boolean，是否签名（需要 cookieParser 传入密钥）
//     encode: Function，自定义编码函数
app.get('/set-cookie', (req, res) => {
  res.cookie('username', '张三', {
    maxAge: 24 * 60 * 60 * 1000,  // 过期时间（毫秒），1天
    httpOnly: true,                 // 禁止 JS 读取（防 XSS）
    secure: false,                  // true 时只在 HTTPS 下发送
    path: '/',                      // Cookie 生效路径
    sameSite: 'lax'                 // 防 CSRF：strict/lax/none
  })
  res.json({ message: 'Cookie 已设置' })
})

// 读取 Cookie
app.get('/get-cookie', (req, res) => {
  const username = req.cookies.username
  res.json({ username })
})

// 删除 Cookie
// res.clearCookie(name[, options])
//   name: Cookie 名称
//   options: 与 res.cookie 相同（path 和 domain 必须与设置时一致）
app.get('/clear-cookie', (req, res) => {
  res.clearCookie('username')
  res.json({ message: 'Cookie 已清除' })
})
```

### 3.3 Cookie 的局限性

| 问题 | 说明 |
|------|------|
| 大小限制 | 每个 Cookie 最大 4KB |
| 安全性 | 存在客户端，可被篡改（需要签名） |
| 跨域限制 | 默认不跨域发送 |
| 移动端 | 原生 App 不支持 Cookie |

**签名 Cookie**（防篡改）：

```javascript
// 初始化时传入密钥
app.use(cookieParser('my-secret-key'))

// 设置签名 Cookie
res.cookie('userId', '123', { signed: true })

// 读取签名 Cookie（自动验证签名）
const userId = req.signedCookies.userId
// 如果被篡改，返回 false
```

## 四、Session

### 4.1 Session 的原理

Session 把用户数据存在**服务器端**，只把一个 Session ID 存在 Cookie 里：

```
1. 用户登录 → 服务器创建 Session，生成 SessionID
2. 服务器通过 Set-Cookie 把 SessionID 发给浏览器
3. 浏览器后续请求自动携带 Cookie（含 SessionID）
4. 服务器根据 SessionID 找到对应的 Session 数据
```

```
浏览器 Cookie：connect.sid=s%3Axxx（只存 ID）
服务器内存：{ 'xxx': { userId: 1, username: '张三' } }（存真实数据）
```

### 4.2 在 Express 中使用 Session

```bash
npm install express-session
```

```javascript
const express = require('express')
const session = require('express-session')

const app = express()
app.use(express.json())

app.use(session({
  secret: 'my-session-secret',  // 签名密钥，必须设置
  resave: false,                 // 每次请求是否强制保存 session
  saveUninitialized: false,      // 未初始化的 session 是否保存
  cookie: {
    maxAge: 24 * 60 * 60 * 1000, // session 过期时间（1天）
    httpOnly: true
  }
}))

// 登录接口
app.post('/login', (req, res) => {
  const { username, password } = req.body

  // 验证用户（实际应查数据库）
  if (username === 'admin' && password === '123456') {
    // 把用户信息存入 session
    req.session.user = { id: 1, username: 'admin', role: 'admin' }
    res.json({ message: '登录成功' })
  } else {
    res.status(401).json({ error: '用户名或密码错误' })
  }
})

// 获取当前用户信息
app.get('/profile', (req, res) => {
  if (!req.session.user) {
    return res.status(401).json({ error: '请先登录' })
  }
  res.json(req.session.user)
})

// 退出登录
app.post('/logout', (req, res) => {
  req.session.destroy((err) => {
    if (err) return res.status(500).json({ error: '退出失败' })
    res.clearCookie('connect.sid')
    res.json({ message: '已退出登录' })
  })
})
```

> **生产环境提醒**：`express-session` 官方 README 明确说明默认 `MemoryStore` 不适合生产环境。开发阶段可以先用内存，真正部署时请尽快换成 Redis 等外部存储。

### 4.3 Session 持久化（存 Redis）

默认 Session 存在内存里，服务器重启就丢失。生产环境用 Redis 持久化：

```bash
npm install connect-redis redis
```

```javascript
const { createClient } = require('redis')
const { RedisStore } = require('connect-redis')

const redisClient = createClient({ url: 'redis://localhost:6379' })
redisClient.connect()

app.use(session({
  store: new RedisStore({ client: redisClient }),
  secret: 'my-secret',
  resave: false,
  saveUninitialized: false,
  cookie: { maxAge: 86400000 }
}))
```

## 五、JWT（JSON Web Token）

### 5.1 JWT 是什么

JWT 是一种**无状态**的认证方案。用户登录后，服务器生成一个 Token 发给客户端，客户端每次请求时携带这个 Token，服务器验证 Token 的合法性即可，**不需要在服务器存储任何状态**。

### 5.2 JWT 的结构

JWT 由三部分组成，用 `.` 分隔：

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VySWQiOjEsInVzZXJuYW1lIjoiYWRtaW4iLCJpYXQiOjE3MTM0MDAwMDAsImV4cCI6MTcxMzQ4NjQwMH0.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

```
Header（头部）.Payload（载荷）.Signature（签名）
```

**Header**（Base64 编码）：

```json
{
  "alg": "HS256",   // 签名算法
  "typ": "JWT"      // token 类型
}
```

**Payload**（Base64 编码，存用户信息）：

```json
{
  "userId": 1,
  "username": "admin",
  "role": "admin",
  "iat": 1713400000,   // 签发时间（issued at）
  "exp": 1713486400    // 过期时间（expiration）
}
```

**Signature**（签名，防篡改）：

```
HMACSHA256(
  base64(header) + '.' + base64(payload),
  secret
)
```

> **注意**：Payload 只是 Base64 编码，不是加密！任何人都能解码看到内容。不要在 Payload 里存密码等敏感信息。

### 5.3 在 Express 中使用 JWT

```bash
npm install jsonwebtoken
```

```javascript
const jwt = require('jsonwebtoken')

const SECRET = 'my-jwt-secret-key'  // 密钥，生产环境放环境变量

// jwt.sign(payload, secretOrPrivateKey[, options]) → string
//   payload: 要编码的数据（对象、Buffer 或字符串）
//     注意：不要存密码等敏感信息（Payload 只是 Base64 编码，不是加密）
//   secretOrPrivateKey: 密钥（HS256 用字符串，RS256 用私钥）
//   options:
//     algorithm: 签名算法（默认 'HS256'）
//       HS256/HS384/HS512: HMAC + SHA（对称，用同一密钥签名和验证）
//       RS256/RS384/RS512: RSA（非对称，私钥签名，公钥验证，更安全）
//       ES256/ES384/ES512: ECDSA（非对称，比 RSA 更短）
//     expiresIn: 过期时间（字符串如 '24h'、'7d'，或秒数）
//     notBefore: 生效时间（字符串或秒数）
//     audience: 受众（string | string[]）
//     issuer: 签发者
//     subject: 主题
//     jwtid: JWT ID（唯一标识）
function generateToken(payload) {
  return jwt.sign(payload, SECRET, {
    expiresIn: '24h',   // 过期时间：24小时
    algorithm: 'HS256'  // 签名算法（默认）
  })
}

// jwt.verify(token, secretOrPublicKey[, options]) → payload
//   token: 要验证的 JWT 字符串
//   secretOrPublicKey: 密钥（HS256 用字符串，RS256 用公钥）
//   options:
//     algorithms: 允许的算法数组（默认根据密钥类型自动选择）
//     audience: 验证受众
//     issuer: 验证签发者
//     ignoreExpiration: boolean，是否忽略过期（默认 false）
//     clockTolerance: number，时钟偏差容忍秒数（默认 0）
//   返回：解码后的 payload 对象
//   抛出：
//     JsonWebTokenError: token 格式错误或签名无效
//     TokenExpiredError: token 已过期
//     NotBeforeError: token 尚未生效
function verifyToken(token) {
  try {
    return jwt.verify(token, SECRET)  // 返回 payload
  } catch (err) {
    if (err.name === 'TokenExpiredError') {
      return { error: 'expired', expiredAt: err.expiredAt }
    }
    return null  // token 无效
  }
}

// jwt.decode(token[, options]) → payload（不验证签名！）
//   仅用于调试，不要用于认证
const decoded = jwt.decode(token)
const decodedWithHeader = jwt.decode(token, { complete: true })
// { header: { alg: 'HS256', typ: 'JWT' }, payload: {...}, signature: '...' }

// 登录接口
app.post('/api/login', async (req, res) => {
  const { username, password } = req.body

  // 查数据库验证用户（示例）
  const user = await User.findOne({ username })
  if (!user || user.password !== password) {
    return res.status(401).json({ error: '用户名或密码错误' })
  }

  // 生成 token
  const token = generateToken({
    userId: user._id,
    username: user.username,
    role: user.role
  })

  res.json({
    message: '登录成功',
    token,
    user: { id: user._id, username: user.username }
  })
})
```

### 5.4 JWT 认证中间件

```javascript
// middleware/auth.js
const jwt = require('jsonwebtoken')
const SECRET = process.env.JWT_SECRET || 'my-jwt-secret'

function authMiddleware(req, res, next) {
  // 从请求头获取 token
  const authHeader = req.headers.authorization
  // 格式：Authorization: Bearer <token>

  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    return res.status(401).json({ error: '请先登录' })
  }

  const token = authHeader.split(' ')[1]

  try {
    const payload = jwt.verify(token, SECRET)
    req.user = payload  // 把用户信息挂到 req 上
    next()
  } catch (err) {
    if (err.name === 'TokenExpiredError') {
      return res.status(401).json({ error: 'token 已过期，请重新登录' })
    }
    return res.status(401).json({ error: 'token 无效' })
  }
}

module.exports = authMiddleware

// 在路由中使用
const auth = require('./middleware/auth')

// 需要登录才能访问的接口
app.get('/api/profile', auth, (req, res) => {
  res.json({ user: req.user })
})

// 路由级别保护
app.use('/api/admin', auth, adminRouter)
```

### 5.5 Token 刷新机制

```javascript
// 双 Token 方案：access token（短期）+ refresh token（长期）
app.post('/api/login', async (req, res) => {
  // ... 验证用户

  const accessToken = jwt.sign(
    { userId: user._id },
    SECRET,
    { expiresIn: '15m' }  // access token 15分钟过期
  )

  const refreshToken = jwt.sign(
    { userId: user._id },
    REFRESH_SECRET,
    { expiresIn: '7d' }   // refresh token 7天过期
  )

  // refresh token 存数据库（可以主动吊销）
  await RefreshToken.create({ token: refreshToken, userId: user._id })

  res.json({ accessToken, refreshToken })
})

// 刷新 access token
app.post('/api/refresh', async (req, res) => {
  const { refreshToken } = req.body

  try {
    const payload = jwt.verify(refreshToken, REFRESH_SECRET)

    // 验证 refresh token 是否在数据库中（未被吊销）
    const stored = await RefreshToken.findOne({ token: refreshToken })
    if (!stored) return res.status(401).json({ error: 'refresh token 无效' })

    // 生成新的 access token
    const accessToken = jwt.sign(
      { userId: payload.userId },
      SECRET,
      { expiresIn: '15m' }
    )

    res.json({ accessToken })
  } catch (err) {
    res.status(401).json({ error: 'refresh token 已过期，请重新登录' })
  }
})
```

## 六、Cookie、Session、JWT 对比

| 对比项 | Cookie | Session | JWT |
|--------|--------|---------|-----|
| 存储位置 | 客户端 | 服务器端 | 客户端 |
| 状态 | 有状态 | 有状态 | 无状态 |
| 扩展性 | 差 | 差（需共享存储） | 好（天然支持分布式） |
| 安全性 | 可被篡改（需签名） | 较安全 | 不可篡改（签名保证） |
| 主动失效 | 可以 | 可以 | 困难（需黑名单） |
| 跨域 | 受限 | 受限 | 友好（Header 传递） |
| 移动端 | 不支持 | 不支持 | 支持 |
| 适用场景 | 传统 Web | 传统 Web | 前后端分离、移动端 |

**如何选择：**

- **传统 Web 应用**（服务端渲染）：Session + Cookie
- **前后端分离 / 移动端 App**：JWT
- **需要主动踢人下线**：Session（或 JWT + Redis 黑名单）
- **微服务 / 分布式**：JWT（无需共享 Session 存储）

## 七、完整实战：用户认证系统

```javascript
// app.js
const express = require('express')
const mongoose = require('mongoose')
const cors = require('cors')
const authRouter = require('./routes/auth')
const userRouter = require('./routes/users')
const auth = require('./middleware/auth')

const app = express()

app.use(cors({ origin: 'http://localhost:5173', credentials: true }))
app.use(express.json())

// 公开路由（不需要登录）
app.use('/api/auth', authRouter)

// 受保护路由（需要登录）
app.use('/api/users', auth, userRouter)

// 错误处理
app.use((err, req, res, next) => {
  console.error(err)
  res.status(500).json({ error: '服务器错误' })
})

mongoose.connect('mongodb://127.0.0.1:27017/myapp')
  .then(() => app.listen(3000, () => console.log('http://localhost:3000')))
```

```javascript
// models/User.js
const { Schema, model } = require('mongoose')
const bcrypt = require('bcrypt')

const userSchema = new Schema({
  username: { type: String, required: true, unique: true, trim: true },
  password: { type: String, required: true, minlength: 6 },
  email: { type: String, required: true, unique: true },
  role: { type: String, enum: ['user', 'admin'], default: 'user' }
}, { timestamps: true })

// 保存前加密密码（Mongoose 9 不再接收 next 参数）
userSchema.pre('save', async function() {
  if (!this.isModified('password')) return
  this.password = await bcrypt.hash(this.password, 10)
})

// 验证密码方法
userSchema.methods.comparePassword = function(plainPassword) {
  return bcrypt.compare(plainPassword, this.password)
}

module.exports = model('User', userSchema)
```

```javascript
// routes/auth.js
const express = require('express')
const router = express.Router()
const jwt = require('jsonwebtoken')
const User = require('../models/User')

const SECRET = process.env.JWT_SECRET || 'dev-secret'

// 注册
router.post('/register', async (req, res) => {
  try {
    const { username, password, email } = req.body
    if (!username || !password || !email) {
      return res.status(400).json({ error: '所有字段不能为空' })
    }

    const user = await User.create({ username, password, email })
    res.status(201).json({
      message: '注册成功',
      user: { id: user._id, username: user.username }
    })
  } catch (err) {
    if (err.code === 11000) {
      return res.status(409).json({ error: '用户名或邮箱已存在' })
    }
    res.status(500).json({ error: err.message })
  }
})

// 登录
router.post('/login', async (req, res) => {
  try {
    const { username, password } = req.body
    const user = await User.findOne({ username })

    if (!user || !(await user.comparePassword(password))) {
      return res.status(401).json({ error: '用户名或密码错误' })
    }

    const token = jwt.sign(
      { userId: user._id, username: user.username, role: user.role },
      SECRET,
      { expiresIn: '24h' }
    )

    res.json({
      message: '登录成功',
      token,
      user: { id: user._id, username: user.username, role: user.role }
    })
  } catch (err) {
    res.status(500).json({ error: err.message })
  }
})

module.exports = router
```

## 八、密码安全：bcrypt

明文存储密码是大忌，必须使用哈希算法加密：

```bash
npm install bcrypt
```

```javascript
const bcrypt = require('bcrypt')

// 加密密码（saltRounds 越大越安全，但越慢，推荐 10）
const saltRounds = 10
const plainPassword = '123456'

// 加密
const hashedPassword = await bcrypt.hash(plainPassword, saltRounds)
console.log(hashedPassword)
// $2b$10$xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

// 验证（比较明文和哈希值）
const isMatch = await bcrypt.compare(plainPassword, hashedPassword)
console.log(isMatch)  // true

const isMatch2 = await bcrypt.compare('wrong-password', hashedPassword)
console.log(isMatch2)  // false
```

> bcrypt 是单向哈希，无法反向解密。即使数据库泄露，攻击者也无法直接得到明文密码。

### 9.1 参数校验（express-validator）

```bash
npm install express-validator
```

```javascript
const { body, validationResult } = require('express-validator')

// 定义校验规则
const registerValidation = [
  body('username')
    .trim()
    .isLength({ min: 2, max: 20 }).withMessage('用户名长度 2-20 个字符')
    .matches(/^[a-zA-Z0-9_]+$/).withMessage('用户名只能包含字母、数字和下划线'),
  body('email')
    .isEmail().withMessage('邮箱格式不正确')
    .normalizeEmail(),
  body('password')
    .isLength({ min: 6 }).withMessage('密码至少 6 个字符')
    .matches(/\d/).withMessage('密码必须包含数字')
]

// 在路由中使用
router.post('/register', registerValidation, async (req, res) => {
  // 检查校验结果
  const errors = validationResult(req)
  if (!errors.isEmpty()) {
    return res.status(400).json({
      error: errors.array().map(e => e.msg).join(', ')
    })
  }

  // 校验通过，继续处理
  const { username, email, password } = req.body
  // ...
})
```

```javascript
const rateLimit = require('express-rate-limit')

// 全局限流：每 15 分钟最多 100 次请求
const globalLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 100,
  message: { error: '请求过于频繁，请稍后再试' }
})

// 登录接口限流：每 15 分钟最多 5 次
const loginLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5,
  message: { error: '登录尝试次数过多，请 15 分钟后再试' }
})

app.use(globalLimiter)
app.post('/api/auth/login', loginLimiter, loginHandler)
```

## 十、小结

| 知识点 | 核心要点 |
|--------|----------|
| RESTful | URL 表示资源（名词），HTTP 方法表示操作，状态码语义化 |
| apipost | 接口测试工具，支持 GET/POST/PUT/DELETE，可设置 Headers 和 Body |
| Cookie | 存客户端，浏览器自动携带，`httpOnly` 防 XSS，`signed` 防篡改 |
| Session | 数据存服务器，只把 ID 存 Cookie，生产环境用 Redis 持久化 |
| JWT | 无状态 token，Header.Payload.Signature，适合前后端分离 |
| bcrypt | 密码单向哈希，`hash()` 加密，`compare()` 验证 |
| 认证中间件 | 从 `Authorization: Bearer <token>` 提取并验证 JWT |
| 参数校验 | express-validator 校验请求参数 |
| 限流 | express-rate-limit 防止暴力攻击 |
| 安全头 | helmet 自动设置安全响应头 |
