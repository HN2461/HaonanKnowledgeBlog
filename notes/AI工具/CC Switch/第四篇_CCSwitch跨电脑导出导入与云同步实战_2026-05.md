# 第四篇：CC Switch 跨电脑导出导入与云同步实战（2026-05核对）

> 资料核对时间：2026-05-01  
> 结论先说：**有这个功能。**CC Switch 官方明确支持 `导出配置`、`导入配置`、`自定义应用配置目录` 以及 `WebDAV 云同步`。  
> 适合人群：像主人这样会在家里电脑、公司电脑、其他设备之间来回切换，而且手里有多家中转 Provider，不想每次都重新手配的人。

[[toc]]

---

## 1. 先给结论：主人的思路是可行的

如果主人的目标是：

- 在一台电脑上维护一份最新配置
- 需要时把这份配置带到其他电脑
- 尽量避免手改 Claude Code、Codex、Gemini CLI 等工具的配置文件

那 CC Switch 官方提供的 **手动导出 / 导入** 就是合适方案。

官方手册在 `Settings -> Advanced -> Data Management` 里明确写了：

- 可以导出备份
- 可以从备份文件导入
- 导入时会**覆盖现有配置**

所以从产品能力上说，主人的方案不是“曲线救国”，而是 **CC Switch 本来就支持的正式用法**。

---

## 2. 官方到底支持哪些跨设备方式

按官方资料，CC Switch 目前至少有 3 种相关能力：

| 方式 | 适合谁 | 特点 |
|------|--------|------|
| 手动导出 / 导入 | 想保留一份“最新版配置包”的人 | 最稳、最容易理解、最好控风险 |
| 自定义应用配置目录 | 想把数据放进 OneDrive / Dropbox / iCloud / NAS 的人 | 更接近“文件夹级同步” |
| WebDAV 云同步 | 想让多设备直接同步数据库的人 | 适合长期多设备使用 |

如果只看主人的需求，我更推荐先用 **手动导出 / 导入**，因为它最贴近“我只保留最新一份，然后其他电脑直接导入”这个思路。

---

## 3. 手动导出 / 导入，官方保证能带走什么

根据 CC Switch 中文设置手册，导出备份文件时，官方明确列出的内容包括：

- 所有 Provider 配置
- MCP 服务器配置
- Prompts 预设
- 应用设置

这意味着：

- 你配置过的中转站列表
- 各 Provider 的地址、Key、模型相关配置
- MCP
- 提示词预设

这些核心内容，理论上都可以跟着导出文件一起迁移。

---

## 4. 但有两个限制，主人一定要先知道

这两条很关键，很多人以为“导入”就是“自动合并”，或者以为“所有设置都会一模一样跟过去”，实际不是。

### 4.1 导入是覆盖，不是合并

官方手册明确提示：

- 导入时会覆盖现有配置
- 建议导入前先导出当前配置做备份

所以如果主人现在多台电脑上的配置并不完全一样，最稳的做法不是直接选一台覆盖全部，而是：

1. 先把每台电脑都各自导出一份留底
2. 选一台作为“主配置机”
3. 把别的机器里独有的 Provider / MCP / Prompt 补齐到这台主配置机
4. 最后只从这台机器导出一份“最终最新版”

换句话说，**CC Switch 导入适合做“统一版本下发”，不适合做“多机自动合并”**。

### 4.2 设备级设置不会跨设备同步

官方 FAQ 明确写了：

- `~/.cc-switch/settings.json` 存的是设备级设置
- 这些设置 **not synced across devices**

它举的内容包括：

- 语言
- 主题
- 窗口行为
- 开机自启
- 各 CLI 工具的自定义配置目录

所以主人要理解成：

- **核心 Provider / MCP / Prompt 配置可以迁移**
- **这台电脑自己的界面偏好、目录路径之类，不保证跟着一起过去**

如果你在不同电脑上把 Claude / Codex / Gemini 的配置目录改到了非默认位置，那目标机器上仍然要再检查一次。

---

## 5. 主人最该用的操作流程

下面这套是我结合官方文档后，给主人的 **最稳方案**。

### 5.1 第一次整理时，不要急着只留一份

先在每一台旧电脑上都做一次导出备份，命名可以像这样：

- `cc-switch-home-2026-05-01.json`
- `cc-switch-office-2026-05-01.json`
- `cc-switch-laptop-2026-05-01.json`

这样做的目的很简单：

- 防止后面发现某台机器上有独有 Provider，却已经被覆盖掉
- 给自己留回滚空间

### 5.2 选一台主配置机，整理成最终版本

建议选“配置最全、最新”的那台电脑做主配置机。  
如果别的机器上还有它没有的中转，就先补到主配置机里。

等确认无误后，再导出一份最终版，例如：

- `cc-switch-master-2026-05-01.json`

以后主人只维护这一个“主版本”就行。

### 5.3 把主版本放到你常用的同步位置

例如：

- OneDrive
- Dropbox
- iCloud Drive
- NAS 同步目录
- 加密 U 盘

这样你到其他电脑时，只要拿到这一个 JSON 就够了。

## 6. 具体怎么导出

### 6.1 在主配置机上操作

1. 打开 CC Switch
2. 点击左上角 `⚙️` 进入设置
3. 进入 `高级` / `Advanced`
4. 找到 `数据管理` / `Data Management`
5. 点击 `导出`
6. 保存生成的 JSON 备份文件

官方说明里还提到，这个备份文件是 JSON 格式，可以直接用文本编辑器查看。

### 6.2 导出前建议主人多做一步

如果你刚刚改过很多 Provider，建议在导出前快速检查一下：

- 当前常用 Provider 是否都还在
- Base URL、API Key、模型 ID 是否是最新值
- MCP / Prompt 是否齐全

因为你之后分发给其他电脑的，其实就是这一份“标准答案”。

---

## 7. 具体怎么导入

### 7.1 在目标电脑上操作

1. 先安装好 CC Switch
2. 打开 CC Switch
3. 进入 `设置 -> 高级 -> 数据管理`
4. 点击 `选择文件`
5. 选中你保存的主配置 JSON
6. 点击 `导入`
7. 确认覆盖现有配置

这一段是官方手册明确给出的导入流程。

### 7.2 导入后怎么验证

建议主人至少做 3 件事：

1. 看 Provider 列表是否已经完整出现
2. 随机切换一个你常用的中转 Provider
3. 打开对应 CLI 工具做一次最小验证

例如：

- Claude Code：进项目后看 `/status`
- Codex / Gemini CLI / OpenCode：重开终端后跑一次最小请求

官方 README 还特别说明了一条：

- 大多数 CLI 工具切换 Provider 后需要重启终端或重启该 CLI
- **Claude Code 当前支持热切换，不需要重启**

---

## 8. 为什么我更推荐主人先用“手动导出 / 导入”

因为主人的核心诉求不是“所有机器永远自动同步”，而是：

- 我想维护一份最新配置
- 我去别的电脑时能快速恢复

这类场景下，手动导出 / 导入有 4 个优点：

1. 你永远知道“哪一份是主版本”
2. 不会因为自动同步把错误配置立即传播到所有设备
3. 迁移路径清晰，出问题容易回滚
4. 很适合多中转、多模型、多机器但不一定天天同步的人

所以主人说的“只保留最新的一份，然后其他电脑直接导入”，从方法论上是对的。

---

## 9. 如果你后面嫌手动麻烦，可以升级成官方云同步

CC Switch 官方 README 和设置手册都明确提到两类云同步方向。

### 9.1 方案一：自定义应用配置目录

官方说明里写到，CC Switch 自身数据默认放在：

```text
~/.cc-switch/
```

并且这个应用配置目录可以在设置里自定义。  
README 还明确举了这些典型场景：

- Dropbox
- OneDrive
- iCloud
- NAS

这意味着你可以把 CC Switch 的数据目录放进同步盘，让多台电脑共享同一套应用数据。

不过这里主人要注意：

- “应用配置目录”本身属于设备级设置
- 所以每台电脑第一次仍然要手动指向各自本地的同步盘路径

### 9.2 方案二：WebDAV 云同步

如果你想让 CC Switch 直接通过应用内同步，官方还提供：

- WebDAV 服务器地址
- 用户名 / 密码
- 远程目录
- 配置名称
- 自动同步

并支持：

- 测试连接
- 上传
- 下载
- 自动同步

官方手册也写得很清楚：

- 上传会覆盖远程数据
- 下载会覆盖本地数据
- 下载前会自动创建安全备份

所以 WebDAV 更像“数据库级同步”，适合长期多设备协作使用。

## 10. 哪种方式更适合主人

我给主人的建议很直接：

### 如果你只是偶尔换电脑

优先用：

- **手动导出 / 导入**

这是最稳、最不容易误操作的方案。

### 如果你经常多机切换

可以考虑：

- **主版本 JSON + 云盘保存**

也就是：

1. 仍然用手动导出生成“主版本”
2. 把主版本放在 OneDrive / Dropbox
3. 到别的电脑时直接导入

这样已经能覆盖大部分需求，而且风险比自动同步更低。

### 如果你几乎每天多设备来回切

再考虑：

- **自定义应用配置目录**
- **WebDAV 自动同步**

但这两种方案更适合你已经把配置体系稳定下来之后再上。

---

## 11. 还有一个容易忽略的坑：共享插件配置

官方 FAQ 还提到一个和“完整迁移”很相关的点：

- 如果你在不同 Provider 之间切换后发现插件类配置没跟过去
- 可以使用 `Shared Config Snippet`

路径是：

- `Edit Provider -> Shared Config Panel -> Extract from Current Provider`

创建新 Provider 时，官方说明默认还可以勾选：

- `Write Shared Config`

这类功能的意义是：  
把 API Key、Endpoint 之外的公共配置也一起保存在 Provider 体系里。

所以如果主人不只是配中转地址和 Key，还顺手在某些工具里加了公共扩展配置，那在整理“主配置机”时，最好顺手检查一下这一块。

---

## 12. 我给主人的最终推荐方案

如果让我按“稳、简单、少踩坑”来排序，我会建议主人这样做：

1. 先在每台旧电脑各导出一份备份
2. 选一台电脑作为主配置机
3. 把所有常用中转、MCP、Prompts 整理齐
4. 导出一份 `cc-switch-master-日期.json`
5. 把它放进 OneDrive / Dropbox 之类的位置
6. 在其他电脑安装 CC Switch 后直接导入这份主配置
7. 后续只维护这一份主配置，需要同步时重新导出覆盖旧文件

这套流程最符合主人现在的实际需求。

---

## 13. 两个额外提醒

### 13.1 如果你有自装 Skills，最好额外留意

官方导出清单里明确写了 Provider、MCP、Prompts、应用设置；  
而 FAQ 里又说明 `skills/` 是独立目录。

所以更稳妥的理解是：

- 如果主人非常依赖本地安装的 Skills
- 最好额外备份 `~/.cc-switch/skills/`
- 或者直接改用“应用配置目录同步” / `WebDAV`

这条是我根据官方导出清单和存储结构做出的**保守建议**，目的是避免你以为“所有 Skills 文件一定被装进 JSON 里”。

### 13.2 导入前，CC Switch 本身会做安全备份

官方 FAQ 写到：

- 每次配置导入前会自动创建备份
- 默认保留最近 10 个备份

所以主人即使导错一份，也不是完全没有回头路。
