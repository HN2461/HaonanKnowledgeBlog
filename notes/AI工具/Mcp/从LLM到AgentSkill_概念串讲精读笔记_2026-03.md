## 1. 先把整条链背下来

这期视频最适合用一条线来理解：

```text
LLM
  -> Token
  -> Context
  -> Context Window
  -> Prompt
  -> User Prompt / System Prompt
  -> Tool
  -> MCP
  -> Agent
  -> Agent Skill
```

可以把它理解成 3 层：

1. **模型层**
   - LLM、Token、Context、Context Window
2. **指令层**
   - Prompt、User Prompt、System Prompt
3. **执行与扩展层**
   - Tool、MCP、Agent、Agent Skill

一句话总结：

- **LLM 负责理解与生成**
- **Prompt 负责引导方向**
- **Tool / MCP 负责连接外部能力**
- **Agent 负责把任务跑起来**
- **Skill 负责把经验沉淀成可复用能力**

## 2. 每个概念到底在说什么

### 2.1 LLM 是什么

LLM 是大语言模型，本质上是一个经过大规模训练的“语言规律压缩器”。

它最擅长的事有 3 个：

1. 理解输入中的语义关系
2. 在上下文里推断最合理的后续内容
3. 生成符合目标风格和约束的输出

如果只记一句：

- **LLM 不是业务系统，也不是数据库，它首先是一个会基于上下文做推断的通用语言引擎**

### 2.2 Token 是什么

Token 不是“一个字”也不是“一个单词”，而是模型内部处理文本的最小切分单元。

你可以把它理解成：

- 模型真正“看见”和“计算”的，不是原始句子，而是一串 token

为什么它重要：

1. 计费常常按 token 计算
2. 输入太长会占满上下文窗口
3. 输出越长，成本和延迟通常越高

如果只记一句：

- **Token 决定了模型处理文本的粒度，也直接影响成本和容量**

### 2.3 Context 是什么

Context 就是模型这一次回答时能看到的全部上下文。

它通常包含：

1. 你的问题
2. 之前几轮对话
3. 系统提示词
4. 工具返回结果
5. 代码片段、文档片段、网页内容等额外材料

很多人以为模型“记住了你”，其实很多时候只是：

- **你把相关信息又放回了它这次能看到的 context 里**

### 2.4 Context Window 是什么

Context Window 是模型一次性能处理的上下文容量上限。

超过窗口会出现几种情况：

1. 旧内容被截断
2. 关键约束丢失
3. 回答开始漂移
4. 工具输出太多，把真正重要的信息淹没

如果只记一句：

- **Context 是“当前看到什么”，Context Window 是“最多能看到多少”**

### 2.5 Prompt 是什么

Prompt 就是你喂给模型的指令。

它不只是“问一句话”，更像是在定义：

1. 任务目标
2. 输出格式
3. 约束条件
4. 成功标准

差的 prompt 往往是：

- 目标模糊
- 约束缺失
- 背景不够

好的 prompt 往往会说清：

- 你是谁
- 你要做什么
- 不要做什么
- 最终输出长什么样

### 2.6 User Prompt 和 System Prompt

这两个最容易混。

**User Prompt**

- 用户当前输入的需求
- 更偏“这次我想让你干什么”

**System Prompt**

- 系统级规则和角色设定
- 更偏“你应该以什么原则持续工作”

可以这样理解：

- User Prompt 决定当前任务
- System Prompt 决定长期边界

比如：

- User Prompt：帮我总结这篇文章
- System Prompt：始终用中文回答、优先给结构化结果、不要编造事实

## 3. 从“会说”到“会做”：Tool、MCP、Agent

### 3.1 Tool 是什么

Tool 就是给模型额外接上的“手”和“眼”。

没有工具时，模型主要只能：

1. 基于已有上下文推理
2. 生成文本

有了工具后，模型才能进一步：

1. 查网页
2. 读文件
3. 调 API
4. 执行命令
5. 操作浏览器

如果只记一句：

- **Tool 让模型从“只能回答”变成“可以行动”**

### 3.2 MCP 是什么

MCP 可以理解成模型与外部工具之间的一套标准连接方式。

在实际使用里，MCP 解决的是：

1. 工具怎么被发现
2. 工具参数怎么描述
3. 返回结果怎么规范传回模型
4. 不同工具怎么用统一方式接入

所以它更像：

- **工具接入层**
- **协议层**
- **标准化适配层**

而不是具体某一个工具本身。

一个非常实用的理解方式是：

- Tool 是“能力”
- MCP 是“接能力的标准插座”

### 3.3 Agent 是什么

Agent 不只是“一个更厉害的提示词”。

它更像一个会围绕目标反复推进的执行者，通常具备下面几步：

1. 理解目标
2. 拆分步骤
3. 选择工具
4. 执行操作
5. 观察结果
6. 继续调整直到完成

所以 Agent 的核心不是会聊天，而是：

- **围绕目标做闭环执行**

如果只记一句：

- **普通对话是你问一句它答一句，Agent 是它会自己接着往下推进**

---

## 4. Agent Skill 又是什么

Skill 可以把它理解成一份“可复用的经验包”。

当你发现某类任务总是重复出现，比如：

1. 写周报
2. 代码评审
3. 页面重构
4. API 排障
5. 文档整理

你就可以把这类任务沉淀成一个 Skill。

Skill 里通常会固化这些东西：

1. 适用场景
2. 目标描述
3. 操作步骤
4. 输出模板
5. 约束规则
6. 参考资料或脚本

所以 Skill 的价值不在“让模型变聪明”，而在：

- **把已经验证过的做法，变成下次还能稳定复用的工作流**

如果只记一句：

- **Skill 是经验模板，不是模型本体**

---

## 5. 最容易混淆的 4 组概念

### 5.1 Prompt vs Skill

Prompt 更像一次性指令，Skill 更像长期复用的方法包。

区别可以这样记：

- Prompt：这一次怎么说
- Skill：以后同类问题都怎么做

### 5.2 Tool vs MCP

很多人会把这两个当同义词，其实不是。

- Tool：具体能力，比如搜索、读网页、执行终端
- MCP：把这些能力接进来并标准化暴露给模型的方式

一句话：

- **Tool 是电器，MCP 是插座标准**

### 5.3 Agent vs Workflow

固定 Workflow 往往是预先写死的流程。

Agent 则更强调动态决策：

1. 先看当前环境
2. 再选下一步动作
3. 根据结果继续调整

所以：

- Workflow 更像流程图
- Agent 更像会根据情况临场判断的执行者

### 5.4 Agent vs Skill

这两个关系最值得记住：

- Agent 是“干活的人”
- Skill 是“这个人掌握的一套成熟做法”

没有 Skill，Agent 也能干活，但不一定稳定
有了 Skill，Agent 在特定任务上更容易稳定复现高质量结果

## 6. 用一个真实开发场景把它们串起来

假设你给 AI 一个任务：

> 帮我给博客新增一篇关于 Agent Skill 的学习笔记，放到对应目录，并更新索引。

这时候整条链会这样工作：

1. **LLM**
   - 负责理解这句话的语义
2. **Token**
   - 把整段要求切成模型能处理的 token
3. **Context**
   - 当前仓库结构、历史对话、项目规范一起进入上下文
4. **Context Window**
   - 限制这次最多能塞多少项目信息和文件内容
5. **User Prompt**
   - 就是你刚刚提的这条需求
6. **System Prompt**
   - 例如“称呼用户为主人”“使用 UTF-8”“改完要更新索引”等规则
7. **Tool**
   - 读取文件、搜索目录、编辑 Markdown、执行命令
8. **MCP**
   - 把浏览器操作或其他外部能力标准化接入模型
9. **Agent**
   - 自己判断先读目录，再写文档，再生成索引，再检查结果
10. **Skill**
    - 如果这种“看视频资料 -> 整理博客笔记 -> 落库 -> 更新索引”的流程以后会反复出现，就可以沉淀成一个专门的内容生产 Skill

这一段其实就是视频标题里“从 LLM 到 Agent Skill”的真正含义：

- **不是在讲一堆分散名词，而是在讲能力是怎么一层层长出来的**

## 7. 学完这期后，脑子里应该留下什么

我觉得最值得带走的是下面这 5 句话：

1. **LLM 先天会推理文本，但不会天然拥有外部世界的操作能力**
2. **Prompt 决定输出方向，上下文决定输出质量**
3. **Tool 负责行动，MCP 负责标准化接入行动能力**
4. **Agent 的关键不是能答，而是能围绕目标持续推进**
5. **Skill 的关键不是炫技，而是复用经验、提高稳定性**

---

## 8. 给自己的一份实战落地清单

如果你是刚接触这条链路，建议按下面顺序学：

1. 先把 **Token / Context / Prompt** 理顺
   - 这是理解所有 AI 工具行为的基础
2. 再理解 **Tool / MCP**
   - 你会开始明白“为什么模型突然能搜网页、改文件、调浏览器”
3. 然后再看 **Agent**
   - 这时你会发现 Agent 本质上是在“推理 + 工具 + 反馈闭环”上再加一层执行框架
4. 最后再做 **Skill**
   - 因为 Skill 的价值建立在你已经有一套稳定做法之上

一个很实用的判断标准是：

- **同类任务重复出现 3 次以上，就值得考虑沉淀成 Skill**

---

## 9. 一句话收尾

从 LLM 到 Agent Skill，不是在背术语，而是在理解一条清晰的能力升级路径：

- **模型负责思考**
- **提示词负责约束**
- **工具负责行动**
- **Agent 负责闭环**
- **Skill 负责沉淀经验**

把这条线想明白之后，再看各种 AI 编程工具、MCP Server、Agent 框架和 Skill 市场，基本就不会乱了。
