## Specs 系统概述

在传统的 AI 辅助编程工作流中，开发者往往直接向 AI 描述需求，AI 立即生成代码，整个过程缺乏中间的规划与设计环节。这种方式在处理简单任务时效率很高，但面对复杂功能时，AI 容易产生偏差，生成的代码难以满足真实需求，后续维护成本也居高不下。

Kiro 的 Specs 系统正是为了解决这个问题而设计的。Specs（规格文档）是 Kiro 最核心、最具差异化的功能，它要求开发者在 AI 执行任何代码修改之前，先通过结构化的文档明确三个层次的内容：做什么（需求）、怎么做（设计）、分几步做（任务）。

这种方式的核心价值体现在以下几个方面：

**可预测性**：AI 按照明确的规格执行，而不是凭感觉猜测开发者的意图。每一步操作都有文档依据，行为可预测、可审查。

**可追溯性**：Spec 文件存储在项目目录中，纳入版本控制。三个月后回看某个功能的开发过程，需求变更、设计决策、任务执行记录一目了然。

**可协作性**：团队成员可以在 Spec 文档上进行异步协作，在 AI 开始执行之前就对方案达成共识，避免返工。

**可控性**：任务列表中的每个步骤都可以单独执行、暂停、回滚。开发者始终掌握主导权，AI 是执行者而非决策者。

Kiro 官方将这种开发方式称为"Spec 驱动开发"，与随意的"Vibe Coding"形成鲜明对比。对于需要长期维护的生产项目，Spec 驱动开发能显著提升代码质量和团队协作效率。

## 三层文档结构

每个 Spec 由三个 Markdown 文件组成，分别对应软件开发的三个核心阶段：需求分析、技术设计、任务执行。

### requirements.md — 需求文档

需求文档回答"做什么"的问题。它使用 EARS（Easy Approach to Requirements Syntax）格式描述验收标准，将模糊的需求转化为精确的、可验证的条件语句。

EARS 格式的核心是用结构化的句式描述系统行为，常见模式包括：

- **When**（触发条件）：`当用户点击提交按钮时，系统应验证表单数据`
- **While**（持续状态）：`当用户处于登录状态时，系统应显示个人中心入口`
- **If/Then**（条件响应）：`如果表单验证失败，则系统应在对应字段下方显示错误提示`
- **The system shall**（无条件要求）：`系统应支持中英文双语界面`

需求文档的每一条验收标准都应该是可测试的——你能明确判断这条标准是否被满足。模糊的描述（如"界面要好看"）不是合格的需求，应该转化为具体的标准（如"按钮点击区域不小于 44×44 像素，符合无障碍设计规范"）。

```markdown
# Requirements: 用户认证功能

## 功能需求

### 1. 用户注册

- 1.1 当用户提交注册表单时，系统应验证邮箱格式是否合法
- 1.2 当邮箱已被注册时，系统应显示"该邮箱已被使用"的错误提示
- 1.3 当密码长度少于 8 位时，系统应拒绝注册并提示密码强度要求
- 1.4 当注册成功时，系统应向用户邮箱发送验证邮件

### 2. 用户登录

- 2.1 当用户输入正确的邮箱和密码时，系统应在 2 秒内完成登录
- 2.2 当连续登录失败 5 次时，系统应锁定账号 15 分钟
- 2.3 系统应支持"记住我"功能，有效期 30 天

## 非功能需求

- 3.1 所有密码必须使用 bcrypt 算法加密存储，不得明文保存
- 3.2 登录接口应有频率限制，每 IP 每分钟不超过 10 次请求
```

### design.md — 设计文档

设计文档回答"怎么做"的问题。它描述技术方案、架构决策、数据模型、接口设计，是需求与代码之间的桥梁。

一份好的设计文档通常包含以下部分：

- **概述**：用一段话描述本功能的技术实现思路
- **架构图**：用 Mermaid 或文字描述组件关系、数据流向
- **数据模型**：定义核心数据结构、数据库表结构
- **接口设计**：描述 API 端点、请求/响应格式
- **关键决策**：记录重要的技术选型及其理由
- **错误处理**：描述异常情况的处理策略

```markdown
# Design: 用户认证功能

## 概述

采用 JWT（JSON Web Token）无状态认证方案，前端存储 access token，
后端不维护 session 状态，便于水平扩展。

## 数据模型

### User 表

| 字段       | 类型         | 说明          |
| ---------- | ------------ | ------------- |
| id         | UUID         | 主键          |
| email      | VARCHAR(255) | 唯一索引      |
| password   | VARCHAR(60)  | bcrypt 哈希值 |
| created_at | TIMESTAMP    | 注册时间      |

## 关键决策

- 选择 JWT 而非 Session：便于前后端分离部署，无需共享 session 存储
- access token 有效期设为 1 小时，refresh token 有效期 30 天
```

### tasks.md — 任务列表

任务列表回答"分几步做"的问题。它将设计文档中的方案拆解为原子化的、可独立执行的任务，每个任务都有明确的完成标准和状态标记。

任务列表使用 Markdown 复选框语法表示状态，Kiro 会自动解析并追踪每个任务的执行进度：

```markdown
# Implementation Plan: 用户认证功能

## Tasks

- [ ] 1. 搭建数据库基础设施
  - [ ] 1.1 创建 User 数据表迁移脚本
    - 包含 id、email、password、created_at 字段
    - 为 email 字段添加唯一索引
  - [ ] 1.2 配置数据库连接池

- [x] 2. 实现用户注册接口（已完成）
  - [x] 2.1 创建 POST /auth/register 端点
  - [x] 2.2 实现邮箱格式验证逻辑
  - [x] 2.3 集成 bcrypt 密码加密

- [-] 3. 实现用户登录接口（进行中）
  - [-] 3.1 创建 POST /auth/login 端点
  - [ ] 3.2 实现登录失败次数限制
  - [ ] 3.3 实现 JWT token 生成与返回
```

状态标记说明：

- `[ ]`：未开始（not_started）
- `[-]`：进行中（in_progress）
- `[x]`：已完成（completed）

## Feature Spec vs Bugfix Spec

Kiro 支持两种类型的 Spec，分别针对新功能开发和缺陷修复两种不同场景。

### Feature Spec（功能规格）

Feature Spec 用于开发新功能或对现有功能进行较大改动。它遵循完整的三层文档结构（requirements → design → tasks），适合需要充分规划的场景。

Feature Spec 的典型使用场景：

- 开发一个全新的业务模块（如用户系统、支付流程）
- 对现有功能进行重构或架构升级
- 添加复杂的交互逻辑或数据处理流程
- 需要多人协作、分阶段交付的功能

Feature Spec 的核心价值在于"先想清楚再动手"。通过需求文档和设计文档的撰写过程，开发者往往能提前发现潜在的问题和边界情况，避免在代码层面反复返工。

### Bugfix Spec（缺陷修复规格）

Bugfix Spec 用于修复已知的缺陷。与 Feature Spec 不同，它的文档结构更加精简，核心是准确描述 Bug 的触发条件、根本原因和修复方案。

Bugfix Spec 通常包含：

- **Bug 描述**：复现步骤、预期行为、实际行为
- **根本原因分析**：通过代码审查或日志分析定位问题根源
- **修复方案**：描述如何修复，以及修复后的验证方式
- **任务列表**：修复步骤的拆解

Bugfix Spec 的一个重要特性是支持**属性测试（Property-Based Testing）**工作流。Kiro 鼓励在修复 Bug 之前先编写一个能够复现该 Bug 的测试，确认测试失败（证明 Bug 存在），然后修复代码，最终确认测试通过（证明 Bug 已修复）。这种"先写测试再修复"的方式能有效防止 Bug 复发。

### 两种类型的对比

| 维度     | Feature Spec               | Bugfix Spec               |
| -------- | -------------------------- | ------------------------- |
| 适用场景 | 新功能开发、重构           | 缺陷修复                  |
| 文档结构 | 完整三层（需求/设计/任务） | 精简（Bug描述/原因/修复） |
| 规划深度 | 深度规划，充分设计         | 快速定位，精准修复        |
| 测试策略 | 验收测试                   | 复现测试 + 回归测试       |
| 典型周期 | 数天到数周                 | 数小时到数天              |

## Requirements-First 工作流

Requirements-First（需求优先）是 Kiro 推荐的标准工作流，适合大多数功能开发场景。顾名思义，这种工作流从需求文档开始，逐步推进到设计和任务。

### 工作流步骤

**第一步：描述需求，生成 requirements.md**

在 Kiro Agent 对话框中，用自然语言描述你想要实现的功能。Kiro 会根据你的描述，自动生成结构化的 requirements.md 文件，将模糊的需求转化为 EARS 格式的验收标准。

你不需要一开始就把需求描述得非常精确，Kiro 会在生成文档后与你确认，你可以对每一条验收标准进行修改、补充或删除。这个过程本身就是一次需求澄清的机会。

**第二步：审查需求，确认无误**

仔细阅读生成的 requirements.md，检查每一条验收标准是否准确反映了你的意图。特别注意：

- 是否有遗漏的边界情况？
- 非功能需求（性能、安全、可访问性）是否已覆盖？
- 验收标准是否足够具体，可以被测试？

这一步不要急于跳过。需求文档的质量直接决定后续设计和实现的质量。

**第三步：生成 design.md**

需求确认后，告诉 Kiro 生成设计文档。Kiro 会基于 requirements.md 的内容，结合项目现有的技术栈（通过 Steering 文件获取），生成技术设计方案。

设计文档生成后，同样需要仔细审查：架构是否合理？数据模型是否完整？是否有更好的技术选型？

**第四步：生成 tasks.md**

设计确认后，Kiro 会将设计文档中的方案拆解为具体的任务列表。每个任务都应该足够小，可以在一次 Agent 会话中完成。

**第五步：逐任务执行**

任务列表生成后，你可以让 Kiro 逐个执行任务。每执行完一个任务，Kiro 会将其状态更新为已完成，并等待你的确认再继续下一个任务。这种逐步执行的方式让你始终掌握开发进度，可以在任何时候暂停、审查或调整方向。

### 适用场景

Requirements-First 工作流特别适合以下场景：

- 需求相对清晰，但技术方案尚未确定
- 团队协作项目，需要在开始编码前对需求达成共识
- 功能较为复杂，需要充分的前期规划
- 对代码质量和可维护性有较高要求的生产项目

## Design-First 工作流

Design-First（设计优先）工作流适合另一种场景：你已经有了清晰的技术方案，但需要从设计反推需求文档，或者你更习惯从技术角度思考问题。

### 工作流步骤

**第一步：描述技术方案，生成 design.md**

直接向 Kiro 描述你的技术设计思路，包括架构方案、数据模型、关键接口等。Kiro 会生成结构化的设计文档。这种方式特别适合有丰富经验的开发者——他们往往在脑海中已经有了清晰的技术方案，不需要经过需求分析阶段，可以直接从设计开始。

**第二步：从设计反推需求**

设计文档生成后，Kiro 可以帮助你从技术设计中提炼出对应的需求文档。这个过程有助于确保设计与业务需求的一致性，也便于后续的需求变更管理。

**第三步：生成并执行任务列表**

后续步骤与 Requirements-First 工作流相同：生成任务列表，逐步执行。

### 两种工作流的选择建议

| 场景                       | 推荐工作流         |
| -------------------------- | ------------------ |
| 需求明确，技术方案待定     | Requirements-First |
| 技术方案明确，需求文档待补 | Design-First       |
| 初级开发者，需要引导       | Requirements-First |
| 资深开发者，快速原型       | Design-First       |
| 团队协作，需要需求评审     | Requirements-First |
| 个人项目，追求开发速度     | Design-First       |

两种工作流并没有绝对的优劣之分，关键是根据具体场景灵活选择。Kiro 对两种工作流都提供完整支持。

## 任务执行与状态追踪

任务执行是 Spec 工作流的最后一个阶段，也是 AI 真正"动手"的阶段。Kiro 提供了精细的任务状态追踪机制，确保整个执行过程透明可控。

### 任务状态机制

Kiro 使用 Markdown 复选框语法追踪任务状态，共有三种状态：

- `[ ]`（not_started）：任务尚未开始
- `[-]`（in_progress）：任务正在执行中
- `[x]`（completed）：任务已完成

当 Kiro 开始执行某个任务时，会自动将其状态从 `[ ]` 更新为 `[-]`；任务完成后，再更新为 `[x]`。这些状态变更会实时写入 tasks.md 文件，你可以随时查看当前进度。

### 任务执行的最佳实践

**一次只执行一个任务**：Kiro 默认每次只执行一个任务，执行完毕后等待你的确认。这种方式让你有机会在每个步骤后审查代码变更，及时发现问题。

**任务粒度要适中**：任务不应该太大（一个任务涉及多个文件的大量修改），也不应该太小（一个任务只改一行代码）。理想的任务粒度是：一次 Agent 会话可以完成，且完成后有明确的可验证结果。

**利用子任务分解复杂步骤**：对于较复杂的任务，可以在 tasks.md 中使用缩进创建子任务。Kiro 会先完成所有子任务，再将父任务标记为完成。

**任务失败时的处理**：如果某个任务执行失败（如测试不通过、编译报错），Kiro 会保持任务状态为 `[-]`（进行中），并在对话中说明失败原因。你可以选择让 Kiro 重试、手动修复，或者调整任务描述后重新执行。

### 任务执行的交互模式

Kiro 支持两种任务执行模式：

**手动模式**：每次只执行一个任务，执行完毕后等待用户确认再继续。适合需要仔细审查每个步骤的场景。

**自动模式**：连续执行多个任务，直到遇到需要用户决策的情况才暂停。适合对 Spec 内容已经充分审查、希望快速推进的场景。

## Spec 文件存储位置

Kiro 的所有 Spec 文件都存储在项目根目录的 `.kiro/specs/` 目录下，每个 Spec 对应一个以功能名称命名的子目录。

### 目录结构

```
.kiro/
└── specs/
    ├── user-authentication/
    │   ├── requirements.md
    │   ├── design.md
    │   └── tasks.md
    ├── payment-flow/
    │   ├── requirements.md
    │   ├── design.md
    │   └── tasks.md
    └── todo-list/
        ├── requirements.md
        ├── design.md
        └── tasks.md
```

### 命名规范

Spec 目录名建议使用小写字母和连字符（kebab-case），与功能的英文名称对应。例如：

- `user-authentication`（用户认证）
- `payment-flow`（支付流程）
- `data-export`（数据导出）
- `notification-system`（通知系统）

避免使用中文目录名，以确保跨平台兼容性。

### 版本控制建议

`.kiro/specs/` 目录应该纳入 Git 版本控制（不要添加到 `.gitignore`）。这样做的好处是：

- Spec 文档的变更历史可以追溯
- 团队成员可以通过 Pull Request 对 Spec 进行评审
- 任务执行状态的变更也会被记录，便于回顾开发过程

## 实战示例

下面通过一个具体示例，演示如何从零开始创建并执行一个完整的 Feature Spec。

### 场景描述

假设我们要为一个博客系统添加"文章标签"功能：用户可以为文章添加多个标签，并通过标签筛选文章。

### 第一步：在 Kiro 中发起 Spec 创建

在 Agent 对话框中输入：

```
我需要为博客系统添加文章标签功能。
具体需求：
- 创建文章时可以添加多个标签（最多 5 个）
- 标签名称长度 2-20 个字符
- 首页可以按标签筛选文章列表
- 标签支持中英文

请帮我创建一个 Feature Spec。
```

Kiro 会自动在 `.kiro/specs/article-tags/` 目录下生成三个文件。

### 第二步：审查并完善 requirements.md

Kiro 生成的 requirements.md 示例（已简化）：

```markdown
# Requirements: 文章标签功能

## 功能需求

### 1. 标签管理
- 1.1 当用户创建或编辑文章时，系统应允许添加最多 5 个标签
- 1.2 当标签名称少于 2 个字符或超过 20 个字符时，系统应拒绝并提示
- 1.3 系统应支持中文和英文标签名称
- 1.4 当用户尝试为同一篇文章添加重复标签时，系统应忽略重复项

### 2. 标签筛选
- 2.1 首页文章列表应显示标签筛选区域
- 2.2 当用户点击某个标签时，系统应只显示包含该标签的文章
- 2.3 当用户取消标签筛选时，系统应恢复显示全部文章
```

### 第三步：确认设计方案并执行任务

需求确认后，让 Kiro 生成 design.md，审查数据模型和接口设计是否合理，确认无误后生成任务列表，逐步执行：

```
请执行 article-tags Spec 中的第一个任务
```

Kiro 会开始执行，完成后展示代码变更，等待你确认后继续下一个任务。

## 注意事项与常见问题

### Spec 文档的质量直接影响实现质量

Kiro 的 AI 能力再强，也无法弥补需求文档的缺陷。如果 requirements.md 中的验收标准模糊或不完整，生成的代码很可能无法满足真实需求。建议在确认需求文档之前，多花时间思考边界情况和非功能需求。

### 不要跳过设计文档

有些开发者习惯跳过 design.md，直接从需求跳到任务执行。这在简单功能上可行，但对于复杂功能，缺少设计文档会导致 AI 在实现时做出不一致的技术决策，最终产生难以维护的代码。

### 任务粒度的把握

任务列表中的每个任务应该足够小，以便在一次 Agent 会话中完成，同时又足够有意义，完成后有可验证的结果。经验法则：一个任务对应一个文件或一个功能点的修改，而不是整个模块的重写。

### Spec 与实际代码的同步

随着开发推进，需求可能发生变化。当需求变更时，应该先更新 requirements.md，再更新 design.md，最后更新 tasks.md，保持三层文档的一致性。不要让 Spec 文档与实际代码脱节，否则它就失去了追踪和协作的价值。

### 多个 Spec 并行时的注意事项

如果项目中同时有多个 Spec 在进行，要注意不同 Spec 之间可能存在的依赖关系和代码冲突。建议在开始执行某个 Spec 的任务之前，确认其依赖的其他 Spec 已经完成或处于稳定状态。

### Bugfix Spec 的特殊处理

对于 Bugfix Spec，强烈建议遵循"先写复现测试，再修复代码"的原则。这不仅能确保 Bug 被真正修复，还能防止同类问题在未来再次出现。Kiro 的 Bugfix 工作流对这种模式有专门的支持。

### 常见问题

**问：Spec 文件可以在多台机器之间同步吗？**

答：可以。Spec 文件是普通的 Markdown 文件，通过 Git 同步即可。团队成员可以在各自的机器上查看和编辑 Spec，通过 Pull Request 进行协作。

**问：一个 Spec 可以对应多个功能吗？**

答：不建议。每个 Spec 应该对应一个独立的功能或修复任务，保持职责单一。如果功能过于复杂，可以拆分为多个相互关联的 Spec。

**问：Spec 任务执行到一半，能暂停后继续吗？**

答：完全可以。任务状态保存在 tasks.md 文件中，下次打开 Kiro 时，Agent 可以读取当前状态，从上次中断的地方继续执行。

**问：如何删除一个不再需要的 Spec？**

答：直接删除 `.kiro/specs/{feature_name}/` 目录即可。Kiro 不会对删除操作有任何限制。

**问：Spec 文档必须用英文写吗？**

答：不是必须的。Kiro 支持中文 Spec 文档，包括中文的需求描述、设计说明和任务名称。但 Spec 目录名建议使用英文，以避免跨平台路径问题。
