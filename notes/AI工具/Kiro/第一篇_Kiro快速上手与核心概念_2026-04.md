## Kiro 是什么

Kiro 是 Amazon Web Services（AWS）于 2025 年推出的一款 Agentic IDE（智能体集成开发环境）。它基于 VS Code 内核构建，因此对于熟悉 VS Code 的开发者来说，上手几乎没有门槛——界面布局、快捷键、插件生态都高度兼容。

与市面上其他 AI 编程工具最大的不同在于，Kiro 不仅仅是一个"代码补全"或"对话式编程"工具，它的核心理念是**规格驱动开发（Spec-Driven Development）**。Kiro 鼓励开发者在动手写代码之前，先通过结构化的文档（Specs）明确需求、设计方案和任务拆解，再由 AI Agent 按照规格逐步执行。这种方式让 AI 的行为更加可预测、可追溯、可审查。

Kiro 底层使用 Anthropic 的 Claude Sonnet 4 模型，具备强大的代码理解与生成能力。AWS 将其定位为面向专业开发者的生产力工具，而非面向初学者的"玩具"。

### Kiro 与 VS Code 的关系

Kiro 是一个独立安装的 IDE，并非 VS Code 插件。它 fork 自 VS Code 开源内核（Code - OSS），在此基础上深度集成了 AI Agent 能力。这意味着：

- 你可以直接使用 VS Code 的主题、字体、键位绑定
- 大多数 VS Code 扩展可以在 Kiro 中安装使用
- 项目文件结构、终端、Git 集成与 VS Code 完全一致
- Kiro 特有的功能（Specs、Steering、Hooks、MCP）以独立面板的形式叠加在 VS Code 体验之上

### 发布背景

2025 年是 AI 编程工具爆发的一年。Cursor、GitHub Copilot、Claude Code、Codex 等工具相继推出，各有侧重。AWS 推出 Kiro 的核心差异化在于：将软件工程的规范性（Spec 文档、任务追踪）与 AI 的执行能力结合，试图解决"Vibe Coding"带来的代码质量不可控问题。

## 安装与启动

### 下载安装包

访问 [https://kiro.dev](https://kiro.dev) 官网，点击下载按钮，根据操作系统选择对应安装包：

- **macOS**：下载 `.dmg` 文件，拖拽到 Applications 文件夹即可
- **Windows**：下载 `.exe` 安装程序，按向导完成安装
- **Linux**：下载 `.deb`（Debian/Ubuntu）或 `.rpm`（Fedora/RHEL）包，或使用 `.tar.gz` 解压版

Kiro 不需要通过命令行安装，全程 GUI 操作。安装完成后，在应用列表中找到 Kiro 图标启动即可。

### 首次启动与登录

首次启动 Kiro 时，需要使用 AWS Builder ID 登录（免费注册）。登录后即可使用 Kiro 的全部 AI 功能，包括 Agent 对话、Specs 生成、Hooks 配置等。

登录步骤：

1. 启动 Kiro，点击欢迎页面的"Sign in"按钮
2. 浏览器会自动打开 AWS Builder ID 登录页
3. 使用邮箱注册或登录 AWS Builder ID
4. 授权后浏览器提示"返回 Kiro"，点击确认
5. Kiro 界面刷新，显示已登录状态

### 打开项目

与 VS Code 完全一致：通过菜单 `File → Open Folder` 打开本地项目目录，或直接将文件夹拖拽到 Kiro 窗口。Kiro 会自动识别项目类型（Node.js、Python、Java 等），并在 Agent 对话中提供上下文感知的建议。

## 界面布局

Kiro 的界面在 VS Code 基础上新增了几个专属面板，整体布局如下：

### 左侧活动栏

左侧活动栏除了 VS Code 原有的文件浏览器、搜索、Git、扩展图标外，新增了 Kiro 专属图标：

- **Kiro Agent 图标**：打开 Agent 对话面板，这是与 AI 交互的主入口
- **Specs 图标**：查看和管理当前项目的所有 Spec 文档（位于 `.kiro/specs/` 目录）
- **Steering 图标**：管理 Steering 文件（位于 `.kiro/steering/` 目录），用于向 Agent 注入持久知识
- **Hooks 图标**：查看和配置自动化 Hook（位于 `.kiro/hooks/` 目录）
- **MCP 图标**：管理 MCP（Model Context Protocol）服务器连接

### 右侧 Agent 对话面板

点击 Kiro Agent 图标后，右侧会展开对话面板。这里是与 AI 进行自然语言交互的地方。你可以：

- 直接输入需求，让 Agent 生成代码
- 要求 Agent 创建 Spec（规格文档）
- 让 Agent 执行 Spec 中的任务
- 询问代码问题、请求代码审查

对话面板支持多轮对话，Agent 会记住当前会话的上下文。

### 编辑器区域

与 VS Code 完全一致的代码编辑区域，支持语法高亮、代码补全、多光标编辑等所有 VS Code 功能。当 Agent 修改文件时，会在编辑器中以 diff 形式展示变更，你可以逐个接受或拒绝。

### 底部终端

与 VS Code 集成终端完全一致，Agent 执行命令时会在此处显示输出。

## 核心概念速览

Kiro 有四个核心概念，理解它们是用好 Kiro 的关键：

### Specs（规格文档）

Specs 是 Kiro 最核心的概念。每个 Spec 对应一个功能或修复任务，由三个 Markdown 文件组成：

- `requirements.md`：需求文档，用 EARS 格式描述验收标准
- `design.md`：设计文档，描述技术方案、架构决策
- `tasks.md`：任务列表，将设计拆解为可执行的步骤，带状态追踪

Specs 存储在项目的 `.kiro/specs/{feature_name}/` 目录下，是人类可读、可版本控制的文档。

### Steering（上下文引导）

Steering 是向 Agent 注入持久知识的机制。通过在 `.kiro/steering/` 目录下创建 Markdown 文件，你可以告诉 Agent：项目的技术栈、代码规范、业务背景、禁止事项等。

Steering 文件有四种加载模式：`always`（始终加载）、`fileMatch`（文件匹配时加载）、`manual`（手动引用时加载）、`auto`（Agent 自动判断是否加载）。

### Hooks（自动化钩子）

Hooks 是事件驱动的自动化机制。你可以配置：当某个 IDE 事件发生时（如文件保存、Spec 任务完成、Agent 停止），自动触发一个动作（运行命令或让 Agent 执行某个任务）。

例如：每次保存 TypeScript 文件时自动运行 `tsc --noEmit` 检查类型错误。

### MCP（Model Context Protocol）

MCP 是 Anthropic 提出的开放协议，允许 AI 模型连接外部工具和数据源。Kiro 内置了 MCP 支持，你可以通过配置 `.kiro/mcp.json` 接入各种 MCP 服务器，为 Agent 扩展能力——如数据库查询、文件系统操作、API 调用等。

## Spec 驱动开发 vs Vibe Coding

这是理解 Kiro 设计哲学的关键对比。

### 什么是 Vibe Coding

"Vibe Coding"是 2025 年流行起来的一个词，指的是一种随意、即兴的 AI 辅助编程方式：开发者凭感觉向 AI 描述需求，AI 生成代码，开发者直接接受，不做深入审查，遇到问题再让 AI 修复，如此循环。

这种方式的优点是速度快、门槛低，适合快速原型验证。但缺点也很明显：

- 代码质量难以保证，技术债务快速积累
- AI 的行为不可预测，同样的需求描述可能产生完全不同的实现
- 缺乏文档，后续维护困难
- 在复杂项目中，AI 容易"跑偏"，偏离原始意图

### Spec 驱动开发的理念

Kiro 倡导的 Spec 驱动开发，要求在 AI 执行任何代码修改之前，先完成三个层次的文档：

1. **需求层**（requirements.md）：明确"做什么"，用结构化的验收标准描述功能边界
2. **设计层**（design.md）：明确"怎么做"，描述技术方案、数据模型、接口设计
3. **任务层**（tasks.md）：明确"分几步做"，将设计拆解为原子化的可执行任务

这三层文档不仅是给 AI 看的，更是给人类看的。它们让整个开发过程透明、可审查、可追溯。

### 两种方式的本质区别

| 维度     | Vibe Coding        | Spec 驱动开发      |
| -------- | ------------------ | ------------------ |
| 需求表达 | 自然语言，随意     | 结构化文档，精确   |
| AI 行为  | 不可预测           | 按规格执行，可追溯 |
| 代码质量 | 参差不齐           | 有设计约束，更稳定 |
| 适用场景 | 快速原型、个人项目 | 团队协作、生产项目 |
| 文档     | 几乎没有           | 自动生成，版本控制 |

Kiro 并不完全排斥 Vibe Coding——它的 Agent 对话模式也支持即兴交互。但对于需要长期维护的项目，Kiro 强烈建议使用 Spec 驱动开发流程。

## 第一次对话

打开 Kiro 并登录后，让我们通过一个简单示例体验 Agent 对话。

### 基础对话示例

在 Agent 对话面板中输入以下内容，感受 Kiro 的响应方式：

```
帮我创建一个简单的 Todo List 功能，使用 Vue 3 + Composition API 实现。
要求：
1. 支持添加、删除、标记完成
2. 数据存储在 localStorage
3. 界面简洁，使用原生 CSS
```

Kiro 会先分析你的需求，然后提出一个实现方案，并询问是否需要先创建 Spec 文档。对于简单任务，你可以直接让它生成代码；对于复杂功能，建议先创建 Spec。

### 创建第一个 Spec

在对话框中输入：

```
请为上面的 Todo List 功能创建一个 Spec 文档
```

Kiro 会自动在 `.kiro/specs/todo-list/` 目录下生成三个文件：`requirements.md`、`design.md`、`tasks.md`。你可以在 Specs 面板中查看和编辑这些文件，确认无误后，让 Agent 按任务列表逐步执行。

### 执行 Spec 任务

在对话框中输入：

```
请执行 todo-list Spec 中的第一个任务
```

Agent 会读取 `tasks.md`，找到第一个未完成的任务，执行相应的代码修改，并将任务状态更新为已完成。整个过程透明可见，你可以随时暂停、审查或回滚。

## 注意事项与常见问题

### 登录问题

**问：启动 Kiro 后无法登录，浏览器跳转失败怎么办？**

答：确认系统默认浏览器可以正常访问 AWS 服务。如果在中国大陆使用，可能需要配置网络代理。Kiro 本身不内置代理，需要在系统层面配置。

**问：AWS Builder ID 是否需要绑定信用卡？**

答：不需要。AWS Builder ID 是免费账号，与 AWS 账号不同，不需要绑定支付方式。Kiro 目前（2026-04）提供免费使用额度。

### 性能问题

**问：Kiro 比 VS Code 占用更多内存吗？**

答：是的，由于集成了 AI 功能，Kiro 的内存占用通常比纯 VS Code 高 200-400MB。建议在 8GB 以上内存的机器上使用，16GB 以上体验更佳。

### Spec 相关问题

**问：必须使用 Spec 才能让 Agent 写代码吗？**

答：不是必须的。Kiro 支持直接在对话中让 Agent 修改代码，不需要先创建 Spec。Spec 是推荐的最佳实践，适合复杂功能开发，但不是强制要求。

**问：Spec 文件可以手动编辑吗？**

答：完全可以，而且鼓励这样做。Spec 文件是普通的 Markdown 文件，存储在项目目录中，可以用任何编辑器修改，也可以通过 Git 进行版本控制。

### 与 VS Code 扩展的兼容性

**问：我的 VS Code 扩展在 Kiro 中能用吗？**

答：大多数扩展可以直接安装使用。但由于 Kiro 是独立的 IDE，VS Code 中已安装的扩展不会自动迁移，需要在 Kiro 中重新安装。部分依赖 VS Code 特定 API 的扩展可能不兼容。

### 网络与隐私

**问：Kiro 会将我的代码发送到 AWS 服务器吗？**

答：是的，当你使用 Agent 功能时，相关代码片段会发送到 AWS 后端进行处理。Kiro 提供了隐私设置，可以控制哪些内容会被发送。对于涉及敏感信息的项目，建议仔细阅读官方隐私政策。
