# 第16篇：前端模块导入与导出完全指南

> 适合人群  
>
> + 学过 JS，但一到 import / export 就模糊  
> + 知道怎么用，但**不知道为什么这么设计**  
> + 看源码 / 面试被追问就慌的人
>

## 一、模块化到底在解决什么问题？
### 1️⃣ 没有模块之前（全局时代）
```javascript
// a.js
var count = 0;

// b.js
count++;
```

问题不是「能不能跑」，而是 **不可控**：

+ 所有变量都在 `window` 上
+ 不知道谁改了谁
+ 文件顺序一变就报错
+ 大项目 = 灾难

👉 **模块化的本质目标只有一句话：**

👉「**限制作用域 + 显式暴露接口**」

---

## 二、什么是“模块”？（非常重要）
在 ES Module 中：

**一个文件 = 一个模块**

### 模块的天然特性
+ 自动拥有 **独立作用域**
+ 内部变量默认 **对外不可见**
+ 只有 `export` 的内容，外部才能用

```javascript
// math.js
const PI = 3.14;
function add(a, b) {
  return a + b;
}
```

此时：

+ `PI` ❌ 外部访问不到
+ `add` ❌ 外部访问不到

👉 除非你 **主动导出**

---

## 三、export：我允许外部使用什么
---

### 1、命名导出（Named Export）
#### 1.1 最直观写法
```javascript
export const PI = 3.14;

export function add(a, b) {
  return a + b;
}
```

特点：

+ 一个模块可以导出 **多个**
+ 每一个导出都有 **明确名字**

---

#### 1.2 工程中最推荐写法 ⭐
```javascript
const PI = 3.14;
function add(a, b) {
  return a + b;
}

export { PI, add };
```

为什么推荐？

+ 逻辑定义和导出 **分离**
+ 一眼能看到「这个模块对外提供什么」

---

#### 1.3 导出时改名（防冲突）
```javascript
export { PI as circlePI };
```

📌 常见场景：

+ 多个模块都有 `config`
+ 多个工具库都有 `init`

---

### 2、默认导出（Default Export）
---

#### 2.1 什么是 default？
default 的设计目标是：  
**「这个模块主要就干这一件事」**

```javascript
export default function add(a, b) {
  return a + b;
}
```

或：

```javascript
const add = (a, b) => a + b;
export default add;
```

---

#### 2.2 default 的铁规则（必记）
+ ❗ 一个模块 **只能有一个 default**
+ ❗ default 导出 **不要求名字**
+ ❗ 导入时名字你随便起

```javascript
import myAdd from './add.js';
```

---

### 3、Named vs Default 深度对比（重点）
| 对比点 | Named Export | Default Export |
| --- | --- | --- |
| 数量 | 多个 | 只能一个 |
| 导入是否结构 | 是 | 否 |
| 导入名是否固定 | 是 | 否 |
| 可维护性 | ⭐⭐⭐⭐ | ⭐⭐ |
| Tree Shaking | 更好 | 一般 |


📌 **工程经验总结：**

+ 工具函数 / 常量 → named
+ 组件 / 类 / 单一职责 → default

---

## 四、import：我从模块里拿什么
---

### 4.1 导入"命名导出"（结构赋值）
```javascript
import { PI, add } from './math.js';
```

本质是：

从模块导出的接口表里，**按名字精确取值**

❗ 名字 **必须完全一致**

---

### 4.2 导入并重命名
```javascript
import { PI as circlePI } from './math.js';
```

---

### 4.3 导入默认导出
```javascript
import add from './add.js';
```

⚠️ 注意：  
这里 **没有 **`**{}**`，因为不是结构

---

### 4.4 default + named 一起导入
```javascript
import add, { PI } from './math.js';
```

规则：

+ **default 一定写前面**
+ **named 写在 {} 里**

---

### 4.5 整个模块导入（命名空间）
```javascript
import * as math from './math.js';

math.add(1, 2);
math.PI;
```

📌 常见于：

+ 工具库
+ 第三方 SDK
+ 不确定具体导出内容的场景

---

## 五、import / export 是“静态”的（面试高频）
---

### 5.1 什么叫“静态”？
在代码 **执行之前**  
JS 引擎就要知道：

+ 你依赖了哪些模块
+ 模块之间的关系图

---

### 5.2 为什么 import 不能写在 if 里？
❌ 错误：

```javascript
if (true) {
  import { add } from './math.js';
}
```

原因不是「语法不让」，而是：

**模块依赖关系必须在编译阶段确定**

---

### 5.3 那怎么动态加载？
```javascript
import('./math.js').then(module => {
  module.add(1, 2);
});
```

特点：

+ 返回 Promise
+ 运行时加载
+ 用于懒加载 / 路由拆分

---

## 六、import 不是“值拷贝”（非常容易错）
---

### 示例
```javascript
// a.js
export let count = 0;

export function inc() {
  count++;
}
```

```javascript
// b.js
import { count, inc } from './a.js';

inc();
console.log(count); // 1
```

❗ 说明：

import 拿到的是 **模块内部变量的引用**  
不是一份拷贝

所以：

+ 可以读
+ 不能直接改（只读绑定）

---

## 七、CommonJS（Node 旧方案，理解用）
```javascript
// 导出
module.exports = {
  add
};

// 导入
const { add } = require('./math');
```

### 和 ES Module 的本质区别
| 对比 | CommonJS | ES Module |
| --- | --- | --- |
| 加载时机 | 运行时 | 编译时 |
| 导入结果 | 值拷贝 | 引用 |
| Tree Shaking | ❌ | ✅ |
| 浏览器 | ❌ | ✅ |

## 八、真实工程里的写法（Vue / React）
### Vue 组件
```javascript
export default {
  name: 'MyComponent'
};
```

```javascript
import MyComponent from './MyComponent.vue';
```

---

### 工具模块（强烈推荐）
```javascript
// utils/math.js
export function add() {}
export function sub() {}
```

```javascript
import { add, sub } from '@/utils/math';
```

## 十、终极理解口诀（背这个）
**export 决定“我给外界什么接口”**  
**import 决定“我用哪些接口”**

**default：单一职责出口**  
**named：多个明确出口**

**ES Module 是结构声明，不是普通代码**

****

## **十一、CommonJS 与 ES Module（ESM）**
### **一、核心本质（所有差异的根源）**
+ **CommonJS****：Node.js 自定义的「动态运行时模块系统」，适配服务器端（本地文件、同步需求）**
+ **ESM****：ECMAScript 官方标准的「静态编译时模块系统」，适配浏览器+服务器跨环境**

### **二、基础概念（先搞懂再看差异）**
| **概念** | **通俗解释** | **核心作用** |
| --- | --- | --- |
| **编译** | **代码执行前的「备菜阶段」：检查语法、整理依赖（不执行逻辑）** | **提前排错、准备执行环境** |
| **运行** | **真正执行代码的「炒菜阶段」：赋值、调用、加载（实际干活）** | **执行业务逻辑** |
| **静态** | **编译阶段完成的操作（提前定死，不可变）** | **支持优化、提前校验** |
| **动态** | **运行阶段完成的操作（临时决定，灵活可变）** | **适配动态需求** |


### **三、核心差异对比**
| **对比维度** | **CommonJS** | **ESM** |
| --- | --- | --- |
| **解析时机** | **运行时动态解析（执行到才处理）** | **编译时静态解析（执行前定依赖）** |
| **导入导出本质** | **导出「模块.exports 对象的快照引用」（值/引用拷贝，不同步更新）** | **导出「变量动态绑定」（实时引用，内部修改外部同步）** |
| **加载方式** | **默认同步加载（require 是同步操作）** | **同步+异步兼容（浏览器异步、Node 双支持）** |
| **静态优化** | **不支持 Tree-Shaking** | **支持 Tree-Shaking、类型检查等优化** |
| **顶层作用域** | **函数作用域（this 指向 module.exports，有 __filename/__dirname）** | **独立模块作用域（this 是 undefined，无 __filename/__dirname）** |
| **if 里写导入** | **require(...) 正常执行（动态兼容）** | **import ... 语法报错；需用 import() 函数（动态补充语法）** |
| **循环依赖处理** | **可能拿到未完全初始化的快照** | **动态绑定，后续初始化会同步更新** |


### **四、实用关键要点**
1. **ESM 条件加载****：不能直接在 if 里写 **`**import**`**，需用 **`**import('./module.js').then(...)**`**（返回 Promise，支持 async/await）**
2. **CommonJS 缓存****：模块首次 require 后缓存，后续重复 require 直接取缓存，不重新加载**
3. **ESM 只读限制****：导入的变量默认只读，不能直接重新赋值（需通过导出函数修改内部变量）**
4. **Tree-Shaking****：仅 ESM 支持（编译时剔除未使用代码，打包体积更小）**

### **五、总结**
+ **CommonJS 核心：****动态灵活****（运行时处理，支持动态路径、条件加载），牺牲优化能力**
+ **ESM 核心：****静态标准****（编译时处理，支持优化、跨环境），牺牲部分动态灵活性**
+ **关键记住：静态=编译时做，动态=运行时做；import 是静态指令（怕 if），require 是动态函数（爱 if）**

## 十二、为什么 `import componentsMap from './widgets/components'` 不是 `index.vue`？

你今天这个疑惑非常典型，很多人第一次看到都会懵。

### 1）这行代码到底在干嘛

```javascript
import componentsMap from './widgets/components'
```

这是 **默认导入**，并且导入的是一个“目录路径”（没有写具体文件名）。

关键点在这里：  
“目录路径最终会解析到哪个文件”不是 ES 语法自己拍板，而是由当前运行环境（Node / Webpack / Vite）决定。

### 2）为什么你会以为是 `index.vue`

因为在 Vue 项目里我们经常看到 `xxx/index.vue` 作为页面入口，所以直觉会认为目录导入也会优先找 `index.vue`。

但多数构建工具的默认规则并不是这样，通常是：

1. 先看是否有 `index` 这个主文件名（`mainFiles` 常见默认就是 `index`）
2. 再按可解析扩展名列表去补全（`extensions`）
3. 默认扩展名里一般会有 `.js`、`.json`，不一定包含 `.vue`

因此，如果目录里有 `index.js`，大多数情况下会先命中它，而不是你脑补的 `index.vue`。

### 3）分环境看，结论更稳

1. Node.js ESM：相对导入通常要求写完整扩展名，`import './widgets/components'` 这种目录省略写法不一定按你想的工作。
2. Webpack：可以配置 `resolve.mainFiles` 和 `resolve.extensions`，很多项目默认会命中 `index.js`。
3. Vite：支持 `resolve.extensions`，但官方建议对 `.vue` 这类文件尽量显式写扩展名，减少解析歧义。

所以不要背“固定查找顺序”，要背“先看当前工具链配置”。

### 4）你这个场景为什么能跑通

你描述里提到 `./widgets/components/index.js` 使用了 `require.context`。

这类 `index.js` 通常会做两件事：

1. 扫描当前目录下所有 `.vue` 文件
2. 组装成 `componentsMap` 并默认导出

于是外部拿到的是一个“组件名 -> 组件对象”的映射表，自然可以这样动态渲染：

```vue
<component v-if="componentsMap[element]" :is="componentsMap[element]" />
```

### 5）一眼定位这种问题的排查清单

1. 先看导入路径是否省略了扩展名和文件名
2. 立即去同目录找 `index.js` / `index.ts` / `index.vue`
3. 检查构建配置的 `resolve.extensions` 与 `resolve.mainFiles`
4. 如果是老 Webpack 项目，再确认是否用了 `require.context`
5. 如果是 Vite 项目，优先考虑 `import.meta.glob` 替代 `require.context`

### 6）给自己的一句口诀

“目录导入不是找 Vue 页面入口，而是找工具链定义的模块入口；先看 `index.js`，再看配置，不要靠感觉猜。”
