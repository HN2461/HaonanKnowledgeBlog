## uni-app 工程结构、配置分层与文件职责边界

> 很多 uni-app 项目一开始看起来还能跑，但写一阵子后就开始越来越乱。  
> 这种乱，很多时候不是业务真的复杂，而是工程结构和配置职责从第一天就没有立住。  
> 页面注册、平台配置、全局样式、分包、静态资源、应用入口这些东西一旦混在一起，后面谁都不敢改，项目会越来越重。

## 一、这篇最核心的一句话
`uni-app` 项目最怕的，不是文件多，而是“配置放错层”。`

常见会混掉的东西有：

+ 应用入口
+ 页面注册
+ 页面窗口样式
+ tabBar
+ 分包
+ 应用名称和版本
+ appid 和平台打包参数
+ 网络超时
+ 全局样式变量
+ 静态资源目录
+ 平台专属配置

这些东西如果没有固定边界，最后就会出现：

1. 页面明明写了却跳不过去
2. 配置明明写了却不生效
3. tabBar 行为异常
4. 某个平台能力始终打不开
5. 资源打进了不该进的包

## 二、先把几个最核心的文件固定理解清楚
我建议你先记住下面这张表：

| 文件 | 核心职责 | 常见内容 |
| --- | --- | --- |
| `main.js` | 程序启动入口 | 创建应用、注册插件、挂全局能力 |
| `App.vue` | 应用级容器 | 应用生命周期、全局样式、应用级初始化 |
| `pages.json` | 页面组织中心 | 页面注册、窗口样式、tabBar、分包、easycom |
| `manifest.json` | 应用和平台配置中心 | 应用信息、appid、版本号、网络超时、平台打包参数 |
| `uni.scss` | 全局样式变量层 | 主题色、字号、边距、圆角等变量 |
| `static` | 静态资源目录 | 图片、图标、音视频等固定资源 |

## 三、`main.js` 和 `App.vue` 的区别
这两个文件很容易被初学者混掉。

### 1. `main.js`
它更偏“程序入口”。

通常适合放：

+ 创建应用实例
+ 注册全局插件
+ 注入全局方法
+ 挂全局状态能力

示例：

```javascript
import App from './App'
import { createSSRApp } from 'vue'

export function createApp() {
  const app = createSSRApp(App)
  return {
    app
  }
}
```

不适合放：

+ 页面配置
+ tabBar 配置
+ 某个具体页面业务逻辑

### 2. `App.vue`
它更偏“应用级容器”。

通常适合放：

+ 应用生命周期
+ 全局样式
+ 应用级初始化逻辑

```javascript
export default {
  onLaunch() {
    console.log('应用启动')
  },
  onShow() {
    console.log('应用进入前台')
  }
}
```

要记住：

`main.js` 更偏程序启动，`App.vue` 更偏应用级容器和应用级时序。`

## 四、`pages.json` 为什么是最重要的配置文件之一
如果只从配置文件里挑一个最需要先吃透的，我会优先选 `pages.json`。

因为它直接决定了很多最核心的事：

+ 哪些页面会被编译进项目
+ 页面默认窗口怎么显示
+ tabBar 怎么组织
+ 分包怎么拆
+ 组件自动引入规则怎么配置

换句话说，页面能不能进工程、能不能跳、窗口表现对不对、tabBar 工作是否正常，都和它高度相关。

## 五、`pages.json` 里最重要的几个配置块
### 1. `pages`
用来定义页面路径。

你可以先把它理解成：

`页面要想被路由识别、被编译进项目，先看 pages 里有没有。`

示例：

```json
{
  "pages": [
    {
      "path": "pages/index/index",
      "style": {
        "navigationBarTitleText": "首页"
      }
    }
  ]
}
```

### 2. `globalStyle`
定义默认窗口样式。

常见会放：

+ 导航栏标题颜色
+ 导航栏背景色
+ 页面背景色

它解决的是“全局默认长什么样”，不是每个页面都必须单独重复配置。

### 3. `tabBar`
定义底部 tab。

这里有两个特别容易混的点：

+ `tabBar` 页面必须先在 `pages` 中注册
+ 跳转到 `tabBar` 页面时要用 `uni.switchTab`

不能用：

+ `uni.navigateTo`
+ `uni.redirectTo`

### 4. `subPackages`
负责分包。

很多人把它当成“性能优化补丁”，其实它更像页面组织的一部分。  
因为它影响的不是单点 API，而是页面被如何打包和如何进入主包 / 分包结构。

### 5. `easycom`
这是组件自动导入能力。

价值在于：

+ 页面不需要每次手动注册大量组件
+ 组件使用体验更顺
+ 中大型项目里可以减少重复样板代码

但前提是：

+ 组件命名要稳定
+ 目录结构要统一
+ 页面名和组件名不要混淆

## 六、`manifest.json` 更像应用与平台配置中心
和 `pages.json` 不同，`manifest.json` 不是页面组织中心，而更像应用信息和平台发行配置中心。

它更适合承载：

+ 应用名称
+ 版本号
+ appid
+ 网络超时
+ 平台打包参数
+ 平台专属能力开关

一句话总结：

+ `pages.json` 管页面层
+ `manifest.json` 管应用层和平台层

## 七、`pages.json` 和 `manifest.json` 为什么特别容易写混
因为它们都叫“配置文件”，但职责完全不同。

### 1. `pages.json` 关注的是

+ 页面
+ 路由
+ 窗口
+ tabBar
+ 分包

### 2. `manifest.json` 关注的是

+ 应用身份
+ 应用版本
+ 平台发行
+ 平台能力
+ 构建参数

所以如果你遇到：

+ 页面跳不过去
+ tabBar 异常
+ 页面窗口表现不对

先优先想 `pages.json`。

如果你遇到：

+ appid 配置
+ 打包参数
+ 平台专属能力不生效
+ 发行配置有问题

先优先想 `manifest.json`。

## 八、`uni.scss` 和 `static` 也不能随便理解
### 1. `uni.scss`
它的价值不只是“放几个变量”，而是给整个项目提供统一样式基础。

更适合放：

+ 主色
+ 边距
+ 字号
+ 圆角
+ 阴影
+ 常用层级变量

如果项目里这些东西散在每个页面里，后期统一主题和样式一致性会很痛苦。

### 2. `static`
适合放不会参与业务逻辑计算、但要作为资源直接使用的文件，比如：

+ 图片
+ 图标
+ 音频
+ 视频
+ 默认占位图

如果资源组织不清，常见问题包括：

+ 引用路径混乱
+ 主包被塞太多资源
+ 平台专属资源和通用资源混在一起

## 九、一个更稳的 uni-app 项目目录思路
如果是长期维护项目，我更推荐下面这种结构：

```text
src/
  pages/          页面
  components/     通用组件
  layouts/        布局壳层
  services/       登录、请求、上传、缓存等服务封装
  stores/         状态管理
  utils/          纯工具函数
  styles/         通用样式
  platform/       平台差异适配
static/           静态资源
App.vue
main.js
pages.json
manifest.json
uni.scss
```

这种结构的核心不是“看起来规范”，而是：

+ 页面归页面
+ 组件归组件
+ 服务归服务
+ 平台差异单独收口
+ 配置各司其职

## 十、最常见的配置边界错误有哪些
我把最常见的坑整理成下面几类：

### 1. 把页面相关配置写进 `manifest.json`
结果就是页面问题一直查平台配置，越查越偏。

### 2. 把平台专属配置散在页面业务里
后面业务和平台规则强耦合，越来越难维护。

### 3. 分包和页面注册边界没理清
最后页面打包结构和预期不一致。

### 4. 样式变量没有统一管理
页面各写各的，后期改主题非常痛苦。

### 5. 静态资源没有做组织规划
资源引用混乱，包体也容易失控。

## 十一、这一篇最后要沉淀成哪些共识
如果只记重点，我建议你至少记住下面这些：

1. `main.js` 负责程序启动，`App.vue` 负责应用级容器和应用级时序。
2. `pages.json` 是页面组织中心，`manifest.json` 是应用和平台配置中心。
3. 页面注册、窗口样式、tabBar、分包优先看 `pages.json`。
4. appid、版本号、平台打包参数、平台能力配置优先看 `manifest.json`。
5. `uni.scss` 应该承担全局样式变量层，`static` 应该承担静态资源层。
6. 真正稳定的工程结构，不是文件名记住了就行，而是每一层都知道“该放什么，不该放什么”。
