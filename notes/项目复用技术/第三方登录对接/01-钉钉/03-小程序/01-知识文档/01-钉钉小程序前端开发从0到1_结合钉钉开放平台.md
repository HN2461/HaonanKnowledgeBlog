# 钉钉小程序前端开发从0到1（结合钉钉开放平台）

> 适合谁看：
> 你是前端，之前主要写 `uni-app`、微信小程序、App，刚开始接触钉钉小程序。
>
> 这篇文章不是只讲平台配置，也不是只讲代码，而是把“钉钉开发者平台 + 前端怎么写页面和调 API”串成一条完整线。

## 一、先用一句人话说清楚：钉钉小程序到底是什么

如果你写过微信小程序，你可以先把钉钉小程序理解成：

`运行在钉钉客户端里的小程序页面 + 钉钉自己的 JSAPI 能力 + 钉钉开放平台上的应用配置和发布体系`

从前端视角看，它大概由 3 部分组成：

1. 页面层
   你要写页面、样式、交互、路由。
2. 容器能力层
   你要调钉钉的 JSAPI，比如免登、选人、选部门、客户端能力。
3. 平台配置层
   你要去钉钉开放平台创建应用、绑定小程序、配置安全域名、上传和发布。

所以钉钉小程序不是“只有前端代码”，也不是“只有后台配置”，而是：

`前端代码 + 容器 API + 开放平台配置`

## 二、先记住这件事：前端写钉钉小程序，流程不是先写代码，而是先把平台准备好

很多新手一听“写钉钉小程序”，就立刻去找 `dd.xxx` API。

但官方流程其实更像这样：

1. 判断应用类型
2. 获取开发权限
3. 创建应用
4. 添加小程序能力
5. 下载开发工具
6. 创建或导入项目
7. 关联应用
8. 写页面
9. 调 JSAPI
10. 配安全域名
11. 预览、上传、发布

你如果前面 1 到 7 没打通，后面很多前端代码根本没有运行环境。

## 三、第一步：先判断你要做的是哪种应用

这一步非常重要，因为它直接决定你后面走哪条路线。

钉钉官方“应用类型介绍”把常见情况分成了几类，其中对你最重要的是这两类：

### 1. 企业内部应用

适合：

- 只给某一个组织自己内部用
- 比如某一所学校自己开发给本校老师、学生、行政人员用

特点：

- 主要是内部部署
- 不强调对外卖给很多客户

### 2. 第三方企业应用

适合：

- 你们公司是产品提供方
- 产品要给多个企业客户使用
- 比如教育公司把系统卖给多家学校、培训机构、教育机构

特点：

- 你们是方案提供方
- 学校是客户方
- 后面通常会涉及安装、授权、客户开通、分发、运营

## 四、按你现在的背景，优先把自己理解成“第三方企业应用开发者”

因为你前面补充过，你们是教育公司，学校或教学机构想把你们微信小程序的一些功能搬到钉钉上用。

这就更像：

- 你们不是学校内部技术团队
- 你们是产品方案商
- 学校是客户

所以从业务理解上，你更应该优先按下面这个角色进入钉钉：

`我们公司 = 第三方企业应用提供方`

这会影响你后面很多思路：

- 不是“我公司员工登录”
- 而是“每个学校自己的钉钉组织安装后使用”
- 不是“一个组织一个系统”
- 而是“多个学校租户共用一套产品能力”

这点你脑子里一定要立住。

## 五、第二步：去钉钉开放平台把开发入口打通

### 第 1 小步：拿到开发权限

钉钉官方“获取开发者权限”文档说明了两种方式：

- 管理员直接给你开发权限
- 你自己申请，由管理员审批

所以你先确认这几件事：

1. 你有没有钉钉账号
2. 你有没有加入公司对应的钉钉组织
3. 你能不能登录钉钉开发者后台
4. 你有没有应用开发权限

如果权限不够，你连应用都创建不了。

### 第 2 小步：创建应用

钉钉官方文档里，“创建应用”是单独的一步。

你可以先理解成：

`先有应用，后有小程序能力`

也就是说，小程序不是凭空出现的，它要挂在某个应用下面。

创建时一般会涉及：

- 应用名称
- 应用描述
- 应用图标

官方文档对名称、描述、图标都有基本要求，比如最小长度、图标尺寸和大小限制。

### 第 3 小步：添加“小程序能力”

这一步特别关键。

钉钉官方“添加应用能力”文档明确说了：

- 创建完应用以后
- 还需要去“应用能力”里添加小程序能力

如果你没加这一步，后面你虽然有一个应用，但它不具备小程序入口和小程序发布能力。

## 六、第三步：下载开发工具，创建或导入项目

钉钉官方“开发小程序前端”文档给出的前端路线很清楚：

1. 下载并安装小程序开发工具
2. 在左侧菜单选择“小程序”
3. 选择“钉钉”端
4. 选择模板
5. 创建项目
6. 登录并关联应用

官方还建议：

- 可以选空白模板
- 也可以选官方模板
- 对刚开始的人来说，空白模板最容易理解

### 你可以怎么理解“关联应用”

这个动作很像：

`把你本地写的小程序项目，挂到钉钉开放平台上的那个应用记录上`

这样后面：

- 预览
- 上传
- 发布

才能和平台后台对应起来。

## 七、第四步：前端先搞懂“钉钉小程序项目长什么样”

这里我先说一个很重要的点：

钉钉官方“开发小程序前端”和“小程序使用网页应用客户端 API”文档里，明确出现了这些文件和目录：

- `app.js`
- `app.json`
- `pages/`
- `index.axml`
- `index.js`

所以可以整理出一个前端最常见的项目骨架。

下面这个结构，是我根据官方示例归纳出来的前端理解版，不是逐字摘抄：

```text
mini-app-demo/
├─ app.js
├─ app.json
├─ pages/
│  └─ index/
│     ├─ index.axml
│     ├─ index.js
│     ├─ index.acss
│     └─ index.json
└─ node_modules/
```

你可以把它这样记：

### 1. `app.js`

全局入口。

通常你会在这里做：

- 全局初始化
- 引入 `dingtalk-jsapi`
- 应用启动时的公共逻辑

### 2. `app.json`

全局配置文件。

通常会放：

- 页面路由
- 窗口配置
- 全局页面顺序

### 3. `pages/xxx/index.axml`

页面结构层。

如果你写过微信小程序，可以把它理解成：

- 类似模板层
- 相当于页面结构

### 4. `pages/xxx/index.js`

页面逻辑层。

通常会写：

- `Page({...})`
- 页面事件
- 生命周期
- 调接口
- 调 JSAPI

### 5. `pages/xxx/index.acss`

页面样式层。

可以先把它理解成“当前页面的样式文件”。

### 6. `pages/xxx/index.json`

页面级配置。

如果这个页面有特殊配置，通常会放在这里。

## 八、第五步：先写出你的第一个页面

这一节我不想只说概念，我直接按前端能理解的方式讲。

钉钉官方“小程序使用网页应用客户端 API”文档里，给了一个很典型的页面例子，里面出现了：

- `index.axml`
- `index.js`
- `<view>`
- `<button onTap="...">`
- `Page({...})`

这说明你可以先把写页面理解成下面这件事：

`先写结构，再写事件，再在事件里调 API`

### 一个最小页面例子

下面这个示例，是按官方页面写法整理出的前端入门版。

#### `pages/index/index.axml`

```html
<view class="page">
  <view class="title">这是我的第一个钉钉小程序页面</view>
  <button type="primary" onTap="handleAuth">获取免登授权码</button>
  <button type="default" onTap="handlePickUser">选择部门和人</button>
</view>
```

#### `pages/index/index.js`

下面的代码不是照抄某一页文档，而是把钉钉官方文档里的两个核心示例拼成一个更好理解的最小例子：

- 一个是 `dd.getAuthCode`
- 一个是 `complexPicker`

```js
import complexPicker from 'dingtalk-jsapi/api/biz/contact/complexPicker'

Page({
  handleAuth() {
    dd.getAuthCode({
      success: (res) => {
        console.log('authCode:', res.authCode)
      },
      fail: (err) => {
        console.error('getAuthCode fail:', err)
      }
    })
  },

  handlePickUser() {
    complexPicker({
      title: '选择老师或部门',
      corpId: '这里替换成你的 corpId',
      multiple: true,
      maxUsers: 100,
      pickedUsers: [],
      pickedDepartments: [],
      disabledUsers: [],
      disabledDepartments: [],
      requiredUsers: [],
      requiredDepartments: [],
      permissionType: 'GLOBAL',
      responseUserOnly: false,
      startWithDepartmentId: 0
    })
  }
})
```

### 这个页面到底做了什么

第一颗按钮：

- 调 `dd.getAuthCode`
- 用来拿钉钉免登授权码

第二颗按钮：

- 调 `complexPicker`
- 用来唤起“选择部门和人”的界面

这两个例子足够让你理解钉钉小程序页面开发的基本套路：

`按钮触发 -> 页面 JS 方法 -> 调钉钉 JSAPI`

## 九、第六步：学会用 `dingtalk-jsapi`

这一步是钉钉小程序前端最核心的地方之一。

钉钉官方“客户端 SDK 介绍”文档明确写了：

- 推荐通过 `npm install dingtalk-jsapi --save` 安装
- 推荐用 npm 引入，不推荐靠浏览器 CDN 方式

钉钉官方“小程序使用网页应用客户端 API”文档又补充了一个要点：

- 钉钉小程序项目内要安装 `dingtalk-jsapi`
- 在 `app.js` 顶部引入 `dingtalk-jsapi/entry/mobile`

### 你可以照着这样做

#### 安装

```bash
npm install dingtalk-jsapi --save
```

#### 在 `app.js` 里初始化

```js
import 'dingtalk-jsapi/entry/mobile'
```

### 为什么这一句很重要

因为官方文档的意思是：

`小程序里如果想使用基于 dingtalk-jsapi 的网页应用客户端能力，需要先做这层初始化`

你可以把它理解成：

- 先把钉钉客户端 SDK 接进来
- 后面你页面里再按需 import 具体 API

## 十、第七步：前端怎么理解“免登”

这是钉钉小程序最常见、最重要、也是最容易乱的地方。

你先别想太复杂，先理解一条主线：

### 前端视角下的免登流程

1. 用户在钉钉里打开小程序
2. 前端调用 `dd.getAuthCode`
3. 前端拿到 `authCode`
4. 前端把 `authCode` 发给你自己的后端
5. 后端再去和钉钉服务端接口交互，换用户身份信息
6. 后端再给你自己系统的登录态

你会发现：

`前端负责拿 authCode，后端负责换身份和建立你自己系统的登录态`

### 为什么这点你一定要理解

因为很多刚接触钉钉的人会误以为：

- `dd.getAuthCode` 一调
- 前端就已经登录完成了

其实不是。

它更像是：

`先拿到“去后端换用户身份”的凭证`

### 官方资料里怎么说

钉钉官方“小程序应用免登”文档给出的主流程就是：

- 前端调用 `dd.getAuthCode`
- 后端获取应用级访问凭证
- 后端再根据免登授权码获取用户信息

### 对你这种教育行业场景，再多加一层理解

如果你们是第三方企业应用，那后端通常还要继续处理：

- 这个用户属于哪所学校
- 这个学校是不是已经开通
- 这个学校对应哪个租户
- 这个用户在学校里是什么角色

所以前端不要把免登理解成“一个按钮就结束”，它只是登录链路的第一段。

## 十一、第八步：页面写好了，怎么发请求

这一节你要和“安全域名”一起记。

钉钉官方“配置安全域名”文档明确提到：

- 如果小程序前端需要和服务端通信
- 必须配置 `HTTP 可信域名`

官方“小程序应用免登”文档也提醒：

- 如果在小程序中使用 `dd.httpRequest`
- 请求域名必须在安全设置里配置

### 你作为前端应该怎么理解

不是你代码能发请求就行，而是：

`你的代码 + 后台安全域名配置，两个都得对`

### 你最应该记住的 4 件事

1. 接口域名必须提前加到安全域名里
2. 如果配置变了，要重新构建、重新上传
3. 本地调试和线上域名可能不是同一个
4. 如果有 `web-view`，网页域名也要单独配置

### 最容易踩的坑

前端觉得：

- 我接口写得没问题
- 后端也正常

结果在钉钉里还是报错。

很多时候不是代码有问题，而是：

- 没配安全域名
- 配了但没重新上传版本

## 十二、第九步：怎么调试和预览

钉钉官方“开发小程序前端”文档里给了比较完整的调试说明。

### 本地调试时你一般会经历这几种状态

1. IDE 模拟器里看页面
2. 手机钉钉扫码预览
3. PC 钉钉里访问

### 官方提到的预览方式

- 手机端：
  点击“预览”，生成二维码，然后用手机钉钉扫码
- PC 端：
  点击“预览”，复制二维码链接，到 PC 钉钉聊天框里访问

### 如果你们学校场景还要支持 PC

钉钉官方“添加小程序地址”文档提到：

- 可以开启小程序支持 PC 端
- 可以配置 `pc 端应用访问地址`

所以如果学校老师很多是在电脑上办公，你后面别忘了这块。

## 十三、第十步：怎么上传和发布

钉钉官方“上传小程序”文档说得很明确：

- 在开发者工具里点击“上传”
- 上传后得到的是“开发版本”
- 开发版本可以做体验和测试
- 这还不等于正式上线

### 从前端视角你可以这样理解版本流转

1. 本地开发版本
2. 上传到平台后的开发版本
3. 体验版
4. 正式发布版本

### 你做前端时最该注意什么

#### 1. 版本号别乱写

尽量用：

- `1.0.0`
- `1.0.1`
- `1.1.0`

不要写：

- `test`
- `最新`
- `老板要的`

#### 2. 每次上传只验证一类问题

比如：

- 第一次先验证首页打开
- 第二次验证登录
- 第三次验证选人和请求

这样出问题时你才知道是哪一步炸了。

### 发布时还有一个差异点

如果你走的是企业内部应用路线，官方“发布应用”文档里有明确的版本管理和发布入口。

如果你走的是第三方企业应用路线，那你后面还会碰到：

- 客户怎么安装
- 是否定向分发
- 是否上架
- 审核和运营要求

所以：

`上传是技术动作，发布是产品和平台动作`

## 十四、如果你是 `uni-app` 前端，该怎么理解钉钉小程序

这一节专门写给你这种人。

因为你不是完全的“原生小程序新人”，你已经有 `uni-app` 思维了。

### 你最容易犯的错

就是上来就想：

- `uni.navigateTo` 在钉钉里叫什么
- `uni.request` 在钉钉里叫什么
- `wx.login` 换成什么

这个思路不是完全错，但容易把自己搞碎。

### 更好的理解顺序

#### 先学“钉钉原生小程序是怎么工作的”

也就是先搞懂：

- 应用怎么创建
- 项目怎么组织
- 页面怎么写
- `dd.getAuthCode` 是干嘛的
- `dingtalk-jsapi` 怎么接

#### 再回到你的 `uni-app` 项目

这时你再看迁移问题，会清楚很多：

- 哪些页面能直接复用
- 哪些 API 可以继续用 `uni.*`
- 哪些钉钉能力必须单独适配

### 你可以把两边这样对照着记

#### `uni-app` 里你熟悉的是

- Vue 单文件组件
- `template`
- `script`
- `style`
- `uni.*`

#### 钉钉原生小程序里你现在要认识的是

- `axml`
- `acss`
- `Page`
- `app.js`
- `app.json`
- `dd.*`
- `dingtalk-jsapi`

### 一句总结

`先理解钉钉原生，再做 uni-app 迁移，会比一上来硬改现有项目稳很多。`

## 十五、前端最该先学会的 5 个能力

如果你现在时间有限，我建议你优先把下面 5 个点学会。

### 1. 会判断应用类型

内部应用和第三方企业应用路线不一样。

### 2. 会创建应用并加小程序能力

这是所有开发的入口。

### 3. 会写最小页面

至少要会：

- `axml`
- `Page`
- 按钮事件

### 4. 会调 `dd.getAuthCode`

这是登录链路的第一步。

### 5. 会配置安全域名并上传测试版本

这是你把代码真正跑到钉钉里的关键。

## 十六、你接下来最适合怎么练

如果我是带你做这个项目，我会让你按下面顺序练：

1. 先在钉钉开放平台创建一个测试应用
2. 给它加上小程序能力
3. 用空白模板建一个最小项目
4. 写一个首页，只放两个按钮
5. 第一个按钮调用 `dd.getAuthCode`
6. 第二个按钮调用 `complexPicker`
7. 配一个测试域名
8. 写一个最简单的请求给后端
9. 上传成开发版本
10. 用手机钉钉和 PC 钉钉都测一遍
