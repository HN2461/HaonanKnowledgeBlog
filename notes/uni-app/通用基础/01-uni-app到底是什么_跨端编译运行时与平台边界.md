# uni-app 到底是什么：跨端编译、运行时与平台边界

> 这是这一组 `uni-app 通用基础` 笔记的第 1 篇。  
> 这一组不先讲微信、钉钉、支付宝这些单独平台，而是先把所有 `uni-app` 项目都会反复碰到的共性知识抽出来。  
> 如果后面再去看 `微信小程序`、`钉钉` 等子目录，默认这里是前置基础。

## 一、先说结论：`uni-app` 不是一个“平台”，而是一套“跨端开发约定 + 编译体系”

很多人一上来会有两个错觉：

1. 觉得自己写的是 Vue，所以本质就是 Web 项目。
2. 觉得自己最终发的是小程序或 App，所以 `uni-app` 只是换了个语法壳。

这两个理解都不完整。

更准确的说法是：

`你写的是一套遵循 uni-app 规范的跨端代码，编译器再把它转成目标平台能运行的产物。`

也就是说，`uni-app` 的核心价值不是“替你抹平一切差异”，而是：

- 给你一套尽量统一的页面、组件、API 和工程组织方式
- 在能统一的地方尽量统一
- 在不能统一的地方，允许你用条件编译和平台专属能力收口差异

所以做 `uni-app`，最重要的不是背 API，而是先建立对“层”的理解。

## 二、建议你把 uni-app 项目固定理解成 4 层

| 层级 | 你平时接触到的东西 | 典型问题 | 优先查哪里 |
| --- | --- | --- | --- |
| 业务代码层 | 页面、组件、状态、请求、表单、列表 | 逻辑写错、状态没更新、事件没绑对 | 你自己的业务代码 |
| uni-app 规范层 | `uni.*` API、页面生命周期、`pages.json`、条件编译、组件规范 | 某个平台不兼容、写法超出跨端边界 | uni-app 官方文档 |
| 编译产物层 | 编译后的平台工程、平台配置映射、资源打包 | 页面没编进去、配置没生效、资源丢失 | 编译目标与配置 |
| 平台宿主层 | 小程序宿主、H5 浏览器、App WebView/原生容器 | 权限、登录、路由、性能、宿主能力差异 | 对应平台官方文档 |

这 4 层分清楚以后，排错会快很多。

比如你以后遇到：

- 页面写了但跳不过去
- 某个 API 在 H5 能用、到小程序不行
- 某段代码只在一个平台报错
- 同样的列表，在一个端很顺、另一个端很卡

你先不要急着说“是 uni-app 有问题”，而是先问：

`这是业务写法的问题、uni-app 规范边界的问题、编译配置的问题，还是目标平台宿主的问题？`

## 三、uni-app 能“共用”的，到底是什么

`uni-app` 不是“所有东西都一套代码不改”，它真正适合共用的是下面这些：

### 1. 页面与组件组织方式

- 页面文件怎么组织
- 组件怎么拆
- `props / 事件 / slot` 这类常规通信
- 大多数列表、表单、卡片、弹窗、工具页

### 2. 一大批通用 API

例如：

- `uni.navigateTo`
- `uni.request`
- `uni.setStorage`
- `uni.uploadFile`
- `uni.downloadFile`
- `uni.showToast`
- `uni.getSystemInfo`

这些 API 的价值不在于“每个平台一模一样”，而在于：

`你先用统一入口写，再在必要时补平台差异。`

### 3. 一套相对统一的工程配置思路

- `App.vue` 放应用级内容
- `pages.json` 管页面与窗口配置
- `manifest.json` 管应用信息和平台发行配置
- `static` 放静态资源
- 条件编译负责平台差异收口

## 四、uni-app 不能假装“完全一样”的地方，也要尽早承认

跨端真正容易踩坑的，往往不是“不会写”，而是“不承认差异存在”。

最常见的差异有 4 类：

### 1. 宿主能力差异

比如：

- 小程序有自己的登录、支付、分享、订阅消息
- H5 受浏览器和跨域约束
- App 有原生能力、打包配置和权限声明

这些东西不能指望只靠 `uni.*` 自动抹平。

### 2. 渲染与性能差异

同一份页面：

- 在浏览器里可能没问题
- 到小程序里会更在意通信成本
- 到 App 某些渲染模式下又有别的约束

所以 `uni-app` 的性能优化思路，始终要带着“目标端”去看。

### 3. 配置差异

有些配置是共性的，有些是平台专属的。

例如：

- 页面路由和窗口样式更偏 `pages.json`
- 应用名称、版本、包名、appid、网络超时等更偏 `manifest.json`
- 某个平台特有的权限或能力，通常要落到对应平台配置里

### 4. 第三方生态差异

传统 Vue Web 生态里很多库默认依赖：

- `window`
- `document`
- DOM 事件
- 浏览器布局或脚本运行假设

这类库不一定能直接用在 `uni-app` 的多端场景里。

稳妥的选择应该是：

`优先选明确支持 uni-app 或明确支持多端的小程序 / App 生态方案。`

## 五、做 uni-app，最稳的写法不是“到处兼容”，而是“先分层”

我更推荐你把项目写成下面这个思路：

### 1. 通用层优先

先写：

- 通用页面
- 通用组件
- 通用接口封装
- 通用缓存、路由、提示、上传下载逻辑

也就是优先使用 `uni.*` 和 uni-app 规范能覆盖的部分。

### 2. 平台差异层单点收口

如果必须接某个平台专属能力：

- 不要散着写在所有页面里
- 不要业务里到处直接写平台 API
- 最好集中收口到 `platform`、`adapters`、`services` 或条件编译代码块里

### 3. 业务层只关心“我要什么能力”

例如业务层只关心：

- 获取登录凭证
- 发起上传
- 获取定位
- 打开分享

至于底层到底走 `uni.*` 还是某个平台专属 API，应该由适配层处理。

## 六、以后判断“该不该写平台专属代码”，先问自己 3 个问题

### 1. `uni.*` 能不能满足

能满足就先不要引入平台专属写法。

### 2. 这是业务核心能力，还是宿主附加能力

如果是登录、支付、订阅消息、某平台特殊授权、某平台专属组件，这类天然更可能需要专属处理。

### 3. 这段差异能不能收口在一个地方

如果一段平台代码一旦写散，后面项目越大，维护越痛苦。

所以真正长期可维护的做法是：

`允许差异存在，但要把差异写得可控。`

## 七、把这篇压成 8 条最重要的共识

1. `uni-app` 不是单一平台，而是一套跨端开发规范与编译体系。
2. 先把“业务层、规范层、编译层、宿主层”分清楚，再去排错。
3. 能用 `uni.*` 的地方，优先用 `uni.*`。
4. 不要假设所有 Vue Web 组件库都能直接跨端复用。
5. 配置要分层，不要把页面配置、应用配置、平台配置混在一起。
6. 平台专属能力不可怕，可怕的是把它们散着写满全项目。
7. 做跨端不是追求“零差异”，而是追求“差异可控”。
8. 平台篇应该建立在通用基础之上，而不是每篇都从头重复一遍。

## 八、这一组通用基础后面准备怎么拆

这组内容我建议固定拆成下面几层：

1. `uni-app 到底是什么`：先建立跨端编译和平台边界心智
2. `应用生命周期、页面生命周期与页面事件`
3. `工程结构与配置基础`：`App.vue`、`main.js`、`pages.json`、`manifest.json`
4. `跨端开发通用方法`：路由、请求、缓存、上传下载、条件编译与平台适配

这样后面去写微信小程序、钉钉小程序、H5、App，都能直接复用这套前置知识。
