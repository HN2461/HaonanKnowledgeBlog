# uni-app 做微信小程序前必须先搞懂的底层逻辑：运行机制、生命周期、路由、登录与分包

> 这是这一组 `uni-app 微信小程序` 笔记的第 1 篇。  
> 截至 `2026-03-31`，我对照了 `uni-app` 官方文档和微信开放文档，先把最影响架构判断的底层知识收拢到一篇里。  
> 这篇不追求把 API 一条条背下来，而是先解决一个更关键的问题：`你到底是在什么环境里写代码，它最终又是怎么跑起来的。`

> 建议先读 `uni-app/通用基础`：
> 1. 第 1 篇《uni-app 到底是什么：跨端编译、运行时与平台边界》
> 2. 第 2 篇《uni-app 应用生命周期、页面生命周期与常见页面事件》
> 这一篇默认你已经知道通用的 `uni-app` 概念，下面我会重点补 `mp-weixin` 目标下的微信宿主差异、页面时序误判，以及登录和分包这些更偏微信落地的问题。

## 一、先说结论：uni-app 写微信小程序，本质上不是“写 H5”

很多人刚接手 `uni-app` 项目时，脑子里会有两个错觉：

1. 觉得自己写的是“Vue 页面”，所以问题本质上都是前端网页问题。
2. 觉得自己发的是“微信小程序”，所以 `uni-app` 只是换了一层语法皮。

这两个理解都不完整。

更准确的说法是：

`你写的是 uni-app 风格的跨端代码，编译目标是 mp-weixin，最终运行环境是微信小程序宿主。`

也就是说，中间至少有三层：

1. 你平时写的 `template / script / style`、`uni.*` API、Vue 风格组件逻辑
2. `uni-app` 把这些内容编译成 `mp-weixin` 工程
3. 编译产物最终在微信客户端提供的宿主环境里运行

所以你以后在项目里看到一个问题，不要马上下结论说：

- “这是 Vue 的问题”
- “这是微信 API 的问题”
- “这是 uni-app 抽风”

你先要问自己一句：

`这个问题发生在写代码这一层、编译这一层，还是微信运行时这一层？`

只要这个判断做对了，排错速度会快很多。

## 二、先把 4 层分清楚：以后排错就不容易乱

| 层级 | 你平时接触到的东西 | 典型问题 | 应该优先查哪里 |
| --- | --- | --- | --- |
| 业务代码层 | 页面、组件、状态、接口调用、表单、列表 | 逻辑写错、状态没更新、事件没绑对 | 你自己的项目代码 |
| uni-app 编译层 | `uni.*`、条件编译、`pages.json`、跨端兼容 | 某平台有代码没生效、平台差异没处理、页面没注册 | uni-app 官方文档 |
| 微信小程序框架层 | 路由、页面栈、登录、分包、基础库能力 | `switchTab` 行为不符合预期、登录态误判、包体结构不合理 | 微信开放文档 |
| 微信宿主运行层 | 逻辑层、渲染层、启动机制、后台挂起、内存回收 | 白屏、卡顿、切后台回来状态异常、内存被清 | 微信运行机制和性能文档 |

你可以把这 4 层当成一个固定脑图。

以后凡是遇到：

- 页面切换后为什么会重新触发 `onShow`
- 为啥 `switchTab` 和 `navigateTo` 行为不一样
- 为什么列表一大就卡
- 为什么切后台回来有的状态没了

本质上都不是一句“前端问题”能解释完的。

## 三、为什么同一套代码到了微信里，会突然有很重的“小程序味”

微信官方文档把小程序运行环境拆成两块：

1. `JS` 跑在逻辑层
2. `WXML / WXSS` 跑在渲染层

而且这两层不是直接一体的，它们之间需要通过微信客户端中转通信。

这件事对 `uni-app` 开发非常重要，因为你虽然写的是 Vue 风格代码，但最后发到微信小程序时，依然要遵守微信这套运行模型。

所以你要尽快接受一个现实：

`uni-app 的响应式更新，到了微信小程序里，最终还是要落到小程序视图层更新成本上。`

这会直接带来几个实际影响：

1. 不要频繁更新超大的响应式对象。
2. 不要把深层嵌套、超长列表、复杂模板都堆在一个页面首屏里。
3. 不要以为“JS 里改完数据就等于页面无成本刷新”。
4. 页面白屏、滚动卡顿、首屏慢，很多时候不是接口慢，而是页面结构太重。

如果你现在维护的是公司里的 `uni-app` 项目，这一条尤其关键：

`很多看起来像“Vue 渲染问题”的现象，最后真因其实是微信小程序渲染层通信成本和页面结构设计问题。`

## 四、在 uni-app 里，页面不是随便 import 就能跑，而是由 `pages.json` 驱动

`uni-app` 官方文档明确说明，页面需要注册在 `pages.json` 里，没注册的页面会在编译阶段被忽略。

这意味着：

1. 你不能拿 Web 项目那种“组件写好后按需 import 即可”的心态理解页面。
2. 页面路径、首页、`tabBar`、页面样式，本质上都受 `pages.json` 管理。
3. `uni-app` 的页面路由思路，更接近小程序，而不是 `Vue Router`。

你可以把常见跳转先记成下面这张表：

| API | 常见作用 | 关键限制 |
| --- | --- | --- |
| `uni.navigateTo` | 打开新页面 | 只能去非 `tabBar` 页面 |
| `uni.redirectTo` | 替换当前页面 | 只能去非 `tabBar` 页面 |
| `uni.switchTab` | 切到 `tabBar` 页面 | 只能去 `tabBar` 页面 |
| `uni.reLaunch` | 清空页面栈后进入新页面 | 可以去任意页面 |
| `uni.navigateBack` | 返回上级页面 | 依赖当前页面栈 |

这里最容易踩坑的是 `switchTab`。

因为它不是简单的“跳到一个页面”，而是会牵扯：

- 旧页面是否销毁
- 目标 `tabBar` 页是不是第一次创建
- 页面栈是否要清到只剩一个根页面

所以你在业务里做“登录成功后跳首页”“详情页回列表”“从分享页回主 Tab”时，一定要先分清：

`你现在是在普通页面之间跳，还是在普通页和 tabBar 页之间切。`

## 五、把生命周期想成“路由驱动的页面状态机”

如果你还没把应用生命周期、页面生命周期的基础定义记牢，建议先回看 `通用基础` 第 2 篇。下面这一节不再重复背所有概念，而是专门抓 `uni-app -> mp-weixin -> 微信宿主` 这条链路里最容易误判的时序点。

`uni-app` 页面支持自己的页面生命周期，微信小程序页面也有自己的页面生命周期。两套概念在微信小程序目标下，大方向是能对上的。

你平时最常用、也最容易搞混的是这几个：

### 1. `onLoad`

适合做的事：

- 接收路由参数
- 发起首屏接口请求
- 初始化页面状态

不适合做的事：

- 直接操作节点
- 做很重的同步计算

因为从 `uni-app` 页面时序文档来看，`onLoad` 发生得很早，这时页面还没到“可以安心操作节点”的阶段。

### 2. `onShow`

它不是只进页面时触发一次，而是：

- 页面第一次显示会触发
- 从下级页面返回时会再次触发
- `tabBar` 页面重新露出时也会触发

所以如果你在列表页里要做“从详情页返回后刷新数据”，很多时候更适合放在 `onShow`，而不是只放在 `onLoad`。

### 3. `onReady`

这个阶段可以理解成：

`页面第一次渲染已经完成了。`

适合做的事：

- 获取节点
- 做需要依赖页面已经渲染出来的操作

### 4. `onHide`

页面暂时不可见时触发。

适合做的事：

- 暂停轮询
- 暂停动画或音视频
- 收口页面级监听

### 5. `onUnload`

页面被销毁时触发。

适合做的事：

- 清理定时器
- 解绑全局事件
- 销毁页面私有资源

如果你平时总分不清这些钩子到底该放什么，一个列表页可以先写成下面这个最小骨架：

```js
export default {
  data() {
    return {
      query: {},
      list: [],
      hasEnteredOnce: false
    }
  },
  onLoad(query) {
    this.query = query
    this.fetchList()
  },
  onShow() {
    if (!this.hasEnteredOnce) {
      this.hasEnteredOnce = true
      return
    }

    this.refreshPendingState()
  },
  onReady() {
    this.initPageObserver()
  },
  onHide() {
    this.stopPolling()
  },
  onUnload() {
    this.stopPolling()
    this.destroyPageChannel()
  }
}
```

这段代码建议你这样读：

1. `onLoad` 只做参数接收和首屏请求，因为这时页面逻辑刚建立，最适合把首屏数据先拉起来。
2. `onShow` 负责“回来以后要不要补刷新”，所以我用 `hasEnteredOnce` 把“第一次进入”和“后续返回”分开。
3. `onReady` 只做依赖首渲染完成的事，比如节点查询、`IntersectionObserver`、滚动容器初始化。
4. `onHide` 不做重活，只暂停轮询、动画、音视频这种会持续消耗资源的任务。
5. `onUnload` 才是最终销毁点，所以事件解绑、页面私有资源清理要放在这里兜底。

如果你想把时序彻底记住，可以先背一个最常见的链路：

`列表页进入 -> 列表页 onLoad -> 列表页 onShow -> 列表页 onReady -> 打开详情页 -> 列表页 onHide -> 详情页 onLoad -> 详情页 onShow -> 返回 -> 详情页 onUnload -> 列表页 onShow`

只要你把这条链记住，大多数“为什么又进了这个生命周期”的困惑都会少很多。

## 六、登录链路里，最容易混的是 `code`、`openid`、`unionid`、`session_key`

这一段如果没想清楚，项目越做越容易乱。

在 `uni-app` 里做微信小程序登录时，前端常见写法会是这样：

```js
uni.login({
  success: ({ code }) => {
    if (!code) return

    uni.request({
      url: '/api/auth/wechat/login',
      method: 'POST',
      data: { code }
    })
  }
})
```

这段代码你不要只记“能跑”，而要记清每一步在负责什么：

1. `uni.login` 做的事情很窄，它只是向微信宿主拿一个短时有效的登录凭证 `code`。
2. `if (!code) return` 这一步是在第一时间把无效结果拦住，避免你后面还拿空凭证去打后端。
3. `uni.request` 不是在“拿微信用户信息”，而是在把 `code` 立刻交给你自己的服务端做 `code2Session` 或业务换登。
4. 前端这一层最应该期待后端返回的是你的业务 `token`、用户状态、是否需要补资料，而不是把 `session_key` 暴露出来。
5. 如果这段代码以后要长期维护，最好继续往前收一层 service，比如 `loginByWechatMiniProgram`，不要让每个页面都自己写一份。

但你一定要知道，这里前端真正拿到的核心不是“用户信息”，而是：

`登录凭证 code`

根据微信官方 `wx.login` 文档，`code` 的有效期是 `5 分钟`。前端把它交给你自己的服务端后，服务端再去调用微信的登录凭证校验接口，换回：

- `openid`
- `unionid`
- `session_key`

你可以这样记：

### 1. `code`

一次性的临时登录凭证。

它的职责不是长期标识用户，而是让你的服务端去向微信证明：

`这个用户刚刚确实在这个小程序里发起了一次登录。`

### 2. `openid`

用户在“当前小程序”下的唯一标识。

如果你的业务只是这个小程序自身识别用户，`openid` 很常用。

### 3. `unionid`

用户在“同一个微信开放平台账号体系”下的唯一标识。

微信官方说明里提到，前提是小程序已经绑定到微信开放平台账号。  
如果你未来要做：

- 多端账号打通
- 小程序和公众号用户归并
- 同一公司多个应用识别同一个人

那 `unionid` 会比 `openid` 更有战略价值。

### 4. `session_key`

这是微信登录态里的会话密钥。

它很重要，但它不是给你当前端业务 token 用的。

你要记住两件事：

1. `session_key` 主要服务于微信登录态和相关数据处理
2. 你自己的业务系统，仍然应该签发自己的登录态，比如 `session` 或 `jwt`

不要把“微信登录态”和“你自己系统的登录态”混成一个东西。

## 七、`checkSession` 不是万能续命药，微信登录态也不是永久不变

微信官方对 `wx.checkSession` 的说明里有两个特别值得记住的点：

1. 登录态是否过期，具体时效逻辑由微信维护，对开发者透明
2. `checkSession` 校验的是“最后一次获取 code 对应的登录态”

这意味着你不能简单粗暴地把微信登录态理解成：

`固定几个小时不过期`

更稳妥的理解是：

1. 前端可以在合适时机做 `uni.checkSession` 或微信侧等价校验
2. 失效了就重新走 `uni.login`
3. 业务系统自己的鉴权，和微信登录态分开管理

也就是说：

`微信负责证明“这个人刚刚在微信里完成了登录”，你自己的后端负责管理“这个人现在是不是你的业务已登录用户”。`

## 八、分包和启动性能，不是后期再想的事

微信官方文档对小程序运行机制和分包都讲得很明确，这里你至少要先记住 6 件事。

### 1. 小程序有冷启动和热启动

- 冷启动：第一次打开，或者小程序已经被销毁后再次打开
- 热启动：小程序还活着，只是从后台回到前台

所以你不能默认每次进入页面都是“全新环境”，也不能默认页面内存会一直留着。

### 2. 切后台后，JS 不会永远活着

微信运行机制文档提到：

- 小程序进入后台一段时间后会被挂起
- 挂起足够久后，可能被销毁

这会直接影响你对页面状态的预期。

如果某个状态很重要，不要只放内存里，应该考虑：

- 本地缓存
- 服务端回填
- 页面重进时重新拉取

### 3. `tabBar` 页面必须在主包

这是分包设计里很容易忘的硬规则。

如果你的项目首页、工作台、我的这些页面属于 `tabBar`，那它们天然要在主包，不可能全部丢进分包里。

### 4. 分包之间不是想怎么互相引用都行

微信官方说明里提到：

- `packageA` 不能直接引用 `packageB` 的 JS
- 也不能直接使用 `packageB` 的模板和资源

所以公共资源到底放主包，还是单独整理共享结构，要在项目早期想清楚。

### 5. 主包应该尽量轻

经验上，你可以把主包理解成：

- 启动必需页面
- `tabBar` 页面
- 全局基础组件
- 真正高频复用的公共资源

而那些：

- 报表页
- 复杂表单页
- 大图上传页
- 低频运营页

更适合拆到分包。

### 6. 首屏性能先看“页面结构”，再看“接口快不快”

如果你维护的是公司业务项目，这个判断非常实用。

很多首屏慢问题，根因是：

- 首屏模板太重
- 列表太长
- 一次性塞了太多响应式数据
- 页面进入时做了很多同步计算

而不是单纯“接口多了 200ms”。

## 九、在 uni-app 里，平台专属能力别直接铺满全项目

`uni-app` 官方给了条件编译能力，目的就是让你把跨端共性和平台特性拆开。

对维护业务项目来说，更推荐的思路是：

1. 能用 `uni.*` 的，先用 `uni.*`
2. 只有微信专属能力，再写 `wx.*`
3. 写 `wx.*` 时，用 `#ifdef MP-WEIXIN` 收口

例如：

```js
uni.request({
  url: '/api/list'
})

// #ifdef MP-WEIXIN
wx.requestSubscribeMessage({
  tmplIds: ['你的模板 id']
})
// #endif
```

这段代码表面看只是多了两行条件编译，实际上是在做“能力分层”：

1. 普通请求继续走 `uni.request`，因为它本来就是跨端也有意义的通用能力。
2. 只有 `requestSubscribeMessage` 这种微信专属协议，才放进 `#ifdef MP-WEIXIN`。
3. 这样别的平台在编译时根本看不到 `wx.*` 代码，不会出现“运行到一半才报未定义”的问题。
4. 真正做公司项目时，最好把这种微信专属调用继续收口到单独模块里，不要散在页面里到处写。

这比“全项目到处直接写死 `wx.*`”稳定得多。

因为你一旦把微信专属 API 写散了，后面不管是迁别的平台，还是做多端兼容，都会非常痛苦。

## 十、如果你是公司里长期维护 uni-app 项目的人，先记住这 8 条

1. 默认先用 `uni.*` 理解能力边界，不要上来就满项目写 `wx.*`
2. 平台差异代码必须收口到 `#ifdef MP-WEIXIN`
3. 页面思维先看 `pages.json`，不要用纯 Web 路由心智去想小程序
4. `onLoad` 适合早点请求数据，`onReady` 才适合做节点相关操作
5. `onShow` 会重复触发，很多“返回后刷新”逻辑应放这里
6. 微信登录态和你自己系统登录态必须分开
7. 不要假设页面内存和小程序进程会一直活着
8. 分包、首屏结构、列表复杂度，应该在项目中前期就考虑

## 十一、这一组笔记后面我准备怎么拆

为了从深到浅把 `uni-app` 的微信小程序知识梳理清楚，这一组现在已经拆成下面几篇：

1. 第 2 篇：`uni-app` 微信小程序工程结构实战，包括 `pages.json`、页面组织、条件编译、开发者工具联调
2. 第 3 篇：微信小程序常见能力在 `uni-app` 里的接法，包括登录、用户信息、上传下载、订阅消息、支付与分享
3. 第 4 篇：性能优化专题，包括分包、长列表、首屏、图片、缓存、`setData` 成本意识
4. 第 5 篇：组件设计与跨端差异处理，包括 `props`、事件、`behaviors`、样式隔离、条件编译与原生组件避坑
5. 第 6 篇：登录态、缓存、前后台恢复与隐私授权，包括 `session_key`、业务 `token`、本地存储与稳定性治理
6. 第 7 篇：特殊能力实战，包括 `web-view`、页面通信、登录联动、分享落地与微信专属能力边界
7. 第 8 篇：常用功能全景手册，把页面交互、跳转、请求、缓存、登录、手机号、上传下载、定位、分享、支付与 `web-view` 串成一份新手可查的实战指南

如果前面这一篇是“建立世界观”，那后面几篇就是“把世界观落到工程做法里”。
