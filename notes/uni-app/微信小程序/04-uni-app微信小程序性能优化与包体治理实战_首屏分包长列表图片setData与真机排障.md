# uni-app 微信小程序性能优化与包体治理实战：首屏、分包、长列表、图片、setData 与真机排障

> 这是这一组 `uni-app 微信小程序` 笔记的第 4 篇。  
> 截至 `2026-03-31`，我继续对照 `uni-app` 官方文档和微信开放文档，把“项目为什么会慢、该从哪里下手治理”这一层单独拆出来讲。  
> 这篇不追求把所有优化点堆成清单，而是先回答一个更实用的问题：`公司里的 uni-app 项目做成微信小程序后，为什么开发者工具里看着还行，一上真机就开始慢、卡、白屏，甚至包体超限？`

> 建议先读 `uni-app/通用基础`：
> 1. 第 2 篇《uni-app 应用生命周期、页面生命周期与常见页面事件》
> 2. 第 3 篇《uni-app 工程结构与配置基础：App.vue、main.js、pages.json、manifest.json、uni.scss 与 static》
> 3. 第 4 篇《uni-app 跨端开发通用方法：路由、请求、缓存、上传下载与条件编译》
> 这一篇默认你已经知道通用路由、缓存、配置和页面生命周期，下面只抓微信小程序真机会慢的几个核心成本：包体、首屏、逻辑层到渲染层通信、资源和内存。

## 一、先说结论：首屏慢，通常不是一个点的问题，而是 4 类成本叠加

很多人一提性能优化，第一反应是：

- 接口是不是慢
- 设备是不是差
- 图片是不是太大
- 要不要上骨架屏

这些都可能有关系，但如果你是长期维护公司项目，我更建议你先用下面这 4 类成本来判断：

| 成本类型 | 典型表现 | 真正该查什么 |
| --- | --- | --- |
| 包体成本 | 冷启动慢、首次打开慢、审核前突然超包 | 主包是否过重、低频页面有没有硬塞进主包 |
| 初始化成本 | 页面打开白一下、首屏骨架出来慢、`onLoad` 很重 | 页面一上来做了多少同步计算、请求、数据拼装 |
| 通信成本 | 列表卡、滚动卡、筛选一变就顿 | `data` 太大、`setData` 太频繁、一次更新传太多 |
| 资源与内存成本 | 真机发热、滑动掉帧、图片白块、偶发黑屏 | 大图、多图、节点太多、定时器和监听没清掉 |

所以这一篇最重要的一句话是：

`微信小程序性能问题，很多时候不是“某个接口慢”，而是包体、首屏初始化、逻辑层到渲染层通信、资源与内存压力一起叠上来了。`

也正因为是叠加问题，所以你会看到一个很常见的现象：

- 开发者工具里还凑合
- 高端机问题不明显
- 低端机、弱网、长列表、老页面回切时最容易出问题

这不是玄学，而是因为真机上会真实经历：

1. 主包下载
2. 页面注入
3. 数据请求
4. 视图层渲染
5. 图片下载与解码
6. 页面切换和后台回前台

任何一个环节设计得太重，最后都会汇总成“这个页面怎么这么卡”。

## 二、包体治理要先做，因为它决定了启动下限

微信官方关于分包加载的当前口径里，有几个数字你最好先记住。

截至 `2026-03-31`，以微信官方当前《分包加载》文档为准：

- 整个小程序所有分包大小不能超过 `30M`
- 服务商代开发小程序不能超过 `20M`
- 单个主包或分包不能超过 `2M`

这里要特别提醒一句：

`uni-app` 的 `pages.json` 文档里，你有时还能看到一些旧时代的包体说明痕迹；如果和微信官方当前文档有时间差，做微信小程序时一律以微信官方当前规则为准。

### 1. 为什么主包这么关键

因为小程序启动时，默认先下载主包。

这意味着主包里塞了什么，基本就决定了首次启动成本。  
所以主包应该尽量只放：

- 首页
- `tabBar` 页面
- 启动必经页
- 少量全局公用且首屏必须的资源

而下面这些东西，更适合放分包：

- 详情页
- 报表页
- 设置页
- 活动页
- 低频后台管理页
- 体积较大的专用组件和专用静态资源

如果你把“首页不会立即用到的大图、报表组件、低频业务页面、复杂富文本模板”都塞进主包，那还没开始做首屏优化，启动就已经输了。

### 2. 在 uni-app 里，分包治理主要看 `pages.json`

在 `uni-app` 里，分包还是通过 `pages.json` 来组织，核心是：

- `subPackages`
- `preloadRule`

一个常见结构可以写成这样：

```json
{
  "pages": [
    {
      "path": "pages/home/index",
      "style": {
        "navigationBarTitleText": "首页"
      }
    },
    {
      "path": "pages/user/index",
      "style": {
        "navigationBarTitleText": "我的"
      }
    }
  ],
  "subPackages": [
    {
      "root": "pages-report",
      "pages": [
        {
          "path": "list/index",
          "style": {
            "navigationBarTitleText": "报表列表"
          }
        },
        {
          "path": "detail/index",
          "style": {
            "navigationBarTitleText": "报表详情"
          }
        }
      ]
    },
    {
      "root": "pages-setting",
      "pages": [
        {
          "path": "index",
          "style": {
            "navigationBarTitleText": "设置"
          }
        }
      ]
    }
  ]
}
```

这段 `pages.json` 结构你可以按“启动成本怎么被分摊”来读：

1. `pages` 里放的是主包页面，所以首页、我的页这类高频入口天然会跟着首启一起被下载。
2. `subPackages` 里的 `pages-report`、`pages-setting` 被拆出来，本质上是在告诉微信“这些低频功能不用让每个用户第一次启动都付费”。
3. `root` 不是装饰字段，它决定了这个分包下面页面和资源的实际归属边界。
4. 这类配置最怕的不是写法错，而是业务判断错，比如把大报表页、营销页、复杂详情页继续留在主包。

这类拆法背后的原则不是“为了分包而分包”，而是：

`让用户第一次进入时，只为当前最有可能访问到的页面付费。`

### 3. 分包预下载不是银弹，但很适合“高概率下一跳”

如果你已经知道用户大概率会从首页跳到某几个页面，那就可以评估 `preloadRule`。

```json
{
  "preloadRule": {
    "pages/home/index": {
      "network": "all",
      "packages": [
        "pages-report"
      ]
    }
  }
}
```

`preloadRule` 这段配置最容易被误解成“多配就更快”，其实它更像一个受约束的提前投资：

1. 触发点是 `pages/home/index`，说明只有用户进入首页后，框架才会考虑预拉后续分包。
2. `packages: ['pages-report']` 表达的是“我判断用户下一跳大概率会去报表分包”，而不是无脑全预下。
3. `network: 'all'` 会直接影响弱网体验，所以配之前一定要先确认首页本身是不是已经足够轻。
4. 只有当主包已经压得比较健康时，预下载才是在锦上添花；否则它很容易变成和首屏请求抢资源。

这里的思路是：

- 用户还停留在首页时
- 框架提前把高概率下一跳的分包拉下来
- 等用户真正点进去时，切页成本更低

但你不要把预下载理解成“多配几个就一定更快”。  
如果首页本身已经很重、网络又差，你再无脑预下载，反而会把启动阶段搞得更挤。

更稳的判断方法是：

- 用户下一跳概率高不高
- 这个分包体积大不大
- 首页阶段是否已经有很多核心请求
- 弱网下会不会抢关键资源

`preloadRule` 适合高概率下一跳，不适合“我怕慢所以全都预下”。

## 三、首屏优化，核心不是“先请求”，而是“先让页面尽快可用”

很多项目首屏慢，不是慢在网络，而是慢在页面一打开就想把所有事情一次做完：

- 拉首页全部接口
- 初始化筛选条件
- 还原复杂缓存
- 组装大对象
- 计算多个列表
- 预热弹窗
- 顺手再做埋点和权限判断

最后就是 `onLoad`、`onShow`、首个渲染周期一起爆了。

更稳的做法是把首屏拆成两段：

1. 先把页面框架和首屏必要数据渲染出来
2. 再补次级内容、低优先级请求和非关键计算

### 1. 首页和 `tabBar` 页，只放首屏必须内容

如果某块内容不是用户一打开就必须看到，那它就不应该参与首屏争夺。

例如：

- 首屏先显示顶部信息、核心卡片、首屏列表前几项
- 次屏内容滚动到附近再请求或再渲染
- 营销弹窗、推荐位、统计图表放到首屏稳定后再补

你可以把首页理解成一条装配线：  
先让“用户有感知的第一眼”出来，再慢慢把“后面那一截”补齐。

### 2. 页面跳转前，就可以为下一页提前做准备

微信官方关于页面切换优化的建议里，核心思路之一就是：

- 下个页面需要的数据，可以提前请求
- 页面之间可以通过 `EventChannel` 传数据
- `onHide`、`onUnload` 阶段不要做耗时操作和高频更新

在公司项目里，这通常比“页面打开后从零开始加载”要稳很多。

比如一个列表点详情的链路，可以这样拆：

1. 列表页点击卡片
2. 先拿到当前卡片的基础数据
3. 通过 `EventChannel` 或参数把首屏可展示信息带过去
4. 详情页先渲染基础内容
5. 再补完整详情接口

这样用户看到的就不是“跳过去后白一下再开始转圈”，而是“先出来一版可用内容，再继续补全”。

### 3. `onHide` 和 `onUnload` 不要顺手干重活

很多卡顿不是发生在你“进入页面”的时候，而是发生在你“离开页面”的时候。

例如有人会在 `onHide`、`onUnload` 里：

- 强行保存大块状态
- 触发埋点并同步等待
- 再做一次 `setData`
- 关闭页面前清一堆复杂结构

这会直接拖慢切页体验。

微信官方在页面切换优化里明确提醒过，离场阶段要避免耗时操作、同步调用和不必要的 `setData`。  
所以更稳的原则是：

- 能前置就前置
- 能异步就异步
- 能不在离场时做，就不要卡在离场做

### 4. `handleWebviewPreload` 是高级项，不是默认解法

微信官方提供了 `handleWebviewPreload`，适合部分页面切换场景做更深一层的预处理。  
但它有前提：

- 基础库 `2.15.0+`
- 仅安卓
- 还分 `static / auto / manual`

所以我更建议把它当成：

`在主包治理、首屏拆层、分包预下载都做好后，再评估要不要上的高级手段。`

不要一上来就指望靠它救整个项目。

## 四、理解 `setData` 成本，是做微信小程序性能优化最关键的一步

很多维护 `uni-app` 项目的人，会把“响应式更新”想得太轻。

但到了微信小程序里，你必须重新接受一件事：

`逻辑层和渲染层是分开的，数据通信是异步的，每一次更新都有成本。`

这就是为什么微信官方会反复强调合理使用 `setData`。

### 1. `data` 里只放渲染相关的数据

如果某些数据根本不参与页面渲染，就不要硬塞进 `data`。

例如下面这些更适合挂在实例上，而不是塞进响应式数据里：

- 原始接口返回缓存
- 临时映射表
- 防抖节流句柄
- 大型中间计算结果
- 只给逻辑判断用的状态

错误思路往往是：

```js
data() {
  return {
    pageNo: 1,
    pageSize: 20,
    requestCacheMap: {},
    rawList: [],
    hugePermissionMap: {},
    filterForm: {},
    visibleList: []
  }
}
```

这段“错误示例”最该看的不是字段名，而是它暴露出来的坏习惯：

1. `requestCacheMap`、`hugePermissionMap` 这类纯逻辑数据被塞进 `data`，意味着它们每次变化都有机会进入渲染通信链路。
2. `rawList` 和 `visibleList` 同时放进页面数据，往往会让你一次更新时把原始数据和展示数据都拖着传过去。
3. 一旦页面上还有筛选、滚动、局部交互，这种“什么都进 data”的方式会让 `setData` 体积和频率一起升高。

更稳的思路是：

```js
data() {
  return {
    pageNo: 1,
    visibleList: [],
    filterForm: {}
  }
},
created() {
  this.rawList = []
  this.requestCacheMap = new Map()
  this.hugePermissionMap = {}
}
```

这个“更稳示例”的价值就在于它把“展示态”和“逻辑态”拆开了：

1. `pageNo`、`visibleList`、`filterForm` 这几个字段会直接影响视图，所以保留在 `data` 里是合理的。
2. `rawList`、`requestCacheMap`、`hugePermissionMap` 只是给逻辑判断和中间计算用，就应该移到实例字段。
3. 这样做的收益不是代码更短，而是渲染层再也不用为这些大对象付通信成本。

一句话记住：

`渲染无关的数据，不要参与渲染层通信。`

### 2. 高频更新要合并，不要一口气打很多次

下面这种写法，在列表、滚动、输入联动场景里特别伤：

```js
this.setData({ loading: true })
this.setData({ pageNo: nextPageNo })
this.setData({ list: newList })
this.setData({ finished: newList.length >= total })
```

更好的方式是一次合并：

```js
this.setData({
  loading: true,
  pageNo: nextPageNo,
  list: newList,
  finished: newList.length >= total
})
```

这两段代码放在一起看，差别不是“写法风格”，而是一次页面更新到底被拆成几次跨层通信：

1. 左边连续 4 次 `setData`，意味着逻辑层可能要向渲染层发 4 次更新指令。
2. 右边把同一轮状态变化合并成一次提交，渲染层就只需要接收和处理一轮更新。
3. 在列表翻页、滚动联动、表单联动这种高频场景里，这种合并差异会直接体现在掉帧和卡顿上。

连续多次更新，最终吃掉的是通信和渲染成本。

### 3. 只更新变更字段，不要把整棵对象重新扔过去

微信官方明确不建议做这种事：

```js
this.setData(this.data)
```

也不要动不动就把整个对象或整个数组完整替换。  
如果只是某个字段变了，就用路径更新：

```js
this.setData({
  ['form.name']: form.name,
  ['list[' + index + '].checked']: true
})
```

路径更新这段代码真正体现的是“只更新必要最小面”：

1. `['form.name']` 只告诉渲染层名字字段变了，不会把整份表单对象一起重新传。
2. `['list[' + index + '].checked']` 只影响当前列表项，不会把整个 `list` 数组整批替换。
3. 这在长列表和复杂表单里特别重要，因为你越精确描述变更，渲染层要做的工作就越少。

这样做的核心价值是：

- 传输体积更小
- 视图层变更范围更明确
- 不容易把无关数据一起拖着更新

### 4. 单次 `setData` 也有体积上限

微信官方 `Page` 文档里明确提到，单次 `setData` 传输的数据不能超过 `1024kB`。

所以你看到下面这些现象时，要立刻往这方向怀疑：

- 大列表整批塞入页面
- 整个表单对象、整份树结构一起更新
- 把后端大响应原封不动塞进页面数据

这类问题很多时候不是“代码能不能跑”，而是“就算能跑，也已经不适合上线了”。

### 5. 高频局部状态，尽量下沉到小组件

如果某个区域变化特别频繁，比如：

- 倒计时
- 某个商品卡片内部状态
- 列表项局部交互
- 局部吸顶态

那就不要每次都让整个页面跟着刷新。  
把高频更新状态收进小组件，能明显降低页面级通信负担。

### 6. 后台页面不要高频更新

微信官方也提醒过，页面切到后台后，不要继续做高频 `setData`。  
所以页面 `onHide` 后：

- 能停的轮询就停
- 能停的定时器就停
- 能暂停的动画和局部更新就暂停

不然用户切回来时，很容易遇到状态错乱、补帧卡顿，甚至内存压力变高。

## 五、长列表优化，重点不在“滚动快不快”，而在“你到底渲染了多少东西”

很多公司项目卡顿最明显的地方，不是首页，而是：

- 通讯录
- 消息列表
- 商品流
- 数据报表
- 审批记录

因为这些地方最容易把三个问题叠在一起：

1. 列表长
2. 节点多
3. 图片和交互还复杂

### 1. 没必要就不要定义 `onPageScroll`

微信官方对这一点讲得很直白：

- 不需要就不要定义空的 `onPageScroll`
- 更不要在 `onPageScroll` 里频繁 `setData`

因为一旦你监听滚动，很多人就会顺手在里面做：

- 吸顶切换
- 返回顶部按钮显隐
- 进度条变化
- 查询节点位置

如果这些更新又直接带页面级 `setData`，卡顿会很明显。

### 2. 元素曝光、懒渲染，优先 `IntersectionObserver`

如果你的目标是：

- 到视口再加载图片
- 滑到某区域再请求次级数据
- 统计模块曝光

那优先考虑 `IntersectionObserver`，不要总想着：

- `onPageScroll`
- `SelectorQuery`
- 自己算滚动距离

微信官方渲染性能文档已经明确给出方向：  
曝光判断场景，优先观察器，不要靠持续滚动监听硬算。

### 3. 长列表一定要有“分段渲染”思维

如果一个接口一次返回 200 条、500 条，你不要因为接口给了，就一股脑全渲进去。

更稳的做法通常是：

- 分页加载
- 分批 append
- 先渲染首屏若干条
- 再在滚动过程中按段补齐

如果业务允许，再进一步：

- 折叠次要信息
- 减少每个列表项里的层级
- 降低每项里的图片和复杂组件数量

性能优化的底层逻辑从来不是“让框架更努力”，而是“别让页面一次背太多东西”。

### 4. WXML 节点数量本身就要控制

微信官方对页面结构也给了明确建议：

- 页面节点数尽量控制在 `< 1000`
- 层级深度尽量 `< 30`
- 单个节点子节点数尽量 `<= 60`

这些数字不需要你机械背诵，但你应该形成习惯：

如果一个列表项里已经有很多层：

- 外层卡片
- 条件渲染块
- 多套图标
- 多个插槽
- 再嵌几层自定义组件

那列表一长，性能一定会吃亏。

## 六、图片和资源治理，很多时候比接口优化更直接

微信小程序里很常见的一种误判是：

`接口都不慢，为什么页面还是慢？`

答案往往在图片。

因为图片的成本不只是下载，还包括：

- 解码
- 绘制
- 占用内存
- 滑动时反复参与渲染

### 1. 图片要按展示尺寸来，不要按原图尺寸来

如果页面上最终展示的是一个 200px 左右的小图，你却给它上一张几千像素的大图，那下载、解码、绘制、内存都在白白花钱。

所以更稳的原则是：

- 给前端的图尽量按展示尺寸生成
- 优先用合适格式和压缩比
- 列表缩略图和详情大图分开处理

### 2. 不要滥用 `widthFix` 和 `heightFix`

微信官方资源优化文档里也明确提醒过，不要滥用 `image` 的 `widthFix / heightFix`。

因为这类模式如果配合长图、大图、多图，很容易让布局和解码成本一起变重。  
更好的方式通常是：

- 提前给容器尺寸
- 确保占位稳定
- 让图片按已知盒子去显示

这样既减少抖动，也更利于首屏稳定。

### 3. 主包里不要塞太多本地大图

有些项目为了“稳”，会把很多资源直接打进包里。  
但对微信小程序来说，这会直接挤占包体预算，尤其会压主包。

所以你最好先问自己：

- 这张图是不是首屏必须
- 是不是所有用户都会立刻用到
- 能不能放 CDN
- 能不能缩成缩略图
- 能不能晚点再加载

如果答案是否定的，就别让它抢主包位置。

### 4. 长列表多图场景，要同时控制数量和节奏

一个商品流里如果每条都有：

- 大图
- 多个角标
- 多个状态图标
- GIF 或高分辨率素材

那你就不是在渲染一个列表，你是在让设备背着一堆图一路滑。

更稳的做法通常是：

- 首屏只出必要图片
- 视口附近再加载
- 先缩略图，必要时再切高清
- 尽量让每个列表项的图片尺寸统一

## 七、为什么开发者工具不明显，真机却更容易出问题

因为真机上性能问题会被放大到真实环境里：

- 网络真实波动
- 设备内存有限
- 图片解码真实发生
- 前后台切换真实发生
- 分包下载真实发生

这也是为什么很多问题一定要上真机看。

### 1. 真机调试要看 4 类信息

我更建议你把排查顺序固定成下面这样：

1. 看包体结构是不是合理
2. 看页面首屏阶段到底请求了什么、算了什么、渲了什么
3. 看滚动时有没有高频更新和大图
4. 看内存、弱网、前后台切换时有没有异常

工具层面可以优先配合：

- 微信开发者工具性能面板
- 真机调试 2.0
- 内存调试
- FPS 面板

如果一个问题只在真机复现，你就不要继续拿开发者工具里的“感觉还行”安慰自己了。

### 2. 内存问题，常常不是“内存大”，而是“东西没释放”

微信官方内存优化文档里提到过几个很典型的问题源头：

- 页面实例被全局变量或闭包继续引用
- 事件监听没解绑
- 定时器没清掉

这类问题在公司项目里非常常见，因为很多老页面会这么写：

```js
onShow() {
  this.timer = setInterval(() => {
    this.refreshBadge()
  }, 1000)
},
onHide() {
  // 什么都不做
},
onUnload() {
  // 什么都不做
}
```

更稳的做法至少要保证离场可清理：

```js
onHide() {
  if (this.timer) {
    clearInterval(this.timer)
    this.timer = null
  }
},
onUnload() {
  if (this.timer) {
    clearInterval(this.timer)
    this.timer = null
  }
}
```

这两段代码放一起看，能非常直观地说明“内存问题很多时候就是清理问题”：

1. 第一段在 `onShow` 开启定时器，却没有任何离场清理，页面反复进出后很容易叠出多个轮询。
2. 第二段同时在 `onHide` 和 `onUnload` 清理，是因为页面“暂时不可见”和“最终销毁”都可能成为资源回收点。
3. 这种写法不只是为了省电，更是为了避免页面切后台后继续高频更新，回来时出现补帧卡顿和状态错乱。

如果项目对稳定性要求高，还可以结合 `wx.onMemoryWarning` 做主动降级，例如：

- 释放大列表缓存
- 清空不必要的大对象
- 暂停次级页面预加载

### 3. 弱网和前后台切换会让很多“平时没事”的逻辑暴露出来

微信官方网络文档里有一个很实战的点：

`小程序切到后台 5 秒后，网络请求可能被中断，并回调 interrupted。`

这意味着你不能把很多关键链路写成：

- 发出去就不管
- 回来继续假设它一定成功
- 页面恢复时也不重新校验

更稳的做法是：

- 关注 `wx.onNetworkStatusChange`
- 在合适场景下评估 `wx.onNetworkWeakChange`
- 页面回到前台时，对关键请求和关键状态做兜底校验

如果是纯展示类、非强时效的数据，还可以评估 `wx.createCacheManager`。  
但这里一定要分清边界：

`缓存可用于弱网兜底，不等于所有数据都适合缓存。`

像下面这些带强时效和安全含义的数据，就不要靠缓存假装“还能用”：

- `wx.login`
- `wx.checkSession`
- 依赖当前登录态的关键鉴权结果

## 八、把这一篇压成 10 条最实用规则

如果你只想先记住最能直接落地的版本，那就先记这 10 条：

1. 主包只放启动必需页面和资源，低频功能尽量进分包。
2. 截至 `2026-03-31`，微信官方当前口径是总包 `30M`、服务商代开发 `20M`、单包 `2M`，包体规则先按微信官方看。
3. 首页和 `tabBar` 页只保留首屏必需内容，次级内容延后。
4. 页面跳转时尽量提前准备下一页首屏数据，不要让详情页总是从零开始。
5. `data` 只放渲染相关数据，逻辑缓存和大对象不要硬塞进去。
6. 高频更新要合并，别连续多次 `setData`，更不要 `this.setData(this.data)`。
7. 没必要就不要监听 `onPageScroll`，曝光和懒加载优先 `IntersectionObserver`。
8. 长列表要分页、分段、减节点，不要一次渲染整批大数据。
9. 图片按展示尺寸和场景来，不要让大图、多图和主包一起把性能拖垮。
10. 真机一定要看内存、弱网、前后台切换，不要只在开发者工具里判断“应该没问题”。
