# uni-app 微信小程序迁移到钉钉小程序超详细指南

> 适用人群：
> 你现在维护的是一个用 uni-app 写的微信小程序，老板要求你把它改成钉钉小程序，而且你之前几乎没碰过钉钉开放平台。
>
> 这篇文章会尽量按照“像教婴儿走路一样”的方式来写，不追求一句话讲完，而是追求你真的能动手做出来。

## 一、先说结论：这件事不是“重写一个项目”，而是“先判断，再迁移，再补差异”

如果你的项目真的是 `uni-app` 项目，而不是“原生微信小程序项目外面套了一层 uni-app 名字”，那这件事通常不是从零重写。

你真正要做的事，一般是这 4 类：

1. 在钉钉开放平台先判断应用类型；如果是教育公司把能力提供给多家学校/机构用，更可能是“第三方企业应用”，然后再给它加上“小程序能力”。
2. 在 `uni-app` 里新增一个“钉钉小程序”编译目标。
3. 把项目里所有直接写死的微信能力，替换成：
   - 能跨端的 `uni.*`
   - 只能钉钉用的 `dd.*`
   - 通过条件编译拆开的平台差异代码
4. 在钉钉开发工具里调试、上传、发布。

一句大白话总结：

`页面和大部分业务逻辑尽量复用，平台能力的不同点单独处理。`

## 二、我查到的官方结论和路线

截至 `2026-03-29`，我把钉钉官方和 uni-app 官方资料串起来后，可以得出下面几个非常关键的结论：

### 1. 结合真实业务背景，你这个场景更像“第三方企业应用”

我前面第一次判断时，默认你是“自己公司内部用”，所以偏向企业内部应用。

但如果真实情况是：

- 你们是教育公司
- 产品要给多家学校、培训机构、教学机构使用
- 对方想把你们原来的微信小程序能力，接到他们自己的钉钉里

那这个业务形态，和钉钉官方“应用类型介绍”里说的“第三方企业应用”更接近。

钉钉官方对两类应用的区分，核心可以这样理解：

- 企业内部应用：只给单一企业自己的内部人员使用
- 第三方企业应用：产品方案商把同一套产品提供给多个企业客户使用

而钉钉官方“第三方企业应用学习指南”也明确写了：

- 产品方案商可以开发第三方企业应用，并推广给企业客户
- 第三方企业应用既可以做 H5，也可以做小程序
- 开发前需要按服务商/合作伙伴路线做准备、权限、规范和推广配置

所以，按你现在补充的场景，我建议把路线改判为：

`教育公司给多所学校接入 -> 优先按“第三方企业应用”理解`

只有一种情况，你才更适合继续按“企业内部应用”理解：

`你们只是给某一所学校做单独定制，而且只装在这一家学校自己的钉钉组织里。`

### 2. uni-app 官方确实给了钉钉小程序方案

uni-app 官方 `package.json` 配置页明确给了“示例：钉钉小程序”：

- 自定义平台名可以叫 `mp-dingtalk`
- 实际基准平台用的是 `mp-alipay`
- 可以通过自定义常量 `MP-DINGTALK` 做条件编译
- CLI 模式下可运行 `npm run dev:custom mp-dingtalk` 和 `npm run build:custom mp-dingtalk`
- HBuilderX 会根据 `package.json` 生成“钉钉小程序”的运行/发行菜单

这个信息非常重要，因为它告诉我们：

`uni-app 并不是完全不支持钉钉，而是通过基于 mp-alipay 的自定义平台方式支持。`

### 3. 钉钉官方的小程序流程已经很完整

钉钉官方文档现在已经把流程拆得很细：

- 获取开发者权限
- 创建应用
- 添加应用能力
- 开发小程序前端
- 配置安全域名
- 上传小程序
- 添加小程序地址
- 发布应用

也就是说，你不是在“猜怎么做”，而是在“按官方流程走”。

## 三、先别急着改代码，先做 3 个判断

很多新手一上来就开始改 `wx`，这很容易把自己改晕。正确顺序是先判断。

### 判断 1：你手上的项目到底是不是标准 uni-app 项目

请先看项目里有没有这些典型文件：

- `pages.json`
- `manifest.json`
- `App.vue`
- `main.js`
- `uni.scss`
- 页面里主要用的是 `uni.navigateTo`、`uni.request`、`uni.setStorageSync` 这类 API

如果有这些，基本就是标准 uni-app 项目。

如果你看到项目里虽然是 uni-app 工程，但代码里大量是下面这些：

- `wx.login`
- `wx.request`
- `wx.getUserProfile`
- `wx.chooseMessageFile`
- `open-type="getPhoneNumber"`
- `open-data`
- `wx.cloud`
- 各种只在微信里存在的组件和授权流程

那就说明：

`这个项目虽然壳子是 uni-app，但业务实现明显偏微信，需要补一轮平台改造。`

### 判断 2：你们是在“自己内部用”，还是“给多家学校开通”

你现在应该先确认的，不再是“老板想不想做”，而是业务模式本身：

`这套功能，是不是要给多个学校/机构分别开通到他们各自的钉钉组织里？`

如果答案是“是”，那更像：

- 第三方企业应用
- 多租户产品
- 需要学校管理员安装、授权、开通

如果答案是“不是，只服务于一个组织”，那才更像：

- 企业内部应用

对你这个教育行业场景，我建议默认先按下面这个认知理解：

`你们公司是方案提供方，学校/机构是客户方，因此优先走第三方企业应用路线。`

### 判断 3：当前项目用没用微信专属能力

这是迁移难度的关键。

你可以先全局搜这些关键词：

```bash
rg 'wx\\.' src pages components
rg 'mp-weixin|weixin|WeixinJSBridge|open-type|getPhoneNumber|open-data' src pages components
rg '订阅消息|subscribe|支付|分享|客服|手机号' src pages components
```

搜出来以后，把结果分成 3 类：

1. `纯 uni 能力`
   例如：`uni.request`、`uni.navigateTo`、`uni.showToast`
   这类通常可以直接复用。

2. `微信写法，但有 uni 替代`
   例如：`wx.request` 改成 `uni.request`
   这类属于好改。

3. `微信专属业务能力`
   例如：
   - 微信登录
   - 微信手机号授权
   - 微信支付
   - 微信订阅消息
   - 微信开放数据组件
   - 微信分享卡片能力

   这类不能机械替换，必须重新设计钉钉版本方案。

## 四、你应该怎么走：完整执行路线图

你可以把整个任务理解成 5 个阶段：

1. 平台开通阶段
2. uni-app 编译目标接入阶段
3. 代码差异改造阶段
4. 本地调试与真机验证阶段
5. 上传发布阶段

下面我们一阶段一阶段走。

## 五、第 1 阶段：先把钉钉开放平台环境打通

### 第 0 步：先判断你们公司在钉钉里的角色

这一点非常非常重要。

如果你们是教育公司，学校只是你们的客户，那你们在钉钉里的角色更像：

- 产品方案商
- 服务商
- 第三方应用提供方

这意味着你后面要处理的不只是“把前端跑起来”，还包括：

- 学校怎么安装你们的应用
- 学校管理员怎么授权
- 一套系统怎么服务多个学校
- 每个学校的数据怎么隔离

所以从现在开始，你脑子里要把“一个公司一个应用”换成“多个学校多个租户”。

### 第 1 步：让管理员给你开发权限

钉钉官方“获取开发者权限”文档说得很清楚：

- 管理员可以在管理后台给成员添加开发者权限
- 开发者也可以自己发起申请，让管理员审批

所以你第一件事不是下载工具，而是找公司里有权限的人确认下面这些：

1. 你们公司有没有钉钉企业组织
2. 你是否已经加入该企业组织
3. 你是否有开发者权限
4. 谁是企业管理员，谁能帮你创建或授权应用

如果你连权限都没有，后面很多页面你根本进不去。

### 第 2 步：按“第三方企业应用”路线创建应用

基于你现在补充的业务背景，这里不建议再默认按“企业内部应用”走。

更贴近你真实场景的理解应该是：

- 你们公司是产品提供方
- 学校和教学机构是使用方
- 同一套能力要给多个组织开通

这时你应该优先看钉钉官方的“第三方企业应用学习指南”，因为它和企业内部应用相比，多了这些准备动作：

- 服务商/合作伙伴入驻
- 第三方企业应用的开发规范和审核规范
- 多企业客户的安装、推广、定向分发
- 后续可能涉及上架、自运营、客户管理

如果你只是先做一个试点版本，也可以先把“小程序前端能不能跑起来”这件事单独验证，但正式产品路线最好尽早按第三方企业应用来设计。

如果后面确认真的只是给某一所学校单独定制，不会复制给第二家学校，那你再退回企业内部应用路线也不迟。

### 第 2.5 步：把“每一所学校”当成一个独立租户

这是做教育行业钉钉接入时最容易忽略，但实际上最关键的点。

在微信小程序里，你可能更多是在想：

- 用户是谁
- 用户有没有登录

但到了钉钉第三方企业应用场景，你还必须多想一层：

- 这个用户属于哪一所学校
- 这所学校有没有安装你们的应用
- 这所学校对应的钉钉组织标识是什么
- 这所学校能看到哪些菜单、班级、课程、学生数据

也就是说，你后端很可能需要有这样的概念：

- `school_id`
- `tenant_id`
- `corp_id`
- 学校管理员授权记录
- 学校级配置

你可以把它想成一句大白话：

`以后不是“一个小程序服务所有人”，而是“一个产品服务多个学校租户”。`

这会直接影响：

- 登录后的身份绑定
- 接口传参
- 数据权限
- 菜单权限
- 学校级个性化配置

### 第 3 步：给应用加上“小程序能力”

创建应用不等于它已经能跑小程序。

钉钉官方“添加应用能力”文档要求你继续做：

`目标应用 -> 应用能力 -> 添加应用能力 -> 选择小程序`

这一步非常关键。

因为如果你没加小程序能力，后面：

- 小程序开发入口没有
- 上传入口没有
- 相关配置项也没有

所以你要记住一个顺序：

`先创建应用，再添加小程序能力。`

## 六、第 2 阶段：在 uni-app 里接入钉钉小程序编译目标

这一部分是整件事里最容易让新手迷惑的地方，我慢慢讲。

### 1. 为什么 uni-app 里不是直接写一个 `mp-dingtalk` 原生平台

uni-app 官方 `package.json` 页面给出的钉钉方案是“自定义平台”，示例核心配置如下：

```json
{
  "uni-app": {
    "scripts": {
      "mp-dingtalk": {
        "title": "钉钉小程序",
        "env": {
          "UNI_PLATFORM": "mp-alipay"
        },
        "define": {
          "MP-DINGTALK": true
        }
      }
    }
  }
}
```

你先别被 `mp-alipay` 吓到。

这不代表你在发支付宝小程序，而是：

`uni-app 官方当前是把钉钉小程序放在基于阿里系小程序适配的自定义平台方案里处理。`

### 2. 这段配置是什么意思

- `mp-dingtalk`
  这是你自定义的平台名字
- `title`
  这是给 HBuilderX 菜单展示用的名字
- `UNI_PLATFORM: 'mp-alipay'`
  表示编译底层基于支付宝小程序目标
- `MP-DINGTALK: true`
  表示你可以在代码里写 `#ifdef MP-DINGTALK`

### 3. 你应该把这段配到哪里

一般放到项目根目录的 `package.json`。

注意两个坑：

1. uni-app 官方文档明确说了，`package.json` 里不要写注释，否则扩展配置无效。
2. 如果公司项目已经有 `uni-app.scripts` 配置，你要合并，不要覆盖。

### 4. 代码里怎么区分微信和钉钉

uni-app 官方文档明确支持条件编译。

你可以这么理解：

- `MP-WEIXIN`：微信小程序专属代码
- `MP-ALIPAY`：支付宝小程序通用代码，钉钉因为走该基准平台，很多兼容代码也会走到这里
- `MP-DINGTALK`：你自己定义的钉钉专属代码

官方 `package.json` 页面还给了类似思路：

```js
// #ifdef MP
// 小程序平台通用代码（含钉钉）
// #endif

// #ifdef MP-ALIPAY
// 支付宝平台通用代码（含钉钉）
// #endif

// #ifdef MP-DINGTALK
// 钉钉平台特有代码
// #endif
```

这就是你以后改平台差异的核心武器。

### 5. 如果你用 HBuilderX，怎么运行

uni-app 官方文档说：

`HBuilderX 会根据 package.json 的扩展配置，在“运行”“发行”菜单下生成自定义菜单（钉钉小程序）`

所以如果你的项目是 HBuilderX 打开的，你可以：

1. 配好 `package.json`
2. 重启或重新打开项目
3. 看顶部“运行”菜单里是否出现“钉钉小程序”

### 6. 如果你用 CLI 工程，怎么运行

uni-app 官方文档给的命令是：

```bash
npm run dev:custom mp-dingtalk
npm run build:custom mp-dingtalk
```

这适合 CLI 方式的 uni-app 工程。

### 7. 一个很容易把人搞晕的点：到底用钉钉开发者工具，还是支付宝开发者工具

我查到的官方资料里，这里存在一个你必须知道的“现实差异”：

1. 钉钉官方“开发小程序前端”文档：
   - 让你下载并安装“钉钉开发者工具”
   - 选择“小程序”
   - 选择“钉钉”端

2. uni-app 官方 `package.json` 文档：
   - 明确写了钉钉小程序编译目录依然是 `mp-alipay`
   - 提示“需通过支付宝开发者工具，选择‘钉钉小程序’，然后打开该目录进行预览及发布”

这两份官方资料并存时，新手最容易慌。

我的建议是这样执行：

1. 优先按钉钉官方的新工具链走
   - 先安装并尝试使用钉钉开发者工具导入产物
2. 如果你的 uni-app 输出目录、导入识别或发布链路和钉钉工具不顺
   - 再按 uni-app 官方文档的说明，使用它给出的 `mp-alipay` 产物路径方案

你要记住一件事：

`看到 mp-alipay 不是你做错了，而是 uni-app 对钉钉的适配方式如此。`

## 七、第 3 阶段：开始改代码，但只改“差异点”

### 先定一个原则

迁移不是“把整个项目改成钉钉写法”。

迁移应该是：

- 能保留的页面结构保留
- 能保留的 `uni.*` 保留
- 只有平台不一样的部分才拆开

### 你要优先排查的 6 种高风险代码

#### 1. 直接写死的 `wx.*`

例如：

```js
wx.request()
wx.login()
wx.navigateTo()
wx.setStorageSync()
```

处理原则：

- 能改成 `uni.*` 的优先改成 `uni.*`
- 真的是平台专属能力，再拆成微信版和钉钉版

#### 2. 微信登录

这是最常见的大坑。

微信端常见是：

```js
wx.login()
```

但钉钉官方客户端 SDK 文档给的示例是：

```js
import * as dd from 'dingtalk-jsapi'

dd.getAuthCode({
  corpId: 'your_corp_id',
  onSuccess: (res) => {
    console.log(res.authCode)
  }
})
```

也就是说：

`微信登录 -> 钉钉免登 / authCode 流程`

这不是简单改个函数名就完了，后端通常也要跟着改，因为服务端要根据钉钉的 `authCode` 去换员工身份信息。

#### 3. 微信手机号、开放数据、订阅消息

这些往往是微信专属业务能力。

你要重点检查有没有：

- `open-type`
- `open-data`
- 订阅消息授权
- 微信手机号快捷授权

如果有，千万别幻想“换个 API 就全通了”。

这类功能通常需要：

- 钉钉侧找等价能力
- 或者改成手机号表单输入
- 或者改成员工身份自动识别
- 或者用钉钉消息/通知体系重新设计

#### 4. 微信支付

如果项目里有微信支付，那迁移到钉钉基本一定不是简单替换。

这类需求要单独确认：

- 钉钉里到底还要不要付款
- 用什么支付体系
- 是否其实变成内部审批、内部报销、内部业务单据

#### 5. 分享能力

微信里常见：

- 分享到好友
- 分享到群
- 分享卡片

钉钉的社交语义和微信不一样，这类需求要重新确认。

#### 6. 只在微信里存在的组件或样式习惯

例如：

- 某些微信专属标签
- 某些只在微信里支持的 `open-type`
- 某些依赖微信容器表现的样式

这些都要单独排查。

## 八、最推荐的改法：给平台能力包一层适配

不要在业务页面里到处写：

- 这里 `wx.login`
- 那里 `dd.getAuthCode`
- 另一处 `uni.login`

这样后面你一定会乱。

更稳妥的写法是自己封一层，比如 `src/utils/platformAuth.js`：

```js
// #ifdef MP-DINGTALK
import * as dd from 'dingtalk-jsapi'
// #endif

export function getPlatformAuthCode(corpId) {
  return new Promise((resolve, reject) => {
    // #ifdef MP-WEIXIN
    wx.login({
      success: (res) => resolve(res),
      fail: (err) => reject(err)
    })
    // #endif

    // #ifdef MP-DINGTALK
    dd.getAuthCode({
      corpId,
      onSuccess: (res) => resolve(res),
      onFail: (err) => reject(err)
    })
    // #endif
  })
}
```

这样业务页只关心：

```js
const result = await getPlatformAuthCode(corpId)
```

而不关心底层到底是微信还是钉钉。

这就是新手最该学会的“适配层思维”。

## 九、第 4 阶段：网络请求和安全域名一定要配

钉钉官方“配置安全域名”文档说得非常明确：

- 小程序前端如果要和服务端通信，必须配置 `HTTP 可信域名`
- 如果要使用 `web-view` 内嵌网页，必须配置 `Webview 可信域名`

后台路径是：

`目标应用 -> 开发配置 -> 安全设置`

你至少要确认这几件事：

1. 接口域名有没有加进去
2. H5 页面域名有没有加进去
3. 是不是 HTTPS
4. 改完域名后，有没有重新构建并重新上传

钉钉官方还特别提醒：

`安全域名更新后，必须重新构建并上传小程序包，否则更改不会生效。`

这句话你一定要记住。

很多人以为后台一保存就好了，结果前端还是报错，最后排查半天。

## 十、第 5 阶段：本地调试与真机验证

钉钉官方“开发小程序前端”文档给出的流程是：

1. 下载并安装小程序开发工具
2. 选择“小程序”
3. 选择“钉钉”端
4. 关联应用
5. 进行预览和真机调试

官方文档还提到：

- 手机端可扫码预览
- PC 端也可以做预览验证

所以你调试时不要只看模拟器。

最少要测这几类终端：

1. 手机钉钉
2. PC 钉钉
3. 不同角色账号
4. 不同网络环境

## 十一、第 6 阶段：上传小程序

钉钉官方“上传小程序”文档说明：

- 上传后得到的是“开发版本”
- 这不是正式上线版本
- 你可以基于它设置体验版进行测试

官方给的关键操作是：

1. 在开发者工具中确认项目编译通过
2. 点击“上传”
3. 填写版本号和版本说明
4. 到开发者后台确认上传是否成功

这里有两个非常实用的建议：

### 建议 1：版本号别乱写

比如：

- `1.0.0`
- `1.0.1`
- `1.1.0`

不要今天写 `test1`，明天写 `最新版本`，后面你会完全不知道哪个版本改了什么。

### 建议 2：每次上传只做一类改动

例如：

- 第一次只验证首页能不能打开
- 第二次只验证登录和接口
- 第三次再验证复杂业务

不要一口气把所有代码都扔上去，不然出问题时你根本不知道是哪一块炸了。

## 十二、第 7 阶段：发布应用

钉钉官方“发布应用”文档有几个你必须知道的点。

### 1. 先分清：你现在到底走的是内部应用发布，还是第三方应用发布

这一节是很多人最容易混淆的地方。

我前面查到的“版本管理与发布”这条文档，明确是企业内部应用发布路径。

如果你当前做的是：

- 只给某一所学校单独交付
- 只在那一个组织里用

那你可以继续参考这条内部应用发布路径：

`目标应用 -> 应用发布 -> 版本管理与发布 -> 创建新版本`

但如果你现在做的是：

- 教育公司把产品提供给多家学校
- 多家学校分别安装和开通

那你更应该按第三方企业应用路线理解发布，它不仅是“传一个版本上去”，还会涉及：

- 学校怎么安装
- 学校管理员怎么开通
- 是否上架
- 是否定向分发
- 后续客户管理和运营

### 2. 版本号有格式要求

官方文档明确提到：

- 格式类似 `1.0.0`
- 每段最大 99

所以不要自己造奇怪格式。

### 3. 发布时要注意“工作台应用”

钉钉官方文档特别提醒：

`在发布非工作台应用（如小程序应用或网页应用）时，必须同时选择“工作台应用”进行发布，否则可能导致应用默认下线。`

这句话非常像考试重点。

很多新手以为“我只选小程序就行了”，结果应用在客户端里根本找不到。

### 4. 可以设置应用可见范围

你可以控制：

- 哪些成员可见
- 哪些部门可见

这对公司内部灰度测试特别有用。

## 十三、如果老板还要求 PC 端也能用，你别忘了这一步

钉钉官方“添加小程序地址”文档说：

- 如果希望小程序不仅在移动端用，也能在钉钉 PC 端访问
- 需要去小程序能力配置里开启 PC 端支持
- 可以配置 `pc 端应用访问地址`

如果你不配，可能出现：

- 手机能打开
- PC 钉钉搜得到但打不开
- 或者打开后不是你想要的页面

所以如果老板提了“电脑上也要能用”，你要把这一步单独列进上线清单。

## 十四、你最应该优先验证的测试清单

你不要一上来就测 100 个点。

先按这个顺序测：

### 第一轮：先活过来

1. 应用能在钉钉里打开
2. 首页不白屏
3. 静态资源能加载
4. 路由跳转正常

### 第二轮：核心链路

1. 登录或身份识别正常
2. `uni.request` 请求能通
3. token、缓存、状态管理正常
4. 列表、详情、表单正常

### 第三轮：平台差异

1. 所有直接写微信逻辑的地方都测一遍
2. 所有 `#ifdef MP-DINGTALK` 代码都测一遍
3. 所有 `web-view` 页面都测一遍
4. 所有上传、下载、定位、扫码等能力都测一遍

### 第四轮：终端差异

1. Android 钉钉
2. iPhone 钉钉
3. PC 钉钉

## 十五、迁移时最容易踩的坑，我提前给你标出来

### 坑 1：以为“uni-app = 完全不用改”

不对。

`uni-app` 的价值是帮你复用大部分代码，但它不能替你抹平所有平台业务差异。

### 坑 2：看到 `mp-alipay` 就以为方向错了

不对。

这是 uni-app 官方给钉钉做自定义平台时的适配方案。

### 坑 3：只改前端，不改后端登录逻辑

如果你原来是微信登录，现在改成钉钉登录，后端大概率也要接钉钉身份识别流程。

### 坑 4：后台配了域名，但没有重新构建上传

钉钉官方已经明确提醒，这种情况下配置不会生效。

### 坑 5：业务页里到处写平台判断

这样写 3 天就乱。

正确做法是：

- 抽适配层
- 页面只调统一方法
- 平台差异收口到少量文件

### 坑 6：没有先搜微信专属代码

你不先排查，就会以为“明明编译成功了，为什么到钉钉全坏了”。

实际上问题往往不是编译，而是业务能力不兼容。

## 十六、给你一个最适合新手的执行顺序

如果你现在就要开始做，我建议你严格按下面顺序执行：

1. 先开一个新分支

```bash
git checkout -b feature/mp-dingtalk
```

2. 全局搜索微信专属代码

```bash
rg 'wx\\.|open-type|open-data|WeixinJSBridge|mp-weixin' src pages components
```

3. 找管理员开通钉钉开发权限
4. 按真实业务场景创建应用：如果是给多家学校/机构使用，优先按第三方企业应用创建；如果只是某一所学校单独定制，才考虑企业内部应用
5. 给应用添加“小程序能力”
6. 在项目 `package.json` 里加 `mp-dingtalk` 配置
7. 先不改复杂业务，只先把首页编译到钉钉跑起来
8. 首页跑通后，再逐个替换微信能力
9. 配置安全域名
10. 真机调试
11. 上传开发版本
12. 设体验版给老板和测试同事验收
13. 最后再正式发布

这个顺序的核心思想是：

`先证明链路通，再补业务细节。`

不要反过来。

## 十七、如果你问我：这件事最关键的能力是什么

我会回答你两个词：

### 1. 条件编译

你以后会经常用到：

```js
// #ifdef MP-WEIXIN
// 微信代码
// #endif

// #ifdef MP-DINGTALK
// 钉钉代码
// #endif
```

### 2. 适配层

把这些能力都包起来：

- 登录
- 请求头
- 上传
- 下载
- 扫码
- 选择人员
- 选择部门

然后业务页面只调你自己的统一方法。

这样以后你就不是“会修一个平台”，而是真的开始具备“多端项目维护能力”了。

## 十七点五、基于你现在这个教育行业场景，我给你的推荐结论

如果按你刚补充的背景来判断，我现在最推荐你的落地方向是：

`第三方企业应用 + 钉钉小程序`

原因很简单：

1. 你们是教育公司，不是学校内部技术团队
2. 你们的产品不是只给自己公司员工用，而是给多个学校/机构客户用
3. 你当前最熟的是 uni-app 小程序体系，迁到钉钉小程序复用率更高

如果后面学校还有更强的 PC 场景需求，再补：

`第三方企业应用 + H5/网页应用`

这样会更稳。

## 十九、最后给你一句最务实的话

如果你是刚入行的前端，不要把这件事理解成“我要一下子学会钉钉开放平台的全部知识”。

你只需要按顺序完成下面三件事：

1. 先把应用和权限环境打通
2. 先把首页编译并在钉钉里跑起来
3. 再一个功能一个功能迁移微信差异

只要你做到这三点，这个任务就已经不是“不会做”，而是“按清单推进”了。

## 二十、补一句最现实的经验判断

如果你的项目：

- 页面多，但平台能力少
- 大部分用的是 `uni.*`
- 没有微信支付、订阅消息、开放数据、复杂分享

那这次迁移大概率是“中等工作量”。

如果你的项目：

- 强依赖微信登录
- 强依赖微信开放能力
- 后端也绑死了微信身份体系

那这次迁移就不是单纯前端改造，而是“前后端联动迁移”。

先把这个认知建立起来，你后面和老板沟通排期时，心里就不会发虚。
