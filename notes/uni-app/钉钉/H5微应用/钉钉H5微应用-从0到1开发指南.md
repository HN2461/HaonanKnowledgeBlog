# 钉钉 H5 微应用 - 从 0 到 1 开发指南

> 适合谁看：你已经有 uni-app 项目，老板要求先做一个钉钉 H5 微应用版本试试水，不想一上来就搞钉钉小程序。
>
> 说明：本文是“从零搭建”的通用接入指南；当前仓库的实际实现请优先对照 `pages/login/index.vue`、`secondary/dingTalkH5Login/index.vue`、`secondary/dingTalkLogin/index.vue`、`util/platform/auth.js`、`util/platform/runtime.js`、`util/auth/dingtalkLogin.js`。当前项目采用的是“登录链路内按需动态加载钉钉 JSAPI SDK”的方案，不要直接按本文默认一定存在 `dingtalk-jsapi` npm 依赖或 `util/platform/dingtalkH5.js`。

---

## 一、先搞清楚：H5 微应用到底是什么

H5 微应用本质上就是一个网页，套在钉钉的 webview 里跑。

和钉钉小程序的核心区别：

| 对比项 | 钉钉小程序 | H5 微应用 |
|---|---|---|
| 运行方式 | 钉钉原生小程序运行时 | 钉钉内嵌 webview（网页） |
| 代码产物 | `mp-alipay` 编译产物 | `h5` 编译产物，部署到你自己服务器 |
| 部署位置 | 上传到钉钉开放平台 | 你自己的服务器 + 域名 |
| 更新方式 | 上传新版本，走审核 | 直接更新服务器文件，实时生效 |
| 性能体验 | 接近原生，流畅 | webview 网页，相对慢 |
| JSAPI 调用 | `dd.*` 直接调 | 需要先做 JSAPI 鉴权，再调 `dd.*` |
| 开发门槛 | 较高，需要适配小程序语法 | 低，就是写网页 |
| 适合场景 | 正式交付、体验要求高 | 快速试验、内部工具、首次接入验证 |

**结论：H5 微应用更简单，适合首次接入钉钉时先跑通链路。**

## 二、整体流程概览

```
钉钉开放平台
  └─ 创建应用（企业内部应用）
       └─ 添加 H5 微应用能力
            └─ 配置首页地址（你的服务器域名）
            └─ 配置可信域名

你的服务器
  └─ uni-app 编译 H5 产物
       └─ 部署到服务器
            └─ 配置 Nginx（HTTPS + SPA 路由）

前端代码
  └─ 引入 dingtalk-jsapi
       └─ dd.ready 等钉钉容器初始化完成
            └─ dd.runtime.permission.requestAuthCode 获取免登码
                 └─ 发给后端换系统 token
```

## 三、第一步：钉钉开放平台配置

### 3.1 创建应用

1. 进入 [钉钉开放平台](https://open-dev.dingtalk.com)
2. 进入「应用开发」→「企业内部开发」→「H5 微应用」→「创建应用」
3. 填写应用名称、描述、图标

> 注意：选「企业内部开发」，不是「第三方企业应用」。H5 微应用试水阶段先用内部应用，流程最简单。

### 3.2 配置开发信息

创建完成后进入应用详情，找到「开发管理」：

- **应用首页地址**：填你部署好的 H5 地址，例如 `https://your-domain.com/`
- **PC 端首页地址**（可选）：如果要支持 PC 钉钉，也填上

### 3.3 配置可信域名

在「开发管理」→「服务器出口 IP」或「安全设置」里：

- 添加你的域名到「可信域名」列表
- 钉钉强制要求 HTTPS，HTTP 域名不可用

### 3.4 配置应用权限

在「权限管理」里，至少开通：

- `通讯录个人信息读权限`（获取用户信息需要）

不配权限，后续后端调接口会报权限错误。

### 3.5 发布应用

配置完成后点「发布」，不发布的话配置不生效。

## 四、第二步：uni-app 编译 H5 产物

### 4.1 HBuilderX 方式

「发行」→「网站-H5 手机版」→ 编译

产物在：`unpackage/dist/build/h5/`

### 4.2 命令行方式

```bash
npm run build:h5
```

产物同样在 `unpackage/dist/build/h5/`

### 4.3 注意事项

uni-app H5 默认是 hash 路由（`/#/pages/...`），钉钉 webview 对 hash 路由兼容性好，**建议先保持 hash 模式**，不要改成 history 模式，省去 Nginx 额外配置。

如果确实需要 history 模式，在 `manifest.json` 里配置：

```json
"h5": {
  "router": {
    "mode": "history",
    "base": "/"
  }
}
```

## 五、第三步：部署到服务器

### 5.1 把产物上传到服务器

把 `unpackage/dist/build/h5/` 目录下的所有文件上传到服务器，例如放到 `/var/www/dingtalk-h5/`。

### 5.2 Nginx 配置（hash 路由版，推荐）

```nginx
server {
    listen 443 ssl;
    server_name your-domain.com;

    ssl_certificate     /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    root /var/www/dingtalk-h5;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    # 接口代理（如果前端需要跨域）
    location /api/ {
        proxy_pass http://your-backend-server/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

# HTTP 强制跳 HTTPS
server {
    listen 80;
    server_name your-domain.com;
    return 301 https://$host$request_uri;
}
```

### 5.3 Nginx 配置（history 路由版）

如果用了 history 模式，`location /` 里的 `try_files` 必须加上 fallback 到 `index.html`，否则刷新页面会 404：

```nginx
location / {
    try_files $uri $uri/ /index.html;
}
```

## 六、第四步：前端接入 dingtalk-jsapi

### 6.1 安装

```bash
npm install dingtalk-jsapi --save
```

### 6.2 H5 微应用的 JSAPI 鉴权流程

H5 微应用里要区分两类 JSAPI：

1. **通用业务 JSAPI**：比如通讯录、分享、打开链接、选人等，这类接口仍然需要后端签名 + `dd.config`。
2. **免登 JSAPI**：`dd.runtime.permission.requestAuthCode` / `dd.getAuthCode` 这一类登录接口，根据钉钉官方文档，**不需要先做 `dd.config`**，但依然建议放在 `dd.ready` 中调用。

也就是说：**如果当前页面只是为了拿 `authCode` 做登录，可以直接走 `dd.ready → requestAuthCode`；只有后面还要继续调用其他需签名的 JSAPI，才需要补 `dd.config`。**

通用业务 JSAPI 的鉴权流程（需要后端配合）：

```
前端加载页面
  → 前端把当前页面 URL 发给后端
  → 后端用 appKey + appSecret 换 access_token
  → 后端用 access_token 换 jsapi_ticket
  → 后端用 ticket + nonceStr + timestamp + url 计算签名
  → 后端把签名信息返回给前端
  → 前端调用 dd.config() 完成鉴权
  → 鉴权成功后才能调用通讯录、分享等需签名的 JSAPI
```

### 6.3 前端鉴权代码（仅当你要调用通用业务 JSAPI 时）

```js
import * as dd from 'dingtalk-jsapi'

// 调用后端接口获取鉴权配置
async function initDDConfig() {
  const url = location.href.split('#')[0] // 当前页面 URL，不含 hash
  const res = await uni.$uv.http.get('/register/dingtalk/jsapi-config', {
    params: { url }
  })
  const { agentId, corpId, timeStamp, nonceStr, signature } = res.data

  dd.config({
    agentId,
    corpId,
    timeStamp,
    nonceStr,
    signature,
    jsApiList: [
      'runtime.info',
      'biz.contact.choose',
      'device.notification.confirm',
    ]
  })

  dd.error((err) => {
    console.error('dd.config 鉴权失败', err)
  })
}
```

> 注意：`url` 必须是当前页面完整 URL，不含 `#` 及后面的部分。每次页面 URL 变化都需要重新鉴权。

### 6.4 免登获取 authCode

如果当前只是做登录，直接在 `dd.ready` 里调用免登接口即可：

```js
function getDingtalkAuthCode(corpId) {
  return new Promise((resolve, reject) => {
    dd.ready(() => {
      dd.runtime.permission.requestAuthCode({
        corpId,
        onSuccess: (info) => resolve(info.code || info.authCode),
        onFail: reject
      })
    })

    dd.error(reject)
  })
}
```

> 当前项目 `secondary/dingTalkH5Login/index.vue` + `util/platform/auth.js` 采用的就是这条链路：先等 `dd.ready`，然后直接调用 `requestAuthCode`，没有额外做 `dd.config`，这一点和钉钉官方当前文档是一致的。

### 6.5 完整登录流程

```js
async function loginWithDingtalk() {
  // 1. 获取免登码（登录场景不必先 dd.config）
  const authCode = await getDingtalkAuthCode('your-corpId')

  // 2. 发给后端换系统 token
  const res = await uni.$uv.http.post('/api/auth/dingtalk-login', {
    authCode,
    corpId: 'your-corpId'
  })

  if (res.data.code === 200) {
    // 3. 存储 token，和现有微信登录后的处理保持一致
    uni.setStorageSync('USER_INFO_TOKEN', res.data.token)
    uni.setStorageSync('USER_INFO', res.data.userInfo)
  }
}
```

## 七、后端需要做的事

前端无法单独闭环，后端需要提供：

### 7.1 可选：JSAPI 鉴权配置接口

只有当页面除了登录，还要继续调用通讯录、分享、打开链接等**需要签名的通用业务 JSAPI** 时，才需要这个接口。

如果页面只是为了调用 `requestAuthCode` / `getAuthCode` 做微应用免登，可以跳过这一节。

`GET /register/dingtalk/jsapi-config?url=当前页面URL`

后端处理逻辑：
1. 用 `appKey` + `appSecret` 请求 `https://oapi.dingtalk.com/gettoken` 换 `access_token`
2. 用 `access_token` 请求 `https://oapi.dingtalk.com/get_jsapi_ticket` 换 `jsapi_ticket`
3. 用 `ticket` + `nonceStr`（随机串）+ `timestamp`（时间戳）+ `url`（前端传来的页面 URL）计算签名：
   ```
   签名字符串 = "jsapi_ticket=xxx&noncestr=xxx&timestamp=xxx&url=xxx"（按字母序拼接）
   signature = sha1(签名字符串)
   ```
4. 返回 `{ agentId, corpId, timeStamp, nonceStr, signature }`

> `access_token` 和 `jsapi_ticket` 都有有效期（2 小时），后端需要做缓存，不能每次都重新请求。

### 7.2 免登换用户信息接口

`POST /api/auth/dingtalk-login`，入参：`{ authCode, corpId }`

后端处理逻辑：
1. 用 `access_token` + `authCode` 请求 `https://oapi.dingtalk.com/topapi/v2/user/getuserinfo` 换 `userId`
2. 用 `access_token` + `userId` 请求 `https://oapi.dingtalk.com/topapi/v2/user/get` 换用户详情
3. 根据手机号或其他字段匹配系统用户，下发系统 token

## 八、常见坑位

### 坑 1：dd.config 鉴权失败

- 检查 `url` 是否正确（必须是当前页面完整 URL，不含 `#`）
- 检查可信域名是否配置
- 检查应用是否已发布
- 检查签名计算是否正确（参数必须按字母序排列）

### 坑 2：接口请求失败

- 检查可信域名是否包含接口域名
- 检查 HTTPS 证书是否有效

### 坑 3：hash 路由下 JSAPI 鉴权 URL 问题

hash 路由的 URL 形如 `https://your-domain.com/#/pages/login/index`，鉴权时必须截取 `#` 前面的部分：

```js
const url = location.href.split('#')[0]
// 结果：https://your-domain.com/
```

### 坑 4：每次路由跳转后 JSAPI 失效

hash 路由下，页面 URL 的 `#` 前面部分不变，所以只需要在应用启动时鉴权一次。
history 路由下，每次路由变化 URL 都变，需要在路由守卫里重新鉴权。

### 坑 5：钉钉 PC 端和手机端行为不一致

- PC 钉钉里 `dd.runtime.permission.requestAuthCode` 的 `corpId` 必须传
- 手机钉钉里不传 `corpId` 也能拿到，但建议统一传

## 九、和现有微信小程序代码的关系

H5 微应用走的是 uni-app H5 编译，和微信小程序是两套产物，互不影响。

改造建议：
- 登录页用 `#ifdef H5` 条件编译隔离钉钉 H5 登录逻辑
- 不要动微信小程序的现有代码
- 新项目可以把钉钉能力统一收口到单独平台文件；当前仓库实际收口位置是 `util/platform/auth.js`
- 运行时环境判断单独放在 `util/platform/runtime.js`，不要把 UA 判断散落到业务页
- 登录成功后的会话落盘统一放在 `util/auth/dingtalkLogin.js`，避免两个登录页重复写 storage
