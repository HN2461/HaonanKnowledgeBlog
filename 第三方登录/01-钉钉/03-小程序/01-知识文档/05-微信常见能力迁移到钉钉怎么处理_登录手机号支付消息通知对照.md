# 微信常见能力迁移到钉钉怎么处理（登录、手机号、支付、消息通知对照）

> 这篇文章专门解决一个最真实的问题：
> 你手头已经有一个微信小程序或 `uni-app` 项目，里面有登录、手机号、支付、消息通知这些常见功能。
>
> 现在老板让你迁到钉钉，你最怕的不是“不会写页面”，而是：
>
> `微信这些能力，到了钉钉到底怎么对应？`

## 一、先给你结论版

如果你只想先记住最重要的结论，可以先看这张表。

| 微信常见能力 | 钉钉能不能直接一比一替换 | 我的结论 |
|---|---|---|
| `wx.login` | 不能直接同名替换，但有对应登录链路 | 改成 `dd.getAuthCode` + 后端换身份 |
| 微信手机号授权按钮 | 我目前没查到钉钉小程序前端“一键取手机号”的清晰一比一官方方案 | 不要按“按钮替换”理解，要改成组织身份绑定或服务端授权信息方案 |
| 微信支付 | 我目前没查到你这个钉钉小程序路线下的明确一比一支付文档 | 不要默认能替，必须单独确认产品和架构方案 |
| 微信订阅消息 | 不是同一套机制 | 钉钉更常见的是工作通知、会话消息、机器人消息 |
| `wx.request` | 不需要专门换成钉钉 API | 普通请求继续优先用 `uni.request` |
| 微信选人/组织能力 | 不是原样照搬，但钉钉有自己的组织能力 | 用 `dingtalk-jsapi` 的选人选部门能力 |

## 二、先别急着找“同名 API”，要先换脑子

微信和钉钉最大的差别，不只是 API 名字不一样，而是产品语境不一样。

### 微信常见语境

- 面向 C 端用户
- 强调手机号、支付、订阅消息、分享
- 很多能力围绕“个人用户”展开

### 钉钉常见语境

- 面向组织和企业
- 强调免登、组织身份、选人、选部门、工作通知、机器人消息
- 很多能力围绕“组织关系”和“企业应用”展开

所以你迁移时不要想：

`微信这个按钮，钉钉里是哪一个按钮？`

你更应该想：

`这个业务目的，在钉钉的组织场景里应该怎么实现？`

## 三、第一类：`wx.login` 迁到钉钉，怎么处理

这是最清晰、最应该先改的一块。

### 微信里你常见的是

```js
wx.login({
  success(res) {
    console.log(res.code)
  }
})
```

### 钉钉官方“小程序应用免登”文档里，对应入口是

```js
dd.getAuthCode({
  success: (res) => {
    const { authCode } = res
  }
})
```

### 你要怎么理解这两个东西

它们都不是“前端自己完成登录”。

它们更像：

- 微信：给你一个 `code`
- 钉钉：给你一个 `authCode`

然后你再把这个值发给你自己的后端，让后端去换真正的用户身份。

### 钉钉官方链路大意

根据钉钉官方“小程序应用免登”文档：

1. 前端调用 `dd.getAuthCode`
2. 前端拿到 `authCode`
3. 后端获取应用级访问凭证
4. 后端根据免登授权码获取用户身份信息

### 对你做教育项目时，这一步还要多做一层

因为你们不是单一企业内部工具，而是给学校用。

所以后端在换完钉钉用户身份后，通常还要继续做：

1. 识别这个用户属于哪所学校
2. 判断这所学校是否开通了应用
3. 判断这个用户是老师、管理员、家长还是学生账号映射
4. 生成你们自己系统里的登录态

### 前端怎么封装最稳

```js
export function getPlatformAuthCode() {
  return new Promise((resolve, reject) => {
    // #ifdef MP-WEIXIN
    wx.login({
      success: (res) => resolve(res.code),
      fail: (err) => reject(err)
    })
    // #endif

    // #ifdef MP-DINGTALK
    dd.getAuthCode({
      success: (res) => resolve(res.authCode),
      fail: (err) => reject(err)
    })
    // #endif
  })
}
```

### 这一块的最终结论

`wx.login` 不是不能迁，而是要迁成“钉钉免登链路”，不是机械改函数名。`

## 四、第二类：微信手机号授权，迁到钉钉怎么处理

这一块要特别小心，因为它最容易被误判成“有个对应按钮就完事了”。

### 微信里你可能用过这些

- `open-type="getPhoneNumber"`
- 获取手机号按钮
- 用户点一下，前端拿加密手机号相关信息

### 截至 `2026-03-29`，我查钉钉官方资料后的结论

我目前没有查到一份钉钉官方文档，明确告诉你：

`钉钉小程序前端存在一个和微信 getPhoneNumber 一一对应的按钮式能力`

这一点我不想乱猜，所以我明确告诉你：

`不要把“手机号授权”理解成可以直接做按钮替换。`

### 我查到的更接近官方思路是什么

在钉钉官方“实现网页方式登录应用（登录第三方网站）”文档里，官方提到了：

- 申请 `Contact.User.mobile` 和 `Contact.User.Read` 权限点
- 授权后可获取用户相关个人信息

这说明：

- 钉钉体系里，手机号更像“组织授权和用户信息权限”的一部分
- 它不是我目前查到的“小程序前端一键按钮交互”路线

### 所以你在项目里该怎么处理

这里要分 3 种情况。

#### 情况 1：你原来拿手机号只是为了“识别是谁”

那迁到钉钉后，很多时候不需要再执着于手机号。

因为钉钉组织应用更常见的做法是：

- 先通过 `authCode` 拿组织内用户身份
- 再绑定你们系统账号

也就是说：

`在钉钉里，员工/老师身份本身往往比手机号更重要。`

#### 情况 2：你原来拿手机号是为了“注册或找回”

那钉钉里更适合改成：

- 组织身份直接登录
- 首次登录时引导补充绑定手机号
- 或者后端通过用户信息权限去补齐

#### 情况 3：你业务上真的必须显示手机号

那你要先和后端、产品一起确认：

1. 当前应用类型是不是支持相关权限
2. 当前用户是否属于组织用户
3. 是否能通过用户信息接口合规获取
4. 是否要改成“表单输入 + 短信验证”方案

### 这一块的最终结论

`微信手机号授权在钉钉里不要按“前端按钮能力”迁移，而要按“身份体系和权限体系”重设计。`

## 五、第三类：微信支付，迁到钉钉怎么处理

这一块我要说得非常直接。

### 截至 `2026-03-29`，我这次查官方资料的结果

我没有在当前检索到的钉钉开放平台官方小程序资料里，找到一个和你现在这个路线完全一一对应的：

`钉钉小程序前端支付 = 像微信支付那样直接改一个前端接口`

所以这里我不会给你一个假答案。

### 你应该怎么理解这件事

支付不是一个“前端替换 API”的问题，而是一个：

- 产品流程问题
- 合规问题
- 客户场景问题
- 后端结算问题

### 在教育行业里，你反而要先问产品和老板这几个问题

1. 这个支付是不是还必须发生在钉钉里
2. 付费对象是谁
   - 学校
   - 老师
   - 家长
3. 支付发生在哪种业务里
   - 课程购买
   - 缴费
   - 订单支付
   - 服务续费
4. 它到底是 C 端支付，还是 B 端机构结算

### 为什么这很重要

因为如果你们现在做的是“学校机构用的钉钉应用”，很多业务在钉钉里可能根本不是“用户自己点微信支付”的模式。

它可能会被重设计成：

- 学校后台统一结算
- 合同和线下收款
- 外部 H5 收银台
- 其他业务流程，而不是小程序内支付按钮

### 这一块给你的实战建议

先不要改代码，先开会确认：

1. 钉钉版本到底还要不要支付
2. 如果要，支付入口放在哪
3. 是不是仍然在小程序内做
4. 是否需要换成别的收款方案

### 这一块的最终结论

`微信支付不要默认有钉钉小程序一比一替代，先把它当成一个产品和架构议题，而不是前端 API 迁移题。`

## 六、第四类：微信订阅消息，迁到钉钉怎么处理

这一块很多人也会误判。

### 微信里你熟悉的是

- 用户订阅某类消息
- 到业务节点后系统推送订阅消息

### 钉钉里更常见的不是同一个概念

我这次查到的钉钉官方资料里，和“消息触达”更接近的是这些路线：

#### 1. 工作通知

钉钉官方“发送工作通知文本消息”文档提到：

- 工作通知消息是以某个应用的名义推送给员工
- 可以发送文本、语音、链接等消息类型

#### 2. 企业群消息 / 会话消息

钉钉官方“发送文本消息到企业群”文档提到：

- 可以向企业内部群发送消息通知

#### 3. 企业机器人消息

钉钉官方还有：

- 企业机器人发送群聊消息
- 企业机器人发送单聊消息

这说明钉钉的消息体系更像：

- 工作通知
- 群消息
- 单聊消息
- 机器人消息

而不是你脑子里那个“微信订阅消息按钮 + 模板消息”。

### 还有一个特殊情况

我还查到了钉钉官方“第三方个人应用发送消息”文档，它里面有一套基于 `formId` 的推送思路。

但注意：

这个文档明确适用的是：

`第三方个人应用`

而你现在更像是：

`第三方企业应用 / 学校组织应用`

所以不要把这个个人应用的消息方案直接套到你当前 B 端场景上。

### 你在项目里应该怎么改

先按业务触达对象拆：

#### 场景 1：通知老师本人

优先考虑：

- 工作通知
- 单聊机器人消息

#### 场景 2：通知班级群、部门群、教研群

优先考虑：

- 企业群消息
- 群机器人消息

#### 场景 3：只是页面内提醒

那很多时候根本不需要消息系统，只要：

- 页面红点
- 待办列表
- 应用内消息中心

### 这一块的最终结论

`微信订阅消息在钉钉里不是一比一替换，而是要改成“工作通知 / 群消息 / 单聊消息 / 机器人消息”这一套。`

## 七、第五类：普通接口请求，怎么处理

这一块很多人会问：

`都迁到钉钉了，是不是请求也都要改成 dd.httpRequest？`

我的建议仍然是：

`普通业务接口优先继续用 uni.request。`

### 为什么

因为：

- 普通接口请求本身不是钉钉专属能力
- `uni.request` 更利于你保留多端复用
- 你真正需要单独抽出来的，是钉钉专属容器能力

### 所以你可以这样分层

#### 通用层

- `uni.request`
- `uni.uploadFile`
- `uni.navigateTo`
- `uni.setStorageSync`

#### 钉钉专属层

- `dd.getAuthCode`
- `dingtalk-jsapi`
- 选人
- 选部门
- 组织能力
- 客户端能力

### 这一块的最终结论

`迁钉钉时，不要把通用请求层也一起重写掉。`

## 八、第六类：选人、选部门、组织关系，怎么处理

这个能力在钉钉里反而比微信更重要。

尤其你们是教育公司给学校用，很多功能都离不开组织关系：

- 选老师
- 选班主任
- 选部门
- 选年级组

钉钉官方“小程序使用网页应用客户端 API”文档里给了 `complexPicker` 例子，说明这类能力可以通过 `dingtalk-jsapi` 调。

### 这类能力怎么迁最合理

不是把微信原有选择器硬搬过来，而是：

1. 先接入 `dingtalk-jsapi`
2. 单独封装选人/选部门适配层
3. 业务页面统一调用

### 一个最简封装思路

```js
// #ifdef MP-DINGTALK
import complexPicker from 'dingtalk-jsapi/api/biz/contact/complexPicker'
// #endif

export function pickOrgUsers(options = {}) {
  return new Promise((resolve, reject) => {
    // #ifdef MP-DINGTALK
    complexPicker({
      title: options.title || '请选择人员',
      corpId: options.corpId,
      multiple: options.multiple ?? true,
      maxUsers: options.maxUsers ?? 100,
      pickedUsers: options.pickedUsers || [],
      pickedDepartments: options.pickedDepartments || [],
      disabledUsers: [],
      disabledDepartments: [],
      requiredUsers: [],
      requiredDepartments: [],
      permissionType: 'GLOBAL',
      responseUserOnly: false,
      startWithDepartmentId: 0,
      onSuccess: (res) => resolve(res),
      onFail: (err) => reject(err)
    })
    // #endif
  })
}
```

### 这一块的最终结论

`组织能力不是微信那套，但钉钉恰恰很强，建议你把它当成钉钉版的重点增强能力。`

## 九、项目里最建议你先建立的 4 个适配模块

如果你现在就准备开改，我建议你先建这 4 个文件。

### 1. `src/utils/platformAuth.js`

负责：

- `wx.login`
- `dd.getAuthCode`

### 2. `src/utils/platformUser.js`

负责：

- 用户身份换取
- 用户信息读取
- 学校租户信息初始化

### 3. `src/utils/platformPicker.js`

负责：

- 选人
- 选部门

### 4. `src/utils/platformMessage.js`

负责：

- 页面内提醒
- 消息触达参数组织
- 前端和后端消息能力对接约定

这样你的业务页就不会散着写平台判断。

## 十、给你一个真正能落地的改造顺序

如果你现在就动手，我建议顺序是这样的：

1. 先改 `wx.login`
2. 再确认手机号需求到底还要不要保留原样
3. 再把订阅消息重设计成钉钉消息体系
4. 支付最后单独拉出来开会确认

### 为什么支付放最后

因为登录和消息往往是“肯定要改”的。

但支付很可能不是“改接口”，而是“改业务方案”。

## 十一、给你一张更适合实战的对照表

| 原微信能力 | 迁到钉钉后的建议动作 | 优先级 |
|---|---|---|
| `wx.login` | 改 `dd.getAuthCode` + 后端换身份 | 最高 |
| 手机号授权 | 改成组织身份绑定 / 表单补录 / 权限获取方案 | 高 |
| 订阅消息 | 改成工作通知 / 群消息 / 机器人消息 | 高 |
| 支付 | 单独确认是否保留，避免前端先入坑 | 最高但先不改代码 |
| 组织选择 | 用 `dingtalk-jsapi` 增强实现 | 中高 |
| 普通请求 | 保持 `uni.request` | 高 |
